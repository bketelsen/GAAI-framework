# Story E06S20: Security Hardening — Rate Limiting Binding + Turnstile

## User Story

**As a** platform operator,
**I want** bot protection (Turnstile) on prospect submissions and native rate limiting on public endpoints,
**so that** abuse is prevented without manual KV-based rate limiting overhead.

---

## Context

Current rate limiting uses a manual KV-based implementation (`src/lib/rateLimit.ts`) with 10 req/60s per IP. This works but wastes KV reads/writes and doesn't scale well.

Cloudflare Rate Limiting binding (free on Workers Paid) provides native rate limiting with zero storage overhead. Turnstile (free, invisible CAPTCHA) adds bot protection on the most critical public endpoint: prospect submit.

---

## Acceptance Criteria

- [ ] AC1: Rate Limiting binding configured in `wrangler.toml` for all envs
- [ ] AC2: `checkRateLimit()` uses binding API — zero KV reads/writes
- [ ] AC3: Public endpoints rate-limited: 30 req/min per IP
- [ ] AC4: `POST /api/prospects/submit` requires `cf-turnstile-response` in body (422 if missing)
- [ ] AC5: Turnstile verification POST to CF siteverify API
- [ ] AC6: Staging/dev: `TURNSTILE_SECRET_KEY` = test key → always passes
- [ ] AC7: Invalid Turnstile token → 422 `{ error: "Bot verification failed" }`
- [ ] AC8: All tests pass + new tests for Turnstile validation and rate limiting

---

## Out of Scope

- Turnstile on expert registration (already behind auth)
- Custom rate limiting per endpoint (uniform 30/min sufficient for MVP)

---

## Dependencies

- E06S18 (Env type conventions, wrangler config patterns)
- **Human action required:** Create Turnstile site key in CF Dashboard, store secret as Worker secret

## Notes

Rate Limiting binding replaces `src/lib/rateLimit.ts` entirely. The `RATE_LIMITING` KV namespace can be removed after this story ships. Turnstile test keys: `1x0000000000000000000000000000000AA` (always passes) / `2x0000000000000000000000000000000AA` (always fails).
