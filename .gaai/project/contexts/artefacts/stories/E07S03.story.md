# Story E07S03: Core API Server-Side Event Capture (Prospect + Expert Funnels)

## User Story

**As a** platform operator,
**I want** every critical API interaction to emit a PostHog event server-side,
**so that** both prospect and expert conversion funnels are fully tracked end-to-end, independent of client-side ad blockers.

---

## Context

Client-side tracking (E07S02) covers satellite page views and clicks, but the core conversion steps happen via API calls to the `callibrate-core` Worker. Server-side event capture ensures 100% coverage of funnel steps that occur through API handlers — form submission, match viewing, email identification, booking — regardless of browser extensions or JS failures.

PostHog's `posthog-node` SDK is officially supported on Cloudflare Workers with `flushAt: 1`, `flushInterval: 0`, and `captureImmediate()` + `ctx.waitUntil()` to avoid data loss in the stateless Worker runtime.

## Notes

The PostHog client should be created as a shared utility (`src/lib/posthog.ts`) initialized from `env.POSTHOG_API_KEY`. Pattern:

```typescript
// Conceptual — not prescriptive implementation
const ph = new PostHog(env.POSTHOG_API_KEY, { host: 'https://ph.callibrate.io', flushAt: 1, flushInterval: 0 });
ctx.waitUntil(ph.captureImmediate({ distinctId, event, properties }));
```

If `POSTHOG_API_KEY` is not set (e.g. local dev), tracking should silently no-op.

### Event Naming Convention (enforced)

All events follow `object.action` in snake_case:
- `prospect.form_submitted` (not `formSubmit` or `prospect_signup`)
- `booking.confirmed` (not `bookingConfirmed`)
- `expert.gcal_connected` (not `gcal_connected`)

Controlled verb list: `submitted`, `viewed`, `identified`, `checked`, `held`, `confirmed`, `cancelled`, `registered`, `updated`, `connected`, `disconnected`

### Internal Traffic Filtering

All events emitted by founder/dev accounts should include `$set: { is_internal: true }` as a person property. This allows filtering internal traffic out of all dashboards and funnels globally in PostHog.

### LLM Analytics (bonus)

The GPT-4o-mini extraction service (`POST /api/extract`) should also emit:
- `llm.extraction_completed` — properties: `model`, `tokens_used`, `latency_ms`, `satellite_id`

This enables monitoring extraction cost/quality via PostHog's LLM Analytics feature. Same `captureImmediate()` + `ctx.waitUntil()` pattern.
