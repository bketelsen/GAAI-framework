# Story E06S29: Staging — Smoke Test End-to-End

## User Story

**As a** platform operator,
**I want** to validate the full backend loop on staging via real HTTP calls,
**so that** I can confirm the infra, secrets, and code are integrated correctly before investing in Layer 3 (UI).

---

## Context

Toute l'infrastructure backend est théoriquement complète (E06S01 → E06S26 + E07S01 → E07S05 livrés). Mais aucun test end-to-end n'a encore été exécuté contre un environnement réel avec toutes les ressources provisionnées.

Cette story valide le "happy path" de staging : expert s'inscrit → prospect soumet → matches calculés → slot réservé. C'est le prérequis technique de M1 ("Full loop working end-to-end without manual intervention").

Cette story est **founder-executed** : curl + vérification dashboard. La Delivery Agent peut fournir les commandes curl exactes si demandé.

---

## Acceptance Criteria

### Infrastructure

- [ ] AC1: `GET https://callibrate-core-staging.frederic-geens-consulting.workers.dev/api/health` → `{ "status": "ok", "supabase": "connected" }` (HTTP 200)
- [ ] AC2: `GET https://callibrate-satellite-staging.frederic-geens-consulting.workers.dev/health` → 302 vers `callibrate.io` (pas de satellite actif) ou 200 si un satellite est configuré

### Expert flow

- [ ] AC3: `POST /api/experts/register` avec payload valide → HTTP 201, `expert_id` retourné, JWT dans la réponse
- [ ] AC4: `PATCH /api/experts/profile` avec JWT → profil mis à jour (skills, industries, budget_range, etc.)
- [ ] AC5: `GET /api/gcal/auth-url` avec JWT → URL OAuth Google retournée (format `https://accounts.google.com/o/oauth2/...`) — le flow complet ne peut pas être testé sans que Google approuve le scope, mais l'URL doit être générée correctement

### Prospect flow

- [ ] AC6: `POST /api/extract` avec `{ "text": "Je cherche un expert n8n pour automatiser mon CRM, budget 5k€" }` → `ProspectRequirements` JSONB retourné avec `skills`, `budget_range`, `industries`
- [ ] AC7: `POST /api/prospects/submit` (satellite funnel) avec requirements JSON → HTTP 201, `prospect_id` retourné
- [ ] AC8: `GET /api/matches/:prospect_id` → liste de matches avec scores (peut être vide si aucun expert ne matche — mais pas d'erreur 5xx)

### Booking flow (partiel — sans GCal live)

- [ ] AC9: `POST /api/bookings/availability` avec `expert_id` valide → réponse structurée (peut retourner 0 slots si GCal non connecté — vérifier le graceful fallback)

### Observabilité

- [ ] AC10: CF Dashboard → Workers → `callibrate-core-staging` → Analytics → au moins 1 request visible correspondant aux tests AC1–AC9
- [ ] AC11: CF Dashboard → Workers → `callibrate-core-staging` → Analytics Engine → dataset `matching-metrics` → au moins 1 data point visible (si AC7/AC8 ont déclenché du matching)
- [ ] AC12 *(optionnel, si E07S06 done)*: PostHog EU Live Events → au moins `expert.registered` et `prospect.submitted` visibles après AC3 et AC7

## Out of Scope

- Test Google Calendar live (bloqué par scope review E06S30)

## Notes

**Si AC1 retourne supabase: "error" :** vérifier SUPABASE_URL + SUPABASE_ANON_KEY dans staging secrets.

**Si AC7 retourne 0 matches :** normal si l'expert créé en AC3 n'a pas de profil complet ou si son `composite_score` est 0. Ajouter skills + industries via AC4 avant de tester AC7.

**Si AC9 retourne erreur GCal :** expected — Google OAuth non encore approuvé. Le test passe si la réponse est un 200 avec `{ slots: [] }` ou un 4xx métier explicite (pas un 5xx infra).

**Commandes curl :** si le Delivery Agent n'a pas fourni un script de smoke test, demander `/gaai-deliver` pour générer un `scripts/smoke-test-staging.sh`.
