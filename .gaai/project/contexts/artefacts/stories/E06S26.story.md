# Story E06S26: Analytics Engine — Matching Observability

## User Story

**As a** platform operator,
**I want** fire-and-forget matching metrics written to Analytics Engine,
**so that** I can observe matching performance (latency, cache efficiency, scoring quality) without impacting request latency.

---

## Context

There is currently no observability into the matching pipeline. Analytics Engine provides free (100K events/day), fire-and-forget metrics via `writeDataPoint()`. Metrics are queryable via the CF Dashboard or GraphQL API.

---

## Acceptance Criteria

- [ ] AC1: Analytics Engine dataset `matching-metrics` configured in `wrangler.toml`
- [ ] AC2: `writeDataPoint()` with blobs (string labels) + doubles (numeric metrics)
- [ ] AC3: Fire-and-forget (NOT awaited, wrapped in try/catch)
- [ ] AC4: Missing binding → no-op (graceful degradation)
- [ ] AC5: All tests pass

---

## Out of Scope

- Dashboard / alerting (use CF Dashboard directly)
- Custom retention policies (default 90 days)
- Per-expert analytics (aggregate only at MVP)

---

## Dependencies

- E06S18 (wrangler config conventions, Env type patterns)
- **Human action required:** Create dataset via `wrangler analytics-engine create matching-metrics` or CF Dashboard

---

## Notes

Tracked metrics: `latency_ms`, `cache_layer` (L1/L2/L3), `pool_size`, `vectorize_candidates` (when E06S22 ships), `top_score`, `mean_score`. Blobs: `satellite_id`, `endpoint`. Fire-and-forget pattern: `env.MATCHING_ANALYTICS?.writeDataPoint({...})` — safe even if binding is undefined.
