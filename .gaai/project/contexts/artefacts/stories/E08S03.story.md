# Story: Booking hold authentication + rate limiting

## User Story

**As an** expert,
**I want** the booking hold endpoint to require a valid prospect token and enforce rate limits,
**so that** attackers cannot block my calendar slots with fake holds.

---

## Context

SEC-AUDIT-001 finding SEC-02 (CRITICAL). `POST /api/bookings/hold` accepts arbitrary expert_id/prospect_id with zero authentication. An attacker can create unlimited holds, blocking all expert slots for 10 minutes each, renewing indefinitely.

---

## Acceptance Criteria

- [ ] AC1: `POST /api/bookings/hold` requires a `token` field in the request body. The token is verified via `verifyProspectToken(token, prospect_id, env.PROSPECT_TOKEN_SECRET)`. If invalid or expired → 403 Forbidden.
- [ ] AC2: Rate limiting is enforced on `/api/bookings/hold`: maximum 5 holds per hour per IP (via `RATE_LIMITER` binding with a separate namespace_id or key prefix `hold:`). Exceeding → 429 Too Many Requests.
- [ ] AC3: A prospect cannot have more than 3 active holds simultaneously (status = 'held'). Checked via DB query before insert. Exceeding → 409 with error `max_holds_reached`.
- [ ] AC4: The `expert_id` is validated to exist in the `experts` table before creating the hold.
- [ ] AC5: `start_at` and `end_at` are validated as ISO-8601 datetime strings. `start_at` must be in the future. `end_at` must be after `start_at`. Invalid → 422.
- [ ] AC6: All existing vitest tests for hold are updated to include a valid prospect token. New tests: (a) missing token → 403, (b) invalid token → 403, (c) rate limit exceeded → 429, (d) max holds exceeded → 409, (e) invalid dates → 422.
- [ ] AC7: tsc clean.

---

## Out of Scope

- Changing hold duration from 10 minutes (configurable holds is post-MVP)
- Turnstile verification on hold (rate limit + token is sufficient)

---

## Dependencies

- None — standalone fix on existing hold.ts

---

## Notes

- The prospect token is already issued at `POST /api/prospects/submit` and used for `/matches` and `/identify`. Reusing it for `/hold` is consistent.
- The satellite funnel client must be updated to pass the token in the hold request body. This is a frontend concern — note in SETUP.md.
- Remediation priority: CRITICAL — DoS vector on expert calendars.
