# Story E06S37: Outcome-Based Expert Profiles + Extraction Enhancement

## User Story

**As an** expert,
**I want** my profile to showcase the outcomes I deliver (time saved, revenue generated, processes automated) rather than just the tools I use,
**so that** prospects choose me based on what I achieve, not what I click.

**As a** prospect,
**I want** to describe the outcomes I need (save time, automate invoicing, reduce manual errors),
**so that** the matching engine finds experts who have actually delivered those results.

---

## Context

Field data from Discovery (E01S01) reveals a critical framing insight:

- **"Found time" framing converts**: Littlecutsie (r/n8n_ai_agents) — "stopped selling n8n workflows, started selling found time". Experts who frame their offer as outcomes, not tools, attract higher-quality clients.
- **Prospect brief quality = matching quality**: r/AiForSmallBusiness — "good consultants interrogate requirements". The quality of the prospect's expressed needs determines matching accuracy.
- **Tool lists are commoditized**: "prompt engineers with a Zapier account" (r/AiForSmallBusiness) — listing tools is a negative signal. Outcomes are the differentiator.

Currently, expert profiles are tool-oriented (`skills TEXT[]`) and matching scores on skill overlap + semantic similarity. There is no mechanism to match on *outcomes delivered* vs *outcomes desired*.

This story adds outcome tags to expert profiles, extracts desired outcomes from prospect briefs, and introduces an `outcome_alignment` scoring dimension.

**Decision reference:** DEC-2026-02-24-81

## Acceptance Criteria

### Schema

- [ ] AC1: DB migration adds `outcome_tags TEXT[] DEFAULT '{}'` to `experts` table
- [ ] AC2: `outcome_tags` contains short outcome phrases, e.g. `["25h/week saved on RFP processing", "automated lead qualification pipeline", "reduced manual data entry by 80%"]`
- [ ] AC3: PATCH `/api/experts/profile` accepts `outcome_tags` (max 10 items, max 200 chars each, validated)

### Extraction

- [ ] AC4: `ProspectRequirements` type (from E06S08/E06S12) extended with optional `desired_outcomes?: string[]`
- [ ] AC5: GPT-4o-mini extraction prompt updated to extract `desired_outcomes` from freetext — e.g. "I need to save time on invoicing" -> `["reduce invoicing time"]`
- [ ] AC6: Extraction function call schema includes `desired_outcomes` as optional string array (max 5 items)
- [ ] AC7: Existing extractions without `desired_outcomes` remain valid (backward compatible)

### Matching

- [ ] AC8: New scoring function `scoreOutcomeAlignment(expert.outcome_tags, prospect.desired_outcomes): number` returns 0.0-1.0
- [ ] AC9: Scoring uses semantic similarity (via Workers AI embeddings) between each desired outcome and each expert outcome tag — returns max pairwise similarity averaged across desired outcomes
- [ ] AC10: If `desired_outcomes` is empty/undefined OR `outcome_tags` is empty, `scoreOutcomeAlignment` returns `null` (not 0) — does not penalize
- [ ] AC11: `scoreMatch()` integrates outcome alignment: when both sides have data, blend weight is configurable via constant `OUTCOME_WEIGHT` (default 0.10, taken from existing weights — reduce `skill_weight` from current value by 0.10)
- [ ] AC12: `ScoreBreakdown` includes `outcome_alignment: number | null` field

### Quality

- [ ] AC13: `tsc --noEmit` — zero errors
- [ ] AC14: `vitest run` — all tests pass + new tests: outcome alignment scoring (high similarity), no outcomes (returns null, no penalty), partial outcomes (one side empty), extraction with desired_outcomes, extraction without desired_outcomes (backward compat)

## Notes

The outcome alignment scoring reuses the Workers AI embedding infrastructure from E06S21/E06S22. The embedding is computed on-the-fly for prospect `desired_outcomes` (short strings, low latency) and compared against pre-computed expert `outcome_tags` embeddings.

For MVP, expert `outcome_tags` embeddings are NOT pre-computed in Vectorize — they are embedded at scoring time alongside the prospect outcomes. This is acceptable because:
- The candidate pool is already filtered (billing + admissibility + Vectorize top-100)
- Each remaining expert has <=10 outcome tags
- Workers AI embedding is fast for short strings

Pre-computing outcome embeddings in Vectorize is an optimization for when the expert pool exceeds 500+ (future story).

The `OUTCOME_WEIGHT` of 0.10 is deliberately conservative — it should improve matching quality without disrupting the existing scoring balance. Weight tuning happens post-Gate 2 with real conversion data.
