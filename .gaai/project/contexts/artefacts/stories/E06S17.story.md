# Story E06S17: Survey Submission Endpoints (Feedback Loop Activation)

## User Story

**As a** platform operator,
**I want** prospects and experts to submit feedback via dedicated API endpoints,
**so that** the score-computation consumer (E06S09) receives messages and composite_score evolves over time.

---

## Context

The feedback loop chain: E06S09 (consumer ready) → E06S16 (survey emails sent with token-gated links) → **E06S17 (submissions received)** → score-computation queue message dispatched → E06S09 consumer processes → composite_score updated.

Without these endpoints, the score-computation consumer never receives messages. Three feedback streams must be collected (DEC-20):
1. Prospect rates call experience J+7 (1-5 scale)
2. Prospect rates project satisfaction J+45 (1-10 scale)
3. Expert evaluates lead quality J+48 (1-10 + optional conversion_declared)

---

## Acceptance Criteria

- [ ] AC1: `POST /api/surveys/call-experience` — token-gated (JWT from survey email link). Validates `booking_id` + `prospect_id` from token. INSERT into `call_experience_surveys` (booking_id, prospect_id, score 1-5, comment optional). Dispatches `{ type: "feedback.call_experience", expert_id }` to `SCORE_COMPUTATION` queue. Returns 201 on success, 409 if already submitted (idempotent on booking_id + prospect_id)
- [ ] AC2: `POST /api/surveys/project-satisfaction` — same token-gated pattern. INSERT into `project_satisfaction_surveys` (booking_id, prospect_id, score 1-10, comment optional). Dispatches `{ type: "feedback.project_satisfaction", expert_id }` to `SCORE_COMPUTATION` queue. Returns 201 / 409
- [ ] AC3: `POST /api/evaluations/lead` — JWT-authenticated (expert session). Validates `lead_id` + expert ownership (lead.expert_id = authenticated expert). INSERT into `lead_evaluations` (lead_id, expert_id, score 1-10, notes optional, conversion_declared boolean optional). Dispatches `{ type: "feedback.lead_evaluation", expert_id }` to `SCORE_COMPUTATION` queue. Returns 201 / 409
- [ ] AC4: Survey token generation — JWT containing `{ booking_id, prospect_id, survey_type, exp }` with 30-day expiry. Generated by E06S16 BookingCompletedWorkflow and embedded in survey email links. Secret: `SURVEY_TOKEN_SECRET`
- [ ] AC5: Input validation — Zod schemas for all 3 endpoints. Score ranges enforced (1-5 for call_experience, 1-10 for satisfaction and lead_evaluation). Invalid input returns 422
- [ ] AC6: Expert ownership check — `/api/evaluations/lead` verifies `leads.expert_id = req.auth.expert_id` before INSERT. Returns 403 if not owner
- [ ] AC7: Queue dispatch — all 3 endpoints dispatch to `SCORE_COMPUTATION` queue after successful INSERT. Queue message includes `expert_id` resolved from the booking/lead chain
- [ ] AC8: Unit tests for all 3 endpoints — happy path, duplicate submission (409), invalid token (401), invalid input (422), expert ownership violation (403)
- [ ] AC9: `tsc --noEmit` clean, `vitest run` all tests pass

## Out of Scope

- Survey UI / landing page (E03 track — HTML form rendered by satellite Worker)
- Email template design for survey emails (E06S16 scope)
- Batch recomputation of scores (post-MVP maintenance)
- Rate limiting on survey endpoints (post-MVP — token expiry provides sufficient protection)

## Dependencies

- E06S09 (score-computation consumer must be ready to process dispatched messages)
- E06S16 (survey emails with token-gated links must be sent for prospects to reach these endpoints)
- E06S02 (feedback tables: call_experience_surveys, project_satisfaction_surveys, lead_evaluations)
- E06S03 (JWT auth middleware for expert evaluation endpoint)

## Notes

Expert ID resolution for queue dispatch:
- `/api/surveys/*`: booking_id → bookings.match_id → matches.expert_id
- `/api/evaluations/lead`: lead_id → leads.expert_id (direct)

The survey token is separate from the prospect token (E06S07) — it uses a different secret (`SURVEY_TOKEN_SECRET`) and contains survey-specific claims.
