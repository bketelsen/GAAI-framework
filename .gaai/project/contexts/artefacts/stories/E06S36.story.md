# Story E06S36: Expert Admissibility Criteria — Non-Budget Matching Filters

## User Story

**As an** expert,
**I want** to define admissibility criteria beyond budget (methodology alignment, stack requirements, minimum project scope),
**so that** I only receive leads that match my working style and technical preferences — not just my price range.

**As a** prospect,
**I want** to be matched only with experts whose methodology and stack align with my project,
**so that** every match is a genuine fit, not just a budget match.

---

## Context

Field data from Discovery (E01S01) reveals that expert filtering criteria are more sophisticated than initially modeled:

- **Budget** is filter #1 (already implemented via DEC-67/68 — E06S34 `applyBillingFilters`)
- **Methodology alignment** is filter #2: "modular or nothing" (heyiamnickk, r/gohighlevel), "scoping-first" approaches preferred over "jump in and build"
- **Architecture/stack preferences** is filter #3: experts reject projects on incompatible stacks or requiring tools they don't endorse
- **Payment security** is a deal-breaker: "card on file", "money upfront" (MachadoEsq, r/gohighlevel) — already covered by credit billing (DEC-68)

Current matching only filters on billing constraints (E06S34) and scores on skills/industry/semantic overlap (E06S05/E06S22). There is no mechanism for an expert to say "I only take projects with a scoping phase" or "I don't work with clients who won't commit to modular architecture."

This story adds `admissibility_criteria` to the expert profile and applies it at matching time, alongside billing filters.

**Decision reference:** DEC-2026-02-24-80

## Acceptance Criteria

- [ ] AC1: DB migration adds `admissibility_criteria JSONB DEFAULT '{}'::jsonb` to `experts` table
- [ ] AC2: `admissibility_criteria` schema supports the following optional fields:
  ```typescript
  {
    min_project_duration_days?: number;       // reject short gigs below threshold
    required_methodology?: string[];          // e.g. ["scoping-first", "modular"]
    excluded_verticals?: string[];            // verticals the expert refuses
    min_budget?: number;                      // floor below max_lead_price (finer control)
    required_stack_overlap_min?: number;      // minimum shared tools with prospect (default: 1)
    custom_rules?: string[];                  // free-text rules for future AI-based filtering
  }
  ```
- [ ] AC3: PATCH `/api/experts/profile` accepts and validates `admissibility_criteria` (JSON schema validation, reject unknown keys)
- [ ] AC4: `applyAdmissibilityFilters(experts, prospect, leadPrice)` pure function in `src/lib/admissibilityFilter.ts` — runs AFTER `applyBillingFilters` but BEFORE scoring
- [ ] AC5: Filter logic:
  - `min_project_duration_days`: exclude if prospect `timeline` (extracted) is shorter
  - `required_methodology`: exclude if none of the expert's required methodologies appear in prospect `description` or `challenge` (case-insensitive substring match — semantic matching deferred to E06S22 integration)
  - `excluded_verticals`: exclude if prospect `industry` matches any excluded vertical
  - `min_budget`: exclude if `prospect.budget_max < expert.admissibility_criteria.min_budget`
  - `required_stack_overlap_min`: exclude if `intersection(expert.skills, prospect.tools).length < min` (default 1 if not set)
- [ ] AC6: Expert excluded by admissibility criteria receives fire-and-forget notification via EMAIL_NOTIFICATIONS queue with reason (`'methodology_mismatch'` | `'vertical_excluded'` | `'budget_floor'` | `'stack_mismatch'` | `'duration_too_short'`) and missed lead context. Dispatched via `ctx.waitUntil()` — non-blocking.
- [ ] AC7: Admissibility filters apply in BOTH matching paths: `POST /api/matches/compute` (core) and `POST /api/prospects/submit` (satellite funnel)
- [ ] AC8: If `admissibility_criteria` is empty `{}` or not set, expert passes all admissibility filters (backward compatible — no implicit exclusion)
- [ ] AC9: Matching Worker response includes `admissibility_excluded` count alongside existing `billing_excluded`
- [ ] AC10: `tsc --noEmit` — zero errors
- [ ] AC11: `vitest run` — all tests pass + new tests: methodology match (included), methodology mismatch (excluded + notification), excluded vertical (excluded), stack overlap insufficient (excluded), empty criteria (passes all), multiple criteria active simultaneously

## Notes

The matching pipeline with this story becomes:
1. Load expert pool (Cache API -> D1 -> Hyperdrive chain)
2. Filter by billing constraints (`applyBillingFilters` — E06S34)
3. **NEW: Filter by admissibility criteria (`applyAdmissibilityFilters` — this story)**
4. Score remaining experts via `scoreMatch()`
5. Return ranked results

The `required_methodology` substring match is intentionally simple for MVP. When Vectorize embeddings include methodology-related text (from expert bios), semantic matching becomes possible as a future enhancement without schema changes.

The `custom_rules` field is write-only for now — stored in the profile but not evaluated at matching time. This prepares for future AI-based filtering where an LLM evaluates free-text rules against prospect briefs.
