# Story E02S05: Lead Pipeline Page + Actions

## User Story

**As a** registered expert,
**I want** to see all my incoming leads with their status and take actions (confirm, flag, evaluate),
**so that** I can manage my pipeline efficiently, flag bad leads within the 7-day window, and provide feedback that improves my future matches.

---

## Context

The lead pipeline is the core expert interaction with the platform. It's where the expert sees the value they're paying for: qualified leads with prospect context, match scores, and pricing transparency.

Lead lifecycle: `new` → `confirmed` (auto after 7 days or manual) | `flagged` (within 7 days, credit restored) → `evaluated` (score 1-10 + notes + optional conversion). This page consumes `GET /api/experts/:id/leads` (E06S38) and provides actions via `POST /api/leads/:id/confirm`, `POST /api/leads/:id/flag` (E06S32).

**RR7 idiom:** Filters and pagination are URL search params (`?status=new&page=1`). Status filter tabs navigate via `<Link>` — no client-side state, full server-side loader revalidation on each navigation. Actions use React Router form actions with auto-revalidation on success.

## Acceptance Criteria

### Lead List

- [ ] AC1: Leads route (`/dashboard/leads`): loader fetches `GET /api/experts/:id/leads?status={status}&page={page}&per_page=20` via `api.server.ts`. URL search params drive the query.

- [ ] AC2: Status filter tabs: Tous / Nouveaux / Confirmés / Signalés. Each tab is a `<Link>` that sets `?status=all|new|confirmed|flagged`. Active tab visually highlighted. Count badge per status tab showing total leads in that status (from API response or separate count query).

- [ ] AC3: Lead list table/cards with columns: prospect (name/email if available, or "Prospect anonyme"), match score (percentage badge), price (€ amount with "standard"/"premium" tag), status (colored badge), date (relative: "il y a 2h"), actions (action buttons per status).

- [ ] AC4: Pagination: page controls at bottom. "Précédent" / "Suivant" links updating `?page=N` in URL. Total count displayed: "Affichage de {start}-{end} sur {total} leads".

### Lead Detail

- [ ] AC5: Clicking a lead row opens a detail drawer (shadcn Sheet/Drawer) or navigates to a sub-route (`/dashboard/leads/:id`). Detail view shows:
  - Prospect requirements (from `prospect.requirements` JSONB — structured display of budget, timeline, skills_needed, description)
  - Match score breakdown (per-criterion bars — same format as prospect-facing E03S03 AC3)
  - Booking info (if exists: date, time, status, meeting URL)
  - Lead price with breakdown (budget tier + qualification level)
  - Flag window status: "X jours restants pour signaler" countdown badge (if status = `new` and within 7 days)
  - Evaluation form (if status = `confirmed` and not yet evaluated)

### Actions

- [ ] AC6: **Confirm action:** Button visible on leads with status `new`. Calls `POST /api/leads/:id/confirm` via RR7 action. On success: toast "Lead confirmé", lead status updates to `confirmed` (auto-revalidation). No confirmation dialog — confirming is the normal flow.

- [ ] AC7: **Flag action:** Button visible on leads with status `new` AND within 7-day flag window (`flag_window_expires_at > now`). Opens dialog with:
  - Required: `flag_reason` textarea (min 20 chars) — "Décrivez pourquoi ce lead ne correspond pas à vos critères"
  - "Signaler ce lead" submit button
  Calls `POST /api/leads/:id/flag` with `{ reason }` via action. On success: toast "Lead signalé — crédit restauré", status updates to `flagged`. Credit balance updates on next page load.

- [ ] AC8: **Flag window badge:** On leads with status `new`, display a countdown badge: "{N} jours restants" (calculated from `flag_window_expires_at`). When < 24h: red badge "Dernier jour". When expired: badge hidden, flag button disabled/hidden.

- [ ] AC9: **Evaluate action:** Available on `confirmed` leads that haven't been evaluated (`evaluation_score IS NULL`). Form in lead detail:
  - Score: 1-10 slider or number input with labels (1 = "Très mauvais lead", 5 = "Correct", 10 = "Excellent lead")
  - Notes: optional textarea (max 500 chars)
  - Conversion: checkbox "Ce lead a mené à un projet" (optional, updates `conversion_declared`)
  Calls `POST /api/leads/:id/evaluate` via action. On success: toast "Évaluation enregistrée — merci, vos futurs matchs s'amélioreront."

### Empty State

- [ ] AC10: When no leads exist: illustration placeholder + "Vous n'avez pas encore de leads" + explanation text: "Dès qu'un prospect correspondant à votre profil sera identifié, il apparaîtra ici." + CTA to check profile completeness → `/dashboard/settings`.

### Optional Enhancement

- [ ] AC11: Client-side Supabase Realtime subscription for new lead notifications. When a new lead arrives: browser notification (if permitted) + badge count update on "Leads" nav item + new lead appears at top of list with highlight animation. Realtime subscription on `leads` table filtered by `expert_id`. Graceful degradation: if Realtime fails, leads appear on next page load.

### Quality

- [ ] AC12: `tsc --noEmit` — zero errors.

- [ ] AC13: Vitest tests: loader with filters, confirm action, flag action (within window + expired window), evaluate action, empty state render.

## Notes

- **Evaluate endpoint:** The plan references `POST /api/leads/:id/evaluate` but this endpoint may not exist yet (E06S17 covers survey submission, not expert lead evaluation). Delivery should check if a dedicated endpoint exists or needs to be created. If missing, flag as a mini Layer 2 gap (could be a small addition to E06S38 or a standalone micro-story).
- **7-day flag window** is the key UX element. The countdown badge must be prominent — experts need to understand they have a limited time to flag bad leads and get credits back.
- **Anti-gaming framing (DEC-68):** The evaluate form should frame evaluation positively: "Vos évaluations améliorent la qualité de vos futurs leads" — never "We're monitoring your behavior."
- **RR7 pattern:** No `useState` for filter state. URL search params are the source of truth. Each filter change triggers a new loader fetch. This is the React Router v7 recommended approach — URL-driven state eliminates sync bugs.
- **PostHog events (Delivery must include):** `expert.leads_viewed` (page load, status_filter), `expert.lead_confirmed` (lead_id, price), `expert.lead_flagged` (lead_id, reason_length), `expert.lead_evaluated` (lead_id, score, conversion_declared). Server-side capture in actions. These feed the Expert Activation Funnel (E07S04).
