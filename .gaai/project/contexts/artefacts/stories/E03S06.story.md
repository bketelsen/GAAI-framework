# Story E03S06: Inline Booking Widget

## User Story

**As a** prospect who has found a matching expert,
**I want** to see their real-time availability and book a discovery call without leaving the page,
**so that** I can take immediate action on the best match while my intent is highest.

---

## Context

The booking widget is the conversion endpoint of both prospect flows: quiz funnel results (E03S03) and directory profiles (E03S04). It renders inline — no redirect, no popup, no external page. The widget consumes the booking engine APIs (E06S11): availability → hold → confirm.

The widget is a reusable component instantiated per expert. It handles the full booking lifecycle: date selection, time slot picking, hold (10min TTL), prospect details form (name, email, description), confirmation (creates Google Calendar event + Meet link), and success display.

Cal.com integration was superseded (DEC-41). All booking is handled by the custom booking engine (E06S11).

---

## Acceptance Criteria

- [ ] AC1: Booking widget renders inline below the expert card/profile when the prospect clicks "Réserver un appel". No page navigation, no modal — the widget expands in place with a smooth open animation. Widget can be collapsed by re-clicking the CTA or a close affordance.

- [ ] AC2: **Step 1 — Date selection:** Widget calls `GET {CORE_API_URL}/api/experts/{expert_id}/availability` and displays available dates for the next 14 days as a calendar view. Days with available slots are highlighted (primary color). Days without slots are grayed out. Loading state: calendar skeleton with shimmer. Timezone display: "Horaires en {timezone}" (detected via `Intl.DateTimeFormat().resolvedOptions().timeZone`).

- [ ] AC3: **Step 2 — Time slot selection:** On date click, display available 20-minute slots for that day as clickable time chips (e.g., "09:00", "09:30", "10:00"). Slots displayed in the prospect's local timezone. Slots already held are filtered server-side (E06S11 AC1).

- [ ] AC4: **Step 3 — Hold:** On slot selection, immediately call `POST {CORE_API_URL}/api/bookings/hold` with `{ expert_id, start_at, end_at, prospect_id }`. On success (`{ booking_id, held_until }`): display countdown timer "Créneau réservé pendant {mm:ss}" + show confirm form. On `409` (slot taken): "Ce créneau vient d'être réservé. Choisissez un autre horaire." + refresh available slots automatically.

- [ ] AC5: **Step 4 — Confirm form:** After successful hold, display:
  - Name input (required)
  - Email input (required, pre-filled if prospect already identified via E03S03)
  - Brief project description textarea (optional, max 500 chars, placeholder: "Décrivez brièvement votre besoin...")
  - "Confirmer la réservation" button
  - Countdown timer visible throughout this step

- [ ] AC6: On confirm submit: call `POST {CORE_API_URL}/api/bookings/{booking_id}/confirm` with `{ prospect_name, prospect_email, description }`. Loading state on button.

- [ ] AC7: **Success state:** On successful confirmation, widget displays:
  - Checkmark visual + "Réservation confirmée !"
  - Expert name + date + time (prospect's local timezone)
  - Google Meet link: clickable `{meeting_url}`
  - "Un email de confirmation avec l'invitation Google Calendar a été envoyé à {prospect_email}."
  - "Préparer mon appel" link → prep page URL (`https://{satellite_domain}/prep/{booking_prep_token}`, from E06S11 AC13)

- [ ] AC8: **Hold expiry:** If the 10-minute hold expires before confirmation (countdown reaches 00:00): "Le créneau a expiré." + "Choisir un nouveau créneau" button that resets widget to Step 2 (date selection). Slot becomes available again server-side (E06S11 AC9 cron cleanup).

- [ ] AC9: **Error handling:**
  - `409` on confirm (race condition — slot taken between hold and confirm, E06S11 AC6): "Le créneau n'est plus disponible." + reset to Step 2
  - `422` on confirm (validation — name/email missing): display field-level errors
  - `502` on confirm (Google Calendar error, E06S11 AC10): "Erreur de connexion au calendrier. Veuillez réessayer dans quelques instants." + retry button
  - `422 gcal_not_connected` on availability: "Cet expert n'a pas encore configuré son calendrier." + hide booking widget, display "Contactez-nous" fallback
  - Network errors: generic retry message

- [ ] AC10: PostHog events:
  - `satellite.booking_widget_opened` (satellite_id, expert_id, source: "match_results" | "directory")
  - `satellite.booking_date_selected` (satellite_id, expert_id, date)
  - `satellite.booking_slot_selected` (satellite_id, expert_id, slot_start)
  - `satellite.booking_held` (satellite_id, expert_id, booking_id)
  - `satellite.booking_hold_expired` (satellite_id, expert_id, booking_id)
  - `satellite.booking_confirmed` (satellite_id, expert_id, booking_id)
  - `satellite.booking_error` (satellite_id, expert_id, error_type)

- [ ] AC11: Accessibility: keyboard navigable (Tab through dates → slots → form fields), focus moves to the widget on open and returns to CTA on close, `aria-label` on calendar dates and time slots, error messages linked to form fields via `aria-describedby`.

- [ ] AC12: Mobile-responsive: on viewports < 480px, calendar renders as a scrollable week view (not full month). Time chips wrap to multiple lines. Confirm form is full-width. All interactive elements have minimum 44px touch targets.

## Notes

- **Hold TTL is 10 minutes (E06S11 AC3).** The countdown timer is a UX affordance — the actual TTL is enforced server-side. If client and server drift, the server returns 409 and the UI recovers gracefully via AC9.
- **prospect_id requirement:** Booking hold (E06S11 AC3) requires a `prospect_id`. For prospects from the quiz funnel (E03S01–S03), this already exists. For directory prospects (E03S04), a prospect record must be created before booking — either via the identify flow or via a lightweight creation at booking time. Delivery to assess the cleanest integration path.
- **Email pre-fill:** If the prospect already identified via E03S03 email gate, the confirm form pre-fills the email field and makes it read-only. No double identification.
- **Timezone handling:** All API times are UTC. The widget converts to/from prospect's local timezone using `Intl.DateTimeFormat` + `Date` operations. The `?tz=` parameter on availability is for metadata echoing only — slots are always UTC.
- **Multiple widgets:** A prospect might open booking widgets for multiple experts simultaneously (e.g., comparing availability). Only one hold can be active per prospect at a time (E06S11 AC3 — max 3 active holds/prospect, E08S03 AC3). The UI should gracefully handle 409s when the prospect has too many active holds.
