# Story E06S40: Input Pre-Validation & Abuse Prevention

## User Story

**As a** platform operator,
**I want** multi-layered defenses against bot submissions, gibberish input, and repeated abuse of the extraction endpoint,
**so that** LLM costs are controlled, the matching engine receives quality data, and the funnel is resilient against automated exploitation.

---

## Context

The `POST /api/extract` endpoint invokes Workers AI (or OpenAI) for every submission — cost ~$0.002-0.004/request. Without pre-validation, a bot can trigger thousands of LLM calls. Industry research (DEC-120) revealed Turnstile only catches 33% of bots, and a real incident cost $12,000 in a weekend from an unprotected LLM endpoint. The current stack has Turnstile on `POST /api/prospects/submit` (E06S20) but NOT on `POST /api/extract` — the most expensive endpoint.

This story implements a 6-layer defense (DEC-120) plus abuse prevention for repeated human submissions (DEC-122). The layers are ordered cheapest-first: reject garbage before it costs anything.

---

## Acceptance Criteria

### Layer 1: Input pre-validation (before LLM)

- [ ] AC1: New `lib/inputValidation.ts` module:
  - `validateFreetext(text)` returns `{ valid: boolean, reason?: string }`.
  - Checks: (1) min 30 characters, (2) min 3 distinct words (split on whitespace + deduplicate), (3) no keyboard mashing patterns (regex: 4+ consecutive identical chars, e.g. `aaaa`, `!!!!`), (4) no character repetition patterns (e.g. `asdfasdf`), (5) no lorem ipsum (case-insensitive match on "lorem ipsum").
  - If invalid → returns reason string for client-facing error message.
  - Cost: ~0ms CPU. Pure synchronous function.

- [ ] AC2: `POST /api/extract` calls `validateFreetext()` BEFORE invoking any LLM. If invalid → 422 with `{ error: "invalid_input", message: <reason> }`. No LLM call, no cost incurred.

### Layer 2: Honeypot + timing

- [ ] AC3: `POST /api/extract` accepts an optional field `_hp` (honeypot). If `_hp` is non-empty → 200 with fake success response `{ extracted: true }` (silent rejection — do not reveal detection to bot). Satellite frontend renders a hidden CSS field (`display: none; position: absolute; left: -9999px`) with `name="_hp"`.

- [ ] AC4: `POST /api/extract` accepts a `_ts` field (page load timestamp, set by satellite JS on page load). If `Date.now() - _ts < 3000` (less than 3 seconds between page load and submit) → 200 with fake success response (same as honeypot — silent rejection). If `_ts` is missing or unparseable → proceed normally (backwards compatibility with existing clients).

### Layer 3: Tiered rate limiting

- [ ] AC5: Rate limiting on `POST /api/extract` (in addition to existing E06S20 limits on `/api/prospects/submit`):
  - Tier 1: 4 requests/minute/IP → Cloudflare managed challenge (429 with Retry-After header).
  - Tier 2: 10 requests/10 minutes/IP → hard block (429).
  - Tier 3: Exceeds Tier 2 → block 24h (429).
  - Implementation: Use existing `RATE_LIMITER` binding (E06S20) or add a second rate limiter scoped to `/api/extract`. Check both IP and session cookie (if present) to handle shared IPs.

### Layer 4: Turnstile placement correction

- [ ] AC6: Move Turnstile verification from its current placement to the confirmation action (when prospect confirms extracted requirements on Page 2, i.e., the submit to matching step). Turnstile on `POST /api/extract` is REMOVED (it was blocking legitimate users for only 33% bot detection). Turnstile on `POST /api/prospects/submit` is KEPT (existing E06S20 — this is the highest-intent moment). Update satellite frontend to render Turnstile widget on the confirmation step, not the freetext input step.

### Layer 5: Hard spending cap documentation

- [ ] AC7: Add `SETUP.md` section documenting the mandatory LLM provider spending cap: $50/day, $500/month. This is a non-negotiable operational control set at the provider dashboard level (OpenAI/Anthropic), not in application code. Include step-by-step instructions for setting caps on OpenAI dashboard and Workers AI billing.

### Layer 6: Circuit breaker

- [ ] AC8: New `lib/circuitBreaker.ts` module:
  - Tracks extraction cost per hour via KV counter (`extract-cost:{hour}` → accumulated cost in cents, TTL 7200s).
  - After each successful extraction, increment counter by estimated cost (configurable `EXTRACT_COST_CENTS` env var, default 0.4 cents = $0.004).
  - Before each extraction, check counter. If > 1000 (= $10/hour) → 503 with `{ error: "service_overloaded", message: "Service temporairement surcharge. Reessayez dans quelques minutes.", retry_after: 300 }`.
  - Auto-resume: next hour's counter starts at 0 (KV TTL handles this).

### Abuse prevention (DEC-122)

- [ ] AC9: **Repeated submissions without email:** After 1 extraction without email identification:
  - Set cookie `_extract_count=1` (HttpOnly, SameSite=Lax, TTL 86400s).
  - On subsequent `POST /api/extract`: if `_extract_count >= 1` AND prospect not identified (no valid prospect JWT) → 403 with `{ error: "identification_required", message: "Identifiez-vous pour continuer." }`.
  - Cookie is primary control. IP-based fallback: KV counter `extract-ip:{ip}` (TTL 86400), max 3 extractions per IP per 24h without identification.

- [ ] AC10: **Max output tokens:** `POST /api/extract` LLM call must set `max_tokens: 500`. The extraction response is structured JSON, not prose. This limits worst-case cost per request.

### Flow token

- [ ] AC11: After successful extraction + Turnstile verification, server generates a signed flow token (HMAC-SHA256 using `PROSPECT_TOKEN_SECRET`, payload: `{ prospect_id, extraction_id, exp: now + 1800 }`). Token returned to client. `POST /api/prospects/submit` requires this token in `Authorization` header or body field `flow_token`. If missing or invalid → 403 `invalid_flow_token`. Prevents direct API calls to `/submit` with fabricated extraction data.

### Tests

- [ ] AC12: Unit tests for `lib/inputValidation.ts` (all patterns: valid text, too short, too few words, keyboard mashing, lorem ipsum). Unit tests for `lib/circuitBreaker.ts` (under threshold, over threshold, counter reset). Integration tests for honeypot, timing, rate limit, cookie-based repeated submission, flow token. All existing tests pass. `tsc` clean across all workers.

## Notes

- **Backwards compatibility:** The honeypot (`_hp`) and timing (`_ts`) fields are optional. Existing clients that don't send them are not blocked. Cookie-based extraction limit applies to new sessions only.
- **Silent rejection pattern:** Honeypot and timing return `200` with fake data, not `4xx`. This prevents bots from detecting the defense and adapting. The fake response mimics a real extraction response shape but with empty/generic values.
- **Turnstile is supplementary, not primary.** The 6-layer design ensures that even if Turnstile fails (which it does 67% of the time), layers 1-3 and 5-6 catch most abuse. The order matters: cheapest checks first.
- **Flow token vs. Turnstile:** The flow token (AC11) is a stronger defense than Turnstile for the submit endpoint. It proves that the client went through a legitimate extraction → confirmation → submit flow, not a direct API call. Turnstile on submit is kept as an additional layer but the flow token is the primary gate.
