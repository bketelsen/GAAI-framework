# Story E02S10: Expert Profile Mandatory Fields + Credit Milestones

## User Story

**As an** expert signing up on Callibrate,
**I want** clear guidance on which profile fields are required to start receiving leads, and visible progress toward unlocking my 100EUR welcome credits,
**so that** I complete my profile efficiently and see immediate value from the platform.

---

## Context

New experts receive 100EUR of free credits (DEC-68/E06S33 — LemonSqueezy `subscription_created` webhook credits 10000 centimes). Currently, credits are released all at once with no completion gate. DEC-116 defines 4 mandatory fields. DEC-117 splits credits into 3 milestones mapped to platform value (Matchable 40EUR, Bookable 40EUR, Trustworthy 20EUR). This story implements both the validation enforcement and the milestone credit allocation.

The onboarding wizard (E02S02) already collects most fields. This story adds: (1) backend validation of mandatory fields with milestone tracking, (2) dashboard UI showing milestone progress, (3) anti-gaming protections.

## Acceptance Criteria

### Schema changes

- [ ] AC1: DB migration adding 3 columns to `experts` table:
  - `milestone_matchable_at` (TIMESTAMPTZ, nullable) — set when: display_name present + bio >= 50 chars + 3+ skill tags.
  - `milestone_bookable_at` (TIMESTAMPTZ, nullable) — set when: availability rules exist (>= 1 active rule in `expert_availability_rules`, see E02S11).
  - `milestone_trust_at` (TIMESTAMPTZ, nullable) — set when: photo_url is set OR portfolio_url is set OR any external link is set.
  - All 3 columns are immutable once set: `CHECK` constraint or application-level enforcement (IF column IS NULL AND conditions met THEN SET + credit).

### Backend milestone logic

- [ ] AC2: New `lib/milestones.ts` module:
  - `checkMilestones(expert)` evaluates all 3 milestone conditions against the current expert record. Returns `{ matchable: boolean, bookable: boolean, trust: boolean }`.
  - `processMilestoneCredits(expert_id, milestoneResults)`:
    - For each milestone where `condition = true` AND `milestone_X_at IS NULL`:
      - Sets the timestamp (`UPDATE experts SET milestone_X_at = NOW() WHERE id = expert_id AND milestone_X_at IS NULL`).
      - Credits the expert via `credit_transactions` INSERT (same pattern as E06S33 welcome credit):
        - Matchable: +4000 centimes, type `milestone_matchable`, description "Profil matchable - 40EUR offerts".
        - Bookable: +4000 centimes, type `milestone_bookable`, description "Profil reservable - 40EUR offerts".
        - Trust: +2000 centimes, type `milestone_trust`, description "Profil de confiance - 20EUR offerts".
    - Returns list of newly unlocked milestones (for UI toast notifications).
  - **Atomicity:** Each milestone unlock is a single SQL transaction (UPDATE expert + INSERT credit_transaction). Use Hyperdrive + postgres.js for ACID.

- [ ] AC3: `PATCH /api/experts/:id/profile` (existing endpoint) calls `processMilestoneCredits()` after successful profile update. If any milestones newly unlocked → include `{ milestones_unlocked: ["matchable", "trust"] }` in response body.

- [ ] AC4: **Anti-gaming: email normalization at signup.** Modify `POST /api/experts/register`:
  - Normalize email before creating expert record (strip `+tag`, normalize Gmail dots, lowercase).
  - Check uniqueness on normalized email. If duplicate → 409 `email_already_registered`.
  - Store both `email` (original) and `normalized_email` (computed) on expert record.

- [ ] AC5: **Anti-gaming: disposable email check at signup.** Reuse `isDisposableEmail()` from E06S39's `lib/emailValidation.ts`. If disposable → 422 `disposable_email` with message "Les adresses email temporaires ne sont pas acceptees."

- [ ] AC6: **Anti-gaming: 72h activation delay.** Milestone credit transactions have a `available_at` timestamp = `created_at + 72h`. The `credit_balance` calculation in `lib/billingFilter.ts` (E06S34) must only count transactions where `available_at <= NOW()`. New column `available_at` (TIMESTAMPTZ, default NOW()) on `credit_transactions` table + migration.

### Dashboard UI

- [ ] AC7: New "Profile Completion" section on dashboard home (`/dashboard`):
  - 3 milestone cards, each showing: milestone name, status (locked/unlocked/pending), credit amount, conditions remaining.
  - Visual: unlocked = green checkmark + "40EUR credites" with date. Locked = grey lock icon + "Completez: [missing items]". Pending (72h delay) = amber clock icon + "Disponible le [date]".
  - Progress bar showing overall completion (0/3, 1/3, 2/3, 3/3).

- [ ] AC8: **Profile page mandatory field indicators.** On `/dashboard/profile` (E02S03), fields that are mandatory for milestones show a badge: "Requis pour devenir matchable" (or bookable/trustworthy). Fields missing for the next locked milestone are highlighted with a subtle border or icon.

- [ ] AC9: **Toast notification on milestone unlock.** When `PATCH /api/experts/:id/profile` returns `milestones_unlocked`, display a celebratory toast: "Felicitations ! Vous etes maintenant matchable. 40EUR credites sur votre compte." Use shadcn/ui `toast` component.

### Validation enforcement

- [ ] AC10: **Mandatory fields validation on profile save.** `PATCH /api/experts/:id/profile` validates:
  - `display_name`: required, min 2 chars.
  - `bio`: required, min 50 chars (server-side count — client may show character counter).
  - `skill_tags`: required, min 3 items from predefined list.
  - These fields CANNOT be cleared once set (the profile update rejects attempts to null them or set below minimum). Other fields remain optional.

### LemonSqueezy welcome credit adjustment

- [ ] AC11: Modify E06S33 welcome credit logic. Currently: `subscription_created` → +10000 centimes (100EUR) all at once. New behavior: `subscription_created` → +0 centimes (no blanket credit). Credits are now exclusively allocated via milestones (AC2). If an expert already has a welcome credit from the old logic (early adopters), milestones still apply — the milestone check is `IF milestone_X_at IS NULL`, not "IF balance < 100EUR".

### Tests

- [ ] AC12: Unit tests for `lib/milestones.ts` (each milestone condition, credit allocation, idempotency — calling twice doesn't double-credit). Integration tests for profile save + milestone unlock flow. Anti-gaming tests (normalized email duplicate, disposable email, 72h delay). All existing tests pass. `tsc` clean.

## Notes

- **Milestone 2 "Bookable" depends on E02S11.** The `checkMilestones()` function queries `expert_availability_rules` table. If E02S11 is not yet delivered, Milestone 2 is permanently locked. Delivery should sequence E02S11 before or alongside E02S10. Alternatively, AC2 can check for the table's existence and skip the bookable milestone if the table doesn't exist yet.
- **72h activation delay (AC6):** This prevents create-account-grab-credits-delete-account cycling. The 72h window gives time to detect multi-account abuse patterns. The delay applies only to milestone credits, not to purchased credits.
- **LemonSqueezy adjustment (AC11):** This is a breaking change for the existing welcome credit flow. Ensure the migration handles in-flight signups gracefully. Experts who signed up before this change and already received 100EUR keep their credits — milestones are additive checks, not retroactive.
