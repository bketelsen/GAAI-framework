# Story E02S02: Expert Signup + Onboarding Wizard

## User Story

**As a** new AI integration expert,
**I want** to create an account and complete a guided onboarding wizard,
**so that** my profile is complete enough for the matching engine to start sending me qualified leads.

---

## Context

The signup and onboarding flow is the expert's first interaction with the platform. It must be fast (< 5 minutes total), progressive (no overwhelming forms), and result in a profile that's immediately matchable.

The flow: `callibrate.io` CTA → `/signup` → email/password → `/onboarding` (4 steps) → `/dashboard`. If an expert signs up but doesn't complete onboarding, they're redirected to `/onboarding` on next login (detected by missing `display_name` on the expert record).

**Supabase Auth PKCE flow:** Signup creates a Supabase user, then calls `POST /api/experts/register` (E06S03) to create the expert record. Login uses `signInWithPassword`. Session managed via cookies (E02S01 AC4).

---

## Acceptance Criteria

### Signup

- [ ] AC1: Signup page (`/signup`): email + password fields + "Créer mon compte" button. Password requirements: min 8 characters, displayed inline. Calls `supabase.auth.signUp({ email, password })`. On success: sets session cookie, calls `POST {CORE_API_URL}/api/experts/register` via server action, redirects to `/onboarding`. On error: inline error message (email taken, weak password, network error).

- [ ] AC2: Login page (`/login`): email + password fields + "Se connecter" button + "Mot de passe oublié ?" link. Calls `supabase.auth.signInWithPassword()`. On success: sets session cookie, checks if expert has `display_name` — if yes → `/dashboard`, if no → `/onboarding`. "Mot de passe oublié" → `/forgot-password` page that calls `supabase.auth.resetPasswordForEmail()`.

- [ ] AC3: Already authenticated users visiting `/signup` or `/login` are redirected to `/dashboard` (or `/onboarding` if incomplete profile) via loader check.

### Onboarding Wizard

- [ ] AC4: Onboarding page (`/onboarding`): 4-step wizard with progress indicator (step dots or progress bar showing 1/4, 2/4, etc.). Back/Next navigation. Current step persisted in URL search param (`?step=1`). Browser back button works (navigates to previous step).

- [ ] AC5: **Step 1 — Basic Info:** `display_name` (required, text), `headline` (required, max 200 chars, placeholder: "Ex: Expert n8n & automatisation IA"), `bio` (optional, max 2000 chars, textarea). All validated with zod. "Suivant" saves via `PATCH /api/experts/:id/profile` action.

- [ ] AC6: **Step 2 — Expertise:** `skills` (multi-select from predefined list + freetext add, min 1, max 15), `verticals` (checkboxes from known verticals list), `rate_min` / `rate_max` (number inputs, EUR/hour), `outcome_tags` (freetext tags, max 10, placeholder: "Ex: 25h/semaine économisées sur le traitement RFP"). Saved via PATCH profile action.

- [ ] AC7: **Step 3 — Preferences:** Market positioning fields saved as `preferences` JSONB:
  - `career_stage`: radio (junior/medior/senior/high-ticket)
  - `work_mode`: radio (remote/on-site/hybrid)
  - `availability`: select (full-time freelance/side projects/X days per week)
  - `budget_range`: range selector (min/max budget accepted)
  - `project_stage`: checkboxes (exploration/defined scope/urgent execution)
  - `languages`: multi-select
  - `geo_zones`: multi-select (regions)
  - `industries`: multi-select
  Saved via PATCH profile (preferences JSONB merge).

- [ ] AC8: **Step 4 — Google Calendar:** Connection status display. If not connected: "Connecter Google Calendar" button → calls `GET /api/experts/:id/gcal/auth-url` → opens OAuth flow (same as E06S10). If connected: shows connected email + "Passer cette étape" option. Skip button available — GCal is optional at onboarding.

- [ ] AC9: After Step 4 completion (or skip): redirect to `/dashboard` with a success toast "Votre profil est prêt ! Vous commencerez à recevoir des leads qualifiés."

- [ ] AC10: Incomplete onboarding detection: the auth layout loader checks if the expert's `display_name` is null. If so, any `/dashboard/*` route redirects to `/onboarding` with the appropriate step (inferred from which fields are missing).

### Pricing Model Transparency (DEC-131 §6)

- [ ] AC15: **Pricing callout on `/signup` page.** Below the form, before the submit button, display a permanent inline block (not a modal, not a tooltip — always visible) with the pricing rule in plain language. Required copy (or equivalent clarity):
  > *"Callibrate vous apporte un prospect → vous payez par booking réussi.*
  > *Vous partagez votre lien direct → le prospect vient à vous → **gratuit**."*

  Displayed in a neutral info-card style (not alarming, not a warning). Link "En savoir plus sur la facturation" → `callibrate.io/experts#tarifs` (E01S06 pricing anchor).

- [ ] AC16: **"Voici comment ça marche" explainer screen** as the first screen of `/onboarding` (step 0, not counted in the 4-step progress bar). Shown only on first visit (flag: `onboarding_explainer_seen` in localStorage). Cannot be skipped — single "Compris, on commence !" CTA proceeds to Step 1. Content:
  - **Block 1 — Le matching :** Callibrate analyse vos compétences et trouve des prospects qui ont besoin exactement de vous.
  - **Block 2 — Facturation :** Vous payez uniquement par booking réussi, uniquement pour les prospects que Callibrate vous apporte. *Si le prospect vient via votre propre lien direct — c'est gratuit.*
  - **Block 3 — Votre lien direct :** Une fois votre profil complété, vous recevez un lien de réservation personnalisé à partager où vous voulez. Aucun frais pour ces leads.

  PostHog event: `expert.onboarding_explainer_viewed`.

## Notes

- **Predefined skills list:** Delivery maintains a hardcoded array of common AI/automation skills (n8n, Make, Zapier, Python, Claude, GPT, LangChain, React, Docker, etc.). Experts can also type custom skills. List is not fetched from API — it's a static constant in the frontend.
- **Preferences JSONB:** The preferences object mirrors the `expert.preferences` JSONB used by the matching engine. The onboarding wizard is the primary way experts populate this field. Step 3 generates the exact structure the matching engine expects.
- **Progressive disclosure:** 4 steps instead of one long form. Research shows multi-step forms have 86% higher completion rates than single-page equivalents (Typeform data). Each step feels achievable.
- **Welcome credit (100€):** Not displayed in onboarding — the expert discovers it in the billing page after completing onboarding. Mentioning money during onboarding shifts focus from value to cost.
- **signUp + register race condition:** If `supabase.auth.signUp()` succeeds but `POST /api/experts/register` fails (network/server error), the Supabase user exists without an expert record. Recovery: on subsequent login, the auth layout loader detects missing expert record (no `display_name`) and retries the register call before redirecting to `/onboarding`. Display a clear error message if the initial register call fails so the expert knows to retry.
- **PostHog events (Delivery must include):** `expert.signup_completed` (on successful signUp + register), `expert.onboarding_step_completed` (step number, fields filled — per step), `expert.onboarding_completed` (on final step done / redirect to dashboard), `expert.login` (on successful signIn). Server-side capture via `captureEvent()` in actions, same pattern as E07S03. These feed the Expert Activation Funnel dashboard (E07S04).
