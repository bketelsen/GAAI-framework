# Story E06S18: Hyperdrive + postgres.js + Smart Placement + Bug Fixes (Foundation)

## User Story

**As a** platform operator,
**I want** all database queries to run over pooled TCP connections via Hyperdrive with postgres.js,
**so that** query latency drops ~6× (especially multi-query paths like score-computation), connection exhaustion risk is eliminated, and the codebase is ready for Vectorize/D1/Analytics Engine integration.

---

## Context

The current stack uses `@supabase/supabase-js` (PostgREST/HTTP) for all 48 database queries across 11 files. Each Worker invocation opens a new HTTP connection — no pooling, no query caching. The score-computation consumer (7 sequential queries) pays ~210ms in HTTP overhead alone.

Hyperdrive (free on Workers Paid) provides TCP connection pooling + automatic query caching at the edge. Combined with `postgres` (postgres.js — tagged template SQL, zero sub-dependencies, serverless-native), this yields ~35ms for the same 7-query path.

This story also fixes two structural bugs:
- **B1:** `prospects.ts` inserts match rows for ALL experts instead of top-20
- **B2:** `matches.ts` bypasses KV cache with a direct DB query for experts

Supabase JS is retained exclusively for `auth.getUser()` in the auth middleware.

## Notes

Migration pattern is mechanical (query-by-query):
```typescript
// BEFORE (Supabase JS):
const { data, error } = await supabase.from('experts').select('id, profile').eq('id', id).single();
if (error) throw error;

// AFTER (postgres.js):
const [row] = await sql<ExpertRow[]>`SELECT id, profile FROM experts WHERE id = ${id}`;
if (!row) throw new Error('Not found');
```
