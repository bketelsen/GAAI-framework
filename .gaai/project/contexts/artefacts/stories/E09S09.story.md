# Story: Fix main vs production branch inconsistency in deploy workflows

## User Story

**As a** founder managing deployments,
**I want** all production deploy workflows to use consistent branch triggers,
**so that** production deploys are predictable and not split across two different branch names.

---

## Context

The repo's default branch is `production`. However:

- `deploy-production.yml` triggers on tags `v*` (correct pattern for backend workers)
- `deploy-dashboard.yml` triggers on push to `main` for production deploys
- `deploy-landing.yml` triggers on push to `main` for production deploys

If `main` and `production` diverge (or if `main` doesn't exist), dashboard and landing production deploys silently stop working. This is a governance gap — the branch naming is inconsistent with the documented convention.

---

## Acceptance Criteria

- [ ] AC1: `deploy-dashboard.yml` production job triggers on push to `production` branch (not `main`), path-filtered to `workers/frontend/dashboard/**`
- [ ] AC2: `deploy-landing.yml` production job triggers on push to `production` branch (not `main`), path-filtered to `workers/frontend/landing/**`
- [ ] AC3: Both workflows' staging jobs continue to trigger on push to `staging` (unchanged)
- [ ] AC4: No changes to `deploy-staging.yml`, `deploy-production.yml`, or `preview-deploy.yml`
- [ ] AC5: If `main` branch exists as an alias, document whether it should be deleted or kept as a mirror

> All acceptance criteria must be verifiable. Vague criteria are invalid.

---

## Out of Scope

- Changing the backend production deploy trigger (tag `v*` is intentional)
- Renaming the default branch
- Branch protection rules (requires GitHub Pro)

---

## Dependencies

- None.

---

## Notes

- The `main` branch may be a GitHub default that was never renamed. Check if `main` and `production` are the same ref or diverged.
- If they're the same (identical SHA), this is cosmetic but still a governance risk. If they've diverged, this is a silent deploy failure waiting to happen.
