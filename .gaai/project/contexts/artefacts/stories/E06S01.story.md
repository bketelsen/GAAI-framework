# Story E06S01: Project Bootstrap — Supabase + Cloudflare Workers + Queues + KV

## User Story

**As a** developer starting a new delivery session,
**I want** all infrastructure services configured and connected,
**so that** I can build and deploy any feature without setup overhead.

---

## Context

This is the foundation story. Every subsequent E06 story depends on it. No feature can be built until the Supabase project, Cloudflare Workers project, Queues, and KV namespaces are initialized and interconnected. Includes CI/CD so that every git push deploys automatically.

Stack: Supabase (PostgreSQL + Auth + Storage) · Cloudflare Workers · Cloudflare Queues · Cloudflare KV · TypeScript · Wrangler.

---

## Acceptance Criteria

- [ ] AC1: Supabase project initialized — PostgreSQL DB, Auth (magic link enabled), Storage bucket `avatars` created
- [ ] AC2: Cloudflare Workers project initialized with Wrangler, TypeScript configured, `wrangler.toml` defines three environments: `dev`, `staging`, `production`
- [ ] AC3: Two Cloudflare Queues created — `callibrate-core-queue-email-notifications-{env}` and `callibrate-core-queue-lead-billing-{env}` — each with a corresponding DLQ (`*-dlq`). Convention: `{scope}-{entity}-{resource}-{env}` (DEC-32). No `matching-jobs` queue (DEC-33).
- [ ] AC4: Three KV namespaces created with explicit names and bound in `wrangler.toml`: `callibrate-core-kv-sessions-{env}` (binding: `SESSIONS`), `callibrate-core-kv-rate-limiting-{env}` (binding: `RATE_LIMITING`), `callibrate-core-kv-feature-flags-{env}` (binding: `FEATURE_FLAGS`)
- [ ] AC5: All secrets bound to Workers via environment secrets (not hardcoded): `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_KEY`
- [ ] AC6: `GET /api/health` endpoint returns `200` with JSON: `{ status: "ok", supabase: "connected", queues: ["email-notifications", "lead-billing"] }`
- [ ] AC7: CI/CD configured — git push to `main` branch triggers auto-deploy to `staging`; git tag `v*` triggers deploy to `production`
- [ ] AC8: Local `wrangler dev` runs without errors and proxies to Supabase dev project

## Notes

Use `supabase init` for local dev + `supabase link` to connect to remote project. Use Wrangler secrets for all credentials — never commit secrets to git. The health endpoint must verify Supabase connectivity with a lightweight ping (e.g., `SELECT 1` via Supabase REST).
