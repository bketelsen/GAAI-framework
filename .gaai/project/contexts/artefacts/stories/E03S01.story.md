# Story E03S01: Prospect Freetext Input Page

## User Story

**As a** prospect arriving on a satellite site,
**I want** to describe my AI integration project in my own words,
**so that** the platform can understand my needs without forcing me through a rigid quiz.

---

## Context

The entry point to the prospect funnel. Prospect clicks "Décrire mon projet" on the landing page (E06S14 AC5) → arrives at `/match`. The page presents a branded freetext textarea. On submit, the satellite frontend calls `POST /api/extract` (Core API, E06S08/E06S12) which returns structured requirements with per-field confidence. The page then transitions to the confirmation step (E03S02) — client-side state transition, no full page reload.

All UI rendering follows the satellite Worker pattern established in E06S14: server-rendered HTML with design tokens from `satellite_configs.theme`, inline CSS, PostHog tracking. Client-side JavaScript handles form submission and API calls.

---

## Acceptance Criteria

- [ ] AC1: `GET /match` route on satellite Worker renders a full HTML page with: branded header (logo + brand name from config), headline from `config.content` (fallback: "Décrivez votre projet"), `<textarea>` with max 2000 characters and placeholder text, submit button "Analyser mon projet". All styles use design tokens from `config.theme` (same pattern as landing page).

- [ ] AC2: Client-side character counter below textarea shows `{current}/{max}` and changes color (warn state) when >1800 characters. Client-side validation: min 30 characters, max 2000. Submit button disabled when validation fails. Inline error message below textarea when validation fails on submit attempt.

- [ ] AC3: On form submit, client-side JS calls `POST {CORE_API_URL}/api/extract` with `{ text: string, satellite_id: config.id }` via `fetch()`. The Core API URL is injected into the HTML by the satellite Worker as a `data-api-url` attribute or inline `<script>` config variable. CORS is already configured on Core API (E06S07 AC9).

- [ ] AC4: During API call: submit button shows spinner + disabled state, textarea becomes readonly, "Analyse en cours..." message displayed below the form. No page reload.

- [ ] AC5: On successful extraction response: client-side state stores the full response `{ requirements, confidence, needs_confirmation, confirmation_questions, ready_to_match }`. Page transitions to the confirmation step (E03S02 UI) without server roundtrip — DOM replacement or same-page state change.

- [ ] AC6: On extraction error (network error, 4xx, 5xx): inline error message "Une erreur est survenue. Veuillez réessayer." with a retry button that re-enables the form. Error details logged to console. No page reload, no user data lost.

- [ ] AC7: PostHog events (follow E07S02 pattern, cookieless persistence:memory):
  - `satellite.match_page_viewed` on page load (satellite_id, referrer, UTM params)
  - `satellite.match_form_submitted` on form submit (satellite_id, text_length)
  - `satellite.extraction_completed` on successful extraction (satellite_id, ready_to_match, needs_confirmation_count)
  - `satellite.extraction_error` on extraction failure (satellite_id, error_status)

- [ ] AC8: `<head>` includes: `<meta name="robots" content="index, follow">`, `<link rel="canonical" href="https://{config.domain}/match">`, OG tags (og:title, og:description, og:url), `<title>` from config.content (fallback: "{brand.name} — Décrivez votre projet"). JSON-LD WebPage structured data.

- [ ] AC9: Mobile-responsive: textarea and button stack vertically on viewports < 640px. Minimum touch target 44px for all interactive elements. Textarea height auto-expands with content on mobile (min 120px, max 400px).

- [ ] AC10: Back navigation: if the prospect navigates back from E03S02, the textarea retains their text (browser history state or sessionStorage). No re-entry from scratch.

---

## Out of Scope

- File/image upload (text only at MVP)
- Voice input
- Multi-language support (MVP is French only — internationalization post-MVP)
- Autocomplete or suggestion dropdown while typing
- Rate limiting on extract (handled server-side on Core API, E06S20)

## Notes

- The Core API URL must be injected by the satellite Worker at render time from an environment variable (`CORE_API_URL`) — not hardcoded. Different environments (staging/production) use different Core API URLs.
- The extraction API was migrated from Claude to GPT-4o-mini in E06S12. The frontend is API-agnostic — it only cares about the response shape.
- No Turnstile on this step. Turnstile is on `POST /api/prospects/submit` (E06S20), which happens in E03S02.
- The `/match` route is already referenced by the landing page CTA (`<a href="/match" class="cta">`), by `sitemap.xml` (E06S14 AC7), and expected by PostHog funnel dashboards.
- Implementation approach (vanilla JS, lightweight framework, or MPA) is a Delivery decision. ACs define user-facing behavior only.
