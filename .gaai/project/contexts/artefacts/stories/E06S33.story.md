# Story E06S33: LS Usage-Based Subscription + Welcome Credit + Usage Reporting Cron

## User Story

**As an** expert,
**I want** to activate my lead pipeline by entering my card once, receive a 100 EUR welcome credit, and have confirmed leads automatically billed at the end of each billing cycle,
**so that** I can start receiving leads immediately without per-lead checkout friction.

---

## Context

DEC-68 specifies LS usage-based subscription: 0 EUR/month base, card on file, auto-charge end-of-cycle based on reported usage. The platform reports usage to LS only AFTER the 7-day flag window expires (leads auto-confirmed by cron or explicitly confirmed by expert). This decouples the flag/restore mechanism (E06S32) from actual LS billing.

Flow:
1. Expert subscribes via LS overlay (triggered by E05 UI, out of scope here) → LS webhook → subscription tracked + 100 EUR welcome credit
2. Leads deducted/flagged/confirmed internally (E06S32)
3. Hourly cron: auto-confirm leads past 7-day flag window
4. After confirmation: report usage to LS
5. End of LS billing cycle: LS auto-charges the saved card for total reported usage

---

## Acceptance Criteria

- [ ] AC1: `POST /api/webhooks/lemonsqueezy` endpoint — no auth middleware (external webhook). HMAC-SHA256 signature verification using `LEMON_SQUEEZY_WEBHOOK_SECRET` env secret. Invalid signature → 401. Missing header → 401.
- [ ] AC2: Webhook handles `subscription_created` event: UPDATE experts SET ls_subscription_id, ls_subscription_status='active'. INSERT credit_transactions (type='welcome_credit', amount=+10000, balance_after). UPDATE experts SET credit_balance = credit_balance + 10000. Expert identified by custom data or email match.
- [ ] AC3: Webhook handles `subscription_updated` event: UPDATE experts SET ls_subscription_status from event payload (maps to: active, past_due, cancelled, unpaid).
- [ ] AC4: Webhook handles `subscription_payment_failed` event: UPDATE experts SET ls_subscription_status='past_due'. Queue email notification via EMAIL_NOTIFICATIONS queue (type 'expert.billing.payment_failed').
- [ ] AC5: Webhook is idempotent — uses LS event ID as dedup key via KV (`idem:ls-webhook:{event_id}`, TTL 86400s). Same pattern as queue consumers.
- [ ] AC6: Cron trigger `0 * * * *` (hourly): query leads WHERE status='pending' AND created_at < now() - INTERVAL '7 days'. Batch UPDATE status='confirmed', confirmed_at=now(). Use staging-accelerated interval via `FLAG_WINDOW_MS` env var (same sleep acceleration pattern as E06S16).
- [ ] AC7: After batch confirmation: for each confirmed lead, report usage to LS API (usage-based billing endpoint) with amount=lead.amount. On success: UPDATE lead SET status='usage_reported', usage_reported_at=now().
- [ ] AC8: If LS usage reporting fails for a lead: leave status='confirmed' (do not revert), log error with lead_id and LS error. Cron retries on next run (picks up leads WHERE status='confirmed' AND usage_reported_at IS NULL).
- [ ] AC9: Webhook route registered BEFORE authenticated blocks in `src/index.ts` (same placement rule as E06S10 OAuth callbacks — external webhooks carry no JWT)
- [ ] AC10: `LEMON_SQUEEZY_WEBHOOK_SECRET` added to Env interface in `src/types/env.ts`
- [ ] AC11: Cron declared in `wrangler.toml` at top-level `[triggers]` AND `[env.staging.triggers]` AND `[env.production.triggers]` (same pattern as E06S11 booking crons)
- [ ] AC12: `tsc --noEmit` — zero errors
- [ ] AC13: `vitest run` — all tests pass + new tests: webhook signature validation (valid/invalid/missing), subscription_created (welcome credit + status update), subscription_payment_failed (status + notification queued), cron auto-confirm (respects flag window, skips already flagged), usage reporting (happy path + LS failure → retry on next run)

## Notes

**LS Usage-Based Billing API:** The Delivery Agent should consult LS API docs for the exact usage reporting endpoint. Likely POST `/v1/subscription-items/{id}/usage-records` or equivalent. The usage report requires the LS subscription item ID (derived from ls_subscription_id on experts table).

**Webhook route placement:** same pattern as E06S10 OAuth callback — unauthenticated external callback registered BEFORE JWT-gated blocks in index.ts.

**Cron scheduling:** hourly is sufficient for 7-day flag windows. The staging env var `FLAG_WINDOW_MS` (e.g., 60000 = 1 minute) accelerates the window for smoke testing — same acceleration pattern as E06S16 `SURVEY_DELAY_7D_MS`.
