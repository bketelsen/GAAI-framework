# Story: Matching-engine unit test suite

## User Story

**As a** developer working on the matching-engine,
**I want** unit tests covering the core matching logic,
**so that** regressions in scoring, pricing, expert pool, and vectorize are caught before deployment.

---

## Context

`callibrate-matching` has 10 source files and zero tests. It contains critical business logic:
- `score.ts` — match scoring algorithm
- `pricing.ts` — pricing/rate logic
- `expertPool.ts` — expert pool queries and filtering
- `d1ExpertPool.ts` — D1 edge database queries
- `vectorize.ts` — embedding and vector search
- `index.ts` — Hono routes and request handling

The core API worker (`callibrate-core`) has 28 test files. The matching-engine — which handles the most business-critical function (matching prospects to experts) — has none.

---

## Acceptance Criteria

- [ ] AC1: `vitest` is added as a devDependency in `workers/backend/matching-engine/package.json`
- [ ] AC2: A `vitest.config.ts` file exists in `workers/backend/matching-engine/`
- [ ] AC3: A `test` script is added to `package.json`: `"test": "vitest run"`
- [ ] AC4: `score.ts` has a corresponding test file with tests covering: scoring formula, edge cases (empty skills, zero rates, missing fields), and score ordering correctness
- [ ] AC5: `pricing.ts` has a corresponding test file with tests covering: rate calculation, boundary values, currency/format edge cases
- [ ] AC6: `expertPool.ts` has a corresponding test file with tests covering: pool filtering, fallback behavior, empty pool handling
- [ ] AC7: All tests pass with `vitest run` — zero failures, zero skipped
- [ ] AC8: `type-check` script (`tsc --noEmit`) passes cleanly
- [ ] AC9: Root `package.json` `test` script is updated to include matching-engine: `npm -w callibrate-matching run test`

> All acceptance criteria must be verifiable. Vague criteria are invalid.

---

## Out of Scope

- Testing `index.ts` routes (requires HTTP mocking — lower priority)
- Testing `d1ExpertPool.ts` and `vectorize.ts` (requires Cloudflare binding mocks — future integration tests)
- Coverage thresholds or coverage reporting
- `@cloudflare/vitest-pool-workers` integration (future epic)

---

## Dependencies

- None. Can be implemented independently.

---

## Notes

- Focus on pure functions first: `score.ts` and `pricing.ts` have deterministic logic that is straightforward to test without mocking CF bindings.
- `expertPool.ts` may need minimal mocking (postgres queries). Use the same patterns as `callibrate-core` tests.
- The Delivery Agent should read existing test files in `workers/backend/api/src/` for conventions (import style, mock patterns, assertion style).
- This story should ideally be delivered before or in parallel with E09S02 so that the CI pipeline has meaningful tests to run.
