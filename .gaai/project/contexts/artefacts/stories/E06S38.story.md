# Story E06S38: Dashboard Read API Endpoints

## User Story

**As the** expert dashboard frontend,
**I need** dedicated API endpoints for leads, bookings, billing, and aggregate stats,
**so that** dashboard pages can load data efficiently with proper pagination and filtering.

---

## Context

The existing Core API has handlers for expert registration, profile CRUD, and Google Calendar OAuth. The dashboard (E02S05–E02S08) needs 4 additional GET endpoints that aggregate data across multiple tables with joins, pagination, and filtering. These endpoints follow the same auth pattern as `handleGetProfile` (JWT + ownership check) and use Hyperdrive/postgres.js for queries.

All 4 endpoints live in `workers/backend/api/src/handlers/experts/` alongside existing handlers. They share the same `AuthUser` middleware and `createSql(env)` pattern.

Also includes `GET /api/experts/public` (public expert listing for directory, E03S04 dependency) — no auth required, anonymized data only.

---

## Acceptance Criteria

### GET /api/experts/:id/leads

- [ ] AC1: New handler `handleGetLeads(request, env, user, expertId)` in `workers/backend/api/src/handlers/experts/leads.ts`. JWT auth + ownership check (same pattern as `handleGetProfile`).

- [ ] AC2: Query parameters: `?page=1&per_page=20&status=new|confirmed|flagged|all` (default: `all`, `per_page` max 50). Returns `{ leads: Lead[], total: number, page: number, per_page: number }`.

- [ ] AC3: Each lead includes: `id`, `status`, `price_cents`, `created_at`, `confirmed_at`, `flagged_at`, `flag_reason`, `flag_window_expires_at`, `evaluation_score`, `evaluation_notes`, `conversion_declared`, `prospect` (embedded: `{ id, email, requirements }` from prospects table via booking), `match_score` (from matches table), `booking` (embedded: `{ id, starts_at, status }` from bookings table if exists).

- [ ] AC4: Results ordered by `created_at DESC`. Pagination is offset-based (`LIMIT per_page OFFSET (page-1)*per_page`).

### GET /api/experts/:id/bookings

- [ ] AC5: New handler `handleGetBookings(request, env, user, expertId)` in `workers/backend/api/src/handlers/experts/bookings.ts`. JWT auth + ownership check.

- [ ] AC6: Query parameters: `?period=upcoming|past|all&page=1&per_page=20` (default: `upcoming`). `upcoming` = `starts_at > NOW() AND status NOT IN ('cancelled')`. `past` = `starts_at <= NOW() OR status = 'cancelled'`.

- [ ] AC7: Each booking includes: `id`, `starts_at`, `ends_at`, `status`, `meeting_url`, `cancel_reason`, `prospect` (embedded: `{ id, email, name }` from prospects table), `created_at`.

- [ ] AC8: Results ordered by `starts_at ASC` for upcoming, `starts_at DESC` for past.

### GET /api/experts/:id/billing

- [ ] AC9: New handler `handleGetBilling(request, env, user, expertId)` in `workers/backend/api/src/handlers/experts/billing.ts`. JWT auth + ownership check.

- [ ] AC10: Returns `{ credit_balance: number, ls_subscription_status: string | null, ls_subscription_id: string | null, spending_limit: number | null, max_lead_price: number | null, monthly_spend: number, transactions: CreditTransaction[], total_transactions: number, page: number, per_page: number }`.

- [ ] AC11: `monthly_spend` = SUM of `amount_cents` from `credit_transactions` WHERE `type = 'debit'` AND `created_at >= first day of current month` AND `expert_id = :id`.

- [ ] AC12: `transactions` query parameter: `?page=1&per_page=20`. Each transaction: `{ id, type, amount_cents, description, lead_id, created_at }`. Ordered by `created_at DESC`.

### GET /api/experts/:id/dashboard

- [ ] AC13: New handler `handleGetDashboard(request, env, user, expertId)` in `workers/backend/api/src/handlers/experts/dashboard.ts`. JWT auth + ownership check.

- [ ] AC14: Returns aggregate stats: `{ unread_leads: number, upcoming_bookings: number, credit_balance: number, composite_score: number | null, quality_tier: string, month_stats: { leads_received: number, leads_confirmed: number, leads_flagged: number, bookings_total: number, bookings_completed: number, conversions_declared: number }, monthly_history: MonthlyStats[] }`.

- [ ] AC15: `unread_leads` = COUNT leads WHERE `status = 'new'`. `upcoming_bookings` = COUNT bookings WHERE `starts_at > NOW() AND status NOT IN ('cancelled')`. `quality_tier` derived from `composite_score` (same thresholds as existing tier logic). `bookings_total` = all bookings this month. `conversions_declared` = COUNT leads WHERE `conversion_declared = true` this month.

- [ ] AC15b: `monthly_history`: array of last 6 months (current month + 5 previous). Each entry: `{ month: string (YYYY-MM), leads_received: number, leads_confirmed: number, leads_flagged: number, bookings_total: number, bookings_completed: number, conversions_declared: number }`. Empty months included with zero counts. Used by E02S08 analytics charts and trend arrows.

### GET /api/experts/public (unauthenticated)

- [ ] AC16: New handler `handleGetPublicExperts(request, env)` in `workers/backend/api/src/handlers/experts/public.ts`. No JWT required. Rate limited via existing Rate Limiter binding.

- [ ] AC17: Query parameters: `?vertical={v}&skills={s1,s2}&page=1&per_page=12`. Returns `{ experts: AnonymizedExpert[], total: number, page: number }`.

- [ ] AC18: `AnonymizedExpert`: `{ slug, headline, skills[], industries[], rate_min, rate_max, composite_score, quality_tier, completed_projects, languages[] }`. No `display_name`, no `email`, no `bio` — anonymized for directory display.

- [ ] AC19: `slug` = `exp-{short_hash}` where `short_hash` = first 8 chars of expert `id`. NOT derived from name. Stable across requests.

### GET /api/experts/public/:slug (unauthenticated)

- [ ] AC20: Returns `AnonymizedExpertDetail` extending `AnonymizedExpert` with `{ bio_excerpt (first 200 chars), availability_status, outcome_tags[] }`. Same handler file.

### POST /api/leads/:id/evaluate (write — expert lead evaluation)

- [ ] AC21: New handler `handleEvaluateLead(request, env, user, leadId)` in `workers/backend/api/src/handlers/experts/leads.ts`. JWT auth + ownership check (lead must belong to the authenticated expert).

- [ ] AC22: Request body validated with zod: `{ score: number (1-10, required), notes?: string (max 500 chars), conversion_declared?: boolean }`. Returns 422 on validation failure.

- [ ] AC23: Lead must have status `confirmed` and `evaluation_score IS NULL` (not already evaluated). Returns 409 if already evaluated, 400 if lead status is not `confirmed`.

- [ ] AC24: Updates lead record: `evaluation_score`, `evaluation_notes`, `conversion_declared`, `evaluated_at = NOW()`. Returns updated lead object.

- [ ] AC25: On success, fires `expert.lead_evaluated` PostHog event via `ctx.waitUntil(captureEvent(...))` with `{ lead_id, score, conversion_declared }`.

### Quality

- [ ] AC26: Route registration in the main router for all 7 endpoints (6 GET + 1 POST).

- [ ] AC27: `tsc --noEmit` — zero errors across all workers.

- [ ] AC28: `vitest run` — all existing tests pass + new tests for each endpoint: valid response, ownership check (403), pagination, status filter, empty results, invalid params (422), evaluate (success, already evaluated 409, wrong status 400).

## Notes

- All queries use Hyperdrive via `createSql(env)` — same pattern as existing handlers.
- The `leads` table joins are the most complex: `leads → bookings (via booking_id) → prospects (via prospect_id on bookings)` and `leads → matches (via expert_id + prospect match)`. Delivery should assess whether a single query with JOINs or two queries is more maintainable.
- The public expert listing (AC16–AC20) serves both E03S04 (directory) and any future external integrations. It's placed in this story because it's a read-only API endpoint, not dashboard-specific UI.
- `quality_tier` derivation logic should be extracted into a shared utility (`lib/qualityTier.ts`) to avoid duplication between dashboard and public endpoints.
