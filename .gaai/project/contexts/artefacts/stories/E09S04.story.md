# Story: Post-deploy automated smoke test job

## User Story

**As a** founder deploying to staging,
**I want** the existing smoke test to run automatically after all workers are deployed,
**so that** infrastructure wiring failures (missing bindings, env vars, route misconfigurations) are caught within minutes of deploy — not during manual browser testing.

---

## Context

`scripts/smoke-test-staging.sh` (339 lines, E06S29) already tests 9 acceptance criteria against the live staging environment: health checks, auth, expert registration, profile update, GCal auth-url, AI extraction, prospect submit, matches, and booking availability. It exits with a non-zero code on failure.

This script is never called in CI — it requires manual execution. The fix is a new GitHub Actions job in `deploy-staging.yml` that runs this script after all deploy jobs complete.

The script requires:
- `curl` and `bash` (available on `ubuntu-latest`)
- `SUPABASE_ANON_KEY` (already hardcoded in script as default, but should be a secret for hygiene)
- A pre-existing smoke test user in staging Supabase (`smoke-runner@callibrate.io`)

---

## Acceptance Criteria

- [ ] AC1: `deploy-staging.yml` includes a new job `smoke-test` with `needs: [deploy-core, deploy-satellite, deploy-posthog-proxy, deploy-matching]` (depends on all deploy jobs)
- [ ] AC2: The `smoke-test` job checks out the repo, runs `bash scripts/smoke-test-staging.sh`, and fails the workflow if the script exits non-zero
- [ ] AC3: The smoke test job uses `SUPABASE_ANON_KEY` from GitHub Secrets (passed as env var), falling back to the hardcoded default in the script
- [ ] AC4: The smoke test job output (PASS/FAIL per AC, summary) is visible in GitHub Actions logs
- [ ] AC5: If the smoke test fails, the overall workflow status is FAILED (red badge on the commit)
- [ ] AC6: The smoke test job has a timeout of 5 minutes (`timeout-minutes: 5`) to prevent hanging on network issues
- [ ] AC7: A `wait-for-deploy` step (30-second sleep or curl-based readiness check) exists before running the smoke test, to account for Cloudflare edge propagation delay
- [ ] AC8: The smoke test script itself is NOT modified — only the CI workflow file changes (except for AC7 readiness wait, which may be added as a wrapper or inline step)

> All acceptance criteria must be verifiable. Vague criteria are invalid.

---

## Out of Scope

- Modifying `smoke-test-staging.sh` to add new test cases
- Adding Playwright browser tests
- Smoke testing production deployments (separate concern)
- Slack/Discord notifications on failure (future enhancement)
- Matching-engine-specific smoke test endpoints (matching-engine is accessed via Service Binding, not direct HTTP — tested indirectly through core's prospect submit + matches endpoints)

## Notes

- Cloudflare Workers deployments propagate to the edge within seconds, but DNS caching or KV eventual consistency can cause transient failures in the first 10-30 seconds. AC7's readiness wait mitigates this.
- The smoke test currently tests 9 ACs (AC1-AC9). AC10-AC12 are manual (CF Dashboard, Analytics Engine, PostHog). This is acceptable — automated smoke tests should cover API-level verification only.
- If the smoke test fails due to a transient issue (network timeout, Cloudflare edge delay), the developer can re-run the job from the GitHub Actions UI without re-deploying.
- Future enhancement: add `continue-on-error: true` to the smoke test job and post results as a PR comment instead of blocking — but start strict (fail-fast) and relax only if false positives become a problem.
