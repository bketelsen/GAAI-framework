# Story E06S42: GDPR Data Retention — Purge Abandoned Funnels + OTP Cleanup

## User Story

**As a** platform operator subject to GDPR,
**I want** abandoned prospect data purged automatically and OTP codes cleaned up after expiry,
**so that** we retain personal data only when justified by a business relationship, minimizing compliance risk.

---

## Context

DEC-129 mandates GDPR-compliant data retention policies. The prospect funnel creates data before the prospect identifies themselves — freetext may contain identifying information ("Je suis Jean Dupont, CEO de X"). Without the prospect providing their email, there's no business relationship justifying data retention. GDPR's data minimization principle requires purging this data.

OTP codes are security-sensitive and must be hard-deleted after use or expiry (stored hashed, never cleartext — E06S39 handles this at the KV level with TTL). This story adds a belt-and-suspenders cron for any edge cases where KV TTL doesn't fire.

---

## Acceptance Criteria

### Abandoned funnel purge

- [ ] AC1: **Cron job:** Scheduled handler running every hour (or piggyback on existing cron).
  - Query: prospects WHERE `email IS NULL` AND `created_at < NOW() - INTERVAL '24 hours'`.
  - For each: delete `freetext`, `requirements` (JSONB), `extraction_confidence` (JSONB) by setting them to NULL (soft purge — keep the prospect record for aggregate analytics: "X abandoned funnels this week").
  - Also delete any associated `matches` records (matching data for unidentified prospects has no business value).
  - Log: PostHog event `system.abandoned_funnel_purged` with count (no PII in event).

- [ ] AC2: **Safety check:** Never purge prospects who HAVE an email (even if email is not verified). The email signals intent to engage. Retention for identified prospects: duration of commercial relationship + 3 years (French prescription period) — this is a documentation note, not a code change at this stage.

### OTP cleanup

- [ ] AC3: **KV cleanup cron:** Scan KV keys matching pattern `otp:*` and `otp-sends:*`. KV TTL should handle this automatically (E06S39 sets TTL on all OTP keys). This cron is a safety net: if any keys exist past their expected lifetime (TTL 600s for OTP, TTL 86400s for rate counters), delete them.
  - Note: KV `list()` with prefix is the only way to scan. If the volume is low (< 1000 keys), a simple scan is sufficient. If volume grows, consider a cursor-based approach.

### Consent mechanism

- [ ] AC4: On the freetext submit button (E03S01 Page 1), add visible text below the button:
  - "En soumettant, vous acceptez notre [politique de confidentialite](/privacy)."
  - Link points to the privacy policy page (E06S30 AC1 — already deployed).
  - No checkbox required (GDPR: consent is implicit in the voluntary action, but information must be accessible).
  - Styling: small muted text, link underlined.

### Audit trail

- [ ] AC5: **Purge log table** (optional but recommended): `data_purge_log(id, purge_type, records_affected, purged_at)`. Each cron execution inserts a row. Provides GDPR audit trail: "We purged X abandoned funnels on Y date."

### Tests

- [ ] AC6: Unit tests for purge query (correctly identifies abandoned prospects, does NOT purge identified prospects). Integration test for cron handler. KV cleanup safety net test. All existing tests pass. `tsc` clean.

## Out of Scope

- Right to erasure (DSAR) flow for identified prospects (requires a dedicated endpoint — post-launch)
- Cookie consent banner (handled by satellite frontend, not backend)
- Data export (GDPR Article 20 portability — deferred)
- Automatic retention enforcement for identified prospects (3-year timer — manual process at launch volume)

---

## Dependencies

- E06S39 (OTP service — OTP keys to clean up, email validation infrastructure)

## Notes

- **Volume at launch:** With < 100 prospects/day, the abandoned funnel purge will process very few records. The cron's primary value is compliance posture, not performance.
- **Soft purge vs hard delete:** Setting freetext/requirements to NULL (soft purge) is preferred over DELETE because it preserves the prospect row for aggregate funnel analytics (conversion rates, drop-off analysis). If full GDPR erasure is required (DSAR), a separate hard-delete endpoint can be built post-launch.
- **Privacy policy update:** Ensure the privacy policy at `/privacy` (E06S30) mentions the 24h retention for unidentified data. Delivery should verify this or flag for founder action.
