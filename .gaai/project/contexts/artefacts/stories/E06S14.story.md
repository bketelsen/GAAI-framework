# Story E06S14: Satellite Worker — Multi-Tenant Scaffold

## User Story

**As a** prospect landing on any satellite domain,
**I want** to see a branded landing page that matches the satellite's visual identity and content,
**so that** the experience feels native to the vertical — without requiring a separate deployment per satellite.

---

## Context

This story creates `callibrate-satellite` — a **separate Cloudflare Worker** from `callibrate-core` — that serves ALL satellite domains from a single deployed codebase. It uses hostname-based routing to identify the satellite, loads its `satellite_configs` row from Supabase (cached in KV), injects design tokens as CSS custom properties, and renders a branded HTML landing page.

This scaffold is the **Layer 3.2 foundation** on which all E03 stories (quiz funnel, directory, match results, booking widget) will be built. At this story's completion, the Worker proves the multi-tenant pattern end-to-end: hostname → config → render → CF edge cache. Functional pages are added in E03.

**Architecture decision (DEC-53):** Multi-tenant Worker, not Astro SSG. One deploy serves N satellites. Adding a satellite = INSERT in `satellite_configs` + DNS configuration. No per-satellite build pipeline.

**Supersedes E03S05 AC2** (which assumed Astro build-time generation). robots.txt and sitemap.xml are generated by this Worker per request, CF-edge-cached for 24h — functionally equivalent to static, no per-satellite deployment required.

## Acceptance Criteria

- [ ] AC1: New CF Worker project at `workers/satellite/` (or top-level sibling to `callibrate-core`): own `wrangler.toml`, `package.json` (Hono + TypeScript), `tsconfig.json` — deployed independently from `callibrate-core`. Worker name convention: `callibrate-satellite` (staging) / `callibrate-satellite` (production env via `wrangler.toml` `[env.production]`)

- [ ] AC2: KV namespace provisioned (human step, documented in `SETUP.md`): `callibrate-satellite-kv-config-staging` and `callibrate-satellite-kv-config-prod` — bound as `CONFIG_CACHE: KVNamespace` in `wrangler.toml`. Secrets declared: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `ADMIN_SECRET`

- [ ] AC3: Hostname-based multi-tenant routing on every request:
  1. `hostname = new URL(request.url).hostname` (strips port if present)
  2. `config = await CONFIG_CACHE.get(`satellite:config:${hostname}`, 'json')`
  3. On KV miss: query Supabase `satellite_configs WHERE domain = hostname AND active = true LIMIT 1`
  4. On DB hit: store in KV with `expirationTtl: 3600`; on DB miss: return `302` redirect to `https://callibrate.io`
  5. Config is passed to all route handlers; no second lookup within the same request

- [ ] AC4: `GET /` renders the satellite landing page — complete HTML document with:
  - `<title>` from `config.content.meta_title`
  - `<meta name="description">` from `config.content.meta_description`
  - Open Graph tags: `og:title`, `og:description`, `og:url` (canonical domain), `og:type: website`
  - Canonical `<link rel="canonical" href="https://${config.domain}/">`
  - JSON-LD `<script type="application/ld+json">` from `config.structured_data`
  - Inline `<style>:root { --color-primary: ${theme.primary}; --color-accent: ${theme.accent}; --font-family: ${theme.font}; --radius-card: ${theme.radius}; }</style>`
  - `<meta name="robots" content="index, follow">`

- [ ] AC5: Landing page `<body>` renders, using only `config` data and CSS custom properties:
  - Logo `<img>` from `config.theme.logo_url` (omitted if null)
  - `<h1>` from `config.brand.name`
  - `<h2>` from `config.content.hero_headline`
  - `<p>` from `config.content.hero_sub`
  - `<ul>` list from `config.content.value_props[]`
  - CTA button: "Décrire mon projet" → `href="/match"` (page added in E03S01)
  - All styles use `var(--color-primary)` etc. — zero hardcoded colors

- [ ] AC6: `GET /robots.txt` returns dynamically-generated robots.txt per E03S05 AC1 spec (training bots blocked, answer bots partial, sitemap link), with `${config.domain}` substituted — `Content-Type: text/plain`, `Cache-Control: public, max-age=86400` — supersedes E03S05 AC2 (no Astro build step)

- [ ] AC7: `GET /sitemap.xml` returns XML sitemap with: `https://${config.domain}/`, `https://${config.domain}/match`, `https://${config.domain}/experts` — `Content-Type: application/xml`, `Cache-Control: public, max-age=86400`

- [ ] AC8: `GET /health` returns `200 { ok: true, domain: hostname, satellite_id: config.id }` — `Cache-Control: no-store`

- [ ] AC9: `POST /admin/cache/purge` — header `x-admin-secret` must match `ADMIN_SECRET` env var; body `{ domain: string }`; deletes `CONFIG_CACHE` key `satellite:config:${domain}`; returns `200 { purged: true, domain }`, `401` on bad secret, `400` on missing domain — allows immediate content refresh without waiting for KV TTL

- [ ] AC10: All rendered HTML routes return `Cache-Control: public, max-age=300, stale-while-revalidate=3600` — CF edge caches across all POPs, functionally equivalent to static serving for repeat visitors

- [ ] AC11: Unknown hostname (no `satellite_configs` row with `active: true`) returns `302` redirect to `https://callibrate.io` with `Cache-Control: no-store`

- [ ] AC12: `SETUP.md` updated with:

  - Secrets provisioning for satellite Worker
  - Custom Domain setup per satellite: Cloudflare Dashboard → Workers & Pages → `callibrate-satellite` → Settings → Domains → Add custom domain → `[satellite domain]`
  - Checklist: create satellite row in `satellite_configs` → configure DNS → add Custom Domain → smoke test → flip `active: true`
  - Cache purge command: `curl -X POST https://[satellite_domain]/admin/cache/purge -H "x-admin-secret: ..." -d '{"domain":"..."}'`

- [ ] AC13: GitHub Actions `deploy-staging.yml` updated to also build and deploy `callibrate-satellite` on push to `production` branch; `deploy-production.yml` updated to deploy `callibrate-satellite --env production` on version tag

- [ ] AC14: Smoke tests against staging (test satellite domain configured in `satellite_configs`):
  - `/health` → 200, correct `satellite_id`
  - `GET /` → 200, HTML contains correct `<title>`, CSS custom property in `<style>`, `<h2>` with hero_headline
  - `/robots.txt` → 200, `text/plain`, contains `User-agent: GPTBot` + `Disallow: /`

## Notes

- **Hono for routing** — consistent with callibrate-core patterns. Satellite Worker uses Hono with JSX or template literal rendering. No full framework (no Next.js, no Astro).
- **No per-satellite CSS file** — all satellite branding lives in the inline `<style>` block injected from `config.theme`. Structural CSS (layout, typography base, component styles) is inlined in the HTML shell. This avoids R2 and static asset serving complexity at MVP.
- **KV TTL vs cache purge** — 3600s TTL means content changes propagate within 1h without intervention. The `/admin/cache/purge` endpoint allows immediate refresh. This is the operator's control surface for MVP; a proper admin UI is post-MVP.
- **E03S05 dependency update** — E03S05 currently lists `E03S01` as its dependency ("satellite site scaffold must exist"). This must be updated to `E06S14` — the scaffold is now this story, not E03S01. E03S05 AC2 (Astro build-time) is superseded by AC6 of this story.
- **Conflict resolved** — E03S05 AC2 stated "statically generated at Astro build time". With the multi-tenant Worker approach, robots.txt is generated dynamically per request and CF-edge-cached for 86400s. The SEO behaviour is identical. E03S05 AC2 is superseded by AC6 of this story.
- **GitHub Actions** — the satellite Worker is a second deployment artifact. Both Workers must deploy together on the same triggers to stay in sync (AC13).
