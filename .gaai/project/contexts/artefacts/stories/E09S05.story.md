# Story: Extended API regression suite

## User Story

**As a** founder deploying to staging,
**I want** the smoke test to cover all API endpoints created by done stories,
**so that** regressions on any backend feature are caught automatically after every staging deploy.

---

## Context

The current `smoke-test-staging.sh` (E06S29) covers 9 checks (AC1-AC9) mapping to ~10 stories. But 12 additional API-testable story groups have endpoints that are never validated post-deploy:

- **E06S38** — Dashboard read API (6 GET + 1 POST): leads, bookings, billing, dashboard stats, public experts, public expert detail, lead evaluation
- **E06S39** — OTP verification (send + verify)
- **E06S40** — Input pre-validation (gibberish rejection, honeypot, flow token HMAC between extract and submit)
- **E06S11 / E08S03** — Booking lifecycle (hold → confirm → cancel — only availability is tested today)
- **E06S17** — Survey submission (call-experience + project-satisfaction)
- **E08S02** — Security headers on all responses (X-Content-Type-Options, X-Frame-Options, HSTS, CSP)
- **E03S05** — Crawler access policy (robots.txt + sitemap.xml on satellite)
- **E06S30** — Legal pages (/privacy + /terms on satellite)
- **E06S34 / E06S36** — Billing enforcement + admissibility filters (verified via match results quality)

The script already has the infrastructure (curl wrappers, pass/fail helpers, JWT auth). Extending it is incremental work, not a rewrite.

---

## Acceptance Criteria

- [ ] AC1: `smoke-test-staging.sh` includes a new check **AC10** that calls `GET /api/experts/:id/leads` (authenticated) and verifies HTTP 200 with a JSON array response
- [ ] AC2: A new check **AC11** calls `GET /api/experts/:id/bookings` (authenticated) and verifies HTTP 200
- [ ] AC3: A new check **AC12** calls `GET /api/experts/:id/billing` (authenticated) and verifies HTTP 200 with `balance` field present in response
- [ ] AC4: A new check **AC13** calls `GET /api/experts/:id/dashboard` (authenticated) and verifies HTTP 200 with `composite_score` field present
- [ ] AC5: A new check **AC14** calls `GET /api/experts/public` and verifies HTTP 200 with a JSON array response (no auth required)
- [ ] AC6: A new check **AC15** calls `POST /api/prospects/:id/otp/send` with the prospect_id from AC7 and verifies HTTP 200 (or 429 if rate-limited — both are valid responses confirming the endpoint works)
- [ ] AC7: A new check **AC16** calls `POST /api/bookings/hold` with expert_id + a future time slot and verifies HTTP 200 (hold created) or HTTP 422 (gcal_not_connected — graceful, same as AC9). Then if 200, calls `DELETE /api/bookings/:id` to clean up
- [ ] AC8: A new check **AC17** verifies that ALL responses from previous checks include security headers: `x-content-type-options: nosniff`, `x-frame-options: DENY` (spot-check on AC1 and AC10 responses using `curl -I`)
- [ ] AC9: A new check **AC18** calls `GET /robots.txt` on the satellite URL and verifies HTTP 200 with `User-agent` directive present
- [ ] AC10: A new check **AC19** calls `GET /privacy` on the satellite URL and verifies HTTP 200 (or 302 if satellite not configured — both valid)
- [ ] AC11: The script summary line is updated to reflect the new total (AC1–AC19 instead of AC1–AC9)
- [ ] AC12: The script exit code remains non-zero if ANY check fails (existing behavior preserved)
- [ ] AC13: All existing AC1–AC9 checks remain unchanged and continue to pass
- [ ] AC14: The E09S04 smoke test CI job runs the extended script without modification (no new secrets or dependencies required beyond what E09S04 already provides)

> All acceptance criteria must be verifiable. Vague criteria are invalid.

## Out of Scope

- Survey endpoint testing (requires a valid survey token signed with SURVEY_TOKEN_SECRET — not feasible in a stateless smoke test without pre-seeding a booking workflow)
- LemonSqueezy webhook testing (requires HMAC signature from LS — not feasible without LS test environment)
- Lead flag/confirm/evaluate actions (require a real lead in the DB — could create data side effects)
- Playwright browser tests (covered by E09S06)
- Modifying the CI workflow (E09S04 already runs the script)

## Dependencies

- E09S04 (smoke test must be automated in CI first — this story extends the script content)
- Smoke test user `smoke-runner@callibrate.io` must exist (already created by E06S29)

## Notes

- AC checks are numbered AC10-AC19 to continue the existing AC1-AC9 sequence. This keeps the output consistent and easy to scan.
- The booking hold check (AC16) may leave orphan holds if the cleanup delete fails. The hold has a 10-minute TTL (E06S11), so it self-cleans. Add a `note` output if cleanup fails rather than failing the entire test.
- Security header check (AC17) is a spot-check, not exhaustive. It verifies the middleware is active, not that every header is correct on every route.
- The OTP send check (AC15) will trigger an actual email to the prospect. Since the prospect from AC7 has no email yet (quiz_answers only), the endpoint should return an error indicating no email — which still validates that the endpoint is live and responding correctly. Adjust the expected response based on the actual API behavior.
- Delivery Agent should read the full current `smoke-test-staging.sh` before implementing to follow the exact conventions (do_get/do_post wrappers, pass/fail helpers, output formatting).
