# Story E06S04: Cal.com Managed Users Integration + Webhook Endpoint

## User Story

**As an** AI integration expert,
**I want** my Google Calendar automatically connected when I register,
**so that** prospects can book a call in my availability without any manual setup on my part.

---

## Context

Cal.com managed users allow Callibrate to create and manage calendar accounts on behalf of experts programmatically — critical for a marketplace model. Each expert gets a Cal.com managed account at registration, their Google Calendar is linked via OAuth, and a "20-min Intro Call" event type is auto-configured with Google Meet. Incoming Cal.com booking webhooks trigger the lead-billing and notification flows.

---

## Acceptance Criteria

- [ ] AC1: `matching-jobs` queue consumer processes `{ type: "cal.setup", expert_id }` messages by calling the Cal.com Managed Users API to create a Cal.com account — stores returned `cal_username` in `experts` table
- [ ] AC2: `GET /api/experts/:id/cal/auth-url` returns a Google Calendar OAuth authorization URL scoped to the Cal.com managed user (used by expert onboarding UI to trigger Google Calendar connection)
- [ ] AC3: `GET /api/experts/:id/cal/callback` handles the OAuth callback, completes Google Calendar connection on the Cal.com managed user — returns `200` on success
- [ ] AC4: After Google Calendar is connected, a "20-min Intro Call" event type is auto-created on the Cal.com managed user: duration 20 min, conferencing set to Google Meet (auto-generated link per booking)
- [ ] AC5: `POST /webhooks/cal` validates the Cal.com webhook signature (HMAC-SHA256 using `CAL_WEBHOOK_SECRET` — bound as Worker secret) — returns `401` on invalid signature
- [ ] AC6: `POST /webhooks/cal` handles `booking.created` event: creates a `bookings` row, pushes `{ type: "booking.created", booking_id, expert_id, prospect_id }` to `lead-billing` queue, pushes `{ type: "booking.confirmed", booking_id }` to `email-notifications` queue
- [ ] AC7: `POST /webhooks/cal` handles `booking.cancelled` event: updates `bookings.status` to `'cancelled'`
- [ ] AC8: Webhook processing is idempotent — if a `cal_booking_id` already exists in `bookings`, the `booking.created` event is ignored (no duplicate row, no duplicate queue messages)
- [ ] AC9: Cal.com setup failure (API error, quota) does not block expert registration — failure is logged and the `cal.setup` queue message is retried (up to 3 times via queue retry logic)
- [ ] AC10: `GET /api/experts/:id/cal/status` returns `{ connected: boolean, cal_username: string | null, event_type_configured: boolean }` — used by onboarding UI to show connection state

---

## Out of Scope

- Teams / Outlook calendar integration (post-MVP — Cal.com supports it but not wired at this stage)
- Custom event type configuration by expert (post-MVP — default 20-min format only)
- Booking cancellation initiated by Callibrate (post-MVP)

---

## Dependencies

- E06S03 (expert registration API must exist — cal.setup queue message is triggered from registration)
- Cal.com account with managed users API access provisioned
- `CAL_API_KEY` and `CAL_WEBHOOK_SECRET` bound as Worker secrets

---

## Notes

Cal.com managed users API docs: `https://cal.com/docs/api-reference/v2/managed-users`. The managed user's booking page URL will be `https://cal.com/{cal_username}/intro-call` — this URL is stored in `experts.cal_username` and returned in match reveal API responses for booking. Google Meet links are auto-generated per booking by Cal.com when conferencing is set to "Google Meet" on the event type.
