# Story E06S23: D1 Edge Serving + Cache API L1 + Cron Sync

## User Story

**As a** platform operator,
**I want** expert pool data served from D1 at the edge with Cache API L1 and automatic sync from Supabase,
**so that** matching reads complete in <5ms without hitting the origin database, with no KV dependency.

---

## Context

Expert pool reads are the hottest read path (every prospect submit + match compute). The current KV-based approach has two problems: no SQL filtering capability and a 25 Mo per-value limit (~5,000 experts max).

D1 provides edge SQL with sub-5ms reads and SQL WHERE filtering. Cache API provides sub-millisecond per-datacenter caching for repeated reads. Combined, the read chain becomes: Cache API (~0ms, 60s TTL) → D1 (~1-5ms, SQL queryable) → Hyperdrive fallback (~30ms, source of truth).

This story replaces the EXPERT_POOL KV namespace entirely for the expert pool read path. A Cron Trigger every 5 minutes incrementally syncs expert data from Supabase (source of truth) to D1 via Hyperdrive. First sync = full load; subsequent syncs = `WHERE updated_at > last_sync_at`.

**Architecture decision:** KV eliminated from expert pool read path (no 25 Mo limit, no blob rewrites, SQL filtering enabled). Cache API requires custom domain (not workers.dev) — production uses `api.callibrate.io`, staging needs a custom domain too. Cache API is per-datacenter and ephemeral — miss is normal and expected, D1 handles it.

- [ ] AC2: Denormalized `expert_pool` table with scoring-relevant fields
- [ ] AC3: `_sync_meta` table stores `last_sync_at` for incremental sync
- [ ] AC4: Cron `*/5 * * * *` triggers `syncExpertPoolToD1()`
- [ ] AC5: Sync reads from Supabase via Hyperdrive (benefits from connection pooling)
- [ ] AC6: Incremental: `WHERE updated_at > last_sync_at` + lead counts
- [ ] AC7: First sync = full load (no `last_sync_at` → load all)
- [ ] AC8: Read chain in `loadExpertPool()`: Cache API L1 (60s TTL) → D1 → Hyperdrive fallback
- [ ] AC9: Cache API TTL = 60 seconds (via `Cache-Control` header on cached Response)
- [ ] AC10: On Cache API miss + D1 hit → return from D1, write-back to Cache API
- [ ] AC11: On both miss → load from Supabase via Hyperdrive, write D1 + Cache API
- [ ] AC12: Sync failure → Hyperdrive fallback (graceful degradation)
- [ ] AC13: `scheduled()` handler dispatches to sync function
- [ ] AC14: `EXPERT_POOL` KV namespace removed from expert pool read path (KV binding kept only if used elsewhere)
- [ ] AC15: All tests pass

## Notes

D1 schema is denormalized for read performance — no joins needed at query time. The sync function handles both INSERT and UPDATE (upsert pattern). Lead counts are aggregated during sync to avoid per-request count queries.

Cache API key: `https://cache.internal/expert-pool` (synthetic URL). Stored as JSON Response with `Cache-Control: max-age=60`. Per-datacenter — cold start on first hit per datacenter is expected and handled by D1 fallback.

This story absorbs the scope of the former E06S19 (Cache API L1) which was eliminated because KV as L2 added no value over D1.
