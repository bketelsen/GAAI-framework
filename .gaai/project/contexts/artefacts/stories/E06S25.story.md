# Story E06S25: Durable Objects — Expert Pool Write Coordinator

## User Story

**As a** platform operator,
**I want** expert pool writes (D1 + Vectorize) coordinated through a single Durable Object,
**so that** concurrent profile updates are batched efficiently and write conflicts are eliminated.

---

## Context

Expert registration and profile updates trigger writes to D1 (edge cache) and Vectorize (semantic index). Without coordination, concurrent writes can create race conditions. A Durable Object acts as a single-writer coordinator: it receives change events, batches them with a 5-second alarm window, then performs batch D1 upserts and Vectorize embeddings.

The cron sync from E06S23 serves as a safety net — if the DO misses an event, the next 5-minute sync catches it.

---

## Acceptance Criteria

- [ ] AC1: `ExpertPoolDO` extends `DurableObject` with `fetch()` and `alarm()` handlers
- [ ] AC2: Receives profile change events, batches with 5-second alarm window
- [ ] AC3: On alarm: batch D1 upsert + batch Vectorize embed/upsert
- [ ] AC4: Single DO instance `"expert-pool-coordinator"` (singleton pattern)
- [ ] AC5: **DO failure does NOT block HTTP response** (async via `ctx.waitUntil`)
- [ ] AC6: Cron sync (E06S23) = safety net if DO fails
- [ ] AC7: All tests pass

---

## Out of Scope

- Multiple DO instances (sharding — post-MVP scale)
- DO-based read path (reads go through Cache API → D1 → Hyperdrive)

---

## Dependencies

- E06S23 (D1 edge serving — DO writes to D1)
- E06S21 (Vectorize infrastructure — DO upserts vectors)

## Notes

Durable Objects billing is based on duration (~$5/month estimated). The singleton pattern ensures a single point of coordination. The 5-second batch window balances latency (new expert visible within 5s) with efficiency (batch D1/Vectorize calls).
