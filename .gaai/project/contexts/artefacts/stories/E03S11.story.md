# Story E03S11: Prospect Waiting State — Credible Feedback & Resilient Errors

## User Story

**As a** prospect who just confirmed my project requirements,
**I want** clear, trustworthy feedback while experts are being matched and actionable guidance when something goes wrong,
**so that** I stay confident the platform is working for me rather than wondering if it's broken.

---

## Context

After clicking "Confirmer et trouver mes experts" on `/confirm`, the prospect enters a wait of 3–25 seconds spanning two pages: Turnstile verification + API submission on `/confirm`, then match fetching (possibly with 202 computing retries) on `/results`.

Current state: a disabled button with tiny grey text ("Recherche d'experts en cours…", 0.875rem, #555). On error: a generic "Une erreur est survenue" red banner with a "Réessayer" link. For a prospect who just disclosed a 15–20k EUR budget, this feedback is disproportionately weak and the error state is a trust killer.

This story addresses three gaps:
- **A.** The visual weight of loading feedback on `/confirm` doesn't match the significance of the action
- **B.** The `/results` 202-computing state gives no reassurance during extended waits
- **C.** Error states are generic, offer no context, and provide no fallback path

---

## Acceptance Criteria

- [ ] AC1: **Confirm page — loading message visibility.** When the prospect clicks "Confirmer et trouver mes experts", the loading message ("Recherche en cours parmi nos experts qualifiés") renders at base font size (not reduced), in body text color (not muted grey), centered below the disabled button. The message must be visually proportionate to the CTA the prospect just clicked.

- [ ] AC2: **Results page — extended wait reassurance.** When the `/results` page receives a 202 (computing) and enters the retry loop, a single message appears below the skeleton cards: "Nous affinons les correspondances, quelques secondes…". If total wait exceeds 10 seconds (cumulative across retries), the message updates to: "Cela prend un peu plus de temps que prévu. Votre recherche est en cours." No rotating messages, no fake progress steps — one update only.

- [ ] AC3: **Results page — silent first retry.** On the first network error or non-202/non-200 response when fetching matches, the page retries once automatically (after 2s delay) without showing the error state to the prospect. The error state only appears if the retry also fails.

- [ ] AC4: **Confirm page — error specificity.** Error messages on `/confirm` are specific to the failure type. Existing 422 and 429 messages are already specific (keep as-is). The catch-all error replaces "Une erreur est survenue" with: "La connexion au serveur a échoué. Vérifiez votre connexion internet et réessayez." Turnstile errors keep their existing specific message.

- [ ] AC5: **Results page — error specificity + fallback.** The `/results` error state (after silent retry exhausted) displays a specific message based on failure type: network error → "Connexion interrompue. Vérifiez votre connexion et réessayez." ; server error (5xx) → "Nos serveurs sont momentanément indisponibles. Réessayez dans quelques instants." The error state includes a secondary action: "Parcourir le répertoire d'experts" link to `/experts` — giving the prospect an alternative path instead of a dead end.

- [ ] AC6: **Results page — max-retry fallback.** When the 202 computing retry loop exhausts all attempts (currently 3), the fallback message changes from the current "Aucun expert disponible pour le moment" to: "Le calcul des correspondances prend plus de temps que prévu. Vous recevrez vos résultats par email dès qu'ils sont prêts." with a secondary link: "En attendant, parcourir le répertoire" → `/experts`. (Note: email delivery is a backend concern outside this story's scope — the message sets the expectation.)

- [ ] AC7: **PostHog error tracking.** Every error displayed to the prospect fires a PostHog event `satellite.matching_error` with properties: `satellite_id`, `prospect_id`, `error_type` (one of: `network`, `server_5xx`, `validation_422`, `rate_limit_429`, `turnstile`, `computing_timeout`), `page` (`confirm` | `results`), `retry_count`.

---

## Out of Scope

- Animated progress steppers, rotating educational messages, or illustrative animations (evaluated and rejected — fake progress for a 3–25s wait with unpredictable duration is dishonest and fragile)
- Backend changes to matching speed or async email delivery of results
- Changes to the skeleton loading cards on `/results` (already adequate)
- Redesign of the `/confirm` summary layout

## Notes

- **No new HTML structure needed on `/confirm`** — the `#loading-msg` div already exists, only its styling and content change.
- **`/results` silent retry** adds one automatic retry before showing error UI. The existing `retryCount` / `MAX_RETRIES` mechanism for 202s is separate and unchanged.
- **AC6 email promise:** The message says "you'll receive results by email" — this is a forward-looking UX promise. If backend email delivery doesn't exist yet, the Delivery Agent should flag this as a gap for Discovery to address in a separate story.
- **Estimated effort:** Low. All changes are frontend text/style updates + one retry mechanism. No new API calls, no new components.
