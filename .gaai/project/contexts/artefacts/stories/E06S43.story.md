# Story E06S43: AI Agent API Endpoints + MCP Server Foundation

## User Story

**As an** AI agent acting on behalf of a verified human,
**I want** dedicated API endpoints to search for experts, request profile reveals, and request booking slots,
**so that** the human I serve can find the right automation expert through their AI assistant without manually navigating the web interface.

---

## Context

DEC-126 establishes that AI agents are ALLOWED to use Callibrate on behalf of verified humans. The current funnel is already agent-compatible by design (freetext-first, email confirmation gates serve as human-in-the-loop). This story adds dedicated agent endpoints and lays the foundation for an MCP server — a first-mover advantage, as no expert marketplace has an MCP server.

The agent API follows a tiered access model:
- **Free actions** (no human gate): search, extract, match (anonymized).
- **Gated actions** (human email confirmation required): reveal (full profiles), booking.

Post-MVP because the web funnel works for agents today (browser-based agents like OpenAI Operator navigate it directly). Dedicated endpoints become valuable when API-native agents (Claude, GPT with function calling) reach mainstream adoption.

## Acceptance Criteria

### Agent authentication

- [ ] AC1: **API key system for prospects.**
  - New `prospect_api_keys` table: `id, prospect_id (FK), key_hash (TEXT, SHA-256 of the API key), name (TEXT, user label), created_at, last_used_at, revoked_at`.
  - `POST /api/prospects/{id}/api-keys` — requires prospect JWT (human must create the key). Generates a random 32-byte hex API key, stores only the hash. Returns the key ONCE in the response (never stored in cleartext).
  - `DELETE /api/prospects/{id}/api-keys/{key_id}` — revoke a key (set `revoked_at`).
  - Maximum 3 active keys per prospect.

- [ ] AC2: **Agent auth middleware.** New middleware `agentAuth()` for `/api/v1/agent/*` routes:
  - Accepts `Authorization: Bearer {api_key}` header.
  - Hashes the key, looks up in `prospect_api_keys`.
  - If valid + not revoked: set `c.set("prospect_id", ...)` + update `last_used_at`.
  - If invalid → 401 `invalid_api_key`.
  - If revoked → 401 `api_key_revoked`.

### Agent endpoints

- [ ] AC3: `POST /api/v1/agent/extract` — Same as `POST /api/extract` but authenticated. Accepts `{ freetext: string, satellite_id?: string }`. Returns extraction result. Rate limit: 10/min per API key (stricter than web). No Turnstile required (API key is the auth).

- [ ] AC4: `POST /api/v1/agent/match` — Extract + match in one call. Accepts `{ freetext: string, satellite_id?: string }`. Returns anonymized match results (same as web funnel anonymized cards — tier, skills, score breakdown, NO names or bios). Creates a `prospect_projects` record (E06S41).

- [ ] AC5: `POST /api/v1/agent/reveal` — Request full profile reveal for a specific match.
  - Accepts `{ project_id, match_id }`.
  - Does NOT immediately reveal. Instead: sends an email to the prospect (the human the agent acts for): "Votre assistant IA souhaite consulter le profil de [Expert anonymized name] pour votre projet. [Confirmer] [Refuser]".
  - Returns `{ status: "pending_human_confirmation", confirmation_sent_to: "{email}" }`.
  - Confirm link: `GET /api/v1/agent/reveal/{match_id}/confirm?token=...` → reveals profile, returns full expert data in API response (stored for agent to retrieve).
  - `GET /api/v1/agent/reveal/{match_id}/status` — polling endpoint. Returns `{ status: "pending" | "confirmed" | "refused" }`. Agent polls until human confirms.

- [ ] AC6: `POST /api/v1/agent/booking/request` — Request a booking slot.
  - Accepts `{ project_id, expert_id, preferred_dates: string[] }` (optional date preferences).
  - Does NOT book directly. Sends email to prospect: "Votre assistant IA souhaite reserver un appel avec [Expert]. [Choisir un creneau]" → link to booking widget (E03S06) pre-filled with expert.
  - Returns `{ status: "pending_human_action", booking_link: "..." }`.
  - The actual booking follows the normal human flow (E03S07 confirmation email).

### Rate limiting

- [ ] AC7: Per-API-key rate limits (separate from web rate limits):
  - `/agent/extract`: 10/min.
  - `/agent/match`: 5/min.
  - `/agent/reveal`: 3/hour.
  - `/agent/booking/request`: 3/hour.
  - Tracked via KV: `agent-rate:{key_hash}:{endpoint}` (TTL per window).
  - On exceed → 429 with `Retry-After` header.

### MCP server foundation

- [ ] AC8: MCP server package at `workers/backend/mcp-server/` (or as a module within core):
  - Exposes 4 tools matching the agent endpoints:
    - `callibrate_describe_project` — wraps `/agent/extract`.
    - `callibrate_find_experts` — wraps `/agent/match`.
    - `callibrate_request_reveal` — wraps `/agent/reveal`.
    - `callibrate_request_booking` — wraps `/agent/booking/request`.
  - Each tool includes a JSON schema describing inputs/outputs.
  - MCP transport: HTTP (Streamable HTTP or SSE, compatible with Claude and other MCP clients).
  - Auth: API key passed as MCP server config parameter.
  - Deployed as a separate CF Worker or as a route group on core.

- [ ] AC9: MCP server README documenting: (1) how to connect from Claude Code / Claude Desktop, (2) available tools + schemas, (3) auth setup (API key creation). Example `.mcp.json` configuration provided.

### PostHog events

- [ ] AC10:
  - `agent.api_key_created` (prospect_id)
  - `agent.extract` (prospect_id, key_id)
  - `agent.match` (prospect_id, key_id, match_count)
  - `agent.reveal_requested` (prospect_id, key_id, match_id)
  - `agent.reveal_confirmed` (prospect_id, match_id)
  - `agent.booking_requested` (prospect_id, key_id, expert_id)

### Tests

- [ ] AC11: Unit tests for API key generation/validation. Integration tests for all 4 agent endpoints (happy path, rate limit, revoked key). MCP server tool invocation tests. Human confirmation flow tests (reveal request → email → confirm → data available). All existing tests pass. `tsc` clean.

## Out of Scope

- Agent billing (agents use the prospect's account — same billing as web)
- Agent identity verification (Visa TAP, Skyfire KYA — monitor but don't implement yet)
- OAuth2 for agents (API key is simpler and sufficient for MVP)
- Agent-to-agent communication (MCP is client-server, not peer-to-peer)
- WebSocket transport for MCP (HTTP is sufficient)

## Notes

- **First-mover advantage:** No expert marketplace has an MCP server. MCP has 97M+ SDK downloads/month and is supported by all major AI companies (Anthropic, OpenAI, Google, Microsoft). Being the first to offer this creates a defensible moat.
- **Human-in-the-loop is non-negotiable.** The agent can search and extract freely, but any action that affects experts (reveal, booking) requires human confirmation via email. This protects experts from agent-initiated spam and preserves billing integrity (DEC-14/30).
- **API key vs OAuth:** API key is chosen for simplicity. The prospect creates a key from a results page or future dashboard. OAuth2 would add significant complexity (token refresh, consent screens, scopes) for minimal benefit at this stage. Can be added later if enterprise agents require it.
- **MCP server deployment:** Can be a standalone Worker or a route group on core (`/mcp/*`). Standalone is preferred for isolation (separate rate limits, separate scaling). Use `@modelcontextprotocol/sdk` npm package.
