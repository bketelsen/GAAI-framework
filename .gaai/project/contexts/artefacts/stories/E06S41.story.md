# Story E06S41: Multi-Project Support — Prospect != Project Data Model

## User Story

**As a** prospect who needs multiple experts for different projects,
**I want** to submit additional project descriptions without repeating the email verification flow,
**so that** I can find experts for each of my needs efficiently from a single identity.

---

## Context

DEC-124 identified that the current data model implicitly assumes 1 prospect = 1 set of requirements. Industry standard is universal: Upwork, Toptal, Expert360, Catalant all support one account = N projects. Forcing a prospect through the full funnel (including email + OTP) for each project is unacceptable UX.

This is a breaking data model change: `matches` currently references `prospect_id` directly. After this story, matches reference `project_id` (from a new `prospect_projects` table), and a prospect can have multiple active projects.

Post-MVP because 1 prospect = 1 project suffices at launch volume.

---

## Acceptance Criteria

### Schema changes

- [ ] AC1: New `prospect_projects` table:
  ```
  id: UUID PRIMARY KEY DEFAULT gen_random_uuid()
  prospect_id: UUID NOT NULL REFERENCES prospects(id) ON DELETE CASCADE
  satellite_id: TEXT NOT NULL
  freetext: TEXT NOT NULL
  requirements: JSONB
  extraction_confidence: JSONB
  status: TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'archived', 'completed'))
  created_at: TIMESTAMPTZ NOT NULL DEFAULT NOW()
  updated_at: TIMESTAMPTZ NOT NULL DEFAULT NOW()
  ```
  - Index: `(prospect_id, status)`.
  - RLS: prospects can only access their own projects.

- [ ] AC2: Migrate `matches` table: add `project_id` column (UUID, FK to `prospect_projects.id`). For existing matches, create a `prospect_projects` record from the prospect's current requirements data and link matches to it. Drop `prospect_id` from `matches` after migration (or keep as denormalized field for query performance — implementation choice).

- [ ] AC3: Migrate existing prospect data:
  - For each prospect with requirements: create a `prospect_projects` row containing their existing `freetext`, `requirements`, `extraction_confidence`.
  - Update their matches to point to the new `project_id`.
  - This is a data migration, not a schema-only change. Must be reversible.

### API changes

- [ ] AC4: `POST /api/extract` — no change (stateless, returns extraction regardless of prospect identity).

- [ ] AC5: `POST /api/prospects/submit` — accepts optional `project_id`. If not provided (new project): creates `prospect_projects` record + runs matching. If provided (edit existing): updates `prospect_projects` record + re-runs matching.

- [ ] AC6: `POST /api/prospects/{id}/identify` — no change (identifies the prospect, not a project).

- [ ] AC7: `GET /api/prospects/{id}/matches` — now requires `project_id` query parameter. Returns matches for a specific project. If omitted → returns matches for the most recent active project (backwards compatible).

- [ ] AC8: New `GET /api/prospects/{id}/projects` — returns list of prospect's projects: `{ projects: [{ id, freetext_preview (first 100 chars), status, match_count, created_at }] }`. Requires prospect JWT.

### Identified prospect fast-track (DEC-124)

- [ ] AC9: **Flow for identified prospect creating project 2+:**
  - Page 1: Freetext textarea (no Turnstile — already verified human). Greeted by name: "Bonjour {prospect_name}, decrivez votre nouveau projet."
  - On submit: extract + submit + match in pipeline (same as DEC-123 ready_to_match flow).
  - Page 2: Full profiles displayed immediately (no email gate, no OTP — already identified). Reveal is free; billing only at booking per DEC-14/30.
  - Total: 2 interactions (write + results). No re-identification.

### Anti-abuse

- [ ] AC10: **Max 5 active projects per prospect.** On `POST /api/prospects/submit` without `project_id`: count active projects. If >= 5 → 429 `max_projects_reached` with message "Vous avez atteint la limite de 5 projets actifs. Archivez un projet pour en creer un nouveau."

- [ ] AC11: **Rate limit: 3 new projects per 24h per prospect.** KV counter `project-rate:{prospect_id}` (TTL 86400). If >= 3 → 429 `project_rate_limit` with message "Vous pouvez creer 3 projets par jour. Reessayez demain."

- [ ] AC12: **Duplicate detection:** Before creating a new project, compute text similarity between new freetext and existing active project freetexts (simple: Jaccard similarity on word sets, or cosine on TF-IDF vectors). If similarity > 85% → 409 `similar_project_exists` with message "Vous avez deja un projet similaire." + link/ID to existing project. Threshold and algorithm are configurable.

### UX — Results page CTA

- [ ] AC13: On the match results page (Page 2), display an "Autre projet ?" CTA at the bottom. Click → mini textarea (inline, no page navigation) for the new project description. Submit → extract + match → new project results displayed (either replace current results with project switcher, or open in new tab).

### Tests

- [ ] AC14: Migration tests (existing data correctly migrated to prospect_projects). CRUD tests for new endpoints. Multi-project flow integration test (create project 1, identify, create project 2 — verify no re-identification). Anti-abuse tests (max projects, rate limit, duplicate detection). All existing tests pass (backwards compatibility). `tsc` clean.

## Notes

- **Breaking change mitigation:** The `matches.prospect_id` → `matches.project_id` migration is the riskiest part. Consider keeping `prospect_id` as a denormalized column (set via trigger or application code) for backwards compatibility during transition. Billing queries (E06S32, E06S34) that join on `prospect_id` need updating.
- **Billing unchanged:** Each booking triggers billing per DEC-14/30, regardless of which project it's associated with. The billing system doesn't care about projects — it cares about bookings.
- **Duplicate detection algorithm:** Jaccard similarity on word sets is the simplest approach. For better accuracy, use the same Workers AI embedding model used for expert matching (E06S21) and compute cosine similarity. This reuses existing infrastructure.
