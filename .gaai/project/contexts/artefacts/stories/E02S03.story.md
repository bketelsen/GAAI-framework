# Story E02S03: Profile Management Page

## User Story

**As a** registered expert,
**I want** to view and edit all my profile fields from my dashboard,
**so that** I can keep my profile accurate as my expertise and positioning evolve.

---

## Context

After onboarding (E02S02), experts need ongoing access to edit their profile. The profile management page (`/dashboard/settings`) displays all profile fields organized in logical sections — mirroring the onboarding wizard structure but in a single editable view. Changes are saved via `PATCH /api/experts/:id/profile` (E06S03).

This page also surfaces the admissibility criteria (E06S36) — the expert's non-budget matching filters (methodology, verticals, stack overlap, duration) — which are not part of onboarding but available for configuration after the expert understands the platform.

---

## Acceptance Criteria

### Page Structure

- [ ] AC1: Profile settings route (`/dashboard/settings`): loader fetches expert profile via `GET /api/experts/:id/profile`. Displays all fields in editable sections.

- [ ] AC2: Sections layout (card-based, one section per card):
  1. **Identité** — display_name, headline, bio
  2. **Expertise** — skills (tag input), verticals, rate_min/rate_max, outcome_tags
  3. **Positionnement marché** — preferences JSONB: career_stage, work_mode, availability, budget_range, project_stage, languages, geo_zones, industries
  4. **Critères d'admissibilité** — admissibility_criteria JSONB: min_project_duration_days, required_methodology, excluded_verticals, min_budget, required_stack_overlap_min, custom_rules
  5. **Google Calendar** — connection status + connect/disconnect (links to E02S04)

### Editing

- [ ] AC3: Each section has an "Enregistrer" button that submits only that section's fields via React Router action → `PATCH /api/experts/:id/profile`. Partial updates — only changed fields sent.

- [ ] AC4: Form validation with zod (same schemas as onboarding). Inline error messages below fields. Server-side validation errors from API (422) displayed per-field.

- [ ] AC5: On successful save: toast notification "Profil mis à jour" (Sonner/shadcn Toast). Section form resets to server state (loader revalidation via RR7 auto-revalidation after action).

- [ ] AC6: On save error: toast notification with error message. Form state preserved — no data loss.

### Admissibility Criteria

- [ ] AC7: Admissibility criteria section displays current values with explanatory help text per field:
  - `min_project_duration_days`: "Durée minimale de projet acceptée (en jours)"
  - `required_methodology`: "Méthodologies requises (ex: agile, waterfall)" — tag input
  - `excluded_verticals`: "Secteurs que vous ne souhaitez pas servir" — multi-select
  - `min_budget`: "Budget minimum du prospect (€)"
  - `required_stack_overlap_min`: "Chevauchement minimum avec votre stack (0-100%)" — slider
  - `custom_rules`: "Règles personnalisées" — freetext list

- [ ] AC8: Admissibility criteria save via `PATCH /api/experts/:id/profile` with `admissibility_criteria` JSONB (full replace, not merge — same API behavior as E06S36).

### Quality

- [ ] AC9: `tsc --noEmit` — zero errors.

- [ ] AC10: Vitest tests: loader returns profile data, section save action (success + validation error), admissibility criteria save.

## Notes

- **Section-level saves** rather than page-level save. Experts editing one field shouldn't worry about accidentally overwriting another section. RR7 actions with `intent` field can distinguish which section was submitted.
- **Admissibility criteria** is an advanced feature — the section should have a brief intro explaining its purpose: "Ces critères filtrent automatiquement les prospects avant le matching. Les prospects qui ne correspondent pas ne vous seront jamais envoyés."
- **Google Calendar section** shows status only (connected email or "non connecté") with a link to the dedicated GCal page (E02S04). Full connect/disconnect flow is on E02S04, not duplicated here.
- **PostHog events (Delivery must include):** `expert.profile_settings_viewed` (page load), `expert.profile_section_saved` (section name, fields_updated). Server-side capture in actions.
