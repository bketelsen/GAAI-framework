# Story E06S24: Service Bindings — Worker Split

## User Story

**As a** platform operator,
**I want** matching computation isolated in a dedicated Worker connected via Service Binding,
**so that** core gateway and matching engine can be deployed, scaled, and debugged independently.

---

## Context

As the matching engine grows (Vectorize, D1, AI bindings, scoring logic), the core Worker accumulates bindings and code. Service Bindings enable zero-overhead RPC between Workers — the core gateway handles auth, routing, CORS, and security, while the matching Worker owns scoring, Vectorize, D1, and expert pool.

---

## Acceptance Criteria

- [ ] AC1: Matching Worker (`callibrate-matching`) deployed independently at `workers/matching/`
- [ ] AC2: Service Binding `MATCHING_SERVICE` configured in core `wrangler.toml`
- [ ] AC3: Matching Worker handles `POST /match` with prospect requirements
- [ ] AC4: Zero-network-hop RPC via Service Binding (`env.MATCHING_SERVICE.fetch()`)
- [ ] AC5: Independent deployment (core and matching deploy separately)
- [ ] AC6: **Fallback:** if binding undefined → local scoring (development/testing)
- [ ] AC7: Core Worker no longer needs D1/Vectorize/AI bindings
- [ ] AC8: All tests pass

---

## Out of Scope

- Multiple matching Workers (single Worker sufficient for MVP scale)
- Custom RPC protocol (standard HTTP via fetch)

---

## Dependencies

- E06S22 (Vectorize scoring integration)
- E06S23 (D1 edge serving)
- E06S18 (Hyperdrive in matching Worker)

---

## Notes

Service Bindings are free and zero-latency (in-process RPC within the same datacenter). The matching Worker receives the full scoring context as a POST body and returns ranked results. This split enables independent scaling and isolates failure domains.
