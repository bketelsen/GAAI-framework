# Story E06S12: Migrate /api/extract from Anthropic to GPT-4o-mini

## User Story

**As a** platform operator,
**I want** the extraction service to use GPT-4o-mini instead of Claude Haiku,
**so that** I eliminate the Anthropic dependency, reduce cost, and use a well-documented provider with mature structured output support.

---

## Context

E06S08 delivered `/api/extract` using `claude-haiku-4-5` via Cloudflare AI Gateway (Anthropic path). The decision was made to migrate to `gpt-4o-mini` for cost efficiency (~$0.15/$0.60 per 1M tokens vs $0.80/$4.00), native JSON Structured Outputs, and reduced provider coupling. CLOUDFLARE_AI_GATEWAY_URL (Anthropic-specific) and ANTHROPIC_API_KEY are no longer needed at this stage — OpenAI is called directly. CF AI Gateway can be re-added as a routing layer later if observability is required (separate story).

External behavior of `POST /api/extract` is **unchanged** — same request shape, same response contract, same error codes. This is a pure provider swap.

---

## Acceptance Criteria

- [ ] AC1: `POST /api/extract` request/response contract is unchanged — `{ text, satellite_id? }` in, `{ requirements, confidence, needs_confirmation, ready_to_match, confirmation_questions? }` out. No breaking change for any caller.
- [ ] AC2: Extraction uses `gpt-4o-mini` via OpenAI Chat Completions API (`https://api.openai.com/v1/chat/completions`) with function calling (`tools` + `tool_choice: { type: "function" }`). Auth header: `Authorization: Bearer ${OPENAI_API_KEY}`.
- [ ] AC3: Response parsed from OpenAI format: `choices[0].message.tool_calls[0].function.arguments` (JSON string). If `tool_calls` is missing or empty → 500 with `{ error: "Unexpected response shape from OpenAI API" }`.
- [ ] AC4: `ANTHROPIC_API_KEY` and `CLOUDFLARE_AI_GATEWAY_URL` removed from `src/types/env.ts` Env interface. `OPENAI_API_KEY: string` added.
- [ ] AC5: SETUP.md staging secrets: `ANTHROPIC_API_KEY` and `CLOUDFLARE_AI_GATEWAY_URL` removed, `OPENAI_API_KEY` added with instructions. Production secrets section updated accordingly.
- [ ] AC6: Satellite vertical context preserved — when `satellite_id` provided, `satellite_configs.vertical` is loaded and injected into the system prompt exactly as before (AC9 of E06S08).
- [ ] AC7: Error surface updated — `'Anthropic API error'` → `'OpenAI API error'`; `'Invalid response from Anthropic API'` → `'Invalid response from OpenAI API'`. `'AI service unreachable'` for fetch failures.
- [ ] AC8: Vitest tests updated — mock env uses `OPENAI_API_KEY` (not `ANTHROPIC_API_KEY`/`CLOUDFLARE_AI_GATEWAY_URL`); mock API response uses OpenAI function_call format; all existing test behaviors maintained (happy path, low confidence, missing fields, 422, 502, etc.). `vitest run` and `tsc --noEmit` PASS.

## Notes

**OpenAI function calling format (vs Anthropic tool_use):**

```
// Request — tools array
tools: [{
  type: "function",
  function: {
    name: "extract_requirements",
    description: "...",
    parameters: { ...same JSON Schema as EXTRACT_TOOL.input_schema... }
  }
}]
tool_choice: { type: "function", function: { name: "extract_requirements" } }

// Response parsing
const args = response.choices[0]?.message?.tool_calls?.[0]?.function?.arguments;
const parsed = JSON.parse(args); // { requirements, confidence, confirmation_questions? }
```

The `parameters` schema is identical to the existing `EXTRACT_TOOL.input_schema` — just needs rewrapping in the OpenAI `function` envelope.

**Env changes summary:**
- Remove: `ANTHROPIC_API_KEY`, `CLOUDFLARE_AI_GATEWAY_URL`
- Add: `OPENAI_API_KEY`
