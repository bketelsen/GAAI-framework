# Story E03S07: Booking Confirmation Email + Reminders + No-Show Tracking

## User Story

**As a** prospect who has selected a time slot with an expert,
**I want** to receive a confirmation email to finalize my booking, and reminder emails before the call,
**so that** my booking is intentional (not accidental or bot-initiated), I don't forget the call, and experts don't waste time on no-shows.

---

## Context

DEC-127 mandates a booking confirmation step after slot selection: the prospect receives an email with confirm/cancel buttons, and the booking is only finalized when the prospect clicks confirm. This prevents phantom bookings from agents and accidental clicks. It also adds automated reminders (J-1 + H-1) and no-show tracking with abuse prevention.

The current booking flow (E03S06 + E06S11) is: select slot → hold (10min TTL) → fill name/email/desc → confirm. This story inserts an email confirmation step: after the inline confirm, the prospect receives an email and must click "Confirmer" within 30 minutes. Until email confirmation, the booking is `status: pending_confirmation`.

Depends on E06S39 for the email sending infrastructure (`lib/email.ts`, Resend integration).

## Acceptance Criteria

### Booking confirmation flow

- [ ] AC1: Modify `POST /api/bookings/:id/confirm` (E06S11 AC6):
  - After the existing confirm logic (which creates the booking record and GCal event), set booking `status = 'pending_confirmation'` instead of `confirmed`.
  - Send confirmation email to prospect via `lib/email.ts` (E06S39): subject "Confirmez votre appel avec {expert_name}", HTML body containing:
    - Expert name + date + time (prospect's timezone, from request or detected).
    - "Confirmer mon appel" button → `GET {CORE_API_URL}/api/bookings/{booking_id}/email-confirm?token={confirmation_token}`.
    - "Annuler" button → `GET {CORE_API_URL}/api/bookings/{booking_id}/email-cancel?token={confirmation_token}`.
    - Footer: "Ce lien expire dans 30 minutes. Sans confirmation, le creneau sera libere."
  - Generate `confirmation_token`: HMAC-SHA256 signed token containing `{ booking_id, action: "confirm", exp: now + 1800 }`.
  - Return `{ status: "pending_confirmation", confirmation_sent_to: "{email}" }` to the booking widget.

- [ ] AC2: `GET /api/bookings/:id/email-confirm?token=...`:
  - Validates token (HMAC + expiry + booking_id match).
  - If valid: update booking `status = 'confirmed'` + `confirmed_at = NOW()`.
  - If booking already confirmed → redirect to success page with "Deja confirme" message.
  - If token expired → redirect to error page with "Le lien a expire. Votre creneau a ete libere."
  - If token invalid → 403.
  - On success: redirect to `https://{satellite_domain}/booking-confirmed/{booking_id}?token={booking_prep_token}` (confirmation page with call details + prep link).

- [ ] AC3: `GET /api/bookings/:id/email-cancel?token=...`:
  - Validates token (same HMAC validation).
  - If valid: cancel the booking (same logic as existing `DELETE /api/bookings/:id` — release slot, remove GCal event if created).
  - Update booking `status = 'cancelled_by_prospect'`.
  - Redirect to `https://{satellite_domain}/booking-cancelled` with "Votre reservation a ete annulee" message.

- [ ] AC4: **Expiry cron:** Bookings with `status = 'pending_confirmation'` AND `created_at + 30min < NOW()` are expired:
  - Set `status = 'expired_no_confirmation'`.
  - Release the held slot (update slot status, same as hold expiry in E06S11 AC9).
  - No email sent (the prospect chose not to confirm — silent expiry).
  - Cron runs every 5 minutes (piggyback on existing booking cleanup cron if available, or new scheduled handler).

### Expert opt-in (configurable)

- [ ] AC5: New column `booking_auto_confirm` on `experts` table (BOOLEAN, DEFAULT true):
  - `true` (default): After prospect email-confirms → booking is immediately `confirmed`. No expert action needed. Optimizes for new experts who want maximum bookings.
  - `false`: After prospect email-confirms → booking becomes `pending_expert_approval`. Expert receives email: "Nouvelle demande de reservation: {prospect_name} pour le {date}. [Accepter] [Refuser]". Expert has 24h to respond. If no response → auto-confirm (bias toward action, not inaction).
  - Dashboard toggle in `/dashboard/settings`: "Approuver manuellement les reservations" (checkbox). Calls `PATCH /api/experts/:id/profile` with `{ booking_auto_confirm: boolean }`.

### Automated reminders

- [ ] AC6: **Reminder cron:** For confirmed bookings:
  - J-1 (24h before `start_at`): Send email to prospect AND expert. Subject: "Rappel : votre appel demain a {time}". Body: date, time, Google Meet link, "Preparer mon appel" link (prospect), "Voir le profil du prospect" link (expert).
  - H-1 (1h before `start_at`): Send email to prospect AND expert. Subject: "Rappel : votre appel dans 1 heure". Same body as J-1 but with urgency framing.
  - Track sent reminders via KV or DB column (`reminder_24h_sent_at`, `reminder_1h_sent_at`) to prevent duplicate sends on cron retries.
  - Cron: run every 15 minutes, query bookings WHERE `status = 'confirmed' AND start_at BETWEEN NOW() + X AND NOW() + X + 15min` (for each reminder window).

### No-show tracking

- [ ] AC7: New column `no_show_count` on `prospects` table (INTEGER, DEFAULT 0):
  - After a booking's `start_at + 20min` (call duration) passes without the prospect joining (no confirmation of attendance — for MVP, this is a manual expert action):
    - Expert marks "No-show" via dashboard (`POST /api/bookings/:id/no-show`). Updates booking `status = 'no_show'`. Increments `prospects.no_show_count`.
  - After 2 no-shows (`no_show_count >= 2`): prospect is flagged. On next booking attempt:
    - If `no_show_count >= 2` → 403 `prospect_flagged` with message "Votre compte a ete signale pour des absences repetees. Contactez-nous pour debloquer." (For post-MVP: require prepayment — out of scope here).
  - Expert dashboard (`/dashboard/bookings`): "Marquer comme absent" button on past bookings where `status = 'confirmed'` and `start_at` has passed.

### Email templates

  - All templates include "Si vous n'avez pas effectue cette reservation..." footer. All include Callibrate branding (text logo).

## Notes

- **Two-phase confirmation:** The inline confirm (E03S06 AC6) becomes "submit booking request". The email confirm (AC2) becomes the true finalization. The hold TTL (10min) and email confirmation TTL (30min) are independent timers. The hold expires and releases the slot if the inline form isn't submitted. The email confirmation expires and releases the booking if the email isn't clicked.
- **GCal event creation timing:** Currently, the GCal event is created on inline confirm (E06S11 AC6). With this story, it should be created on EMAIL confirm (AC2) — otherwise the expert sees a GCal event for an unconfirmed booking. Delivery must move the GCal event creation from inline confirm to email confirm.
- **Expert opt-in UX:** The default is auto-confirm to maximize conversion for new experts. The manual approval option is for senior/premium experts who want to screen prospects before committing their time. The 24h auto-confirm fallback prevents bookings from being stuck in limbo.
