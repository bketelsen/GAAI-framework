# Story E02S06: Bookings Page

## User Story

**As a** registered expert,
**I want** to view my upcoming and past bookings with prospect details and meeting links,
**so that** I can prepare for calls and track my booking history.

---

## Context

Bookings are the platform's core monetizable unit. This page surfaces booking data from `GET /api/experts/:id/bookings` (E06S38) and provides management actions (cancel, reschedule) via existing E06S11 endpoints.

The bookings page complements the lead pipeline (E02S05): leads track the commercial pipeline (qualification, billing), bookings track the operational pipeline (upcoming calls, meeting logistics).

---

## Acceptance Criteria

### Booking List

- [ ] AC1: Bookings route (`/dashboard/bookings`): loader fetches `GET /api/experts/:id/bookings?period={period}&page={page}&per_page=20` via `api.server.ts`. URL search params drive the query.

- [ ] AC2: Period tabs: "À venir" (upcoming, default) / "Passés" (past). Each tab is a `<Link>` setting `?period=upcoming|past`. Active tab visually highlighted. Count badge per tab.

- [ ] AC3: Booking cards display:
  - Date and time (formatted: "Jeu. 6 mars 2026, 14:00 — 14:20", in expert's local timezone)
  - Prospect name and email (from embedded prospect data)
  - Meeting URL (clickable Google Meet link, copy-to-clipboard button)
  - Status badge: "Confirmé" (green) / "En attente" (yellow) / "Annulé" (red) / "Terminé" (gray)
  - "Voir le détail" expand / link

- [ ] AC4: Pagination: same pattern as leads page (AC4 of E02S05).

### Booking Detail

- [ ] AC5: Detail view (drawer or sub-route `/dashboard/bookings/:id`):
  - Full prospect context: name, email, requirements summary (from prospect.requirements JSONB)
  - Prep page link (E06S11 token-gated prep endpoint URL)
  - Meeting URL with "Ouvrir dans Google Meet" button
  - Booking status with timeline (created → confirmed → completed or cancelled)
  - Cancel / Reschedule actions (for upcoming bookings only)

### Actions

- [ ] AC6: **Cancel action:** Available on upcoming, non-cancelled bookings. Opens confirmation dialog: "Êtes-vous sûr de vouloir annuler ce rendez-vous ? Le prospect sera notifié par email." Optional: `cancel_reason` textarea. Calls `POST /api/bookings/:id/cancel` (E06S11) via action. On success: toast "Rendez-vous annulé", status updates.

- [ ] AC7: **Reschedule action:** Available on upcoming, non-cancelled bookings. Opens dialog with datetime picker (date + time selector). New datetime must be in the future and during expert's available slots. Calls `POST /api/bookings/:id/reschedule` (E06S11) with `{ new_starts_at }` via action. On success: toast "Rendez-vous reporté au {new_date}", booking updates.

### Empty State

- [ ] AC8: When no bookings exist: "Aucun rendez-vous pour le moment" + "Vos rendez-vous apparaîtront ici dès qu'un prospect réservera un créneau via votre profil."

### Quality

- [ ] AC9: `tsc --noEmit` — zero errors.

- [ ] AC10: Vitest tests: loader with period filter, cancel action, reschedule action, empty state.

## Out of Scope

- Calendar view (monthly/weekly visual calendar — post-MVP, standard list view at MVP)
- Booking notes / expert pre-call notes feature
- Automatic reminders configuration (handled server-side by E06S11 cron)
- Video call recording or transcript integration
- Multi-participant bookings (1:1 only at MVP)

- E06S11 (POST /api/bookings/:id/cancel + POST /api/bookings/:id/reschedule)

## Notes

- **Timezone handling:** Booking times stored as UTC in DB. Dashboard displays in expert's local timezone (detected via `Intl.DateTimeFormat().resolvedOptions().timeZone`). The loader could pass timezone as a parameter to the API, or the frontend formats UTC dates client-side.
- **Prep page link:** E06S11 generates a token-gated prep page URL for each booking. This URL shows prospect requirements to the expert before the call. The dashboard should display this link prominently for upcoming bookings.
- **Reschedule constraint:** The new datetime must fall within the expert's available slots (per GCal freebusy). The datetime picker should ideally show available slots — but at MVP, a simple datetime input with server-side validation (E06S11 returns 422 if slot unavailable) is acceptable.
- **PostHog events (Delivery must include):** `expert.bookings_viewed` (page load, period_filter), `expert.booking_cancelled` (booking_id), `expert.booking_rescheduled` (booking_id). Server-side capture in actions.
