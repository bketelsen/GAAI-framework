# Story E06S05: Matching Engine Scoring Function + Match API

## User Story

**As a** prospect who has submitted their project needs,
**I want** to receive a ranked list of the most relevant AI experts,
**so that** I can choose the right expert without sifting through irrelevant profiles.

---

## Context

The core matching algorithm. Scores expert profiles against prospect requirements using JSONB field comparison. Bi-directional: also checks if the prospect matches the expert's preferences (deal-breaker logic). Weights are configurable per satellite via `satellite_configs.matching_weights` — not hardcoded — so calibration can happen post-M0 without code changes. Scoring function is a pure function (no side effects, fully testable).

---

## Acceptance Criteria

- [ ] AC1: `scoreMatch(expertProfile: ExpertProfile, expertPreferences: ExpertPreferences, prospectRequirements: ProspectRequirements, weights: MatchingWeights): MatchScore` is a pure TypeScript function returning `{ score: number, breakdown: ScoreBreakdown }`
- [ ] AC2: Score components and default weights: `skills_overlap` (max 40pts), `industry_match` (max 20pts), `budget_compatibility` (max 20pts), `timeline_match` (max 10pts), `language_match` (max 10pts) — total max 100
- [ ] AC3: `skills_overlap` = `(matching skills count / prospect.skills_needed.length) * weight` — 0 if prospect has no skills_needed
- [ ] AC4: `industry_match` = full points if `expert.profile.industries` includes `prospect.requirements.industry`, 0 otherwise
- [ ] AC5: `budget_compatibility` = full points if `expert.rate_min * 20` (20-day month proxy) falls within `prospect.requirements.budget_range`, partial if overlapping, 0 if no overlap
- [ ] AC6: Bi-directional deal-breaker: if `expert.preferences.excluded_industries` includes `prospect.requirements.industry` → total score forced to 0, `breakdown.deal_breaker: true`
- [ ] AC7: `POST /api/matches/compute` accepts `{ prospect_id, satellite_id? }` — loads prospect requirements, loads all experts with `availability != 'unavailable'`, applies `scoreMatch` with weights from `satellite_configs.matching_weights` (falls back to defaults if no satellite_id), stores top-20 results in `matches` table, returns `{ computed: number, top_matches: MatchResult[] }`
- [ ] AC8: `GET /api/matches/:prospect_id` returns top-5 matches from `matches` table (status != 'expired'), paginated, sorted by `score DESC` then `experts.composite_score DESC` as tiebreaker
- [ ] AC12: Each match result includes a `quality_tier` field derived from `experts.composite_score`: `'new'` (composite_score = 0), `'rising'` (1–25), `'established'` (26–60), `'top'` (61–100) — anonymized signal for prospects, no raw score exposed
- [ ] AC9: Match results returned from `GET /api/matches/:prospect_id` are **anonymized** by default: no `display_name`, no `avatar_url`, no `cal_username` — only `skills`, `industries`, `project_types`, `score`, `score_breakdown`
- [ ] AC10: Matching weights can be overridden per request via `satellite_configs.matching_weights` — altering weights changes the ranked order without code changes
- [ ] AC11: `scoreMatch` function has unit tests covering: full match, zero match, deal-breaker, partial skills overlap, missing optional fields

- Prospect-initiated filtering (UI-side — E03)
- Expert ranking by rating or past project count (post-MVP — no rating system yet)

## Notes

Can be developed in parallel with E06S03 and E06S04 — only requires E06S02. The matching weights object shape: `{ skills_overlap: number, industry_match: number, budget_compatibility: number, timeline_match: number, language_match: number }` — values are max point caps, not multipliers. Default weights are defined as a constant in code; satellite_configs overrides them. The 20-day proxy for budget is intentionally simple — refine after M1 data.
