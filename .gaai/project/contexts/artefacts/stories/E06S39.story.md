# Story E06S39: Email OTP Verification Service

## User Story

**As a** platform operator,
**I want** email addresses verified via a 6-digit OTP code before revealing expert profiles,
**so that** billing integrity is preserved (no credits spent on unreachable leads) and fake emails are rejected before they enter the system.

---

## Context

The prospect funnel requires email verification at the gate (DEC-111: email captured after anonymized matches). The current `POST /api/prospects/{id}/identify` accepts any email without verification. This story adds OTP verification as a prerequisite to identification, following the Calendly pattern validated by industry research (DEC-121). It also introduces server-side pre-checks to reject bad emails before sending any OTP (DEC-122: MX check + disposable blacklist).

This is a foundational service — email sending infrastructure (Resend or SES) and OTP logic are reused by E03S07 (booking confirmation), E03S10 (magic link), and all future transactional emails.

## Acceptance Criteria

- [ ] AC1: New `lib/otp.ts` module: `generateOTP()` returns a cryptographically random 6-digit numeric string (`crypto.getRandomValues`, zero-padded). `hashOTP(code)` returns SHA-256 hex digest. `verifyOTP(code, hash)` compares hashed input to stored hash. OTP is NEVER stored in cleartext — only the hash is persisted.

- [ ] AC2: New `lib/email.ts` module: `sendEmail({ to, subject, html })` sends transactional email via Resend API (env `RESEND_API_KEY`). Returns `{ success: boolean, messageId?: string }`. Handles Resend API errors gracefully (429 rate limit → retry-after header, 5xx → log + return `success: false`). SES can be swapped later — interface is provider-agnostic.

- [ ] AC3: New `lib/emailValidation.ts` module:
  - `validateEmailSyntax(email)`: RFC 5322 basic check (regex). Returns `{ valid: boolean, normalized: string }`.
  - `normalizeEmail(email)`: Strip `+tag` suffix, normalize Gmail dots (remove dots before @gmail.com), lowercase domain. Returns normalized email string.
  - `checkMXRecord(domain)`: DNS MX lookup via `fetch('https://dns.google/resolve?name={domain}&type=MX')` (Cloudflare Workers cannot use `dns.resolve`). Returns `{ hasMX: boolean }`.
  - `isDisposableEmail(domain)`: Check domain against an embedded blacklist of ~3000 disposable email providers (open-source list, e.g. `disposable-email-domains`). Returns `boolean`.

- [ ] AC4: **Pre-check pipeline**: Before sending OTP, execute in order: (1) syntax validation → 422 `invalid_email_syntax`, (2) email normalization, (3) MX record check → 422 `no_mx_record` with message "Cette adresse email semble invalide.", (4) disposable blacklist → 422 `disposable_email` with message "Les adresses email temporaires ne sont pas acceptees. Utilisez votre adresse professionnelle." All checks run server-side before any OTP generation or email sending.

- [ ] AC5: `POST /api/prospects/{id}/otp/send` endpoint:
  - Requires valid prospect JWT (same auth as existing prospect endpoints).
  - Accepts `{ email: string }`.
  - Runs pre-check pipeline (AC4). If any check fails → return error, no OTP sent.
  - Generates 6-digit OTP, hashes it, stores in KV: key `otp:{prospect_id}` → value `{ hash, email, attempts: 0, created_at }`, TTL 600 seconds (10 minutes).
  - Sends email via `lib/email.ts`: subject "Votre code de verification Callibrate", body contains the 6-digit code prominently displayed + "Ce code expire dans 10 minutes."
  - Returns `{ sent: true, expires_in: 600 }`.
  - **Rate limits:** Max 3 OTP sends per email per 24h (tracked in KV: `otp-sends:{normalized_email}` → counter, TTL 86400). If exceeded → 429 `otp_rate_limit` with message "Trop de tentatives. Reessayez dans {remaining} heures."

- [ ] AC6: `POST /api/prospects/{id}/otp/verify` endpoint:
  - Requires valid prospect JWT.
  - Accepts `{ code: string }`.
  - Retrieves OTP record from KV (`otp:{prospect_id}`).
  - If no record → 410 `otp_expired` with message "Code expire. Demandez un nouveau code."
  - If `attempts >= 5` → 429 `max_attempts` with message "Trop de tentatives. Demandez un nouveau code." + delete KV record.
  - Increments `attempts` counter in KV.
  - Hashes submitted code, compares to stored hash.
  - If mismatch → 401 `invalid_code` with message "Code invalide. {5 - attempts} tentatives restantes."
  - If match → delete KV record + return `{ verified: true, email: <normalized_email> }`.

- [ ] AC7: Modify `POST /api/prospects/{id}/identify` to require prior OTP verification:
  - After OTP verify succeeds, store `verified:{prospect_id}` in KV (TTL 300 seconds — 5-minute window to complete identification).
  - `identify` endpoint checks for `verified:{prospect_id}` in KV before proceeding. If absent → 403 `email_not_verified` with message "Verifiez votre email avant de continuer."
  - On successful identification: delete `verified:{prospect_id}` from KV. Store `verified_at` timestamp on prospect record.

- [ ] AC8: **Email uniqueness on normalized email:** Before sending OTP, check if a prospect with the same normalized email already exists in DB. If yes → 409 `email_already_used` with message "Un compte existe deja avec cette adresse. Consultez vos emails pour retrouver vos resultats." (links to E03S10 return visit flow, post-MVP).

- [ ] AC9: OTP email template is plain HTML (no external CSS, no images). Works in all major email clients. Contains: Callibrate logo text ("Callibrate"), the 6-digit code in large monospace font, expiry notice, "Si vous n'avez pas demande ce code, ignorez cet email." footer.

- [ ] AC10: All new endpoints registered in `src/index.ts` router. Routes: `POST /api/prospects/:id/otp/send`, `POST /api/prospects/:id/otp/verify`.

- [ ] AC11: Unit tests for `lib/otp.ts` (generation, hashing, verification), `lib/emailValidation.ts` (syntax, normalization, MX mock, disposable check), and integration tests for both OTP endpoints (happy path, expired, max attempts, rate limit, pre-check failures). All existing tests pass. `tsc` clean across all workers.

- [ ] AC12: `RESEND_API_KEY` documented in `wrangler.toml` as required secret (comment). Not committed as plaintext. Setup instruction in notes for human action: `npx wrangler secret put RESEND_API_KEY --env staging`.

## Notes

- **Resend vs SES:** Resend is recommended for MVP — simpler API, free tier (100 emails/day), no domain verification required for testing (sandbox mode). SES requires domain verification + request to leave sandbox. Decision on provider is implementation-level, not architectural — the `lib/email.ts` interface is provider-agnostic.
- **KV for OTP storage:** KV is ideal — built-in TTL handles expiry automatically, no cron needed. Eventual consistency is acceptable (OTP verify is always same region as send within seconds).
- **Disposable email list:** Embed as a Set in `lib/emailValidation.ts` (compile-time). ~3000 domains = ~50KB. Alternative: use a third-party API (e.g., `open.kickbox.com`) but adds external dependency + latency. Embedded list preferred for MVP.
- **Human action required post-delivery:** `npx wrangler secret put RESEND_API_KEY --env staging` (and production at launch). Create Resend account + obtain API key.
