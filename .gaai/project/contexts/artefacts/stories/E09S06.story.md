# Story: Frontend E2E regression suite (Playwright)

## User Story

**As a** founder deploying frontend changes to staging,
**I want** automated browser tests validating the satellite prospect funnel and expert dashboard,
**so that** rendering failures, broken interactions, and JS errors are caught before I discover them manually.

---

## Context

5 done stories created frontend pages that are never tested post-deploy:

**Satellite prospect funnel (E03S01 → E03S02 → E03S03 → E03S06):**
- `/match` — freetext input page (textarea, submit button, loading state)
- `/confirm` — AI-extracted requirements confirmation (structured display, edit, submit to matching)
- `/results` — match results display (anonymized expert cards, email gate, OTP verification, full profile reveal)
- Inline booking widget (availability calendar, hold, confirm)

**Expert dashboard (E02S01 → E02S08):**
- `/login` — Supabase auth (email + password)
- `/onboarding` — 4-step wizard (display name, headline, bio, skills, rate, etc.)
- `/dashboard/leads` — lead pipeline with actions
- `/dashboard/bookings` — upcoming/past bookings
- `/dashboard/billing` — credits, subscription, spending controls
- `/dashboard/analytics` — composite score, charts

These pages are server-rendered (SSR) by Cloudflare Workers. They require a real browser to validate HTML rendering, JavaScript execution, and user interactions.

---

## Acceptance Criteria

### Setup
- [ ] AC1: Playwright is added as a devDependency in the root `package.json` (or a dedicated `tests/e2e/package.json`)
- [ ] AC2: A `playwright.config.ts` exists at the project root (or `tests/e2e/`) with `baseURL` pointing to staging satellite and dashboard URLs, `headless: true`, `retries: 1`, `timeout: 30000`
- [ ] AC3: A `test:e2e` script exists in root `package.json` that runs `npx playwright test`

### Satellite funnel tests
- [ ] AC4: A test file `tests/e2e/satellite-funnel.spec.ts` exists with a test that navigates to `/match` on the staging satellite URL and verifies: page loads (HTTP 200), textarea is visible, submit button is present
- [ ] AC5: A test navigates to `/match`, enters freetext ("Je cherche un expert n8n pour automatiser mon CRM"), submits the form, and verifies navigation to `/confirm` with extracted requirements displayed
- [ ] AC6: A test verifies that `/robots.txt` returns a valid robots file and `/sitemap.xml` returns valid XML
- [ ] AC7: A test verifies that `/privacy` and `/terms` pages load (HTTP 200 or valid redirect)

### Dashboard tests
- [ ] AC8: A test file `tests/e2e/dashboard-auth.spec.ts` exists with a test that navigates to the dashboard login page and verifies: login form is visible (email input, password input, submit button)
- [ ] AC9: A test logs in with the smoke test user credentials (`smoke-runner@callibrate.io` / `SmokeRunner!2026`), and verifies redirect to the dashboard (or onboarding if first visit)
- [ ] AC10: After login, a test navigates to each dashboard page (`/dashboard/leads`, `/dashboard/bookings`, `/dashboard/billing`, `/dashboard/analytics`) and verifies each page loads without JS errors (no uncaught exceptions in console)

### CI integration
- [ ] AC11: `deploy-staging.yml` includes a new job `e2e-test` with `needs: [smoke-test]` (runs after smoke test passes) that installs Playwright browsers and runs the E2E suite
- [ ] AC12: The `e2e-test` job has `timeout-minutes: 10` and uses `continue-on-error: true` with a status annotation (E2E failures are reported but do not block the deploy pipeline initially — can be tightened later)
- [ ] AC13: On test failure, Playwright screenshots are uploaded as GitHub Actions artifacts for debugging

> All acceptance criteria must be verifiable. Vague criteria are invalid.

## Out of Scope

- Testing the full booking flow end-to-end (requires GCal-connected expert — not available in staging smoke test user)
- Testing OTP email verification (requires real email delivery — covered by API smoke test in E09S05)
- Testing LemonSqueezy subscription activation (requires LS test environment)
- Visual regression testing (screenshot comparison) — premature at this stage
- Testing on multiple browsers (Chromium-only is sufficient for staging validation)
- Component-level tests (vitest + jsdom) — those are unit tests, not E2E

## Notes

- **Chromium-only**: Use `projects: [{ name: 'chromium', use: { ...devices['Desktop Chrome'] } }]` in playwright.config.ts. No need for Firefox/WebKit for staging validation.
- **`continue-on-error: true` in AC12**: E2E tests are inherently more flaky than curl checks. Start with soft failure (warning badge, not red badge) and tighten to hard failure once the suite is stable for 2+ weeks.
- **Screenshot artifacts (AC13)**: Use `npx playwright test --reporter=html` and upload `playwright-report/` as an artifact. This gives visual debugging without polluting the repo.
- **Turnstile on satellite forms**: The staging Turnstile secret is the "always pass" test key (`1x0000000000000000000000000000000AA`). Playwright tests must include the Turnstile widget interaction — use `page.waitForSelector('[data-turnstile-response]')` or mock the response.
- **Dashboard auth**: The smoke test user authenticates via Supabase email/password. The dashboard uses PKCE OAuth flow. The E2E test should use the login form directly (email + password fields), not API auth.
- **Delivery Agent** should check if Playwright is already installed anywhere in the monorepo before adding it. If the Playwright MCP server is configured, it runs locally — the CI job needs its own `npx playwright install chromium` step.
- **Estimated CI time**: Playwright install (~30s) + tests (~60-90s) = ~2 min added to pipeline. Acceptable.
