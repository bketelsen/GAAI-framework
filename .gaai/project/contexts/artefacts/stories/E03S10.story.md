# Story E03S10: Return Visit — Cookie Auth + Magic Link

## User Story

**As a** prospect who completed the funnel and verified my email,
**I want** to return to my match results without re-entering my email and verifying again,
**so that** I can review experts and book a call at my own pace across multiple visits.

---

## Context

DEC-128 addresses the return visit problem: a prospect who completes the funnel (email verified, profiles revealed) but doesn't book may return days later. Without persistent state, they'd redo the entire funnel. Two mechanisms work together: (1) cookie/localStorage on same device, (2) magic link in email for cross-device access.

Post-MVP because at launch volume, most prospects will convert or abandon in a single session. Return visits become important as volume grows.

---

## Acceptance Criteria

### Cookie-based authentication

- [ ] AC1: After successful OTP verification (E06S39 `POST /api/prospects/{id}/otp/verify`), generate a `prospect_auth_token`:
  - JWT signed with `PROSPECT_TOKEN_SECRET`, payload: `{ prospect_id, email, purpose: "prospect:session", exp: now + 604800 }` (7 days).
  - Return token in response body AND set as HttpOnly cookie: `Set-Cookie: prospect_session={token}; HttpOnly; Secure; SameSite=Lax; Max-Age=604800; Path=/`.

- [ ] AC2: On satellite page load (any `/results/*` or `/match/results/*` route):
  - Check for `prospect_session` cookie.
  - If present: validate JWT (signature + expiry + purpose claim).
  - If valid: load prospect data + matches from API. Display results page directly (no freetext input, no email gate).
  - If invalid/expired: clear cookie, redirect to `/match` (start fresh).

- [ ] AC3: **API support:** `GET /api/prospects/{id}/matches` (and other prospect endpoints) accepts the `prospect_session` token in the `Authorization: Bearer {token}` header OR from the cookie (forwarded by satellite). Same auth as existing prospect JWT but with the `prospect:session` purpose.

### Magic link in emails

- [ ] AC4: After OTP verification, the confirmation email (or a separate "Vos resultats sont prets" email) includes a magic link at the bottom:
  - "Retrouvez vos resultats de matching : [Voir mes resultats]({satellite_url}/results/{prospect_id}?token={magic_link_token})"
  - `magic_link_token`: JWT signed with `PROSPECT_TOKEN_SECRET`, payload: `{ prospect_id, purpose: "prospect:magic_link", exp: now + 604800 }` (7 days, same TTL as cookie).

- [ ] AC5: Magic link endpoint on satellite: `GET /results/{prospect_id}?token={magic_link_token}`:
  - Validate token (signature + expiry + purpose claim + prospect_id match).
  - If valid: set the `prospect_session` cookie (same as AC1) → display results page. The magic link effectively "logs in" the prospect on this device.
  - If invalid/expired: redirect to `/match` with message "Ce lien a expire. Decrivez a nouveau votre projet pour retrouver vos experts."

### Single token, two channels

- [ ] AC6: Both the cookie and the magic link use the same `prospect_auth_token` format (JWT with `PROSPECT_TOKEN_SECRET`). The only difference is the `purpose` claim (`prospect:session` vs `prospect:magic_link`) for audit purposes. Both are accepted by the API for authentication.

### Expiry and re-identification

- [ ] AC7: After 7 days, both cookie and magic link expire. The prospect must re-identify (email + OTP) to access results again. Match results are still stored server-side indefinitely (retention per DEC-129) — only the auth token expires.

- [ ] AC8: "Renvoyer le lien" endpoint: `POST /api/prospects/{id}/magic-link/resend` — requires active prospect record (email verified). Generates new magic link token and sends email. Rate limit: 3 per 24h (same pattern as OTP sends). Useful when original email is lost or link expired.

### PostHog events

- [ ] AC9:
  - `prospect.return_visit_cookie` (prospect_id) — return via cookie.
  - `prospect.return_visit_magic_link` (prospect_id) — return via magic link.
  - `prospect.session_expired` (prospect_id) — cookie/link expired on attempted return.
  - `prospect.magic_link_resent` (prospect_id).

### Tests

- [ ] AC10: Unit tests for token generation/validation (both purposes). Integration tests: cookie-based return visit (valid, expired), magic link return visit (valid, expired, wrong prospect_id), resend endpoint. All existing tests pass. `tsc` clean.

## Out of Scope

- Prospect account/dashboard (no login system — stateless auth via token)
- Password-based authentication for prospects
- Social login (Google/LinkedIn) for prospects
- Token refresh mechanism (simpler to re-verify after 7 days)

## Notes

- **Security consideration:** The magic link token is equivalent to a session token. If forwarded or intercepted, it grants access to the prospect's results. Mitigations: (1) HttpOnly + Secure cookie prevents XSS extraction, (2) 7-day TTL limits exposure window, (3) results show expert profiles but no sensitive data beyond what the prospect already opted to receive. Risk is acceptable for the data classification level.
- **LocalStorage alternative:** The AC specifies HttpOnly cookie. An alternative is localStorage for the token (more resistant to CSRF but vulnerable to XSS). Cookie is preferred because: (1) HttpOnly prevents JS access, (2) automatically sent with requests (no client-side code needed), (3) consistent with the rest of the auth model.
