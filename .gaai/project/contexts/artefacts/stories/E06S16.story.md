# Story E06S16: Replace n8n Webhooks with CF Workflows (booking.confirmed + booking.completed)

## User Story

**As a** platform operator,
**I want** time-delayed survey workflows to run natively on Cloudflare Workflows instead of n8n,
**so that** the feedback loop operates without external infrastructure dependencies or hosting costs.

---

## Context

E06S06 implemented two n8n webhook triggers in `email-notifications.ts`:
- AC7: `booking.confirmed` → n8n webhook for enriched expert notification with prospect context
- AC10: `booking.completed` → n8n webhook for J+7 call experience survey + J+45 project satisfaction survey

DEC-59 decided to remove n8n from the stack entirely. CF Workflows replaces n8n for all durable, time-delayed workflows. `step.sleep()` supports string durations ("7 days") or milliseconds. Workflows are scoped to the Worker script (staging vs prod isolation via separate Workers, not `[[env.staging.workflows]]` which doesn't exist).

**Dependency chain:** E06S09 (consumer ready) → **E06S16 (survey emails sent)** → E06S17 (submissions received) → score-computation queue → composite_score updated.

---

## Acceptance Criteria

- [ ] AC1: `BookingConfirmedWorkflow` CF Workflow created — triggered after booking confirmation. Sends enriched expert notification email with prospect context (name, requirements summary, booking date/time, Meet link) via `EMAIL_NOTIFICATIONS` queue dispatch
- [ ] AC2: `BookingCompletedWorkflow` CF Workflow created — triggered after booking completion (or scheduled_at passed). Executes: `step.sleep('7 days')` → dispatch `survey.call_experience` email to prospect, `step.sleep('38 days')` → dispatch `survey.project_satisfaction` email to prospect
- [ ] AC3: Survey emails contain a token-gated link to the submission endpoint (JWT with booking_id + prospect_id, short-lived expiry 30 days)
- [ ] AC4: `triggerN8nBookingConfirmed()` removed from `src/queues/email-notifications.ts` (was AC7)
- [ ] AC5: `triggerN8nBookingCompleted()` removed from `src/queues/email-notifications.ts` (was AC10)
- [ ] AC6: `N8N_WEBHOOK_URL` removed from `src/types/env.ts` Env interface
- [ ] AC7: `[[workflows]]` blocks added to `wrangler.toml` (top-level only — not per-env). Workflow class names: `BookingConfirmedWorkflow`, `BookingCompletedWorkflow`
- [ ] AC8: Sleep acceleration for staging: `SURVEY_DELAY_7D_MS` and `SURVEY_DELAY_38D_MS` secrets. If absent (production) → real durations (7 days, 38 days). If present (staging) → accelerated (e.g., 7000ms for smoke tests). Pattern: `parseInt(this.env.SURVEY_DELAY_7D_MS ?? String(7 * 24 * 60 * 60 * 1000))`
- [ ] AC9: `this.env` used in `WorkflowEntrypoint` to access secrets and bindings (same pattern as DurableObject)
- [ ] AC10: Unit tests updated in `email-notifications.test.ts` — n8n webhook tests removed, workflow dispatch tests added
- [ ] AC11: `tsc --noEmit` clean, `vitest run` all tests pass

## Notes

CF Workflows patterns validated against official docs (DEC-59):
- `[[workflows]]` top-level only
- `this.env` accessible in `WorkflowEntrypoint`
- `step.sleep('label', duration)` — string or number
- Vitest recommended for testing; mock `step.sleep` and `step.do`
- `wrangler dev` supports Workflows locally (wrangler >= 3.89.0)
- CLI: `wrangler workflows instances list|describe|terminate|restart|pause|resume [NAME] [ID]`
