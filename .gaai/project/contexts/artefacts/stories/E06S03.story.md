# Story E06S03: Auth Middleware & Expert Registration + Profile API

## User Story

**As an** AI integration expert,
**I want** to register an account and manage my profile via API,
**so that** my profile exists in the system and can be used for matching.

---

## Context

First user-facing API. Establishes the auth pattern (Supabase JWT validated in CF Worker middleware) used by all subsequent authenticated endpoints. Expert registration triggers async Cal.com user creation and welcome email via Queues. Profile supports partial JSONB updates so criteria can evolve without replacing the full object.

---

## Acceptance Criteria

- [ ] AC1: CF Worker middleware validates Supabase JWT on all `/api/experts/*` routes — returns `401` with `{ error: "Unauthorized" }` if token missing or invalid
- [ ] AC2: `POST /api/experts/register` accepts `{ email, display_name, headline?, bio?, rate_min?, rate_max? }`, validates with Zod, creates row in `experts` table, returns `201` with `{ expert_id, display_name }`
- [ ] AC3: `POST /api/experts/register` returns `422` with field-level error details on Zod validation failure
- [ ] AC4: `POST /api/experts/register` returns `409` if email already registered
- [ ] AC5: On successful registration, two Queue messages are pushed: one to `matching-jobs` with `{ type: "cal.setup", expert_id }` and one to `email-notifications` with `{ type: "expert.registered", expert_id, email }`
- [ ] AC6: Rate limiting on `POST /api/experts/register`: max 10 requests per IP per 60 seconds, enforced via `RATE_LIMITING` KV — returns `429` on breach
- [ ] AC7: `GET /api/experts/:id/profile` returns the expert's full profile row — authenticated, only own profile accessible (returns `403` for other expert's profile)
- [ ] AC8: `PATCH /api/experts/:id/profile` accepts a partial update object, performs JSONB merge (not replace) for `profile` and `preferences` fields — returns `200` with updated expert row
- [ ] AC9: `PATCH /api/experts/:id/profile` validates that `availability` value is one of `['available', 'limited', 'unavailable']` — returns `422` otherwise
- [ ] AC10: All endpoints return `Content-Type: application/json` and use consistent error shape `{ error: string, details?: object }`

## Notes

Use Supabase JS client (service role key) inside Workers for DB writes. For JWT validation, use Supabase Auth's `getUser()` method or validate the JWT against Supabase's JWKS endpoint. JSONB merge for `profile` and `preferences`: use PostgreSQL `||` operator via Supabase RPC to deep-merge — not a full replace.
