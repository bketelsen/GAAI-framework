# Story E06S28: Staging — Secrets manquants (Core + Satellite + Turnstile)

## User Story

**As a** platform operator,
**I want** all required secrets configured on staging for every deployed Worker,
**so that** no endpoint silently degrades due to a missing credential.

---

## Context

Après l'ajout de `OPENAI_API_KEY` et `POSTHOG_API_KEY`, le worker `callibrate-core-staging` a 11 secrets. Trois secrets restent manquants :

- `SURVEY_TOKEN_SECRET` — requis par E06S17. Note de livraison : "E06S16 must be updated to use env.SURVEY_TOKEN_SECRET (currently uses PROSPECT_TOKEN_SECRET)". Ces deux tokens servent des usages différents (prospects vs surveys) et ne doivent pas partager la même clé de signature.
- `TURNSTILE_SECRET_KEY` — requis par E06S20 (Turnstile invisible CAPTCHA sur POST /api/prospects/submit).
- Delays CF Workflow (`SURVEY_DELAY_7D_MS`, `SURVEY_DELAY_38D_MS`) — requis par E06S16. En production : 604800000ms (7j) et 3283200000ms (38j). En staging : valeurs courtes pour tests (ex. 60000ms et 120000ms). À configurer comme `vars` dans `wrangler.toml`, pas comme secrets (non sensibles).

Le worker `callibrate-satellite-staging` n'a que `ADMIN_SECRET`. Il lui manque `SUPABASE_URL`, `SUPABASE_ANON_KEY`, et `POSTHOG_API_KEY` — sans quoi il ne peut pas charger les `satellite_configs` ni injecter le tracking PostHog dans le HTML.

Cette story est **founder-executed** : aucun code à écrire.

---

## Acceptance Criteria

### callibrate-core staging

- [ ] AC1: `SURVEY_TOKEN_SECRET` créé — `openssl rand -base64 32 | npx wrangler secret put SURVEY_TOKEN_SECRET --env staging`
- [ ] AC2: `TURNSTILE_SECRET_KEY` créé — depuis CF Dashboard → Turnstile → créer un site (type: invisible) → copier la Secret Key → `npx wrangler secret put TURNSTILE_SECRET_KEY --env staging`
- [ ] AC3: `SURVEY_DELAY_7D_MS=60000` et `SURVEY_DELAY_38D_MS=120000` ajoutés dans `wrangler.toml [env.staging.vars]`, committé sur staging (valeurs courtes pour permettre les smoke tests sans attendre 7 jours)
- [ ] AC4: Vérification que `E06S16` utilise bien `env.SURVEY_TOKEN_SECRET` (et non `PROSPECT_TOKEN_SECRET`) pour signer les tokens survey → rollup fix committé sur staging si nécessaire

### callibrate-satellite staging

- [ ] AC5: `SUPABASE_URL` créé — `cd workers/satellite && npx wrangler secret put SUPABASE_URL --env staging` (même valeur que core)
- [ ] AC6: `SUPABASE_ANON_KEY` créé — `cd workers/satellite && npx wrangler secret put SUPABASE_ANON_KEY --env staging` (même valeur que core)
- [ ] AC7: `POSTHOG_API_KEY` créé — `cd workers/satellite && npx wrangler secret put POSTHOG_API_KEY --env staging` (même valeur que core)

### Turnstile site key

- [ ] AC8: Site key Turnstile (depuis AC2) noté pour injection dans le satellite HTML. La Turnstile site key est publique — à ajouter comme var dans `workers/satellite/wrangler.toml [env.staging.vars]` sous `TURNSTILE_SITE_KEY` si le satellite HTML la nécessite. *(Vérifier si E06S20/E07S02 injectent la site key dans le HTML ou si c'est une story satellite UI.)*

### Smoke tests

- [ ] AC9: `npx wrangler secret list --env staging` (core) → 12 secrets présents (11 précédents + SURVEY_TOKEN_SECRET + TURNSTILE_SECRET_KEY)
- [ ] AC10: `cd workers/satellite && npx wrangler secret list --env staging` → ADMIN_SECRET + SUPABASE_URL + SUPABASE_ANON_KEY + POSTHOG_API_KEY présents

**Ordre recommandé :** AC1 → AC2 → AC3 → AC4 → AC5 → AC6 → AC7 → smoke tests.

**SURVEY_DELAY values :** 60000ms (1 min) + 120000ms (2 min) permettent de tester le flow survey complet en staging sans simulation. En production, supprimer ces vars (CF Workflow utilisera les valeurs hardcodées du code ou des constantes).

**Rollup fix E06S16/E06S17 (AC4) :** Si le code utilise encore `PROSPECT_TOKEN_SECRET` pour signer les tokens survey, créer un mini-commit de fix sur staging directement (pas de nouvelle story nécessaire — c'est une erreur de merge, pas un scope nouveau).
