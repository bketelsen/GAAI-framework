# Story E03S08: Funnel Restructure — Merge Confirmation into Results Page

## User Story

**As a** prospect describing my automation needs,
**I want** to see my extracted requirements and matching results on a single page instead of navigating through 3 separate steps,
**so that** the funnel is faster and I reach expert profiles with fewer clicks.

---

## Context

DEC-123 identified that the current 3-page funnel (E03S01 input → E03S02 confirmation → E03S03 results) includes unnecessary friction. When `ready_to_match: true` (high extraction confidence), the confirmation page (E03S02) adds a wasted screen — the prospect sees a summary and clicks "Confirmer" with no meaningful action.

This story merges E03S02 into E03S03, creating a 2-page funnel:
- **Page 1 (E03S01):** Freetext input + submit.
- **Page 2 (merged E03S02+E03S03):** Extraction summary (collapsible/editable) + confirmation questions (if needed) + match results + email/OTP gate + full profiles.

Post-MVP because the 3-page flow already works. This is a UX optimization.

---

## Acceptance Criteria

### Pipeline optimization

- [ ] AC1: When `POST /api/extract` returns `ready_to_match: true` (all fields confidence >= threshold):
  - Page 1 submits to BOTH extract AND match in sequence (or the backend pipelines them): extract → submit → match.
  - Client receives extraction + matches in one response (or follows a redirect that includes both).
  - Page 2 loads with extraction summary AND match results already displayed. Zero intermediate clicks.

- [ ] AC2: When `POST /api/extract` returns `needs_confirmation`:
  - Page 2 shows confirmation questions at top (same logic as current E03S02: max 3 targeted questions for low-confidence fields).
  - Match results section below shows placeholder or skeleton ("Repondez aux questions ci-dessus pour affiner vos resultats").
  - On question answers → `POST /api/prospects/submit` (with updated requirements) → matches computed → results section updates in-place (AJAX, no page reload).

### Merged Page 2 layout

- [ ] AC3: Page 2 section order (top to bottom):
  1. **Extraction summary** — collapsible section: "Vos besoins identifies [Modifier]". All extracted fields displayed as editable chips/tags. Clicking "Modifier" expands to an editable form (same as current E03S02 edit mode). Changes trigger re-match (AC5).
  2. **Confirmation questions** (if `needs_confirmation`) — 1-3 targeted questions, same format as E03S02 AC3-AC5.
  3. **Match results** — anonymized expert cards (same as current E03S03 AC1-AC4).
  4. **Email + OTP gate** — email input + OTP verification (using E06S39 endpoints). Same position as current E03S03 AC6-AC7.
  5. **Full profiles** — revealed after OTP verification (same as current E03S03 AC8-AC9).
  6. **Booking widget** — per expert (E03S06).

- [ ] AC4: **Collapsible extraction summary.** By default, collapsed (showing "Budget: {X} · Outils: {Y} · ..." as a single-line summary). Click "Modifier" → expands to editable fields. Click "Fermer" → collapses back. No page reload.

- [ ] AC5: **Edit-triggered re-match.** When prospect edits any extraction field (from the collapsible section) and saves:
  - Client sends `POST /api/prospects/{id}/requirements` with updated fields.
  - Server re-runs matching with updated requirements.
  - Match results section updates in-place (AJAX response replaces card list).
  - Loading state on results section during re-match ("Mise a jour de vos resultats...").

### Turnstile placement

- [ ] AC6: Turnstile invisible widget renders on Page 2 at the confirmation/submit action (not on Page 1 freetext). This aligns with DEC-120 (Turnstile at highest-intent moment). The Turnstile token is included in the `POST /api/prospects/submit` request.

### E03S02 deprecation

- [ ] AC7: The standalone E03S02 route (/confirm or equivalent) is removed or redirects to Page 2. The satellite frontend no longer navigates to a separate confirmation page. The E03S02 server-side logic (if any) is absorbed into Page 2's client-side handling.

### Backwards compatibility

- [ ] AC8: If a prospect has a direct link to the old E03S02 page (e.g., from browser history), redirect to Page 2 with their prospect ID. No 404.

### PostHog events

- [ ] AC9: Update event taxonomy:
  - `satellite.funnel_page2_loaded` (replaces separate `confirm_page_loaded` and `results_page_loaded`).
  - `satellite.extraction_edited` (field_name, old_value_hash, new_value_hash).
  - `satellite.rematch_triggered` (prospect_id, edit_source: "extraction_edit" | "confirmation_answer").

### Tests

- [ ] AC10: Integration tests for the merged flow: ready_to_match pipeline, needs_confirmation flow with AJAX re-match, extraction edit + re-match, OTP gate on merged page. All existing E03S02 + E03S03 tests updated or replaced. `tsc` clean.

## Notes

- **This is a refactor, not a rewrite.** All existing E03S02 and E03S03 logic is preserved — it's restructured onto a single page. The backend API endpoints don't change; only the frontend flow changes.
- **Mobile UX:** On mobile, the merged page may be long. Consider using anchor links or a sticky section header to help prospects navigate to the results section quickly.
- **Performance:** The ready_to_match pipeline (AC1) means the extraction + matching happen before Page 2 loads. Ensure the combined request doesn't cause a perceived slowness > 5s. If it does, show progressive results (extraction summary first, then matches loading).
