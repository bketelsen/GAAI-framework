# Story E02S11: Expert Availability — Weekly Recurring Rules

## User Story

**As an** expert on Callibrate,
**I want** to define my weekly availability for discovery calls using recurring time slots,
**so that** the matching engine knows when I'm available and prospects can only book slots that fit my schedule.

---

## Context

DEC-118 defines a hybrid availability model: Callibrate rules (Layer 1) + GCal conflict filter (Layer 2) + date overrides (Layer 3). This story implements Layer 1 — the foundational weekly recurring rules that experts define in their dashboard. Layer 2 (GCal integration) already exists (E06S10/E06S11) and reads FreeBusy data. This story creates the rules that the booking engine combines with GCal data.

Currently, the booking engine (E06S11) queries GCal directly for available slots. With this story, the formula becomes: `Available slots = (Callibrate rules ∩ GCal free time) − existing bookings`. If GCal is not connected, only Callibrate rules are used + expert must manually confirm each booking.

Defining availability rules is a prerequisite for Milestone 2 "Bookable" (DEC-117, E02S10).

## Acceptance Criteria

### Schema

- [ ] AC1: DB migration creating `expert_availability_rules` table:
  ```
  id: UUID PRIMARY KEY DEFAULT gen_random_uuid()
  expert_id: UUID NOT NULL REFERENCES experts(id) ON DELETE CASCADE
  day_of_week: SMALLINT NOT NULL CHECK (day_of_week BETWEEN 0 AND 6)  -- 0=Sunday, 6=Saturday
  start_time: TIME NOT NULL
  end_time: TIME NOT NULL CHECK (end_time > start_time)
  is_active: BOOLEAN NOT NULL DEFAULT true
  created_at: TIMESTAMPTZ NOT NULL DEFAULT NOW()
  updated_at: TIMESTAMPTZ NOT NULL DEFAULT NOW()
  ```
  - Index: `(expert_id, day_of_week, is_active)` for efficient availability queries.
  - RLS policy: experts can only read/write their own rules.
  - Constraint: no overlapping rules for the same expert + day_of_week (application-level check, see AC3).

### API Endpoints

- [ ] AC2: `GET /api/experts/:id/availability/rules` — Returns all active rules for the authenticated expert. Response: `{ rules: [{ id, day_of_week, start_time, end_time, is_active }], timezone: string }`. Requires expert JWT + ownership check.

- [ ] AC3: `POST /api/experts/:id/availability/rules` — Create a new rule. Accepts `{ day_of_week, start_time, end_time }`. Validates:
  - `day_of_week` 0-6.
  - `start_time` and `end_time` in `HH:MM` format (24h).
  - `end_time > start_time` (no midnight-crossing at MVP — simplification).
  - No overlap with existing active rules on the same day for the same expert. If overlap → 409 `overlapping_rule` with message "Vous avez deja un creneau qui chevauche cette plage horaire."
  - Returns created rule with `id`.

- [ ] AC4: `PUT /api/experts/:id/availability/rules/:rule_id` — Update an existing rule. Same validation as AC3. Ownership check (rule belongs to expert).

- [ ] AC5: `DELETE /api/experts/:id/availability/rules/:rule_id` — Soft delete: sets `is_active = false`. Does NOT hard-delete (preserves history for analytics). Ownership check. Returns 204.

### Booking engine integration

- [ ] AC6: Modify `GET /api/experts/:id/availability` (existing endpoint, E06S11):
  - Load expert's active availability rules from DB.
  - If rules exist: generate candidate slots for the requested period (next 14 days) from rules (expand recurring rules into concrete date+time slots).
  - If GCal is connected: intersect candidate slots with GCal free time (existing FreeBusy logic).
  - If GCal is NOT connected: use candidate slots as-is (no conflict filtering).
  - Subtract existing bookings (existing logic).
  - Return available slots in the same format as current endpoint (no breaking change for the booking widget E03S06).
  - **If no rules defined AND GCal not connected:** Return empty slots + `{ availability_status: "not_configured" }` (expert not bookable yet).

- [ ] AC7: **Timezone handling:** Expert's timezone stored on `experts` table (`timezone` column — e.g., "Europe/Brussels"). Rules are stored in expert's local timezone (TIME type). When generating concrete slots for the availability API, convert to UTC for the response (the booking widget handles prospect-side timezone conversion). If `timezone` is not set → default to "UTC" and prompt expert to set their timezone.

### Dashboard UI

- [ ] AC8: New availability management page at `/dashboard/availability` (or section within `/dashboard/settings`):
  - Weekly grid view showing Monday–Sunday (columns) × time blocks (rows, 30-min granularity from 06:00 to 22:00).
  - Existing rules displayed as colored blocks on the grid.
  - Add rule: click an empty block → opens a mini form (day pre-selected, start/end time pickers). Save → POST /api/... → block appears on grid.
  - Edit rule: click an existing block → edit form pre-filled. Save → PUT /api/...
  - Delete rule: click existing block → "Supprimer" button. Confirm → DELETE /api/... → block removed.
  - Timezone selector at top of page. Changing timezone updates the `experts.timezone` field via `PATCH /api/experts/:id/profile`.

- [ ] AC9: **Next 5 available slots preview.** Below the weekly grid, display "Vos 5 prochains creneaux disponibles" computed from rules + GCal (if connected) − bookings. This mirrors what prospects will see. If no slots available → "Aucun creneau disponible. Ajoutez des plages horaires ci-dessus."

- [ ] AC10: **Navigation.** Add "Disponibilites" link to dashboard sidebar navigation (between "Calendrier" and "Facturation" or after "Profil"). Icon: clock or calendar.

### Validation

- [ ] AC11: **Minimum availability gate.** After creating/updating/deleting rules, check if expert has at least 1 active rule. If going from 1+ rules to 0 rules (deleting the last one) → warning toast: "Sans disponibilites, vous ne recevrez plus de reservations." Allow the deletion but warn.

## Notes

- **Slot granularity:** The booking engine (E06S11) uses 20-minute slots. Rules are defined in broader blocks (e.g., 14:00–17:00). The availability endpoint generates 20-minute slots within each rule's time range. This means a rule "14:00–17:00" produces slots at 14:00, 14:20, 14:40, 15:00, ..., 16:40.
- **day_of_week convention:** 0=Sunday, 6=Saturday (JavaScript Date convention). The UI grid should display Monday first (European convention) by reordering visually while keeping the data model consistent.
- **Overlap detection:** Two rules overlap if their time ranges intersect on the same day. Check: `existing.start_time < new.end_time AND existing.end_time > new.start_time`. This is an application-level check (not a DB constraint) because it's only relevant for active rules.
- **Expert profile type update:** The `experts` table may need a `timezone` column if not already present. Check existing schema before migrating.
