# Story E06S10: Google Calendar OAuth Layer

## User Story

**As an** AI integration expert,
**I want** to connect my Google Calendar to Callibrate during onboarding,
**so that** my real availability is reflected and prospects can book calls directly in my agenda without any manual setup.

---

## Context

E06S04 (Cal.com Managed Users) is superseded — Cal.com Platform closed new signups on 15/12/2025 (DEC-41). Callibrate now manages Google Calendar OAuth directly. This story implements the OAuth2 connection layer: auth URL generation, callback handling, token storage (encrypted), refresh utility, status endpoint, and disconnect. It is the mandatory prerequisite for the booking engine (E06S11).

Token security: refresh tokens are stored encrypted at application level using `GCAL_TOKEN_ENCRYPTION_KEY` (AES-256-GCM via Workers Web Crypto). Tokens are never returned in API responses.

---

## Acceptance Criteria

- [ ] AC1: `GET /api/experts/:id/gcal/auth-url` returns a Google OAuth2 authorization URL with scopes `https://www.googleapis.com/auth/calendar.events` and `https://www.googleapis.com/auth/calendar.readonly`, `access_type=offline`, `prompt=consent` (forces refresh token issuance on every authorization), and a `state` parameter containing the `expert_id` — returns `401` if caller is not the authenticated expert
- [ ] AC2: `GET /api/gcal/callback` handles the OAuth2 callback: validates `state` parameter, exchanges `code` for `{ access_token, refresh_token, expiry_date }` via Google Token endpoint, stores both tokens encrypted in `experts` table — redirects to `/onboarding/gcal-connected` on success, `/onboarding/gcal-error` on failure
- [ ] AC3: Refresh token is stored only when present in the token exchange response — if absent (re-authorization without `prompt=consent`), the existing stored refresh token is preserved and not overwritten
- [ ] AC4: `GET /api/experts/:id/gcal/status` returns `{ connected: boolean, google_email: string | null, connected_at: string | null }` — `connected: true` only if a valid (non-null, non-revoked) refresh token exists in DB
- [ ] AC5: `refreshGcalToken(expertId, env)` — exported async utility function: retrieves stored refresh token, calls Google Token endpoint to obtain a fresh access token, updates `experts.gcal_access_token` and `experts.gcal_token_expiry_at` in DB, returns the fresh access token — callable by any Worker handler before Google Calendar API calls
- [ ] AC6: If a Google Calendar API call returns `401` or `403` (token revoked or expired), the caller invokes `refreshGcalToken` once and retries — if still `401`/`403` after refresh, the system sets `experts.gcal_connected = false` and returns `{ error: "gcal_disconnected" }` to the caller — no silent failure
- [ ] AC7: `DELETE /api/experts/:id/gcal/disconnect` calls Google OAuth revocation endpoint (`https://oauth2.googleapis.com/revoke`), clears `gcal_access_token`, `gcal_refresh_token`, `gcal_token_expiry_at` from DB, sets `gcal_connected = false` — returns `200` on success, `404` if expert not found
- [ ] AC8: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GCAL_TOKEN_ENCRYPTION_KEY` are bound as Worker secrets — never hardcoded, never logged, never returned in responses
- [ ] AC9: Schema migration adds to `experts` table: `gcal_connected boolean DEFAULT false`, `gcal_email text`, `gcal_access_token text` (encrypted), `gcal_refresh_token text` (encrypted), `gcal_token_expiry_at timestamptz`, `gcal_connected_at timestamptz`
- [ ] AC10: At least 3 vitest unit tests: (1) auth URL generation — correct scopes, `access_type=offline`, `prompt=consent`, `state` contains `expert_id`; (2) callback token storage — mock Google token exchange, verify encrypted storage; (3) status endpoint — `connected: true` when token present, `connected: false` when null

## Out of Scope

- Microsoft Outlook / Teams OAuth (post-MVP)
- Apple Calendar / iCloud (post-MVP)
- Google Workspace domain-wide delegation (not applicable — experts use personal Google accounts)
- Google app verification process (handled separately by founder — must be initiated before go-live)

## Dependencies

- E06S03 (expert must exist in DB — `experts` table)
- Google Cloud project with Calendar API enabled + OAuth 2.0 credentials configured (Client ID + Secret)
- Authorized redirect URI registered in Google Cloud Console: `https://api.callibrate.io/api/gcal/callback` (staging: `https://callibrate-core.{account}.workers.dev/api/gcal/callback`)

## Notes

- `prompt=consent` is mandatory to force refresh token issuance even when the user has previously authorized the app. Without it, re-authorizations do not return a refresh token, leaving the system unable to act on behalf of the expert after the access token expires.
- Google's sensitive scope review: `calendar.events` is a sensitive scope requiring formal app verification for production use with external (non-Workspace) users. The process involves domain verification, privacy policy review, and potentially a security assessment. **Initiate immediately — process can take several weeks.**
- The callback endpoint is unauthenticated (Google redirects to it) but protected by the `state` parameter validation.
- Encryption: use `crypto.subtle.encrypt` (AES-256-GCM) with `GCAL_TOKEN_ENCRYPTION_KEY` as the encryption key. IV is stored alongside the ciphertext (concatenated, base64-encoded).
- The `cal_*` columns from E06S04 remain in the DB schema — do not drop them in this migration (cleanup deferred to post-MVP).
