# Execution Plan — E08S01: OAuth Security — CSRF State Token + PKCE (SEC-01, SEC-03)

## Approach Rationale

The current `handleGcalAuthUrl` puts the raw `expertId` in the OAuth `state` parameter. This is a textbook CSRF vulnerability: any actor who knows a valid `expert_id` can forge a callback that associates an adversarial Google account with a victim expert. PKCE is absent, leaving the authorization code exposed to interception.

The fix is a two-part change confined entirely to one handler file (`gcal.ts`) and its test file (`gcal.test.ts`):

1. **CSRF fix (SEC-01):** Replace `state: expertId` with a random state token stored in `env.SESSIONS` KV. The callback validates the token and retrieves the expert ID from KV rather than trusting the state parameter directly.
2. **PKCE fix (SEC-03):** Generate a `code_verifier`/`code_challenge` pair per RFC 7636 in `handleGcalAuthUrl`, store `code_verifier` alongside `expert_id` in the same KV entry, and pass it to the token exchange in `handleGcalCallback`.

No new dependencies, no schema changes, no new wrangler bindings (`env.SESSIONS` KV is already bound in all environments). The Web Crypto API (`crypto.subtle`, `crypto.getRandomValues`) is globally available in the Cloudflare Workers runtime.

The `handleGcalStatus` and `handleGcalDisconnect` handlers are not modified.

## Implementation Sequence

### Step 1 — Add crypto helper functions at the top of `gcal.ts`

Insert two pure async helper functions immediately after the existing `redirect()` helper (before the `handleGcalAuthUrl` export). They have no side effects and can be unit-tested in isolation.

**`generateStateToken(): Promise<string>`**

```typescript
async function generateStateToken(): Promise<string> {
  const bytes = new Uint8Array(32);
  crypto.getRandomValues(bytes);
  return Array.from(bytes).map(b => b.toString(16).padStart(2, '0')).join('');
}
```

- Uses `crypto.getRandomValues` (synchronous, available globally in Workers runtime).
- 32 bytes = 256 bits of entropy, hex-encoded to 64 characters.
- No external dependency.

**`generatePkce(): Promise<{ code_verifier: string; code_challenge: string }>`**

```typescript
async function generatePkce(): Promise<{ code_verifier: string; code_challenge: string }> {
  // code_verifier: 43–128 URL-safe chars (RFC 7636 §4.1)
  // Use 32 random bytes → 43 base64url chars (32 * 4/3 rounded up, no padding)
  const verifierBytes = new Uint8Array(32);
  crypto.getRandomValues(verifierBytes);
  const code_verifier = btoa(String.fromCharCode(...verifierBytes))
    .replace(/\+/g, '-')
    .replace(/\//g, '_')
    .replace(/=/g, '');

  // code_challenge = BASE64URL(SHA-256(ASCII(code_verifier))) — RFC 7636 §4.2
  const encoder = new TextEncoder();
  const data = encoder.encode(code_verifier);
  const digest = await crypto.subtle.digest('SHA-256', data);
  const code_challenge = btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-')
    .replace(/\//g, '_')
    .replace(/=/g, '');

  return { code_verifier, code_challenge };
}
```

- `btoa` + character spread: safe for byte arrays up to 65535 items; 32/43 bytes is well within that limit.
- No padding (`=` stripped) per RFC 7636 §4.2.
- `crypto.subtle.digest` is async; helper must be awaited.

**Checkpoint:** Both helpers are TypeScript-clean (no `any`, return types explicit). `tsc --noEmit` passes.

### Step 2 — Rewrite `handleGcalAuthUrl`

Replace the current body with the following logic. The function signature is unchanged.

**Logic:**

```typescript
export async function handleGcalAuthUrl(
  _request: Request,
  env: Env,
  user: AuthUser,
  expertId: string
): Promise<Response> {
  if (user.id !== expertId) return forbidden();

  const sql = createSql(env);
  const [data] = await sql<Pick<ExpertRow, 'id'>[]>`SELECT id FROM experts WHERE id = ${expertId}`;
  if (!data) return notFound();

  // AC1: Generate cryptographically random state token
  const stateToken = await generateStateToken();

  // AC2 + AC3: Generate PKCE pair
  const { code_verifier, code_challenge } = await generatePkce();

  // AC3: Store {expert_id, code_verifier} in SESSIONS KV under oauth-state:{token}, TTL 600s
  await env.SESSIONS.put(
    `oauth-state:${stateToken}`,
    JSON.stringify({ expert_id: expertId, code_verifier }),
    { expirationTtl: 600 }
  );

  // AC2 + AC6: Build auth URL with state = random token (not expertId)
  const params = new URLSearchParams({
    client_id: env.GOOGLE_CLIENT_ID,
    redirect_uri: `${env.WORKER_BASE_URL}/api/gcal/callback`,
    response_type: 'code',
    scope: 'openid email https://www.googleapis.com/auth/calendar.events https://www.googleapis.com/auth/calendar.readonly',
    access_type: 'offline',
    prompt: 'consent',
    state: stateToken,
    code_challenge,
    code_challenge_method: 'S256',
  });

  const authUrl = `https://accounts.google.com/o/oauth2/v2/auth?${params.toString()}`;
  return json({ auth_url: authUrl });
}
```

**Key changes vs. current:**
- `state: expertId` → `state: stateToken` (AC6)
- KV write of `oauth-state:{stateToken}` with TTL 600s (AC1, AC3)
- `code_challenge` + `code_challenge_method=S256` added to URL params (AC2)

**Checkpoint:** Function compiles cleanly. `state` in URL is a 64-char hex token, not the expert UUID.

### Step 3 — Rewrite `handleGcalCallback`

Replace the state-handling logic at the top of `handleGcalCallback`. The token exchange, DB update, and PostHog event sections are unchanged except for adding `code_verifier` to the exchange body.

**Logic changes:**

```typescript
export async function handleGcalCallback(
  request: Request,
  env: Env,
  ctx: ExecutionContext,
): Promise<Response> {
  const url = new URL(request.url);
  const code = url.searchParams.get('code');
  const state = url.searchParams.get('state');

  // AC4: Validate state token presence
  if (!state) return redirect('/onboarding/gcal-error?reason=invalid_state');
  if (!code) return redirect('/onboarding/gcal-error?reason=no_code');

  // AC4: KV lookup — one-time use
  const kvKey = `oauth-state:${state}`;
  const kvRaw = await env.SESSIONS.get(kvKey);
  if (!kvRaw) return redirect('/onboarding/gcal-error?reason=invalid_state');

  // AC4: DELETE immediately (one-time use, replay prevention)
  await env.SESSIONS.delete(kvKey);

  // AC4: Extract expert_id and code_verifier from KV value
  const { expert_id: expertId, code_verifier } = JSON.parse(kvRaw) as {
    expert_id: string;
    code_verifier: string;
  };

  // ... existing DB lookup, token exchange, etc. unchanged below ...
```

**Token exchange change (AC5):** Add `code_verifier` to the POST body:

```typescript
  body: new URLSearchParams({
    code,
    client_id: env.GOOGLE_CLIENT_ID,
    client_secret: env.GOOGLE_CLIENT_SECRET,
    redirect_uri: `${env.WORKER_BASE_URL}/api/gcal/callback`,
    grant_type: 'authorization_code',
    code_verifier,   // <-- ADD THIS
  }).toString(),
```

**Removed from current code:**
- `const expertId = state;` — expert ID now comes from KV
- `SELECT id FROM experts WHERE id = ${expertId}` — this lookup is removed; the expert_id in KV was validated at auth URL generation time. The existing flow then proceeds with that `expertId`.

**Note on expert existence check:** The current callback does a `SELECT id FROM experts WHERE id = ${expertId}` before proceeding. After the change, the `expertId` comes from KV (which was validated at auth URL time). The redundant existence check can be retained for defence-in-depth, or removed since the expert was already verified in `handleGcalAuthUrl`. **Decision: retain it** — the expert could have been deleted between auth URL generation and callback, and the check is cheap.

**Checkpoint:** `handleGcalCallback` no longer trusts `state` as an expert ID. All error paths redirect to `gcal-error?reason=invalid_state`.

### Step 5 — TypeScript clean build

Run `tsc --noEmit` from `workers/backend/api/`.

Areas to verify:
- `JSON.parse(kvRaw) as { expert_id: string; code_verifier: string }` — safe cast since the shape is controlled by `handleGcalAuthUrl`. If `kvRaw` contains unexpected shape (e.g. legacy entry without `code_verifier`), the destructuring will yield `undefined`. The code_verifier passed to the token exchange would then be the string `"undefined"` — Google will reject it with a 400. This is acceptable: the KV entry lifetime is 600s, there is no legacy data to worry about.
- `crypto.getRandomValues` — available globally in Workers runtime; no import needed. TypeScript sees it via `@cloudflare/workers-types`.
- `crypto.subtle.digest` — async, returns `ArrayBuffer`. Cast with `new Uint8Array(digest)` before `btoa`.

**Checkpoint:** `tsc --noEmit` exits 0. No type errors introduced.

## Edge Cases per AC

| AC | Edge case | Handling |
|----|-----------|---------|
| AC1 | `crypto.getRandomValues` not available | Cannot happen in CF Workers runtime — it is part of the spec. No fallback needed. |
| AC1 | KV `put` fails (network/quota error) | Unhandled — will throw, surfaced as 500 to the caller. Acceptable: auth URL generation failing with 500 is correct behaviour if KV is unavailable. |
| AC2 | `crypto.subtle.digest` returns wrong length | Cannot happen for SHA-256 — always 32 bytes. |
| AC2 | `code_challenge` URL encoding | `URLSearchParams` will percent-encode the base64url string correctly. Base64url uses `-`, `_`, and alphanumerics — all URL-safe characters that `URLSearchParams` will NOT percent-encode. Verified by RFC 3986 unreserved character set. |
| AC3 | KV value corrupted or partially written | `JSON.parse` will throw → unhandled exception → 500 response. Expert can retry (new auth URL generates a new KV entry). |
| AC4 | `state` present in URL but not in KV (expired, never existed, already used) | KV `get` returns `null` → redirect `?reason=invalid_state`. KV `delete` is NOT called (nothing to delete). |
| AC4 | `state` present, KV found, but `delete` fails | `await env.SESSIONS.delete(kvKey)` throws → unhandled → 500. The KV entry remains. Expert can retry after TTL expiry (10 min). This is safer than swallowing the delete error and proceeding (which would allow replay within the TTL window). |
| AC5 | `code_verifier` passed to token exchange is correct | Google validates it against the stored `code_challenge`. If they don't match (should be impossible in normal flow), Google returns 400 → callback redirects `?reason=token_exchange_failed`. |
| AC6 | State token contains URL-unsafe characters | Impossible — hex encoding uses `[0-9a-f]` only. |
| AC6 | State token collision | Probability: 1/2^256. Negligible. No collision detection needed. |
| AC7 | Test mock for `crypto` | `crypto.getRandomValues` and `crypto.subtle` are available in the vitest/jsdom environment (or vitest's Workers-compatible runtime). No stubbing needed. If unavailable, stub via `vi.stubGlobal('crypto', { getRandomValues: ..., subtle: { digest: ... } })`. |
| AC8 | `tsc` type error on `env.SESSIONS.put` options | `KVNamespace.put` accepts `{ expirationTtl: number }` — correct per CF types. |

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `btoa(String.fromCharCode(...bytes))` fails for 32-byte arrays due to call stack limit | Very Low | High | `String.fromCharCode.apply(null, bytes)` has a limit (~65535 args). 32 bytes is far below that. Alternatively use a loop-based approach if Workers runtime restricts spread. Plan: use spread; fall back to loop if runtime rejects. |
| `crypto.getRandomValues` or `crypto.subtle` not available in vitest test environment | Low | Medium | Vitest runs in Node.js. Node 18+ exposes `globalThis.crypto` with a Web Crypto-compatible API. If missing, stub with `vi.stubGlobal`. |
| Google rejects PKCE challenge if the algorithm is not exactly RFC 7636 compliant | Low | High | Implementation follows RFC 7636 §4.2 exactly: SHA-256, BASE64URL encoding, no padding. Google explicitly supports `code_challenge_method=S256`. |
| KV `put` with `expirationTtl: 600` is not enforced in local dev (`wrangler dev`) | Low | Low | TTL is enforced in deployed environments. Local dev may keep entries indefinitely — acceptable for development, no security impact in production. |
| State token in the KV key is guessable | None | High | 32 bytes of `getRandomValues` entropy = 2^256 space. Not guessable. |
| `JSON.parse` of KV value causes unhandled exception | Low | Medium | KV value is always written by `handleGcalAuthUrl` in controlled JSON format. Corruption would require a KV infrastructure fault. Accept the risk; 500 is correct behaviour. |
| Existing test `state=${expertId}` assertion not updated | High (if forgotten) | Medium | Step 4a explicitly targets this assertion. Delivery agent must run `vitest run` before considering AC7 met. |
| Token exchange rejection by Google (code_verifier mismatch) | None in normal flow | High | Mismatch is impossible if both functions execute correctly in the same request lifecycle. Only possible if KV is corrupted, which is handled by the 500 path. |

### Boundary 1 — Before Step 2 (pure addition)
After Step 1 (helper functions added), the production behaviour is unchanged — helpers are not yet called. Safe rollback: delete the two helper functions.

### Boundary 2 — After Step 2, before Step 3 (partial implementation — DO NOT DEPLOY)
`handleGcalAuthUrl` writes to KV and generates a state token, but `handleGcalCallback` still reads `state` as `expertId`. This is a broken intermediate state. **Do not deploy to staging at this boundary.** Steps 2 and 3 must be committed together.

### Boundary 3 — After Step 3 (full implementation — safe to deploy)
Both handlers are consistent. The full CSRF + PKCE fix is live. KV TTL means no stale entries accumulate. Safe to deploy to staging for verification.

### Boundary 4 — After Step 4 + Step 5 (tests + TSC passing — PR-ready)
All ACs verified. PR can be opened to staging.

### Git rollback procedure
If a regression is discovered post-deploy:
1. Revert the commit that contains Steps 2+3 changes in `gcal.ts`.
2. The KV namespace will contain `oauth-state:*` entries from the deployed window. These will expire naturally within 600 seconds. No KV cleanup required.
3. Any in-flight OAuth flows at the moment of rollback will land on `gcal-error?reason=invalid_state` (KV entry won't be found by the old code path, which doesn't look up KV). This is a safe failure — expert retries the connection flow.

## Implementation Notes

- **No import changes required.** `crypto` is a global in the CF Workers runtime. `btoa` is also a global. `TextEncoder` is also a global. No new imports needed in `gcal.ts`.
- **`env.SESSIONS.put` third argument:** The `KVNamespace.put` signature accepts an optional options object `{ expirationTtl?: number; expiration?: number; ... }`. Use `expirationTtl: 600` (seconds), not `expiration` (Unix timestamp).
- **KV key prefix convention:** `oauth-state:` follows the existing pattern in the codebase (pattern consistent with any `session:`, `rate:` keys). No namespace conflict expected.
- **Existing DB lookup in callback:** Retain `SELECT id FROM experts WHERE id = ${expertId}` in `handleGcalCallback` after extracting `expertId` from KV. This provides defence-in-depth against deleted experts and maintains existing behaviour for the `expert_not_found` redirect path.
- **Test file mock pattern:** The existing `mockEnv.SESSIONS = {} as KVNamespace` pattern must be replaced with a proper mock object that has callable `get`, `put`, `delete` methods. Define `mockSessions` in the outer `describe` scope and reassign in `beforeEach` to ensure isolation between tests.
- **`vitest` crypto availability:** Vitest v1+ (Node 18+) exposes `globalThis.crypto` natively. `crypto.subtle.digest` and `crypto.getRandomValues` work without stubbing. Confirm with a smoke test if the current vitest version is below 1.0.
