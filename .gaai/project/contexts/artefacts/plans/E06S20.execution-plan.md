## 1. Approach Summary

### What is changing and why

**Rate Limiting (AC1, AC2, AC3):**
The current `checkRateLimit()` in `src/lib/rateLimit.ts` simulates rate limiting via KV reads and writes — two round-trips per request, with no atomicity guarantee. Cloudflare's native Rate Limiting binding provides atomic, hardware-level counters at the edge with zero KV cost. This story migrates to that binding, raising the limit from 10 to 30 req/min/IP for public endpoints.

The `RATE_LIMITING` KV namespace binding is removed from `wrangler.toml` and `src/types/env.ts`. A new `RATE_LIMITER` binding of type `rate_limiting` is introduced for all three environments (dev/staging/production). The `checkRateLimit()` function is rewritten to call `env.RATE_LIMITER.limit({ key: ip })` — one call, no writes.

Rate limiting is also applied at the top of `handleProspectSubmit`, `handleProspectMatches`, and `handleProspectIdentify` in `src/routes/prospects.ts`, because these are public endpoints (no auth middleware). The call site in `src/handlers/experts/register.ts` is unchanged — it already calls `checkRateLimit(request, env)`.

**Turnstile (AC4, AC5, AC6, AC7):**
`POST /api/prospects/submit` is a public, unauthenticated endpoint that is a high-value bot target. Turnstile provides invisible CAPTCHA verification via Cloudflare's siteverify API. The frontend sends a `cf-turnstile-response` token in the request body; the Worker verifies it server-side before any database write.

A module-level (non-exported) `verifyTurnstile()` helper is added to `prospects.ts`. It posts to `https://challenges.cloudflare.com/turnstile/v0/siteverify` and returns a boolean. For staging/dev, the environment will have a test secret key (`1x0000000000000000000000000000000AA`) which always passes, so no production traffic is affected during development.

**Test files (AC8):**
Two existing test files reference `RATE_LIMITING: {} as KVNamespace` in their mock env objects. These must be updated to remove that key and add `RATE_LIMITER` with a mocked `limit` function. A new test file `src/routes/prospects.test.ts` is created with tests for both Turnstile validation and rate limiting.

export async function checkRateLimit(
  request: Request,
  env: Env
): Promise<{ allowed: boolean }> {
  const ip = request.headers.get('CF-Connecting-IP') ?? 'unknown';
  const { success } = await env.RATE_LIMITER.limit({ key: ip });

}

```typescript
// Verify Cloudflare Turnstile token via siteverify API.
// Returns true if the token is valid (or if the secret is a test key).
async function verifyTurnstile(token: string, ip: string, secretKey: string): Promise<boolean> {
  const body = new URLSearchParams({
    secret: secretKey,
    response: token,
    remoteip: ip,
  });
  try {
    const res = await fetch('https://challenges.cloudflare.com/turnstile/v0/siteverify', {
      method: 'POST',
      body,
    });
    if (!res.ok) return false;
    const data = await res.json() as { success: boolean };
    return data.success === true;
  } catch {
    return false;
  }
}
```

## 5. Test Strategy

### What `src/routes/prospects.test.ts` must cover

| Test case | Assertion | AC |
|---|---|---|
| Missing `cf-turnstile-response` in body | 422 + `details['cf-turnstile-response'] === 'required'` | AC4 |
| Empty string `cf-turnstile-response` | 422 + `error === 'Validation failed'` | AC4 |
| `siteverify` returns `{ success: false }` | 422 + `error === 'Bot verification failed'` | AC7 |
| `siteverify` fetch throws | 422 + `error === 'Bot verification failed'` | AC7 |
| `siteverify` returns `{ success: true }` | Status is NOT 422 with Turnstile error | AC5, AC6 |
| `RATE_LIMITER.limit()` returns `{ success: false }` | 429 + `error === 'Too Many Requests'` | AC3 |
| `RATE_LIMITER.limit()` returns `{ success: true }` | Status is not 429 | AC3 |
| `handleProspectMatches` rate limited | 429 | AC3 |
| `handleProspectIdentify` rate limited | 429 | AC3 |

### Mocking strategy

**`fetch` mocking:** Use `vi.spyOn(globalThis, 'fetch')` in `beforeEach` or per-test. Restore with `vi.restoreAllMocks()` in `afterEach`. The spy must discriminate by URL: calls containing `siteverify` get the Turnstile mock response; all others get a generic Supabase response.

**`RATE_LIMITER` mocking:** Inline in `baseMockEnv` as `{ limit: vi.fn().mockResolvedValue({ success: true }) }`. For rate-limit-denied tests, override via `makeMockEnv({ RATE_LIMITER: { limit: vi.fn().mockResolvedValue({ success: false }) } })`. Cast as `unknown as RateLimit` to satisfy TypeScript.

**`TURNSTILE_SECRET_KEY`:** Set to `'1x0000000000000000000000000000000AA'` in `baseMockEnv`. This is the CF always-passes test key. The `verifyTurnstile` function will POST to `siteverify` even in tests, which is why `fetch` must be mocked.

## 6. Risks and Mitigations

### Risk 1 — `namespace_id` for `rate_limiting` binding

**Risk:** The `rate_limiting` binding in `wrangler.toml` requires a real numeric `namespace_id` from Cloudflare. Placeholder IDs (1001, 1002, 1003) will cause `wrangler deploy` to fail.

**Mitigation:** Before deploying, the Implementation Agent must run `npx wrangler rate-limit list` or create namespaces via `npx wrangler rate-limit create`. The test suite (`vitest run`) does not require real CF resources — only the deploy step does. Tests pass with the mocked env. The implementation can be merged and tested locally before CF resources are provisioned.

### Risk 2 — Queue test files may reference `RATE_LIMITING`

**Risk:** `src/queues/lead-billing.test.ts`, `src/queues/score-computation.test.ts`, and `src/queues/email-notifications.test.ts` were not listed in scope but may have mockEnv objects with `RATE_LIMITING`.

**Mitigation:** The Implementation Agent must search for `RATE_LIMITING` across all `*.test.ts` files before Step 7 and apply the same replacement wherever found.

Command: `grep -r "RATE_LIMITING" src/ --include="*.ts" -l`

### Risk 3 — `verifyTurnstile` fetch in test environment

**Risk:** If `vi.spyOn(globalThis, 'fetch')` does not intercept the `verifyTurnstile` internal fetch (e.g., if the module caches a reference to `fetch` at import time), siteverify calls will fail or error in tests.

**Mitigation:** The implementation must call `fetch(...)` directly — not a stored reference. The `verifyTurnstile` function as specified calls `fetch(...)` inline, which is intercepted by `vi.spyOn(globalThis, 'fetch')` in the Vitest environment.

### Risk 4 — `handleProspectSubmit` body type change breaks existing logic

**Risk:** Expanding the body type annotation to include `'cf-turnstile-response'` could cause issues if the TypeScript compiler is strict about hyphenated property names.

**Mitigation:** Using bracket notation `body['cf-turnstile-response']` for extraction is valid TypeScript. The type annotation `'cf-turnstile-response'?: unknown` is a legal computed property name in interface/type literals. The existing destructuring of `{ satellite_id, quiz_answers, utm_source, utm_campaign }` is done after Turnstile extraction, so it remains unaffected — just move it below the Turnstile check as shown in the implementation spec.

### Risk 5 — Rate limiting on `handleProspectMatches` breaks the token-protected flow

**Risk:** `handleProspectMatches` is technically a GET with a JWT token parameter — it is not a purely anonymous endpoint. Rate limiting it at the IP level could affect legitimate users on shared IPs (e.g., behind NAT).

**Mitigation:** The acceptance criteria explicitly require rate limiting for all public (non-auth-middleware) endpoints at 30 req/min. 30 req/min is generous for a single prospect accessing their matches. The limit is per IP with 60-second windows; this is acceptable for MVP. Noted for future review.

### Risk 6 — Missing `TURNSTILE_SECRET_KEY` in `.dev.vars`

**Risk:** The `TURNSTILE_SECRET_KEY` in `[env.staging.vars]` is only used when deploying to staging. Local `wrangler dev` reads from `.dev.vars` — which is gitignored. If not present, the binding will be undefined and cause a runtime error.

**Mitigation:** The Implementation Agent must add `TURNSTILE_SECRET_KEY=1x0000000000000000000000000000000AA` to `.dev.vars` as part of this story. This does not require a commit (`.dev.vars` is gitignored). Document this in the story's implementation report.
