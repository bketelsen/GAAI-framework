# Execution Plan — E06S33: LS Usage-Based Subscription + Welcome Credit + Usage Reporting Cron

## Approach Rationale

Two independent capabilities are delivered together: (1) a LemonSqueezy webhook receiver that handles
subscription lifecycle events and issues a 10,000-credit welcome grant on first subscription, and (2) an
hourly cron that auto-confirms stale pending leads and reports their usage to the LS usage-based billing API.

### Key architectural decisions

**ls_subscription_item_id storage:** LS usage-based billing requires a `subscription_item_id`, not a
`subscription_id`. The only clean path is to fetch the subscription items via
`GET /v1/subscription-items?filter[subscription_id]={sub_id}` immediately during the `subscription_created`
webhook handler, then persist `ls_subscription_item_id` on the `experts` row. Fetching it at cron time
would create N+1 API calls per batch. This requires a small DB migration (Step 1).

**Idempotency key for webhook:** The existing `isAlreadyProcessed` / `markProcessed` utilities use
`idem:{queue}:{messageId}`. For the webhook we call them with `queue='ls-webhook'` and
`messageId=event.meta.event_id` (the LS event UUID). TTL is 86400s, matching the queue consumer pattern (AC5).

**Webhook route placement:** Must be registered at line 87 in `src/index.ts`, immediately after the GCal
callback block (line 87–89) and before the `/api/satellites/` block (line 91). Both are unauthenticated
external-inbound routes registered before any authenticated block.

**FLAG_WINDOW_MS env var:** Added as optional string to `Env` interface. Parsed via
`parseInt(env.FLAG_WINDOW_MS ?? String(7 * 24 * 60 * 60 * 1000))`. This matches the `SURVEY_DELAY_*_MS`
pattern already present in the codebase.

**Cron handler pattern:** New file `src/cron/lsBillingCron.ts` exports `handleLsBillingCron(env)`. The
existing dispatcher in `src/handlers/bookings/cron.ts` gains a new `else if (cron === '0 * * * *')` branch
that delegates to this new function.

**Email notification for payment_failed (AC4):** Adds a new union member
`{ type: 'expert.billing.payment_failed'; expert_id: string }` to `EmailNotificationMessage` in
`src/types/queues.ts`. A stub case is added to the email-notifications consumer switch (same pattern as
`expert.billing.insufficient_balance` stub in E06S31).

**db.ts vs supabase client:** All webhook and cron DB writes use `createSql(env)` (postgres.js via
Hyperdrive) for direct SQL control. The idempotency KV operations use `env.SESSIONS` (same KV namespace
used by all queue consumers).

### Step 5 — `src/handlers/webhooks/lemonsqueezy.ts` (NEW)

Create directory `src/handlers/webhooks/` and file `lemonsqueezy.ts`.

```
POST /api/webhooks/lemonsqueezy

Exports: handleLsWebhook(request, env)

Flow:
  1. verifyLsSignature(request, env) → reads raw body bytes, checks X-Signature header
     → Returns { valid: boolean, bodyText: string }
     → If header missing or signature invalid → return 401
  2. Parse JSON from bodyText
  3. Idempotency check:
     key = `idem:ls-webhook:{event.meta.event_id}` via isAlreadyProcessed(env.SESSIONS, 'ls-webhook', event.meta.event_id)
     If already processed → return 200 { ok: true, duplicate: true }
  4. Dispatch on event.meta.event_name:
     'subscription_created' → handleSubscriptionCreated(event, env)
     'subscription_updated' → handleSubscriptionUpdated(event, env)
     'subscription_payment_failed' → handleSubscriptionPaymentFailed(event, env)
     default → log warn, return 200 (unknown events are acked to avoid LS retries)
  5. markProcessed(env.SESSIONS, 'ls-webhook', event.meta.event_id)
  6. return 200 { ok: true }
```

**`verifyLsSignature(request, env): Promise<{ valid: boolean, bodyText: string }>`**
```
- Clone request to read body twice (or read body once and pass bytes)
- rawBody = await request.arrayBuffer()
- bodyText = new TextDecoder().decode(rawBody)
- signature header = request.headers.get('X-Signature')
- If no signature header → return { valid: false, bodyText: '' }
- Parse hex signature → Uint8Array
- Import HMAC key: crypto.subtle.importKey('raw', encoder.encode(env.LEMON_SQUEEZY_WEBHOOK_SECRET),
    { name: 'HMAC', hash: 'SHA-256' }, false, ['verify'])
- result = await crypto.subtle.verify('HMAC', key, signatureBytes, rawBody)
- return { valid: result, bodyText }
```

Hex decode: `Uint8Array.from(sig.match(/../g)!.map(h => parseInt(h, 16)))`.

**`handleSubscriptionCreated(event, env)`**
```
sql = createSql(env)

1. Identify expert:
   - If event.meta.custom_data?.expert_id → SELECT id FROM experts WHERE id = custom_data.expert_id
   - Else → SELECT id, credit_balance FROM experts WHERE gcal_email = event.data.attributes.customer_email
   - If not found → log warn + return (no-op; LS will not retry if we return 200 in outer handler)

2. Fetch subscription item id from LS API:
   GET https://api.lemonsqueezy.com/v1/subscription-items?filter[subscription_id]={event.data.id}
   Authorization: Bearer {env.LEMON_SQUEEZY_API_KEY}
   Accept: application/vnd.api+json
   → subscriptionItemId = response.data[0]?.id ?? null
   Log warning if null (usage reporting will skip this expert until corrected)

3. Atomic DB operations (in sequence, not a transaction — Hyperdrive does not expose explicit transactions
   via postgres.js template literals at this level; use sequential statements with FOR UPDATE lock):

   a. UPDATE experts SET
        ls_subscription_id = ${event.data.id},
        ls_subscription_status = 'active',
        ls_subscription_item_id = ${subscriptionItemId},
        credit_balance = credit_balance + 10000
      WHERE id = ${expertId}
      RETURNING credit_balance

   b. INSERT INTO credit_transactions
        (expert_id, type, amount, balance_after)
      VALUES
        (${expertId}, 'welcome_credit', 10000, ${balanceAfter})
      ON CONFLICT DO NOTHING  -- idempotency guard (event re-delivery)
```

Note: `ON CONFLICT DO NOTHING` on `credit_transactions` requires a unique constraint on
`(expert_id, type)` WHERE type='welcome_credit', OR we use a separate check. Safer approach:
check idempotency via KV (already handled at outer handler level — the markProcessed call at step 5
ensures this handler only fires once per event_id). The `ON CONFLICT` clause is a secondary guard.
Since `credit_transactions` may not have a unique constraint for this, use a conditional insert:
```sql
INSERT INTO credit_transactions (expert_id, type, amount, balance_after)
SELECT ${expertId}, 'welcome_credit', 10000, ${balanceAfter}
WHERE NOT EXISTS (
  SELECT 1 FROM credit_transactions
  WHERE expert_id = ${expertId} AND type = 'welcome_credit'
)
```

### Step 6 — `src/cron/lsBillingCron.ts` (NEW)

```
Exports: handleLsBillingCron(env)

Flow:
  sql = createSql(env)

  // AC6: compute flag window
  const windowMs = parseInt(env.FLAG_WINDOW_MS ?? String(7 * 24 * 60 * 60 * 1000))
  const cutoff = new Date(Date.now() - windowMs).toISOString()

  // AC6: batch auto-confirm leads past the flag window
  const confirmedLeads = await sql<{ id: string; expert_id: string; amount: number }[]>`
    UPDATE leads
    SET status = 'confirmed', confirmed_at = NOW()
    WHERE status = 'pending'
      AND created_at < ${cutoff}
    RETURNING id, expert_id, amount`

  console.log(`lsBillingCron: auto-confirmed ${confirmedLeads.length} leads`)

  // AC7+AC8: also pick up previously confirmed leads where usage_reported_at IS NULL
  const leadsToReport = await sql<{ id: string; expert_id: string; amount: number }[]>`
    SELECT l.id, l.expert_id, l.amount
    FROM leads l
    WHERE l.status = 'confirmed'
      AND l.usage_reported_at IS NULL`

  console.log(`lsBillingCron: ${leadsToReport.length} leads pending usage reporting`)

  // AC7: for each lead, report usage to LS API
  for (const lead of leadsToReport) {
    await reportLeadUsage(lead, sql, env)
  }
```

**`reportLeadUsage(lead, sql, env)`**
```
1. Fetch expert's ls_subscription_item_id:
   const [expert] = await sql<{ ls_subscription_item_id: string | null }[]>`
     SELECT ls_subscription_item_id FROM experts WHERE id = ${lead.expert_id}`

   If !expert?.ls_subscription_item_id:
     console.warn(`lsBillingCron: expert ${lead.expert_id} has no ls_subscription_item_id — skipping lead ${lead.id}`)
     return  // lead remains status='confirmed', usage_reported_at IS NULL → will retry next run

2. Call LS usage records API:
   POST https://api.lemonsqueezy.com/v1/usage-records
   Headers:
     Authorization: Bearer {env.LEMON_SQUEEZY_API_KEY}
     Content-Type: application/vnd.api+json
     Accept: application/vnd.api+json
   Body:
     {
       "data": {
         "type": "usage-records",
         "attributes": { "quantity": lead.amount, "action": "increment" },
         "relationships": {
           "subscription-item": {
             "data": { "type": "subscription-items", "id": expert.ls_subscription_item_id }
           }
         }
       }
     }

3. AC7 success path:
   If res.ok:
     UPDATE leads SET status = 'usage_reported', usage_reported_at = NOW()
     WHERE id = ${lead.id}

4. AC8 failure path:
   If !res.ok:
     const errorText = await res.text()
     console.error(`lsBillingCron: LS usage reporting failed for lead ${lead.id}`, {
       status: res.status,
       error: errorText,
       expert_id: lead.expert_id
     })
     // Do NOT update status — leave as 'confirmed', usage_reported_at IS NULL
     // Next cron run (0 * * * *) will retry
```

Note: `reportLeadUsage` catches exceptions internally and logs them without re-throwing, so one
lead failure does not abort processing of subsequent leads in the batch.

## Edge Cases per AC

**AC1 (signature verification):**
- LS sends `X-Signature` as lowercase hex of HMAC-SHA256. Web Crypto `verify` operates on raw bytes.
  The hex decode must handle all-zero padding: `parseInt(h, 16)` with `.match(/../g)` regex covers it.
- `request.arrayBuffer()` must be read before parsing JSON — cannot read body twice in CF Workers.
  `verifyLsSignature` returns the decoded body text so the outer handler can parse it without a second read.

**AC2 (welcome credit):**
- If LS subscription-items API call fails (network error), the handler should log the error and continue
  without `ls_subscription_item_id` set (null). The expert will not appear in usage reporting until
  `ls_subscription_item_id` is populated. This is an acceptable degradation — the welcome credit is still
  issued. Log the failure clearly.
- If expert is not found (neither by custom_data.expert_id nor by email), log a warning and return 200
  (do not 500 — LS will retry on 5xx and we cannot identify the expert).

**AC3 (subscription_updated):**
- LS may send status values outside the four mapped values. Safe mapping: only set to one of
  ('active','past_due','cancelled','unpaid'). Unknown values → log warn, skip UPDATE to avoid
  CHECK constraint violation. Do not throw — return 200 to prevent LS retries.

**AC5 (idempotency):**
- KV `get` uses `env.SESSIONS` (same namespace as queue consumers). Key format:
  `idem:ls-webhook:{event_id}`. TTL 86400s. The event_id is the UUID in `event.meta.event_id`
  (distinct from `event.data.id` which is the subscription id).

**AC6 (cron — auto-confirm):**
- Uses `UPDATE ... WHERE status='pending' AND created_at < ${cutoff} RETURNING ...` for atomic
  batch confirm. Sets `confirmed_at = NOW()`. Does not touch leads already in status='confirmed'
  or later statuses.
- `FLAG_WINDOW_MS` absent in production → parseInt falls back to `7 * 24 * 60 * 60 * 1000` = 604800000.

**AC7 (usage reporting):**
- `leadsToReport` query uses `status='confirmed' AND usage_reported_at IS NULL` — this picks up:
  (a) leads just confirmed by the auto-confirm pass in the same cron run, and
  (b) leads that were confirmed on a previous run but whose usage reporting failed.
  The auto-confirmed leads from step (a) are already in the RETURNING set but the query is separate
  to also catch previously-failed leads. Implementation must run these as two distinct SQL statements.

**AC8 (LS failure retry):**
- Do not catch the exception from the `reportLeadUsage` function at the loop level in a way that
  would silently swallow it. Use try/catch inside `reportLeadUsage` that logs and returns (does not
  rethrow). The outer loop continues to the next lead.
- Status stays 'confirmed', usage_reported_at stays NULL → next cron picks it up.

**AC9 (route placement):**
- In `src/index.ts`, the webhook route must appear before the `/api/experts/` block (line 252)
  which calls `authenticate()`. The webhook carries no JWT and must not reach auth middleware.
  Insertion after the GCal callback (line 89) satisfies this requirement.

**AC11 (wrangler.toml cron declaration):**
- `"0 * * * *"` must appear in all three locations: top-level `[triggers]`, `[env.staging.triggers]`,
  and `[env.production.triggers]`. Missing from any one location will cause the cron to not fire in
  that environment.

## Rollback Boundary

Story branch only. The DB migration (`ls_subscription_item_id` column) is additive (`ADD COLUMN IF NOT EXISTS`) and is non-destructive to roll back (column can be dropped or left unused). Reverting the PR leaves the column in place but unused — this is safe. The `wrangler.toml` cron removal would stop the new `0 * * * *` trigger; existing `*/5` and `*/15` crons are unaffected. The `LEMON_SQUEEZY_WEBHOOK_SECRET` secret has no effect if the route is not registered.

No data is destructive: `UPDATE leads SET status='confirmed'` advances lead state (intended); on rollback these leads remain confirmed and would need manual review if the cron is re-deployed, but confirmed is a valid terminal state for billing purposes.
