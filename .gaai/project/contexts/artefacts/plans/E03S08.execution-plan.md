# Execution Plan: E03S08 — Funnel Restructure: Merge Confirmation into Results Page

## Execution Sequence

### Step 1: Backend — Add `POST /api/prospects/{id}/requirements` endpoint

**File:** `workers/backend/api/src/routes/prospects.ts`

Add `handleProspectRequirements` at the bottom:
```typescript
export async function handleProspectRequirements(
  request: Request,
  env: Env,
  prospectId: string,
  ctx: ExecutionContext,
): Promise<Response>
```

Logic:
1. Parse body: `{ requirements: ProspectRequirements, token: string }`
2. `verifyProspectToken(token, prospectId, env.JWT_SECRET)` → 403 if invalid
3. Validate requirements shape (basic — not empty, object)
4. `normalizeRequirements` the incoming requirements using existing helper
5. `UPDATE prospects SET requirements = $normalized WHERE id = $prospectId`
6. Delete existing non-expired matches to force fresh compute:
   `UPDATE matches SET status = 'expired' WHERE prospect_id = $prospectId AND status != 'expired'`
7. Trigger async recompute: `ctx.waitUntil(handleMatchCompute(...))`
8. Return `jsonResponse({ status: 'recomputing' }, 202)`

Rate limit: apply `checkRateLimit` (same as other endpoints). Max 5 re-match per prospect per 24h (use KV key `rematch:${prospectId}`).

**File:** `workers/backend/api/src/index.ts`

Add route registration after existing prospects routes:
```typescript
// POST /api/prospects/:id/requirements — E03S08 (AC5)
if (method === 'POST' && pathname === `/api/prospects/${prospectId}/requirements`) {
  return handleProspectRequirements(request, env, prospectId, ctx);
}
```

Import `handleProspectRequirements` at top.

**JavaScript flow:**

```
On page load:
  1. Read extraction = sessionStorage.getItem('match:extraction')
     → if missing: redirect to /match
  2. Read prospect_id = sessionStorage.getItem('match:prospect_id')
     → may be null (no submit yet)
  3. Read token = sessionStorage.getItem('match:token')
  4. Fire PostHog: satellite.funnel_page2_loaded

  5. renderSummarySection(extraction) — shows extraction summary
  6. If extraction.needs_confirmation.length > 0:
       renderConfirmationQuestions(extraction) → show questions + confirm btn
       renderMatchesPlaceholder() → show placeholder in results section
     Else (ready_to_match: true):
       if prospect_id + token already in sessionStorage (returning user / back nav):
         startFetchMatches() — poll for matches
       else:
         renderMatchesSkeleton()
         renderTurnstile() → auto-fire → doSubmit() → startFetchMatches()

Ready to match flow (auto-submit):
  renderTurnstile (invisible, fires immediately):
    callback: doSubmit(turnstileToken)
      POST /api/prospects/submit {satellite_id, quiz_answers: extraction.requirements, cf-turnstile-response, utm_data}
      → store prospect_id + token in sessionStorage
      → startFetchMatches()

Needs-confirmation flow:
  User fills questions → clicks "Confirmer et trouver mes experts"
  → collectConfirmationAnswers() → mergedReqs
  → if Turnstile not yet rendered: renderTurnstile()
  → Turnstile callback: doSubmit(token, mergedReqs)
  → store prospect_id + token → startFetchMatches()

Edit-triggered re-match (AC5, only after prospect_id exists):
  User clicks "Modifier" on summary → expand edit form
  User saves edit → firePostHog('satellite.extraction_edited', ...)
  → fire PostHog: satellite.rematch_triggered (edit_source: 'extraction_edit')
  → showMatchesRecomputing() → show loading on results section
  → POST /api/prospects/{id}/requirements {requirements: updatedReqs, token}
  → 202 received → startFetchMatches() (polls until new results)

Match fetch (polling, same as existing results.ts):
  GET /api/prospects/{id}/matches?token=xxx
  → 200: renderAnonymizedCards(matches)
  → 202: handle202 (exponential backoff)
  → error: show fetch-error + retry btn

Email gate (shows after matches rendered):
  User enters email → POST /api/prospects/{id}/otp/send
  → show OTP input
  → User enters OTP → POST /api/prospects/{id}/otp/verify
  → success → POST /api/prospects/{id}/identify
  → revealFullProfiles()
```

**PostHog events (AC9):**
- `satellite.funnel_page2_loaded` — on page load (satellite_id, ready_to_match, needs_confirmation_count)
- `satellite.extraction_edited` — when edit form saved (field_name, old_value_hash, new_value_hash)
- `satellite.rematch_triggered` — when re-match fires (prospect_id, edit_source: "extraction_edit" | "confirmation_answer")
- Keep existing: `satellite.matches_viewed`, `satellite.email_gate_submitted`, `satellite.profiles_unlocked`, `satellite.booking_cta_clicked`, `satellite.matching_error`
- Remove: `satellite.confirmation_viewed`, `satellite.confirmation_field_edited`, `satellite.confirmation_submitted` (replaced by unified events)

**Note on OTP gate:** The current results.ts uses old email identify without OTP (E06S39 added OTP, but the frontend wasn't updated for it in E03S03). The merged page should implement the full OTP flow:
1. Email input → POST /api/prospects/{id}/otp/send
2. OTP input (6 digits) → POST /api/prospects/{id}/otp/verify
3. → POST /api/prospects/{id}/identify (now uses verified KV, no email in body needed per E06S39)
Actually, re-reading the identify endpoint: it reads email from the `verified:${prospectId}` KV set by otp/verify. So the identify call needs no email body — just the token.

## Edge Cases

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC1 | ready_to_match + returning user (prospect_id already in sessionStorage) | Skip Turnstile + submit, directly poll matches |
| AC1 | Turnstile timeout | Show error on results section, offer retry |
| AC2 | needs_confirmation with 0 questions (edge: API returns needs_confirmation=[] but not ready_to_match) | Treat as ready_to_match |
| AC4 | Collapsible with undefined fields | Show "Non identifié" same as current confirm.ts |
| AC5 | Edit-triggered re-match before prospect_id exists | Disable edit save button until after initial submit |
| AC5 | Re-match rate limit exceeded | Show inline error on results section |
| AC8 | Old /confirm link with prospect_id in URL or sessionStorage | Redirect to /results; Page 2 JS handles missing extraction gracefully |
| AC10 | OTP expires during flow | Show "Code expiré, renvoyer" UI |

## Rollback Boundaries

- Frontend change is backwards-compatible: `/confirm` redirects to `/results`, old links work
- Backend `POST /api/prospects/{id}/requirements` is additive only — no existing endpoints modified
- If deployment fails: revert commit, `/confirm` reverts to old behavior, `/results` reverts to pre-merge state
