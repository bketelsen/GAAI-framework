# Execution Plan — E06S18: Hyperdrive + postgres.js + Smart Placement + Bug Fixes

## High-Level Approach

Mechanical query-by-query migration of all DB interactions from Supabase JS
(PostgREST/HTTP) to postgres.js tagged templates over Hyperdrive. Supabase JS
is retained exclusively for `auth.getUser()` in `middleware/auth.ts`.

**Approach rationale:** Story explicitly mandates postgres v3.4.4+ with Hyperdrive
binding (`prepare: true`). No approach evaluation needed — both the library and
the pattern are fully specified in the Story (AC1–AC5) and in the migration
notes. The migration is mechanical, not architectural.

---

## Implementation Sequence

### Phase 1 — New infrastructure files (no existing code touched)

**Step 1.1 — Install postgres:**
- `package.json`: add `"postgres": "^3.4.4"` to dependencies

**Step 1.2 — Create `src/types/db.ts`:**
- Export row interfaces for all 9 tables derived from `Database` type:
  `ExpertRow`, `BookingRow`, `ProspectRow`, `MatchRow`, `SatelliteConfigRow`,
  `LeadRow`, `CallExperienceSurveyRow`, `ProjectSatisfactionSurveyRow`,
  `LeadEvaluationRow`
- Also export `Json` re-export for convenience

**Step 1.3 — Create `src/lib/db.ts`:**
```typescript
import postgres from 'postgres';
import type { Env } from '../types/env';

export function createSql(env: Env) {
  return postgres(env.HYPERDRIVE.connectionString, { prepare: true });
}
export type SqlClient = ReturnType<typeof createSql>;
```

**Step 1.4 — Update `src/types/env.ts`:**
- Add `HYPERDRIVE: Hyperdrive;`

**Step 1.5 — Update `wrangler.toml`:**
- Add `[placement]` with `mode = "smart"` BEFORE existing `[[queues.consumers]]`
  (must be top-level, not inside an env block)
- Add `[[hyperdrive]]` binding for dev (top-level, with `localConnectionString`
  placeholder: `"postgres://postgres:postgres@db.PLACEHOLDER.supabase.co:5432/postgres"`)
- Add `[[env.staging.hyperdrive]]` with `binding = "HYPERDRIVE"` and
  `id = "PLACEHOLDER_HYPERDRIVE_CONFIG_ID"`
- Add `[[env.production.hyperdrive]]` with `binding = "HYPERDRIVE"` and
  `id = "PLACEHOLDER_HYPERDRIVE_CONFIG_ID"`
- SETUP.md note: human must run `wrangler hyperdrive create` and replace the placeholder ID

### Phase 3 — Migrate route handlers

**Step 3.1 — `src/routes/satellites.ts`:**
1 query: `satellite_configs` select by id. Remove inline `createClient`.

**Step 3.2 — `src/handlers/experts/register.ts`:**
1 query: `experts` insert + select id. Error code `23505` handling must be
preserved (map to duplicate detection). With postgres.js, unique violations throw
`PostgresError` with `code = '23505'`.

**Step 3.3 — `src/handlers/experts/profile.ts`:**
2 operations:
- `experts` SELECT * → `SELECT * FROM experts WHERE id = ${expertId}`
- `merge_expert_profile` RPC → postgres.js named param call (AC8):
  ```typescript
  const rows = await sql<ExpertRow[]>`
    SELECT * FROM merge_expert_profile(
      p_id := ${expertId},
      p_display_name := ${displayName ?? null},
      p_headline := ${headline ?? null},
      p_bio := ${bio ?? null},
      p_rate_min := ${rateMin ?? null},
      p_rate_max := ${rateMax ?? null},
      p_availability := ${availability ?? null},
      p_profile := ${profile ? JSON.stringify(profile) : null}::jsonb,
      p_preferences := ${preferences ? JSON.stringify(preferences) : null}::jsonb
    )`;
  ```
  Error handling: `error?.code === 'PGRST116'` → not found. With postgres.js,
  empty array means not found; postgres errors throw directly.

**Step 3.4 — `src/handlers/experts/gcal.ts`:**
6 queries across 4 functions. All straightforward SELECT/UPDATE patterns.
Preserve 404 detection: empty array = not found.

**Step 3.5 — `src/handlers/bookings/availability.ts`:**
2 queries. Expert gcal fields + held/confirmed bookings.

**Step 3.6 — `src/handlers/bookings/hold.ts`:**
3 queries: conflict check, match lookup, booking insert.

**Step 3.7 — `src/handlers/bookings/confirm.ts`:**
6 queries (some conditional). Preserve race guard (delete + 409). Note: the
slot-taken guard deletes the hold — with postgres.js:
```typescript
await sql`DELETE FROM bookings WHERE id = ${bookingId}`;
```

**Step 3.8 — `src/handlers/bookings/cancel.ts`:**
2 queries: booking select + booking update.

**Step 3.9 — `src/handlers/bookings/reschedule.ts`:**
4 queries: booking select, conflict check, expert select, booking update.

**Step 3.10 — `src/handlers/bookings/cron.ts`:**
7 queries (some in loops). Key changes:
- DELETE with count: `DELETE FROM bookings WHERE ...` — postgres.js returns count
  via `sql.unsafe` or checking result length; use `result.count` after COUNT query.
  Alternative: use `DELETE ... RETURNING id` and count returned rows.
- For `cleanupExpiredHolds`: `await sql\`DELETE FROM bookings WHERE status = 'held' AND held_until < ${now} RETURNING id\``
  then `count = result.length`.

**Step 3.11 — `src/handlers/bookings/prep.ts`:**
4–5 queries: booking by prep_token, prospect, expert, match.

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|-----------|
| JSONB serialization | Medium | Use `JSON.stringify(val)::jsonb` for all JSONB params |
| PostgresError vs Supabase error codes | Medium | Map `23505` → duplicate detection in register + lead-billing |
| `.single()` semantics | Low | Return `[row]` then check `!row` for not-found |
| `.maybeSingle()` semantics | Low | Destructure `[row]`, undefined = not found (not an error) |
| Array IN clause | Low | Use `WHERE id = ANY(${ids})` pattern |
| Cron delete with count | Low | Use `RETURNING id`, count returned rows |
| RPC optional params | Medium | Pass `null` explicitly for omitted params |
