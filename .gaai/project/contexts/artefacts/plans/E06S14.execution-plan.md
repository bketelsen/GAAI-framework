# Execution Plan: E06S14 — Satellite Worker Multi-Tenant Scaffold

## Tier Assessment

**Tier 2 — Core Team (Planning + Implementation + QA)**
- Complexity: 6 (14 ACs, new Worker project, multi-tenant routing, HTML rendering, GitHub Actions)
- Files affected: ~12 new files + 3 modified existing files
- Specialist triggers: None
- Risk: Low (greenfield project, no existing code modification except SETUP.md + GitHub Actions)

## Approach Rationale

**Hono for routing** — explicitly specified in the Story. While callibrate-core uses regex-based routing, the satellite Worker is an independent project. Hono provides cleaner middleware composition (ideal for the hostname resolution middleware) and JSX/template rendering support.

**Project location: `workers/satellite/`** — AC1 specifies this path. First multi-worker project structure in the repo.

**No Supabase JS dependency** — the satellite Worker only needs to query `satellite_configs` on KV miss. A direct `fetch()` to Supabase REST API (with anon key) is sufficient. No need for the full `@supabase/supabase-js` package, keeping the Worker bundle minimal.

### Step 3: Config Resolution Middleware (AC3, AC11)

**File to create:** `workers/satellite/src/middleware/config.ts`

**Logic:**
1. Extract hostname from `new URL(request.url).hostname`
2. KV lookup: `CONFIG_CACHE.get(\`satellite:config:${hostname}\`, 'json')`
3. On KV hit: return parsed config
4. On KV miss: `fetch(\`${SUPABASE_URL}/rest/v1/satellite_configs?domain=eq.${hostname}&active=eq.true&limit=1\`)` with `apikey` and `Authorization: Bearer` headers (anon key)
5. On DB hit: `CONFIG_CACHE.put(\`satellite:config:${hostname}\`, JSON.stringify(config), { expirationTtl: 3600 })`, return config
6. On DB miss: return `null` (caller handles 302 redirect)

**Edge cases:**
- Hostname with port (e.g. `localhost:8787`): `new URL(request.url).hostname` already strips port
- KV write failure: non-blocking (try/catch), config still served from DB response
- Supabase unreachable: return `null` → 302 redirect (fail-safe)

**Checkpoint:** Config resolution works for known domain, returns null for unknown.

### Step 4: Hono App + Route Registration (AC4-AC11)

**File to create:** `workers/satellite/src/index.ts`

**Hono app setup:**
```typescript
import { Hono } from 'hono';
const app = new Hono<{ Bindings: Env; Variables: { config: SatelliteConfig } }>();
```

**Middleware chain:**
1. Config resolution middleware (Step 3) — sets `c.set('config', config)` or returns 302
2. Route handlers access config via `c.get('config')`

**Routes (in order):**
1. `GET /health` — AC8 (no-store cache, no config required — health should work even without valid hostname)
2. `GET /robots.txt` — AC6
3. `GET /sitemap.xml` — AC7
4. `POST /admin/cache/purge` — AC9 (before config middleware — uses its own auth)
5. Config middleware (all routes below require valid config)
6. `GET /` — AC4 + AC5 (landing page)
7. Fallback: 302 redirect (for unknown paths on valid satellites)

**Note on /health:** Should respond before config middleware — health check should not fail on unknown hostname. Return `200 { ok: true, domain: hostname }` even without config. If config available, include `satellite_id`.

Actually, re-reading AC8: "returns 200 { ok: true, domain: hostname, satellite_id: config.id }". This requires config. But health should be resilient. Two options:
- Health requires config → will 302 on unknown hostname (technically correct per AC)
- Health works without config → more operationally useful

**Decision:** Follow AC8 literally — health requires config resolution. If hostname unknown, 302 applies. The operator will always hit health on a configured satellite domain.

### Step 5: Landing Page Renderer (AC4, AC5, AC10)

**File to create:** `workers/satellite/src/pages/landing.ts`

**Renders complete HTML document using template literals** (not JSX — simpler, no build step):

```html
<!DOCTYPE html>
<html lang="fr">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>${config.content.meta_title}</title>
  <meta name="description" content="${config.content.meta_description}">
  <meta name="robots" content="index, follow">
  <!-- Open Graph -->
  <meta property="og:title" content="${config.content.meta_title}">
  <meta property="og:description" content="${config.content.meta_description}">
  <meta property="og:url" content="https://${config.domain}/">
  <meta property="og:type" content="website">
  <!-- Canonical -->
  <link rel="canonical" href="https://${config.domain}/">
  <!-- JSON-LD -->
  <script type="application/ld+json">${JSON.stringify(config.structured_data)}</script>
  <!-- Design tokens -->
  <style>
    :root {
      --color-primary: ${theme.primary};
      --color-accent: ${theme.accent};
      --font-family: ${theme.font};
      --radius-card: ${theme.radius};
    }
    /* Base structural CSS inlined */
  </style>
</head>
<body>
  ${config.theme.logo_url ? `<img src="${config.theme.logo_url}" alt="${config.brand.name}">` : ''}
  <h1>${config.brand.name}</h1>
  <h2>${config.content.hero_headline}</h2>
  <p>${config.content.hero_sub}</p>
  <ul>${config.content.value_props.map(vp => `<li>${vp}</li>`).join('')}</ul>
  <a href="/match">Décrire mon projet</a>
</body>
</html>
```

**Cache headers:** `Cache-Control: public, max-age=300, stale-while-revalidate=3600` (AC10)

**XSS prevention:** All config values are operator-controlled (inserted via Supabase by the founder), not user-generated. Template literal injection is safe in this context. However, as a defensive measure, escape HTML entities in text content fields (not in URLs or JSON-LD).

**Checkpoint:** Landing page renders correctly with seed data config.

### Step 8: Admin Cache Purge Endpoint (AC9)

**Implemented in:** `workers/satellite/src/index.ts` (inline or separate handler)

**Logic:**
1. Check `x-admin-secret` header against `env.ADMIN_SECRET`
2. On mismatch: 401
3. Parse body `{ domain: string }` — on missing: 400
4. Delete KV key: `CONFIG_CACHE.delete(\`satellite:config:${domain}\`)`
5. Return `200 { purged: true, domain }`

**No config middleware needed** — this route is admin-only, not tenant-specific.

### Step 9: Health Endpoint (AC8)

**Logic:**

## Edge Cases

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC3 | Hostname with port | `new URL().hostname` strips port automatically |
| AC3 | KV write failure | Non-blocking try/catch, serve from DB response |
| AC3 | Supabase unreachable | Return null → 302 redirect (fail-safe) |
| AC4 | `config.content` is null | Fallback to empty strings / omit sections |
| AC4 | `config.structured_data` is null | Omit JSON-LD script tag |
| AC5 | `config.theme.logo_url` is null | Omit `<img>` element (conditional render) |
| AC5 | `config.content.value_props` is empty array | Omit `<ul>` section |
| AC9 | Missing `domain` in body | 400 response |
| AC9 | JSON parse failure | 400 response |
| AC11 | Unknown hostname, first visit | DB miss → 302 redirect, no KV write |
