# Execution Plan — E06S34: Balance / max_lead_price / spending_limit Enforcement at Matching

## Approach Rationale

Three billing filters (balance, max_lead_price, spending_limit) must run before scoring in the
matching pipeline. The filter is applied to every expert in the pool before they are scored, so
that excluded experts are never ranked, never written to the matches table, and never presented
to the prospect.

### Key architectural decisions

**Billing data is live-queried, not cached.** The expert pool cache (Cache API L1 → D1 →
Hyperdrive) deliberately excludes billing columns because credit_balance changes on every debit.
Caching it would produce stale exclusions. Instead, billing data is queried live from Hyperdrive
at match time, using two single-batch SQL statements that cover all experts in the pool at once.

**Two billing queries, not N.** The spending_limit check requires a SUM of `credit_transactions`
per expert for the current UTC month. Rather than N per-expert queries, a single GROUP BY query
covers all pool members. The index on `credit_transactions(expert_id, created_at)` (AC5 from
E06S31) ensures the monthly spend query is efficient.

**Pure filter utility `applyBillingFilters`.** A pure function takes experts + billing data +
leadPrice and returns `{ eligible, excluded }` where each excluded entry carries the exclusion
reason. This function is the single source of truth for billing logic and is tested in isolation.

**The Matching Worker returns `billing_excluded` in its response body.** The Matching Worker has
no EMAIL_NOTIFICATIONS queue binding and cannot queue notifications. Instead, after applying the
billing filter, `handleMatch` in `workers/matching/src/index.ts` includes the excluded list in
the JSON response: `{ computed, top_matches, billing_excluded }`. The Core Worker intercepts
this response, parses it, and queues EMAIL_NOTIFICATIONS entries via `ctx.waitUntil()`.

**Core Worker intercept pattern in `matches.ts`.** Currently, `handleMatchCompute` returns the
MATCHING_SERVICE response directly. This must change to parse the response and queue
notifications for `billing_excluded` entries, then forward a clean response to the caller.

**`ctx` must be added to `handleMatchCompute`.** Currently called from `src/index.ts` without
`ctx`. The function signature must add `ctx: ExecutionContext` and `src/index.ts` must pass it.

**Core Worker fallback paths.** Both `src/routes/matches.ts` and `src/routes/prospects.ts`
must apply the billing filter in their local scoring paths.

**New queue message type `expert.billing.lead_missed`.** Fires at match time when an expert is
excluded. Distinct from `expert.billing.insufficient_balance` (post-debit failure).

**`ExpertWithBilling` local type in Matching Worker.** Do NOT add billing fields to the shared
`ExpertPoolEntry` type — this would break the D1/Cache layer which doesn't store those fields.
Instead, define a local `ExpertWithBilling = ExpertPoolEntry & { credit_balance, max_lead_price, spending_limit }`
inside `handleMatch` for the enriched in-memory representation.

### Step 4 — `src/lib/billingFilter.ts` (NEW)

Complete new file. Exports:

**Interfaces:**
```typescript
export interface BillingRecord {
  expert_id: string;
  credit_balance: number;
  max_lead_price: number | null;
  spending_limit: number | null;
  monthly_spend: number;
}

export interface BillingExclusion {
  expert_id: string;
  reason: 'insufficient_balance' | 'max_lead_price_exceeded' | 'spending_limit_reached';
}
```

**`applyBillingFilters` — pure function:**
```
applyBillingFilters(
  experts: { id: string }[],
  billingMap: Map<string, BillingRecord>,
  leadPrice: number
): { eligible: { id: string }[]; excluded: BillingExclusion[] }

For each expert:
  billing = billingMap.get(expert.id) ?? { credit_balance: 0, max_lead_price: null, spending_limit: null, monthly_spend: 0 }

  // AC2 + AC9: balance < leadPrice (0 < 4900 is true → excluded, no free pass)
  if (billing.credit_balance < leadPrice) → excluded, reason='insufficient_balance'

  // AC3: max_lead_price deal-breaker (skip if null)
  if (billing.max_lead_price !== null && leadPrice > billing.max_lead_price) → excluded, reason='max_lead_price_exceeded'

  // AC4: spending_limit cap (skip if null)
  if (billing.spending_limit !== null && billing.monthly_spend + leadPrice > billing.spending_limit) → excluded, reason='spending_limit_reached'

  → eligible
```

**`loadBillingData` — Hyperdrive query helper:**
```
async loadBillingData(sql, expertIds: string[]): Promise<Map<string, BillingRecord>>

if expertIds.length === 0: return new Map()

Query 1 (experts table):
  SELECT id, credit_balance, max_lead_price, spending_limit
  FROM experts WHERE id = ANY(${expertIds})

Query 2 (credit_transactions — AC4, AC5):
  const monthStart = new Date()
  monthStart.setUTCDate(1); monthStart.setUTCHours(0, 0, 0, 0)

  SELECT expert_id, COALESCE(SUM(ABS(amount)), 0)::integer AS monthly_spend
  FROM credit_transactions
  WHERE type = 'lead_debit'
    AND expert_id = ANY(${expertIds})
    AND created_at >= ${monthStart.toISOString()}
  GROUP BY expert_id

Build and return Map<expert_id, BillingRecord>
```

Note: `COALESCE(SUM(ABS(amount)), 0)` handles the case where SUM returns NULL (no rows this month).
`ABS` converts negative debit amounts to positive for the monthly spend comparison.

Imports: `createSql` type from `'./db'`; `ExpertRow` from `'../types/db'`.

**e. Build billing map and apply filter:**
```typescript
// Build billing map
const billingMap = new Map<string, { credit_balance: number; max_lead_price: number | null; spending_limit: number | null; monthly_spend: number }>();
for (const row of billingRows) {
  billingMap.set(row.id, {
    credit_balance: row.credit_balance,
    max_lead_price: row.max_lead_price,
    spending_limit: row.spending_limit,
    monthly_spend: 0,
  });
}
for (const row of spendRows) {
  const existing = billingMap.get(row.expert_id);
  if (existing) existing.monthly_spend = row.monthly_spend;
}

// Calculate leadPrice once (AC1)
const leadPriceResult = calculateLeadPrice(
  (requirements.budget_range as { max?: number } | undefined)?.max ?? null,
  {
    budget_max: (requirements.budget_range as { max?: number } | undefined)?.max ?? null,
    timeline_days: null,
    skills: requirements.skills_needed ?? [],
  }
);
const leadPrice = leadPriceResult.amount;

// Apply billing filter
const billingExcluded: BillingExclusion[] = [];
const eligible: typeof expertPool = [];
for (const expert of expertPool) {
  const billing = billingMap.get(expert.id) ?? { credit_balance: 0, max_lead_price: null, spending_limit: null, monthly_spend: 0 };
  if (billing.credit_balance < leadPrice) {
    billingExcluded.push({ expert_id: expert.id, reason: 'insufficient_balance' });
  } else if (billing.max_lead_price !== null && leadPrice > billing.max_lead_price) {
    billingExcluded.push({ expert_id: expert.id, reason: 'max_lead_price_exceeded' });
  } else if (billing.spending_limit !== null && billing.monthly_spend + leadPrice > billing.spending_limit) {
    billingExcluded.push({ expert_id: expert.id, reason: 'spending_limit_reached' });
  } else {
    eligible.push(expert);
  }
}

// AC7: all excluded → empty results
if (eligible.length === 0) {
  return jsonResponse({ computed: 0, top_matches: [], billing_excluded: billingExcluded });
}
```

### Step 11 — `src/lib/billingFilter.test.ts` (NEW)

Vitest test file. Imports `applyBillingFilters` and `BillingRecord`. No mocks needed.

**Test groups:**

1. **Balance check (AC2, AC9)**
   - `balance sufficient (>)` → eligible
   - `balance exactly equal (===)` → eligible
   - `balance insufficient (<)` → excluded, reason=insufficient_balance
   - `balance zero → excluded (AC9)` → excluded, no free pass
   - `expert not in billing map → defaults to 0 → excluded` → excluded

2. **max_lead_price check (AC3)**
   - `max_lead_price null → skip check → included`
   - `leadPrice equals max_lead_price → included`
   - `leadPrice exceeds max_lead_price → excluded, reason=max_lead_price_exceeded`

3. **spending_limit check (AC4)**
   - `spending_limit null → skip check → included`
   - `monthly_spend + leadPrice equals spending_limit → included`
   - `monthly_spend + leadPrice exceeds spending_limit → excluded, reason=spending_limit_reached`
   - `zero monthly_spend, spending_limit high → included`

4. **Check ordering (first-failing reason wins)**
   - `balance fails + would also fail max_lead_price → reason=insufficient_balance`
   - `balance passes, max_lead_price fails → reason=max_lead_price_exceeded`
   - `balance passes, max_lead_price passes, spending_limit fails → reason=spending_limit_reached`

5. **Mixed pool (AC7, AC11)**
   - `all experts excluded → eligible=[], excluded=[all]`
   - `partial exclusion → correct split`
   - `multiple constraints simultaneously → each expert excluded for correct reason`

6. **Edge cases**
   - `empty pool → eligible=[], excluded=[]`

## Edge Cases per AC

**AC1:** `calculateLeadPrice` called once per match request, result stored as `leadPrice` constant.

**AC2/AC9:** `credit_balance < leadPrice` (strict less-than). `credit_balance=0` with any leadPrice > 0 → excluded. No bypass.

**AC3:** `max_lead_price !== null && leadPrice > max_lead_price` (strict greater-than). NULL → skip entirely.

**AC4:** Month boundary computed as UTC midnight of first day. `COALESCE(SUM(ABS(amount)), 0)` handles experts with no transactions this month (SUM returns NULL for empty groups).

**AC5:** Index on `credit_transactions(expert_id, created_at)` is used because the WHERE clause includes `expert_id = ANY(...)` AND `created_at >= ...` — both index columns. The `COALESCE` call does not prevent index usage.

**AC6:** All queue sends wrapped in `ctx.waitUntil()`. Non-blocking. Failures isolated to waitUntil promise.

**AC7:** Return HTTP 200 with empty results. In `prospects.ts`, the prospect row is already committed; the token is still issued (prospect can poll for matches later).

**AC8:** Applied in both endpoints via both execution paths (MATCHING_SERVICE bound + fallback).

**AC9:** Covered in `applyBillingFilters` default billing record having `credit_balance: 0`.
