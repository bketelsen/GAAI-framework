# E06S25 Execution Plan: Durable Objects — Expert Pool Write Coordinator

## Approach

**Tier: 2** — Core team (complexity 4, 6 files affected, 7 ACs, new CF primitive).

**Base branch:** `story/E06S23` — provides D1 + Hyperdrive foundation (dependency). Vectorize infrastructure (`vectorize.ts`) added as part of this story since E06S21 is not yet merged to staging. This is a pragmatic parallel-branch strategy: when the founder merges E06S21 and E06S25 to staging, the vectorize.ts file will be identical (trivial conflict).

**No approach evaluation needed:** The DO singleton pattern is explicitly defined in the story, and the `cloudflare:workers` DurableObject class is the only viable pattern for CF DO.

## Implementation Sequence

### Step 1 — `src/lib/vectorize.ts` (NEW, from E06S21)
Import `buildEmbeddingText` and `upsertExpertEmbedding` functions. The DO needs `buildEmbeddingText`. `upsertExpertEmbedding` included for completeness (will be needed by register/profile on E06S21 merge path).

### Step 2 — `src/durable-objects/expertPoolDO.ts` (NEW)
- `ExpertPoolDO extends DurableObject<Env>`
- `pendingEvents: Map<string, ProfileEvent>` — in-memory queue, keyed by `expert_id` (last write wins)
- `fetch(request)` — accepts POST with `ProfileEvent`, stores in map, sets 5s alarm if none exists
- `alarm()` — processes all pending events: (1) batch D1 upsert via `upsertToD1`, (2) per-expert Vectorize embed+upsert
- `notifyExpertPoolDO(env, ctx, event)` — exported helper for handlers to call fire-and-forget via `ctx.waitUntil`; no-op when `EXPERT_POOL_DO` binding absent (graceful degradation)

**Alarm window:** `Date.now() + 5000` — set once, not re-set on subsequent events within the window (idempotent batching).

**Error handling:** Both D1 and Vectorize failures are caught + logged. Non-fatal — cron sync (*/5) acts as safety net (AC6).

### Step 3 — `src/types/env.ts` (MODIFY)
Add optional bindings: `AI?: Ai`, `VECTORIZE?: VectorizeIndex`, `EXPERT_POOL_DO?: DurableObjectNamespace`. Optional ensures graceful degradation before provisioning.

### Step 4 — `wrangler.toml` (MODIFY)
Add for all 3 envs (dev, staging, production):
- `[ai] binding = "AI"`
- `[[vectorize]] binding = "VECTORIZE" index_name = "expert-profiles"`
- `[[durable_objects.bindings]] name = "EXPERT_POOL_DO" class_name = "ExpertPoolDO"`
- `[[migrations]] tag = "v1" new_classes = ["ExpertPoolDO"]` (top-level only)

### Step 5 — `src/index.ts` (MODIFY)
- Add `export { ExpertPoolDO } from './durable-objects/expertPoolDO'` (CF runtime discovers DO by named export)
- Add `ctx: ExecutionContext` to `fetch()` signature
- Pass `ctx` to `handleRegister` and `handlePatchProfile`

### Step 6 — `src/handlers/experts/register.ts` (MODIFY)
- Add `ctx: ExecutionContext` parameter
- After successful DB insert + email queue: call `notifyExpertPoolDO(env, ctx, { id, profile:{}, preferences:{}, rate_min, rate_max, composite_score:null, total_leads:0, availability:null })`

### Step 7 — `src/handlers/experts/profile.ts` (MODIFY)
- Add `ctx: ExecutionContext` parameter to `handlePatchProfile`
- After successful RPC merge: call `notifyExpertPoolDO(env, ctx, { id, profile, preferences, rate_min, rate_max, composite_score, total_leads:0, availability })`

## Edge Cases

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC2 | Same expert writes twice in 5s window | Last write wins (Map key = expert_id) |
| AC2 | Alarm already set | `getAlarm()` check — no re-schedule |
| AC3 | D1 not provisioned (`EXPERT_DB` absent) | Skip D1 block silently |
| AC3 | AI/VECTORIZE not provisioned | Skip Vectorize block silently |
| AC3 | D1 upsert throws | Caught + logged; Vectorize still runs |
| AC3 | Vectorize fails for one expert | Caught + logged; continues to next expert |
| AC3 | AI returns empty vector | `console.warn` + `continue` (no upsert) |
| AC5 | DO unreachable | `notifyExpertPoolDO` catches in `waitUntil` async block; no error propagated |
| AC5 | `EXPERT_POOL_DO` binding absent | Early return in `notifyExpertPoolDO` |
| AC6 | DO alarm never fires | Cron */5 `syncExpertPoolToD1` catches all changes from Hyperdrive |

## Test Checkpoints

- `fetch()`: 405 non-POST, 400 invalid JSON, 400 missing id, 202 valid event
- `fetch()` + `alarm()` integration: alarm set on first event, NOT re-set on second, last-write-wins for duplicate id
- `alarm()` D1: batch call, skip when no EXPERT_DB, D1 failure non-fatal, pending cleared after alarm
- `alarm()` Vectorize: AI.run called with embedding text, metadata fields, empty vector skip, Vectorize failure per-expert non-fatal
- `notifyExpertPoolDO`: non-blocking (returns void + uses waitUntil), uses idFromName('expert-pool-coordinator'), no-op when binding absent, DO failure non-fatal
