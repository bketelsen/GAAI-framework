## Architecture Analysis

### Key Findings from Codebase Exploration

**JWT pattern (lib/jwt.ts):** All tokens use the same HMAC-SHA256 Web Crypto pattern: build `header.payload`, sign with `crypto.subtle`, encode as base64url. The `verifyBookingToken` must return a discriminated result (`'valid' | 'expired' | 'invalid'`) instead of boolean so callers can send different redirects for expired vs. tampered tokens.

**Status constraint gap:** The initial migration defines `CHECK (status IN ('confirmed', 'cancelled', 'completed', 'no_show'))`. However `'held'` is used in the code with no migration to update the constraint. This means the constraint was either dropped directly in the DB or never truly enforced. The E03S07 migration must perform a safe DROP CONSTRAINT + ADD CONSTRAINT covering all valid statuses including `'held'`.

**Cron structure (cron.ts):** `*/5 * * * *` already calls `cleanupExpiredHolds`. Adding `cleanupExpiredConfirmations` and `autoConfirmPendingExpertApproval` is a clean additive pattern. The `*/15 * * * *` cron already handles reminders via queue dispatch.

**Queue consumer (email-notifications.ts):** Uses `sendResendEmail` internally (not the `sendEmail` from lib/email.ts). New reminder cases should use `sendResendEmail` directly inside the consumer. The `default` case has an exhaustive TypeScript check — adding new types to `EmailNotificationMessage` will break compilation until the switch handles them.

**Booking widget (booking-widget.ts):** The `renderSuccess(data, email)` function is the target for AC9. Currently renders "Réservation confirmée!" with a Google Meet link. Must become "check email" state. The confirm API response shape changes from `{ booking_id, meeting_url, start_at, end_at, ... }` to `{ status: "pending_confirmation", confirmation_sent_to: email }`.

**Dashboard (dashboard.bookings.tsx):** Typed `BookingStatus = "held" | "confirmed" | "cancelled"`. Must be extended. The no-show action needs a new `intent: "no-show"` in the action function and a new button visible only on past confirmed bookings.

**Resend endpoint rate limiting:** The `SESSIONS` KV namespace is already used for rate limiting in hold.ts. Use the same KV sliding window pattern (key: `confirm-resend-rate:{booking_id}`, value: count, TTL 3600s, max 3).

**AC5 expert approval emails:** These need HMAC-signed tokens with `action: 'expert-approve' | 'expert-reject'` — handled by the same `bookingToken.ts` but with different action values. The expert approval endpoint is `GET /api/bookings/:id/expert-approve?token=...` and `GET /api/bookings/:id/expert-reject?token=...`. TTL: 24h for expert tokens (vs. 30min for prospect confirmation tokens). Since both use the same signing function with different `action` and TTL, the `signBookingToken` function needs a `ttlSeconds` parameter.

**File:** `supabase/migrations/20260228000005_e03s07_booking_confirmation.sql`

```sql
-- E03S07: Booking Confirmation Email + Reminders + No-Show Tracking
-- ────────────────────────────────────────────────────────────────────

-- 1. Drop existing CHECK constraint on bookings.status (name was auto-generated)
--    Must use pg_constraint catalog because Postgres does not support
--    DROP CONSTRAINT IF EXISTS for inline column constraints.
DO $$
DECLARE
  constraint_name TEXT;
BEGIN
  SELECT conname INTO constraint_name
  FROM pg_constraint
  WHERE conrelid = 'bookings'::regclass
    AND contype = 'c'
    AND conname LIKE '%status%';
  
  IF constraint_name IS NOT NULL THEN
    EXECUTE 'ALTER TABLE bookings DROP CONSTRAINT ' || quote_ident(constraint_name);
  END IF;
END $$;

-- 2. Add new CHECK constraint with all valid statuses
ALTER TABLE bookings ADD CONSTRAINT bookings_status_check
  CHECK (status IN (
    'held',
    'confirmed',
    'cancelled',
    'completed',
    'no_show',
    'pending_confirmation',
    'expired_no_confirmation',
    'cancelled_by_prospect',
    'pending_expert_approval'
  ));

-- 3. Add confirmation_token column to bookings
ALTER TABLE bookings
  ADD COLUMN IF NOT EXISTS confirmation_token TEXT;

-- 4. Add booking_auto_confirm to experts
ALTER TABLE experts
  ADD COLUMN IF NOT EXISTS booking_auto_confirm BOOLEAN NOT NULL DEFAULT true;

-- 5. Add no_show_count to prospects
ALTER TABLE prospects
  ADD COLUMN IF NOT EXISTS no_show_count INTEGER NOT NULL DEFAULT 0;

-- 6. Index for expiry cron query performance
CREATE INDEX IF NOT EXISTS bookings_status_created_at_idx
  ON bookings (status, created_at)
  WHERE status IN ('pending_confirmation', 'pending_expert_approval');
```

**Rationale for pg_constraint approach:** The constraint name generated by Postgres for inline CHECK constraints in CREATE TABLE is implementation-specific (e.g., `bookings_status_check` or `bookings_status_check1`). Using the catalog lookup avoids hardcoding a potentially wrong name.

### Step 2 — `lib/bookingToken.ts` (new file, no deps)

**File:** `workers/backend/api/src/lib/bookingToken.ts`

This module mirrors the jwt.ts pattern exactly but uses `BOOKING_CONFIRM_SECRET` and a simpler payload shape. It must NOT import from jwt.ts to avoid any secret cross-contamination.

```typescript
// ── Booking confirmation/cancellation tokens ──────────────────────────────────
// HMAC-SHA256 signed JWT following the same pattern as lib/jwt.ts.
// Uses BOOKING_CONFIRM_SECRET (separate from PROSPECT_TOKEN_SECRET).
// Audience values prevent cross-use between action types.

export type BookingTokenAction = 
  | 'confirm'           // prospect email-confirms — 30min TTL
  | 'cancel'            // prospect email-cancels — 30min TTL
  | 'expert-approve'    // expert approves (AC5) — 24h TTL
  | 'expert-reject';    // expert rejects (AC5) — 24h TTL

export type BookingTokenResult = 'valid' | 'expired' | 'invalid';

// Internal helpers — same as jwt.ts (not re-exported)
function toBase64Url(data: Uint8Array | ArrayBuffer): string { ... }
function fromBase64Url(str: string): Uint8Array { ... }
async function importHmacKey(secret: string, usage: 'sign' | 'verify'): Promise<CryptoKey> { ... }

export async function signBookingToken(
  bookingId: string,
  action: BookingTokenAction,
  secret: string,
  ttlSeconds = 1800,  // default 30min; pass 86400 for expert tokens
): Promise<string>

// Returns discriminated result — 'expired' vs 'invalid' drive different redirect destinations
export async function verifyBookingToken(
  token: string,
  bookingId: string,
  action: BookingTokenAction,
  secret: string,
): Promise<BookingTokenResult>
```

**Payload structure:**
```json
{ "booking_id": "uuid", "action": "confirm", "exp": 1234567890, "iss": "callibrate", "aud": "booking:confirm" }
```

The `aud` field is derived from `"booking:" + action` to prevent a confirm token from being used as a cancel token.

**signBookingToken implementation:**
1. Compute `exp = Math.floor(Date.now() / 1000) + ttlSeconds`
2. Build header (`{"alg":"HS256","typ":"JWT"}`) and payload (`{booking_id, action, exp, iss:"callibrate", aud:"booking:"+action}`)
3. Sign with `crypto.subtle.sign('HMAC', key, encoder.encode(signingInput))`
4. Return `header.payload.signature` as base64url

**verifyBookingToken implementation:**
1. Split token into 3 parts — if not 3, return `'invalid'`
2. Try `fromBase64Url(sig)` — on exception, return `'invalid'`
3. `crypto.subtle.verify(...)` — if false, return `'invalid'`
4. Parse payload — on exception, return `'invalid'`
5. Check `claims.booking_id !== bookingId` or `claims.aud !== "booking:" + action` — return `'invalid'`
6. Check `claims.exp < Math.floor(Date.now() / 1000)` — return `'expired'`
7. Return `'valid'`

### Step 8 — `handlers/bookings/email-confirm.ts` (NEW FILE)

**File:** `workers/backend/api/src/handlers/bookings/email-confirm.ts`

This handler receives `GET /api/bookings/:id/email-confirm?token=...` from the prospect clicking the confirmation email link. It is NOT CORS-gated (direct browser navigation from email link). It returns HTTP redirects, not JSON.

```typescript
import { Env } from '../../types/env';
import { createSql } from '../../lib/db';
import { verifyBookingToken } from '../../lib/bookingToken';
import { getAccessToken, gcalFreebusy, gcalInsertEvent, GcalApiError } from '../../lib/gcalClient';

export async function handleEmailConfirm(
  request: Request,
  env: Env,
  bookingId: string,
  ctx: ExecutionContext,
): Promise<Response>
```

**Logic flow:**

1. Extract `token` from URL query params. If missing → redirect to expired page.

2. Verify token:
   ```typescript
   const result = await verifyBookingToken(token, bookingId, 'confirm', env.BOOKING_CONFIRM_SECRET);
   ```
   - `'invalid'` → redirect `https://{satelliteDomain}/booking-error` (or WORKER_BASE_URL fallback)
   - `'expired'` → redirect `https://{satelliteDomain}/booking-expired`
   - `'valid'` → proceed

3. Fetch booking: `SELECT id, expert_id, prospect_id, start_at, end_at, status, confirmation_token, prep_token, match_id, duration_min, prospect_name, prospect_email, description, prospect_id FROM bookings WHERE id = {bookingId}`

4. Check `status`:
   - `'confirmed'` → redirect to already-confirmed page: `https://{satelliteDomain}/booking-confirmed/{bookingId}?already=true`
   - NOT `'pending_confirmation'` → redirect to error page
   - `'pending_confirmation'` → proceed

5. Verify token matches stored `confirmation_token` (double-check — the HMAC already validates, but this ensures the token wasn't revoked by a resend):
   - If `booking.confirmation_token !== token` → redirect to expired/invalid page

6. Fetch expert: `SELECT gcal_email, gcal_access_token, gcal_token_expiry_at, display_name, booking_auto_confirm FROM experts WHERE id = {booking.expert_id}`

7. Get satellite domain for redirect:
   ```typescript
   const satelliteDomain = await getSatelliteDomain(sql, booking.prospect_id, env);
   // Falls back to new URL(env.WORKER_BASE_URL).hostname if not found
   ```

8. Get access token via `getAccessToken`. On failure → redirect to gcal error page (rare edge case).

9. GCal freebusy re-check (same as current confirm.ts). If slot taken → update booking status to `'expired_no_confirmation'`, redirect to error page.

10. Build event description from `booking.description`.

11. Create GCal event via `gcalInsertEvent` (moved from confirm.ts). On failure → redirect to gcal error page.

12. Determine next status based on `expert.booking_auto_confirm`:
    - `true` → `status = 'confirmed'`
    - `false` → `status = 'pending_expert_approval'`

13. Update booking:
    ```sql
    UPDATE bookings SET 
      status = {nextStatus},
      gcal_event_id = {gcalResult.eventId},
      meeting_url = {gcalResult.meetingUrl},
      confirmed_at = NOW(),
      confirmation_token = NULL  -- revoke token after use
    WHERE id = {bookingId}
    ```

14. If `booking_auto_confirm = true`:
    - Send queue messages: `LEAD_BILLING.send({ type: 'booking.created', ... })` and `EMAIL_NOTIFICATIONS.send({ type: 'booking.confirmed', ... })`
    - Generate `booking_prep_token` = booking.prep_token
    - Fire PostHog `booking.email_confirmed`
    - Redirect to: `https://{satelliteDomain}/booking-confirmed/{bookingId}?token={booking.prep_token}`

15. If `booking_auto_confirm = false`:
    - Send expert approval email (see below)
    - Fire PostHog `booking.email_confirmed` (prospect confirmed — expert still pending)
    - Redirect to: `https://{satelliteDomain}/booking-pending-approval` (expert must approve)

**Sending expert approval email (AC5):**
```typescript
const approveToken = await signBookingToken(bookingId, 'expert-approve', env.BOOKING_CONFIRM_SECRET, 86400);
const rejectToken = await signBookingToken(bookingId, 'expert-reject', env.BOOKING_CONFIRM_SECRET, 86400);
const approveUrl = `${env.WORKER_BASE_URL}/api/bookings/${bookingId}/expert-approve?token=${approveToken}`;
```

**Helper `getSatelliteDomain`:** Internal async helper that queries `prospects.satellite_id → satellite_configs.domain`. Returns the domain string or null (caller falls back to `new URL(env.WORKER_BASE_URL).hostname`).

### Step 11 — Resend endpoint (new route in `index.ts`)

The resend endpoint `POST /api/bookings/:id/confirmation/resend` is handled by a new inline function (too small for a separate file). It is CORS-gated (called from satellite widget's "Renvoyer l'email" button).

**Handler logic (can be in `confirm.ts` as a named export `handleResendConfirmation`):**

1. Rate limit check using `SESSIONS` KV: key = `confirm-resend-rate:{bookingId}`, max 3 per hour.
   - `const count = parseInt(await env.SESSIONS.get(key) || '0', 10)`
   - If `count >= 3` → return `json({ error: 'Too Many Requests' }, 429)`

2. Fetch booking: `SELECT id, prospect_email, expert_id, status, prospect_name FROM bookings WHERE id = {bookingId}`

3. Check `status === 'pending_confirmation'` → if not, return `json({ error: 'Booking is not pending confirmation' }, 409)`

4. Fetch expert `display_name`.

5. Generate fresh tokens (replaces old tokens — old tokens will now fail the `confirmation_token` DB check in email-confirm):
   ```typescript
   const token = await signBookingToken(bookingId, 'confirm', env.BOOKING_CONFIRM_SECRET, 1800);
   const cancelToken = await signBookingToken(bookingId, 'cancel', env.BOOKING_CONFIRM_SECRET, 1800);
   ```

6. Update `confirmation_token` in DB:
   ```sql
   UPDATE bookings SET confirmation_token = {token} WHERE id = {bookingId}
   ```

7. Send confirmation email.

8. Increment KV counter: `env.SESSIONS.put(key, String(count + 1), { expirationTtl: 3600 })`

9. Return `json({ sent: true })`.

```typescript
async function sendReminderEmail(
  recipientType: 'prospect' | 'expert',
  body: Extract<EmailNotificationMessage, { type: 'booking.reminder_prospect' | 'booking.reminder_expert' }>,
  env: Env,
): Promise<void> {
  const sql = createSql(env);
  
  const [booking] = await sql<{
    id: string;
    start_at: string;
    meeting_url: string | null;
    prospect_email: string | null;
    prospect_name: string | null;
    expert_id: string;
    reminder_j1_sent_at: string | null;
    reminder_h1_sent_at: string | null;
  }[]>`
    SELECT id, start_at, meeting_url, prospect_email, prospect_name, expert_id,
           reminder_j1_sent_at, reminder_h1_sent_at
    FROM bookings WHERE id = ${body.booking_id}`;
  
  if (!booking) throw new Error(`Booking not found: ${body.booking_id}`);
  
  const [expert] = await sql<{ gcal_email: string; display_name: string | null }[]>`
    SELECT gcal_email, display_name FROM experts WHERE id = ${booking.expert_id}`;
  
  if (!expert?.gcal_email) throw new Error(`Expert not found: ${booking.expert_id}`);

  // The cron sets reminder_j1_sent_at before sending the queue message,
  // so we determine type by checking which was JUST set (both null → j1 first)
  const isJ1 = booking.reminder_h1_sent_at === null;
  const reminderType: 'j1' | 'h1' = isJ1 ? 'j1' : 'h1';

    expertName: expert.display_name ?? 'Expert',
    prospectName: booking.prospect_name,
    startAt: booking.start_at,
    meetingUrl: booking.meeting_url,
    reminderType,
  });

    ? booking.prospect_email

  if (!recipientEmail) throw new Error(`No email for ${recipientType} on booking ${body.booking_id}`);

    distinctId: `booking:${body.booking_id}`,
    event: 'booking.reminder_sent',
    properties: { booking_id: body.booking_id, recipient_type: recipientType, reminder_type: reminderType },
  }).catch(() => {});
```

## Key Constraint Decisions

**Token storage in DB:** The `confirmation_token` column stores the raw JWT. This allows token revocation on resend (old token is replaced) and on use (set to NULL). The HMAC signature already prevents forgery — the DB copy provides revocation capability.

**Why `book.confirmation_token !== token` check in email-confirm.ts:** When a prospect resends the email, a new token is generated and stored. The old token URL (from the first email) becomes invalid because `confirmation_token` no longer matches, even though the HMAC is still mathematically valid. This prevents stale link confusion.

**Why 302 redirects from email-confirm/email-cancel:** These endpoints are opened by clicking email links in a browser. They must redirect to user-facing satellite pages, not return JSON. The satellite pages (`/booking-confirmed`, `/booking-cancelled`, `/booking-expired`) are static pages already rendered by the satellite worker.

**GCal freebusy in email-confirm:** The freebusy check is preserved (moved from confirm.ts). This maintains the race condition guard for the window between hold creation and email confirmation click.

**AC5 auto-confirm TTL:** Uses `confirmed_at` as the baseline for the 24h window (set at email-confirm time, not at hold time). The cron checks `confirmed_at < NOW() - 24h` on `pending_expert_approval` bookings.
