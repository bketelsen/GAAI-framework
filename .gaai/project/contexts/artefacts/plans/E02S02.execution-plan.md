# Execution Plan — E02S02: Expert Signup + Onboarding Wizard

## Approach Rationale

No approach evaluation needed — all patterns are established:
- Supabase Auth (signUp/signInWithPassword): established in E02S01 login.tsx
- React Router v7 form actions with SSR: established in E02S01
- Zod validation: established convention (conventions.md, E06S03 pattern)
- Multi-step wizard: URL search param `?step=N` navigation, established React Router pattern
- PostHog server-side capture: direct fetch to `https://eu.i.posthog.com/capture/`

Dependencies confirmed available:
- `GET /api/experts/:id/profile` → handlers/experts/profile.ts (E06S03)
- `PATCH /api/experts/:id/profile` → handlers/experts/profile.ts (E06S03)
- `POST /api/experts/register` → handlers/experts/register.ts (E06S03)
- `GET /api/experts/:id/gcal/auth-url` → E06S10
- `GET /api/experts/:id/gcal/status` → E06S10

---

### Phase 2: Routes

**Step 5** — `app/routes/signup.tsx` [AC1, AC3]
- Loader: check if already authenticated → redirect to /dashboard (or /onboarding if no display_name)
- Action: Zod validate email + password (min 8 chars) → supabase.auth.signUp → POST /api/experts/register → set session cookies → redirect to /onboarding?step=1
- Error handling: email taken → inline error; weak password → inline error; register API fail → inline error + recovery note
- PostHog: `expert.signup_completed` on success
- Component: email + password fields with inline validation messages

**Step 6** — `app/routes/login.tsx` (UPDATE) [AC2, AC3]
- Loader: already authenticated → check display_name (call GET /api/experts/:id/profile) → /onboarding or /dashboard
- Action: after signInWithPassword success → GET expert profile → check display_name → redirect to /onboarding?step=N or /dashboard
- Add "Mot de passe oublié ?" link → /forgot-password
- PostHog: `expert.login` on success
- Keep existing UI structure, add forgot-password link + validation errors

**Step 7** — `app/routes/forgot-password.tsx` [AC2]
- Action: supabase.auth.resetPasswordForEmail → success/error message
- Component: email field + submit → shows success message after submission

**Step 8** — `app/routes/onboarding.tsx` [AC4–AC12]
- Loader: requireSession → GET /api/experts/:id/profile → GET /api/experts/:id/gcal/status → parse ?step param (clamp 1–4) → return { profile, gcalStatus, step, user }
- Action: requireSession → parse formData → validate by step → PATCH /api/experts/:id/profile → PostHog per step → redirect to next step or /dashboard
- Component: 4-step wizard with progress indicator (1/4 dots), back/next nav, URL ?step= param
  - Step 1: display_name (req), headline (req, max 200), bio (opt, max 2000)
  - Step 2: skills (multi-select from PREDEFINED_SKILLS + custom, min 1, max 15), verticals (checkboxes), rate_min/rate_max, outcome_tags (freetext tags, max 10)
  - Step 3: career_stage (radio), work_mode (radio), availability (select), budget_range (range), project_stage (checkboxes), languages (multi-select), geo_zones (multi-select), industries (multi-select)
  - Step 4: GCal connection status + connect/skip buttons
- Form state preserved via useActionData() on error
- Zod client-side: validate on submit, show inline errors, disable submit if invalid
- AC9: After step 4 (complete or skip) → redirect /dashboard + success toast

**Step 9** — `app/routes/_layout.dashboard.tsx` (UPDATE) [AC10]
- After requireSession, call GET /api/experts/:id/profile
- If display_name null → inferOnboardingStep(profile) → redirect /onboarding?step=N
- Pass expert profile to outlet context (for use by sub-routes)

**Step 10** — `app/routes/routes.ts` (UPDATE)
- Add signup, forgot-password to unauthenticated block
- Add onboarding inside layout(_layout.tsx) but outside _layout.dashboard.tsx

## Edge Cases

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC1 | signUp success + register fails | Show error, Supabase user exists — next login: auth layout detects missing expert record + retries register |
| AC3 | Authenticated user visits /signup or /login | Loader redirect to /dashboard or /onboarding |
| AC4 | Step URL param out of range (0, 5, NaN) | Clamp to 1–4 in loader |
| AC8 | GCal already connected | Show connected email + "Passer cette étape" |
| AC10 | Expert record exists but no display_name | All /dashboard/* routes redirect to /onboarding?step=1 |
| AC10 | Expert record does not exist (register race) | Treat same as no display_name → /onboarding?step=1 |
| AC11 | Zod schema mismatch between client + server | Same schemas in same file (both contexts use the same zod module) |
| AC12 | Server validation error | Action returns submitted values + errors, component pre-fills from useActionData() |

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|------------|
| posthog direct fetch EU endpoint unreachable | Low | try/catch wraps capture, non-blocking |
| GET /api/experts/:id/gcal/status errors in onboarding loader | Low | .catch(() => null) — step 4 shows disconnected state |
| display_name check in dashboard layout causes N+1 API calls | Low | Profile cached in loader, one call per dashboard page load |
| zod client-side hydration mismatch | Low | Single file — same schema used server + client side |
