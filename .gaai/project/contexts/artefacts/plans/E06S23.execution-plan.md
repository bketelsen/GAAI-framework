## Overview

This plan adds a two-layer edge caching strategy for the expert pool:
1. **Cache API L1** (in-process, per-datacenter, 60s TTL) — fastest possible read, ~0ms
2. **D1 edge database** (`expert-pool-edge`) — persistent edge store, <5ms reads
3. **Hyperdrive fallback** — existing postgres.js path used when D1 is empty or unavailable

A `syncExpertPoolToD1` cron job (every 5 min) keeps D1 in sync with Supabase. The first run does a full load; subsequent runs are incremental (`WHERE updated_at > last_sync_at`). All sync failures are non-fatal.

## Architecture: 3-Tier Read Chain

```
Cache API L1 (60s TTL, per-datacenter)
        ↓ miss
D1 edge database (expert-pool-edge)
        ↓ miss or EXPERT_DB undefined
Hyperdrive (Supabase PostgreSQL)
        ↓ result
Write-back to Cache API + D1
```

- Cache API uses synthetic URL `https://cache.internal/expert-pool` (silently fails on workers.dev)
- D1 is optional (`EXPERT_DB?`) — graceful fallback to Hyperdrive when not provisioned
- Write-back is non-blocking: failures are caught and swallowed

---

## Cron Sync Strategy

```
*/5 * * * * cron
  → getLastSyncAt(EXPERT_DB)
  → null? → full load (WHERE availability != 'unavailable')
  → ts?   → incremental (WHERE availability != 'unavailable' AND updated_at > ts)
  → aggregate lead counts via JOIN
  → upsertToD1 (INSERT OR REPLACE batch)
  → setLastSyncAt
  → catch: log error, do not propagate (AC12)
```
