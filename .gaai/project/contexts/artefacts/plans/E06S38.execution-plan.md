# E06S38 Execution Plan — Dashboard Read API Endpoints

## Story Summary
7 new API endpoints for the expert dashboard (6 GET + 1 POST):
1. GET /api/experts/:id/leads
2. GET /api/experts/:id/bookings
3. GET /api/experts/:id/billing
4. GET /api/experts/:id/dashboard
5. GET /api/experts/public
6. GET /api/experts/public/:slug
7. POST /api/leads/:id/evaluate

## Key Findings from Codebase Analysis

### Column naming discrepancies (story vs DB)
- Story says `starts_at`/`ends_at` → DB uses `start_at`/`end_at`
- Story says `amount_cents` → DB uses `amount` (integer cents). Alias in SQL: `amount AS price_cents`
- Story says `cancel_reason` on bookings → MISSING, add via migration
- Story says `evaluation_score`, `evaluation_notes`, `conversion_declared`, `evaluated_at` on leads → MISSING, add via migration
- `flag_window_expires_at` → not stored, compute as `created_at + interval '7 days' AS flag_window_expires_at`

### Tier thresholds (from routes/matches.ts)
```ts
if (!compositeScore || compositeScore === 0) return 'new';
if (compositeScore <= 25) return 'rising';
if (compositeScore <= 60) return 'established';
return 'top';
```
Extract `deriveQualityTier` from `routes/matches.ts` into `lib/qualityTier.ts`.

### Profile JSONB fields (from routes/matches.ts)
- `skills` → `profile.skills ?? []`
- `industries` → `profile.industries ?? []`
- `languages` → `profile.languages ?? []`
- `completed_projects` → not directly mapped in existing code; use `profile.completed_projects ?? 0`

### Join strategy
- Leads endpoint: LEFT JOIN prospects ON prospects.id = leads.prospect_id; LEFT JOIN bookings ON bookings.id = leads.booking_id; LEFT JOIN matches ON matches.expert_id = leads.expert_id AND matches.prospect_id = leads.prospect_id (no match_id on leads)
- Bookings endpoint: use denormalized prospect_email + prospect_name already on bookings table + prospect_id
- Billing: separate queries — expert row (credit_balance, ls info, limits) + monthly_spend aggregate + transactions page

### Public endpoints
- `slug = 'exp-' || LEFT(id::text, 8)`
- No auth. Apply rate limiting via `env.RATE_LIMITER.limit()` before processing
- For `/public/:slug`, extract expert_id from slug: `id::text LIKE slug_prefix || '%'` — better: store first 8 chars, or compute: `id::text LIKE (SUBSTRING(slug FROM 5) || '%')`

### Auth pattern
New expert endpoints receive `(request, env, user, expertId)` — the ownership check `if (user.id !== expertId) return forbidden()` is done INSIDE each handler (same as `handleGetProfile`). The authenticate() is done in the router.

### Step 4: handlers/experts/leads.ts

Function signatures:
```ts
export async function handleGetLeads(request, env, user, expertId): Promise<Response>
export async function handleEvaluateLead(request, env, user, leadId): Promise<Response>
```

**handleGetLeads query strategy (single JOIN query):**
```sql
SELECT
  l.id,
  l.status,
  l.amount AS price_cents,
  l.created_at,
  l.confirmed_at,
  l.flagged_at,
  l.flag_reason,
  (l.created_at + interval '7 days') AS flag_window_expires_at,
  l.evaluation_score,
  l.evaluation_notes,
  l.conversion_declared,
  l.evaluated_at,
  p.id AS prospect_id,
  p.email AS prospect_email,
  p.requirements AS prospect_requirements,
  m.score AS match_score,
  b.id AS booking_id,
  b.start_at AS booking_starts_at,
  b.status AS booking_status
FROM leads l
LEFT JOIN prospects p ON p.id = l.prospect_id
LEFT JOIN matches m ON m.expert_id = l.expert_id AND m.prospect_id = l.prospect_id
LEFT JOIN bookings b ON b.id = l.booking_id
WHERE l.expert_id = ${expertId}
  AND (${status} = 'all' OR l.status = ${status})
ORDER BY l.created_at DESC
LIMIT ${per_page} OFFSET ${offset}
```

Separate COUNT query:
```sql
SELECT COUNT(*) FROM leads WHERE expert_id = ${expertId} AND (${status} = 'all' OR status = ${status})
```

Response shape:
```json
{
  "leads": [{
    "id": "...",
    "status": "...",
    "price_cents": 8900,
    "created_at": "...",
    "confirmed_at": null,
    "flagged_at": null,
    "flag_reason": null,
    "flag_window_expires_at": "...",
    "evaluation_score": null,
    "evaluation_notes": null,
    "conversion_declared": false,
    "evaluated_at": null,
    "prospect": { "id": "...", "email": "...", "requirements": {} },
    "match_score": 0.85,
    "booking": { "id": "...", "starts_at": "...", "status": "confirmed" }
  }],
  "total": 42,
  "page": 1,
  "per_page": 20
}
```

Validation: `per_page` max 50, default 20. `status` enum: new|confirmed|flagged|all (default: all). `page` min 1.

**handleEvaluateLead:**
- Ownership check: fetch lead → verify `expert_id = user.id` → 403 if not
- Status check: lead.status must be 'confirmed' → 400 if not
- Already evaluated: `evaluation_score IS NOT NULL` → 409
- Zod schema: `{ score: z.number().int().min(1).max(10), notes: z.string().max(500).optional(), conversion_declared: z.boolean().optional() }`
- Update: `UPDATE leads SET evaluation_score=${score}, evaluation_notes=${notes}, conversion_declared=${conversion_declared}, evaluated_at=NOW() WHERE id=${leadId} RETURNING *`
- PostHog: `expert.lead_evaluated` with `{ lead_id, score, conversion_declared }`

### Step 6: handlers/experts/billing.ts


Query 2 — monthly spend aggregate:
```sql
SELECT COALESCE(SUM(amount), 0) AS monthly_spend

WHERE expert_id = ${expertId}
```

```sql
FROM credit_transactions
WHERE expert_id = ${expertId}
```

Query 4 — transaction count:
```sql
SELECT COUNT(*) FROM credit_transactions WHERE expert_id = ${expertId}
```

Response shape matches AC10.

### Step 7: handlers/experts/dashboard.ts

Function signature:
```ts
export async function handleGetDashboard(request, env, user, expertId): Promise<Response>
```

**Queries:**

Query 1 — expert base data:
```sql
SELECT credit_balance, composite_score FROM experts WHERE id = ${expertId}
```

Query 2 — unread leads count:
```sql
SELECT COUNT(*) FROM leads WHERE expert_id = ${expertId} AND status = 'new'
```

Query 3 — upcoming bookings count:
```sql
SELECT COUNT(*) FROM bookings WHERE expert_id = ${expertId} AND start_at > NOW() AND status NOT IN ('cancelled')
```

Query 4 — current month stats (single query):
```sql
SELECT
  COUNT(*) FILTER (WHERE true) AS leads_received,
  COUNT(*) FILTER (WHERE status = 'confirmed') AS leads_confirmed,
  COUNT(*) FILTER (WHERE status = 'flagged') AS leads_flagged,
  COUNT(*) FILTER (WHERE conversion_declared = true) AS conversions_declared
FROM leads
WHERE expert_id = ${expertId}
  AND created_at >= date_trunc('month', NOW())
```

Query 5 — bookings this month:
```sql
SELECT
  COUNT(*) AS bookings_total,
  COUNT(*) FILTER (WHERE status = 'completed') AS bookings_completed
FROM bookings
WHERE expert_id = ${expertId}
  AND start_at >= date_trunc('month', NOW())
```

Query 6 — monthly history (last 6 months, leads):
```sql
SELECT
  TO_CHAR(date_trunc('month', created_at), 'YYYY-MM') AS month,
  COUNT(*) AS leads_received,
  COUNT(*) FILTER (WHERE status = 'confirmed') AS leads_confirmed,
  COUNT(*) FILTER (WHERE status = 'flagged') AS leads_flagged,
  COUNT(*) FILTER (WHERE conversion_declared = true) AS conversions_declared
FROM leads
WHERE expert_id = ${expertId}
  AND created_at >= date_trunc('month', NOW()) - interval '5 months'
GROUP BY 1
```

Query 7 — monthly history (bookings):
```sql
SELECT
  TO_CHAR(date_trunc('month', start_at), 'YYYY-MM') AS month,
  COUNT(*) AS bookings_total,
  COUNT(*) FILTER (WHERE status = 'completed') AS bookings_completed
FROM bookings
WHERE expert_id = ${expertId}
  AND start_at >= date_trunc('month', NOW()) - interval '5 months'
GROUP BY 1
```

Merge the two monthly history results in TypeScript, filling missing months with zeros.

## Risks & Edge Cases

1. **matches JOIN on expert_id + prospect_id**: Multiple matches may exist for same pair. Use `LIMIT 1` in a subquery or take the latest match score.
2. **monthly_history empty months**: Must fill months with 0 values for months with no data — do in TypeScript after SQL aggregation.
3. **Public rate limiting**: `env.RATE_LIMITER.limit()` requires a `key` — use `request.headers.get('CF-Connecting-IP') ?? 'unknown'`.
4. **Type `QualityTier`**: Already defined in `types/matching.ts`. Import from there or from new `lib/qualityTier.ts`.
5. **`evaluation_score` null check**: In handleEvaluateLead, check `lead.evaluation_score !== null` for 409. After migration, column defaults to NULL.
6. **`per_page` clamp**: `Math.min(parseInt(per_page_param) || 20, 50)`.
7. **`matches` table JOIN**: Consider that a match may not exist for a lead (if lead was created via a different flow). LEFT JOIN handles this.
