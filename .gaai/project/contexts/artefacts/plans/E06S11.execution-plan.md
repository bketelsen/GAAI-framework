# E06S11 — Booking Engine: Execution Plan

## Scope Summary

Headless booking engine replacing Cal.com. Touches: DB schema, TypeScript types, queue message types, GCal API library, availability computation library, 6 handler files, route registration in `src/index.ts`, and cron trigger declarations in `wrangler.toml`. All new booking routes are CORS-gated (satellite funnel consumption). No new `Env` bindings required — `EMAIL_NOTIFICATIONS` and `LEAD_BILLING` queues already exist.

---

## Schema Gap Resolution (Critical Pre-Condition)

The current `bookings` table lacks `expert_id`, `prospect_id`, `start_at`, `end_at`. AC3 requires `POST /api/bookings/hold` to accept `{ expert_id, start_at, end_at, prospect_id }` and AC1 requires querying bookings by `expert_id`. These columns are not listed in AC11 but are functionally required.

**Decision:** Include `expert_id uuid REFERENCES experts(id)`, `prospect_id uuid REFERENCES prospects(id)`, `start_at timestamptz`, `end_at timestamptz` in the migration as necessary implicit additions. The `match_id` FK remains (populated at hold time via `matches WHERE expert_id=X AND prospect_id=Y` lookup). Also: the existing `bookings_status_check` constraint only allows `('confirmed','cancelled','completed','no_show')` — must DROP and recreate to add `'held'`.

## Implementation Sequence

### Step 1 — DB Migration

**File:** `supabase/migrations/20260222000001_booking_engine.sql` (in worktree)
**Applied via:** Supabase MCP `apply_migration`

```sql
-- ── Extend experts table ──────────────────────────────────────────────────────
ALTER TABLE experts
  ADD COLUMN IF NOT EXISTS availability_rules  JSONB,
  ADD COLUMN IF NOT EXISTS reminder_settings   JSONB DEFAULT '{"enabled": true}';

-- ── Extend bookings table ─────────────────────────────────────────────────────
ALTER TABLE bookings
  -- Denormalized FKs (availability queries, hold creation)
  ADD COLUMN IF NOT EXISTS expert_id           UUID REFERENCES experts(id),
  ADD COLUMN IF NOT EXISTS prospect_id         UUID REFERENCES prospects(id),
  -- Explicit time range (replaces scheduled_at for new flow; scheduled_at kept)
  ADD COLUMN IF NOT EXISTS start_at            TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS end_at              TIMESTAMPTZ,
  -- Hold TTL (AC3, AC9)
  ADD COLUMN IF NOT EXISTS held_until          TIMESTAMPTZ,
  -- GCal integration (AC4, AC7)
  ADD COLUMN IF NOT EXISTS gcal_event_id       TEXT,
  ADD COLUMN IF NOT EXISTS meeting_url         TEXT,
  -- Prospect identity at confirm time (AC4)
  ADD COLUMN IF NOT EXISTS prospect_name       TEXT,
  ADD COLUMN IF NOT EXISTS prospect_email      TEXT,
  ADD COLUMN IF NOT EXISTS description         TEXT,
  ADD COLUMN IF NOT EXISTS confirmed_at        TIMESTAMPTZ,
  -- Reminder idempotency (AC12)
  ADD COLUMN IF NOT EXISTS reminder_j1_sent_at TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS reminder_h1_sent_at TIMESTAMPTZ,
  -- Prep page token (AC13)
  ADD COLUMN IF NOT EXISTS prep_token          TEXT;

-- ── Extend bookings status constraint to include 'held' ───────────────────────
ALTER TABLE bookings DROP CONSTRAINT IF EXISTS bookings_status_check;
ALTER TABLE bookings ADD CONSTRAINT bookings_status_check
  CHECK (status IN ('held', 'confirmed', 'cancelled', 'completed', 'no_show'));

-- ── Performance indexes ───────────────────────────────────────────────────────
CREATE INDEX IF NOT EXISTS bookings_expert_status_idx
  ON bookings (expert_id, status)
  WHERE status IN ('held', 'confirmed');

CREATE INDEX IF NOT EXISTS bookings_start_at_idx ON bookings (start_at);

CREATE INDEX IF NOT EXISTS bookings_reminder_idx
  ON bookings (start_at, reminder_j1_sent_at, reminder_h1_sent_at)
  WHERE status = 'confirmed';
```

### Step 4 — GCal API Utility Library

**New file:** `src/lib/gcalClient.ts`

Exports:
- `getAccessToken(expertId, env)` — reads `gcal_access_token` from DB (decrypt via `gcalCrypto`), checks `gcal_token_expiry_at`; calls `refreshGcalToken` only if expired. Returns plaintext token.
- `withGcalRetry<T>(fn: (token: string) => Promise<T>, expertId, env)` — internal retry wrapper: 429 → exponential backoff (1s/4s/16s, max 3 retries); 401 → `refreshGcalToken` + retry once; 404 on delete → success; other errors → throw `GcalApiError`.
- `gcalFreebusy(token, calendarId, timeMin, timeMax)` → `Array<{start: string, end: string}>`
- `gcalInsertEvent(token, calendarId, event, conferenceDataVersion)` → `{ eventId, meetingUrl }`
- `gcalDeleteEvent(token, calendarId, eventId)` → `void` (404 treated as success)
- `gcalPatchEvent(token, calendarId, eventId, patch)` → `{ eventId, meetingUrl }`

`sleep(ms)` internal: `new Promise(resolve => setTimeout(resolve, ms))`

All functions call Google Calendar REST API via raw `fetch` (no SDK — CF Workers only). `withGcalRetry` wraps all public functions.

### Step 5 — Availability Computation Library

**New file:** `src/lib/availability.ts`

Pure computation module — no DB/GCal calls. Enables clean unit testing.

Exports:
- `AvailabilityRules` interface: `{ working_hours: Record<string, [string, string]>; buffer_minutes: number; min_notice_hours: number; booking_window_days: number }`
- `DEFAULT_AVAILABILITY_RULES: AvailabilityRules` — mon-fri 09:00-18:00, buffer 15, notice 4h, window 14 days
- `mergeRules(base: AvailabilityRules, overrides?: Partial<AvailabilityRules> | null)` — merge expert's stored rules with defaults
- `computeFreeSlots(params)` — pure slot computation:
  1. Combine busy intervals (freebusy + held/confirmed bookings)
  2. For each day in [windowStart, windowEnd]: skip weekends (not in working_hours)
  3. Get working start/end times for the day
  4. Enumerate 20-min candidate slots with `buffer_minutes` gap between them
  5. Filter: exclude slots starting before `now + min_notice_hours`, exclude slots overlapping any busy interval (buffer applied around each busy interval)
  6. Return `Array<{ start_at: string; end_at: string }>` (UTC ISO strings)

### Step 6 — Booking Handler Files

**New directory:** `src/handlers/bookings/`

#### `availability.ts` — AC1, AC2

`handleGetAvailability(request, env, expertId)`:
1. Fetch expert (`gcal_access_token`, `gcal_token_expiry_at`, `gcal_email`, `availability_rules`, `gcal_connected`) — 404 if not found
2. Return 422 `{ error: "gcal_not_connected" }` if not connected
3. Get access token via `getAccessToken(expertId, env)`
4. Compute `[timeMin, timeMax]` from now to `now + booking_window_days * 24h`
5. Call `gcalFreebusy(token, expert.gcal_email, timeMin, timeMax)` with `withGcalRetry`
6. Query DB: `bookings WHERE expert_id=X AND status IN ('held','confirmed') AND start_at < timeMax AND end_at > timeMin`
7. Call `computeFreeSlots(...)` with merged rules
8. Return `{ slots, metadata: { tz: url.searchParams.get('tz') ?? 'UTC', generated_at } }`

#### `hold.ts` — AC3

`handleHold(request, env)`:
1. Parse + validate `{ expert_id, start_at, end_at, prospect_id }` — 422 on missing
2. DB conflict check — 409 if held/confirmed booking overlaps slot
3. Lookup `match_id` from `matches WHERE expert_id=X AND prospect_id=Y ORDER BY created_at DESC LIMIT 1` (nullable)
4. Generate `prep_token = crypto.randomUUID()`
5. Insert booking: `{ expert_id, prospect_id, match_id, start_at, end_at, status: 'held', held_until: now+10min, duration_min: 20, prep_token }`
6. Return `{ booking_id, held_until }`

#### `confirm.ts` — AC4, AC5, AC6

`handleConfirm(request, env, bookingId)`:
1. Fetch booking (status=`held`, held_until > now) — 404/410 if not found or expired
2. Parse + validate `{ prospect_name, prospect_email, description? }`
3. Fetch expert (gcal_email, gcal_access_token, gcal_token_expiry_at)
4. Get access token
5. **AC6:** Freebusy re-check on `[booking.start_at, booking.end_at]` — if conflict: DELETE booking row → return 409 `{ error: "slot_taken" }`
6. Fetch prospect (`requirements`) for event description; query `satellite_configs` for domain (non-blocking, fallback to `env.WORKER_BASE_URL`)
7. Build event description (DEC-43 format) with prep URL: `{satellite_domain}/prep/{booking.prep_token}`
8. Call `gcalInsertEvent(token, 'primary', { start, end, attendees, description, conferenceDataVersion: 1 }, env)` with `withGcalRetry`
9. Update booking: `status='confirmed'`, `gcal_event_id`, `meeting_url`, `prospect_name`, `prospect_email`, `description`, `confirmed_at=now()`
10. Push `booking.created` → `LEAD_BILLING`; `booking.confirmed` → `EMAIL_NOTIFICATIONS`
11. Return `{ booking_id, meeting_url, start_at, end_at, expert_gcal_event_id }`

#### `cancel.ts` — AC7

`handleCancel(request, env, bookingId)`:
1. Fetch booking — 404 if not found
2. If `gcal_event_id` present: fetch expert token → `gcalDeleteEvent(token, 'primary', gcal_event_id)` with `withGcalRetry` (404 = success)
3. Update booking: `status='cancelled'`
4. Push `{ type: 'booking.cancelled', booking_id }` → `EMAIL_NOTIFICATIONS`
5. Return 200 `{ cancelled: true }`

Auth: CORS-gated, unauthenticated (MVP). Expert cancellation from dashboard = post-MVP.

#### `reschedule.ts` — AC8

`handleReschedule(request, env, bookingId)`:
1. Fetch booking (must be `status='confirmed'`) — 404 if not found
2. Parse `{ new_start_at, new_end_at }` — 422 on missing
3. DB conflict check (exclude self: `id != bookingId`) — 409 if conflict
4. GCal freebusy re-check on new slot — 409 if busy
5. `gcalPatchEvent(token, 'primary', gcal_event_id, { start: new_start_at, end: new_end_at })`
6. Update booking: `start_at=new_start_at`, `end_at=new_end_at`, `scheduled_at=new_start_at`
7. Push `{ type: 'booking.rescheduled', booking_id }` → `EMAIL_NOTIFICATIONS`
8. Return `{ booking_id, meeting_url, new_start_at, new_end_at }`

#### `prep.ts` — AC13

`handleGetPrep(request, env, prepToken)`:
1. Query `bookings WHERE prep_token=X` with JOINs (or separate selects for `prospect_id`, `expert_id`, `match_id`, `start_at`, `end_at`, `meeting_url`, `duration_min`)
2. 404 if not found
3. Check expiry: `now > start_at + 2h` → 404 `{ error: "prep_token_expired" }`
4. Fetch prospect: `requirements`, `first_name` (from profile or requirements)
5. Fetch expert: `display_name`, `profile.bio_short` (or `bio`), `composite_score`, `profile.skills`
6. Fetch match (by `match_id` or query `matches WHERE expert_id=X AND prospect_id=Y LIMIT 1`): `score`, `score_breakdown`
7. Compute tier: `score < 50 → "rising"`, `50-75 → "established"`, `> 75 → "top"`
8. Build `direction_expert` and `direction_prospect` from `score_breakdown` labels (string concat of top 2 criteria — no AI call)
9. Return full AC13 payload

#### `cron.ts` — AC9, AC12

`handleScheduled(event: ScheduledEvent, env: Env)`:
- Dispatch by `event.cron`:
  - `*/5 * * * *` → `cleanupExpiredHolds(env)`: `DELETE FROM bookings WHERE status='held' AND held_until < now()`
  - `*/15 * * * *` → `dispatchReminders(env)`:
    - J-1: query `bookings WHERE status='confirmed' AND start_at BETWEEN now()+23h AND now()+25h AND reminder_j1_sent_at IS NULL`; for each: push `booking.reminder_prospect`; if `experts.reminder_settings.enabled = true` push `booking.reminder_expert`; update `reminder_j1_sent_at = now()`
    - H-1: same with `start_at BETWEEN now()+50min AND now()+70min AND reminder_h1_sent_at IS NULL` → update `reminder_h1_sent_at`

// DELETE /api/bookings/:id
// POST /api/bookings/:id/reschedule

Pattern matching approach (consistent with existing code):
```typescript
const expertAvailMatch = pathname.match(/^\/api\/experts\/([^/]+)\/availability$/);
if (method === 'GET' && expertAvailMatch) { ... CORS + handleGetAvailability ... }

// /api/bookings/* routing block
if (pathname.startsWith('/api/bookings/')) {
  // GET /api/bookings/:token/prep — public
  const prepMatch = pathname.match(/^\/api\/bookings\/([^/]+)\/prep$/);

  // CORS check for remaining booking routes

  if (corsResult.preflight) return corsResult.preflight;
  if (!corsResult.allowed) return corsForbidden(corsResult.origin);

  if (method === 'POST' && pathname === '/api/bookings/hold') { ... }
  const bookingIdMatch = pathname.match(/^\/api\/bookings\/([^/]+)\//);
  ...
}
```

## Risk Register

| Risk | Mitigation |
|---|---|
| `bookings_status_check` constraint name may differ in actual DB | Use `DROP CONSTRAINT IF EXISTS` — safe if name differs |
| Expert has `gcal_connected=true` but token decryption fails | `getAccessToken` wraps decrypt in try/catch → 502 |
| Race condition: two prospects hold same slot simultaneously | DB overlap check is non-atomic; hold INSERT proceeds, subsequent hold query catches it on next request. For MVP this is acceptable. |
| Google Meet link reuse between bookings | `conferenceDataVersion=1` with unique `requestId` per booking ensures a fresh Meet link per event — never reuse conference data |
| `prep` endpoint token brute-force | UUID is 122 bits of entropy — brute force infeasible at MVP scale |
| `satellite_configs` lookup fails in confirm (non-blocking) | Wrap in try/catch, fallback to `env.WORKER_BASE_URL/prep/...` |
| GCal API rate limits on reminder cron dispatch | Reminder cron fires every 15 min; at MVP scale, 1-10 bookings max — no concern. Add batch limiting post-scale. |

## Rollback Boundary

If QA fails after migration: the DB migration is applied to Supabase production. Columns can be dropped with `ALTER TABLE bookings DROP COLUMN ...` and `ALTER TABLE experts DROP COLUMN ...`. The status constraint can be reverted. No data loss risk since new columns are all nullable (except `reminder_settings` which has a DEFAULT). The old `bookings_status_check` only needs the original values re-added.

Code rollback: story branch — no merge to production until QA PASS.
