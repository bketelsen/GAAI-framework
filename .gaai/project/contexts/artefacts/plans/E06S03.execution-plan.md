## Supabase RPC — JSONB Merge Strategy

**Decision: Use a Supabase RPC stored procedure.**

Rationale: Inline SQL via `.rpc()` with raw SQL strings is not possible with the typed Supabase JS client without casting. A stored procedure is clean, typed, and reusable.

**Function name:** `merge_expert_profile`

**SQL to apply via `mcp__supabase__apply_migration`** (migration name: `add_merge_expert_profile_rpc`):

```sql
CREATE OR REPLACE FUNCTION public.merge_expert_profile(
  p_id         uuid,
  p_display_name text DEFAULT NULL,
  p_headline   text DEFAULT NULL,
  p_bio        text DEFAULT NULL,
  p_rate_min   numeric DEFAULT NULL,
  p_rate_max   numeric DEFAULT NULL,
  p_availability text DEFAULT NULL,
  p_profile    jsonb DEFAULT NULL,
  p_preferences jsonb DEFAULT NULL
)
RETURNS SETOF experts
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  UPDATE public.experts
  SET
    display_name  = COALESCE(p_display_name, display_name),
    headline      = COALESCE(p_headline, headline),
    bio           = COALESCE(p_bio, bio),
    rate_min      = COALESCE(p_rate_min, rate_min),
    rate_max      = COALESCE(p_rate_max, rate_max),
    availability  = COALESCE(p_availability, availability),
    profile       = CASE WHEN p_profile IS NOT NULL THEN COALESCE(profile, '{}'::jsonb) || p_profile ELSE profile END,
    preferences   = CASE WHEN p_preferences IS NOT NULL THEN COALESCE(preferences, '{}'::jsonb) || p_preferences ELSE preferences END
  WHERE id = p_id;

  RETURN QUERY SELECT * FROM public.experts WHERE id = p_id;
END;
$$;
```

**Why `SECURITY DEFINER`:** The Worker uses the service role key, which bypasses RLS regardless — but using `SECURITY DEFINER` ensures the function runs with owner privileges even if the call context changes in future. The function is limited by its `WHERE id = p_id` clause so there is no privilege escalation risk.

**`database.ts` update required after migration:** The `Functions` block in `src/types/database.ts` must be updated to declare `merge_expert_profile` so the typed client can call `.rpc('merge_expert_profile', {...})` without type errors. This is done manually as part of Step 2 (the Supabase type generator may or may not pick up function signatures — manual addition is more reliable for RPC functions).

### Step 5 — Create `src/lib/rateLimit.ts`

**File:** `/Users/fred/Dropbox/Development/MicroSaaS-Projects/The-AI-Integrator-Network/src/lib/rateLimit.ts`

**Purpose:** Rate limiting helper using `RATE_LIMITING` KV namespace.

**Algorithm:**
1. Construct key: `rate_limit:register:{ip}` where `ip` is extracted from `request.headers.get('CF-Connecting-IP') ?? 'unknown'`.
2. GET current count from KV.
3. If count exists and `parseInt(count) >= 10`: return `{ allowed: false }`.
4. If count does not exist: PUT `"1"` with `expirationTtl: 60`.
5. If count exists and `< 10`: PUT `String(parseInt(count) + 1)` with `expirationTtl: 60`.
6. Return `{ allowed: true }`.

**Content:**

```typescript
import { Env } from '../types/env';

const WINDOW_SECONDS = 60;
const MAX_REQUESTS = 10;

export async function checkRateLimit(
  request: Request,
  env: Env
): Promise<{ allowed: boolean }> {
  const ip = request.headers.get('CF-Connecting-IP') ?? 'unknown';
  const key = `rate_limit:register:${ip}`;

  const current = await env.RATE_LIMITING.get(key);

  if (current !== null && parseInt(current, 10) >= MAX_REQUESTS) {
    return { allowed: false };
  }

  const next = current === null ? 1 : parseInt(current, 10) + 1;
  await env.RATE_LIMITING.put(key, String(next), {
    expirationTtl: WINDOW_SECONDS,
  });

  return { allowed: true };
}
```

**Edge case:** `CF-Connecting-IP` is guaranteed present in deployed Workers. In `wrangler dev`, it may be absent — fallback to `'unknown'` causes all local dev requests to share one rate limit bucket, which is acceptable for development.

**Edge case:** KV `put` with `expirationTtl` resets the TTL on every write. This means the window is a sliding 60s from the last request, not a fixed 60s tumbling window. This is acceptable behaviour for anti-abuse rate limiting and matches the AC wording ("per 60 seconds").

### Step 6 — Create `src/middleware/auth.ts`

**File:** `/Users/fred/Dropbox/Development/MicroSaaS-Projects/The-AI-Integrator-Network/src/middleware/auth.ts`

**Purpose:** JWT validation middleware. Validates Supabase JWT and extracts the authenticated user. Returns either the user object or a `401` Response to short-circuit the handler.

**Content:**

```typescript
import { Env } from '../types/env';
import { createServiceClient } from '../lib/supabase';

export type AuthUser = {
  id: string;
  email: string | undefined;
};

export type AuthResult =
  | { user: AuthUser; response?: never }
  | { user?: never; response: Response };

const UNAUTHORIZED = (): Response =>
  new Response(JSON.stringify({ error: 'Unauthorized' }), {
    status: 401,
    headers: { 'Content-Type': 'application/json' },
  });

export async function authenticate(
  request: Request,
  env: Env
): Promise<AuthResult> {
  const authHeader = request.headers.get('Authorization');

  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return { response: UNAUTHORIZED() };
  }

  const token = authHeader.slice(7); // Remove "Bearer " prefix

  const supabase = createServiceClient(env);
  const { data, error } = await supabase.auth.getUser(token);

  if (error || !data.user) {
    return { response: UNAUTHORIZED() };
  }

  return {
    user: {
      id: data.user.id,
      email: data.user.email,
    },
  };
}
```

**Notes:**
- `supabase.auth.getUser(token)` makes a network call to Supabase Auth to validate the JWT — this is the authoritative validation method. It validates signature, expiry, and revocation state.
- `AuthResult` is a discriminated union. Callers check `if ('response' in result)` to short-circuit.
- The `email` field from `data.user.email` is used in AC5 (queue message) — it is only available at this point because the experts table has no email column.

### Step 7 — Create `src/handlers/experts/register.ts`

**File:** `/Users/fred/Dropbox/Development/MicroSaaS-Projects/The-AI-Integrator-Network/src/handlers/experts/register.ts`

**Purpose:** Handles `POST /api/experts/register`. Creates expert row, enqueues notification.

**Zod schema:**

```typescript
import { z } from 'zod';

const RegisterSchema = z.object({
  display_name: z.string().min(1).max(100),
  headline: z.string().max(200).optional(),
  bio: z.string().max(2000).optional(),
  rate_min: z.number().int().positive().optional(),
  rate_max: z.number().int().positive().optional(),
});
```

**Note on the `email` field in AC2:** The story AC2 states the body accepts `{ email, display_name, ... }`. However, the `experts` table has no `email` column, and the authenticated JWT already identifies the user. The `email` field in the request body is therefore **accepted but ignored** — the email for the queue message is sourced from `user.email` (from the JWT via `authenticate()`). The Zod schema does NOT include `email` as a required field to avoid confusion. If the body contains `email`, it is silently discarded. The `display_name` field IS required in the Zod schema.

**AC4 — duplicate detection strategy:** The `experts` table has a UNIQUE constraint on `id` (the PK), which is the Supabase Auth UUID. If the same authenticated user calls register twice, the INSERT will fail with a Postgres unique violation (`23505` error code). Check for this error and return `409`. Do NOT perform a pre-flight SELECT — use the DB constraint as the authoritative source.

**Full handler logic:**

```typescript
import { z } from 'zod';
import { Env } from '../../types/env';
import { AuthUser } from '../../middleware/auth';
import { createServiceClient } from '../../lib/supabase';
import { checkRateLimit } from '../../lib/rateLimit';

const RegisterSchema = z.object({
  display_name: z.string().min(1, 'display_name is required').max(100),
  headline: z.string().max(200).optional(),
  bio: z.string().max(2000).optional(),
  rate_min: z.number().int().positive().optional(),
  rate_max: z.number().int().positive().optional(),
});

export async function handleRegister(
  request: Request,
  env: Env,
  user: AuthUser
): Promise<Response> {
  // AC6: Rate limiting
  const rateCheck = await checkRateLimit(request, env);
  if (!rateCheck.allowed) {
    return new Response(JSON.stringify({ error: 'Too Many Requests' }), {
      status: 429,
      headers: { 'Content-Type': 'application/json' },
    });
  }

  // Parse body
  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return new Response(JSON.stringify({ error: 'Invalid JSON body' }), {
      status: 400,
      headers: { 'Content-Type': 'application/json' },
    });
  }

  // AC3: Zod validation
  const parsed = RegisterSchema.safeParse(body);
  if (!parsed.success) {
    return new Response(
      JSON.stringify({
        error: 'Validation failed',
        details: parsed.error.flatten().fieldErrors,
      }),
      {
        status: 422,
        headers: { 'Content-Type': 'application/json' },
      }
    );
  }

  const { display_name, headline, bio, rate_min, rate_max } = parsed.data;
  const supabase = createServiceClient(env);

  // AC2: Insert expert row
  const { data: expert, error } = await supabase
    .from('experts')
    .insert({
      id: user.id,
      display_name,
      headline: headline ?? null,
      bio: bio ?? null,
      rate_min: rate_min ?? null,
      rate_max: rate_max ?? null,
    })
    .select('id, display_name')
    .single();

  if (error) {
    // AC4: Duplicate detection — Postgres unique violation
    if (error.code === '23505') {
      return new Response(
        JSON.stringify({ error: 'Expert already registered' }),
        {
          status: 409,
          headers: { 'Content-Type': 'application/json' },
        }
      );
    }
    return new Response(
      JSON.stringify({ error: 'Registration failed', details: { message: error.message } }),
      {
        status: 500,
        headers: { 'Content-Type': 'application/json' },
      }
    );
  }

  // AC5: Push queue message to email-notifications
  // Email sourced from JWT (user.email) — not from experts table (no email column)
  await env.EMAIL_NOTIFICATIONS.send({
    type: 'expert.registered',
    expert_id: user.id,
    email: user.email ?? '',
  });

  // AC2 + AC10: Return 201
  return new Response(
    JSON.stringify({
      expert_id: expert.id,
      display_name: expert.display_name,
    }),
    {
      status: 201,
      headers: { 'Content-Type': 'application/json' },
    }
  );
}
```

**Edge cases:**
- `request.json()` throws if body is empty or not valid JSON — caught and returns `400`.
- `user.email` may be undefined if Supabase Auth user was created without an email (phone auth, anonymous auth). Guard with `?? ''` so the queue message is never undefined. The queue consumer in E06S06 must handle empty email gracefully.
- `rate_min` and `rate_max` are numbers in the Zod schema. JSON bodies always send numbers as floats — the `.int()` constraint ensures they are integers. If the client sends `rate_min: 100.5`, Zod returns a `422`.
- Do NOT validate that `rate_min <= rate_max` in this story — no such AC exists. Leave cross-field validation for a future story.

### Step 8 — Create `src/handlers/experts/profile.ts`

**File:** `/Users/fred/Dropbox/Development/MicroSaaS-Projects/The-AI-Integrator-Network/src/handlers/experts/profile.ts`

**Purpose:** Handles `GET /api/experts/:id/profile` (AC7) and `PATCH /api/experts/:id/profile` (AC8, AC9).

**Zod schema for PATCH:**

```typescript
const VALID_AVAILABILITY = ['available', 'limited', 'unavailable'] as const;

const PatchProfileSchema = z.object({
  display_name: z.string().min(1).max(100).optional(),
  headline: z.string().max(200).optional(),
  bio: z.string().max(2000).optional(),
  rate_min: z.number().int().positive().optional(),
  rate_max: z.number().int().positive().optional(),
  availability: z.enum(VALID_AVAILABILITY).optional(),
  profile: z.record(z.unknown()).optional(),
  preferences: z.record(z.unknown()).optional(),
});
```

**GET handler logic:**
1. Extract `:id` from URL path.
2. Compare `id` to `user.id` from JWT — if not equal, return `403`.
3. Query `supabase.from('experts').select('*').eq('id', id).single()`.
4. If no row found, return `404` with `{ error: 'Expert not found' }`.
5. Return `200` with full row.

**PATCH handler logic:**
1. Extract `:id` from URL path.
2. Compare `id` to `user.id` from JWT — if not equal, return `403`.
3. Parse and validate body with `PatchProfileSchema.safeParse()` — return `422` on failure.
4. Call `supabase.rpc('merge_expert_profile', { p_id: id, p_display_name: ..., p_profile: ..., p_preferences: ... })`.
5. Only pass fields that are present in the validated body — use `undefined` for absent optional fields (Supabase RPC will pass `NULL` to Postgres, which the function handles via `COALESCE`).
6. Return `200` with updated expert row.

**Full handler content:**

```typescript
import { z } from 'zod';
import { Env } from '../../types/env';
import { AuthUser } from '../../middleware/auth';
import { createServiceClient } from '../../lib/supabase';
import { Json } from '../../types/database';

const VALID_AVAILABILITY = ['available', 'limited', 'unavailable'] as const;

const PatchProfileSchema = z.object({
  display_name: z.string().min(1).max(100).optional(),
  headline: z.string().max(200).optional(),
  bio: z.string().max(2000).optional(),
  rate_min: z.number().int().positive().optional(),
  rate_max: z.number().int().positive().optional(),
  availability: z.enum(VALID_AVAILABILITY).optional(),
  profile: z.record(z.unknown()).optional(),
  preferences: z.record(z.unknown()).optional(),
});

function forbidden(): Response {
  return new Response(JSON.stringify({ error: 'Forbidden' }), {
    status: 403,
    headers: { 'Content-Type': 'application/json' },
  });
}

export async function handleGetProfile(
  _request: Request,
  env: Env,
  user: AuthUser,
  expertId: string
): Promise<Response> {
  // AC7: Own profile only
  if (user.id !== expertId) {
    return forbidden();
  }

  const supabase = createServiceClient(env);
  const { data, error } = await supabase
    .from('experts')
    .select('*')
    .eq('id', expertId)
    .single();

  if (error || !data) {
    if (error?.code === 'PGRST116') {
      // PostgREST "row not found" code
      return new Response(JSON.stringify({ error: 'Expert not found' }), {
        status: 404,
        headers: { 'Content-Type': 'application/json' },
      });
    }
    return new Response(
      JSON.stringify({ error: 'Failed to fetch profile', details: { message: error?.message } }),
      {
        status: 500,
        headers: { 'Content-Type': 'application/json' },
      }
    );
  }

  return new Response(JSON.stringify(data), {
    status: 200,
    headers: { 'Content-Type': 'application/json' },
  });
}

export async function handlePatchProfile(
  request: Request,
  env: Env,
  user: AuthUser,
  expertId: string
): Promise<Response> {
  // AC7 / AC8: Own profile only
  if (user.id !== expertId) {
    return forbidden();
  }

  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return new Response(JSON.stringify({ error: 'Invalid JSON body' }), {
      status: 400,
      headers: { 'Content-Type': 'application/json' },
    });
  }

  // AC9: Validate availability enum
  const parsed = PatchProfileSchema.safeParse(body);
  if (!parsed.success) {
    return new Response(
      JSON.stringify({
        error: 'Validation failed',
        details: parsed.error.flatten().fieldErrors,
      }),
      {
        status: 422,
        headers: { 'Content-Type': 'application/json' },
      }
    );
  }

  const { display_name, headline, bio, rate_min, rate_max, availability, profile, preferences } =
    parsed.data;

  const supabase = createServiceClient(env);

  // AC8: JSONB merge via RPC
  const { data, error } = await supabase.rpc('merge_expert_profile', {
    p_id: expertId,
    p_display_name: display_name ?? null,
    p_headline: headline ?? null,
    p_bio: bio ?? null,
    p_rate_min: rate_min ?? null,
    p_rate_max: rate_max ?? null,
    p_availability: availability ?? null,
    p_profile: (profile as Json) ?? null,
    p_preferences: (preferences as Json) ?? null,
  });

  if (error || !data || data.length === 0) {
    if (!data || data.length === 0) {
      return new Response(JSON.stringify({ error: 'Expert not found' }), {
        status: 404,
        headers: { 'Content-Type': 'application/json' },
      });
    }
    return new Response(
      JSON.stringify({ error: 'Failed to update profile', details: { message: error?.message } }),
      {
        status: 500,
        headers: { 'Content-Type': 'application/json' },
      }
    );
  }

  return new Response(JSON.stringify(data[0]), {
    status: 200,
    headers: { 'Content-Type': 'application/json' },
  });
}
```

**Edge cases:**
- `PGRST116` is the PostgREST error code for "no rows returned by `.single()`". This is the correct way to detect a missing expert row when using the JS client's `.single()` method.
- `profile` and `preferences` in the Zod schema are `z.record(z.unknown())` — this accepts any JSON object. It does NOT accept JSON arrays or primitives. If the client sends `profile: [1, 2, 3]` (an array), Zod returns `422`. This is intentional — the DB schema expects JSONB objects for these fields.
- In `handlePatchProfile`, all optional fields default to `null` in the RPC call. The Postgres function handles `NULL` via `COALESCE(p_X, column_X)` — `NULL` means "don't change this field".
- The `as Json` cast for `profile` and `preferences` is required because `z.record(z.unknown())` does not exactly match the `Json` type from `database.ts`. This is a type assertion, not a runtime conversion.

```typescript
  if (request.method === 'GET' && profileGetMatch) {
    return handleGetProfile(request, env, user, profileGetMatch[1]);
  }

  // PATCH /api/experts/:id/profile
  const profilePatchMatch = url.pathname.match(/^\/api\/experts\/([^/]+)\/profile$/);
  if (request.method === 'PATCH' && profilePatchMatch) {
    return handlePatchProfile(request, env, user, profilePatchMatch[1]);
  }

  // No expert route matched — authenticated but unknown path
  return new Response(JSON.stringify({ error: 'Not Found' }), {
    status: 404,
    headers: { 'Content-Type': 'application/json' },
  });
}
```

**Full updated `src/index.ts`:**

```typescript
import { Env } from './types/env';
import { authenticate } from './middleware/auth';
import { handleRegister } from './handlers/experts/register';
import { handleGetProfile, handlePatchProfile } from './handlers/experts/profile';

const QUEUES = ['email-notifications', 'lead-billing'] as const;

async function checkSupabase(env: Env): Promise<'connected' | 'error'> {
  try {
    const res = await fetch(`${env.SUPABASE_URL}/rest/v1/`, {
      headers: {
        apikey: env.SUPABASE_ANON_KEY,
        Authorization: `Bearer ${env.SUPABASE_ANON_KEY}`,
      },
    });
    return res.ok ? 'connected' : 'error';
  } catch {
    return 'error';
  }
}

export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const url = new URL(request.url);

    if (request.method === 'GET' && url.pathname === '/api/health') {
      const supabaseStatus = await checkSupabase(env);
      const body = { status: 'ok', supabase: supabaseStatus, queues: [...QUEUES] };
      const statusCode = supabaseStatus === 'connected' ? 200 : 503;
      return new Response(JSON.stringify(body), {
        status: statusCode,
        headers: { 'Content-Type': 'application/json' },
      });
    }

    // Auth middleware + expert routes
    if (url.pathname.startsWith('/api/experts/')) {
      const authResult = await authenticate(request, env);
      if (authResult.response) {
        return authResult.response;
      }
      const user = authResult.user;

      if (request.method === 'POST' && url.pathname === '/api/experts/register') {
        return handleRegister(request, env, user);
      }

      const profileMatch = url.pathname.match(/^\/api\/experts\/([^/]+)\/profile$/);
      if (profileMatch) {
        if (request.method === 'GET') {
          return handleGetProfile(request, env, user, profileMatch[1]);
        }
        if (request.method === 'PATCH') {
          return handlePatchProfile(request, env, user, profileMatch[1]);
        }
      }

      return new Response(JSON.stringify({ error: 'Not Found' }), {
        status: 404,
        headers: { 'Content-Type': 'application/json' },
      });
    }

    return new Response(JSON.stringify({ error: 'Not Found' }), {
      status: 404,
      headers: { 'Content-Type': 'application/json' },
    });
  },
} satisfies ExportedHandler<Env>;
```

## Edge Cases Per Acceptance Criterion

| AC | Edge Case | Handling |
|----|-----------|----------|
| AC1 | Authorization header present but not `Bearer` scheme | `authenticate()` checks `startsWith('Bearer ')` — returns `401` |
| AC1 | Token is expired | `supabase.auth.getUser(token)` validates expiry — returns error — `401` |
| AC1 | Token is from a different Supabase project | `getUser()` validates against current project's keys — returns error — `401` |
| AC1 | Route is `/api/experts/register` — rate limit check happens AFTER auth | Rate limit is inside `handleRegister`, after auth in index.ts. Auth runs first. Intentional: unauthenticated requests don't consume rate limit budget. |
| AC2 | `display_name` missing from body | Zod fails with `422` before DB insert |
| AC2 | Body is empty / not JSON | `request.json()` throws — caught as `400` before Zod runs |
| AC2 | `rate_min` or `rate_max` sent as string in JSON | Zod `z.number()` fails — `422` |
| AC3 | Multiple fields fail validation simultaneously | `parsed.error.flatten().fieldErrors` returns all field errors in one response |
| AC4 | Same user registers twice | Postgres `23505` unique violation on `experts.id` (PK) — `409` |
| AC5 | `user.email` is undefined (non-email auth) | `?? ''` guard prevents undefined in queue message payload |
| AC5 | Queue send fails (CF Queue unavailable) | `await env.EMAIL_NOTIFICATIONS.send(...)` throws — propagates as unhandled exception — `500`. Acceptable: queue availability is an infrastructure concern. |
| AC6 | `CF-Connecting-IP` absent (local dev) | Fallback to `'unknown'` — all local requests share one bucket |
| AC6 | Exactly 10 requests within window | 10th request is allowed (count goes to 10, then GET returns 10 which is `>= MAX_REQUESTS` on the next request). The check is: if `current !== null && parseInt >= 10` then reject. So requests 1–10 are allowed, request 11 is rejected. This matches "max 10 per 60 seconds". |
| AC7 | Expert row does not exist for authenticated user | `PGRST116` error from `.single()` — `404` |
| AC7 | Requesting another user's profile | `user.id !== expertId` check — `403` |
| AC8 | PATCH with empty body `{}` | Zod succeeds (all fields optional). RPC called with all-null args. All `COALESCE(NULL, column)` returns current column values. Row returned unchanged. Status `200`. |
| AC8 | PATCH sends `profile: null` explicitly | `z.record(z.unknown()).optional()` does NOT match `null` — Zod returns `422`. To pass `null` as a profile reset, the client would need a different endpoint or explicit `nullable()` schema. This is intentional — resetting profile to null is not an AC. |
| AC8 | Expert row does not exist when PATCH called | RPC runs UPDATE with no matching rows. Function returns 0 rows. `data.length === 0` check triggers `404`. |
| AC9 | `availability: "on_vacation"` (not in enum) | `z.enum(VALID_AVAILABILITY).optional()` fails — Zod returns `422` with field error for `availability` |
| AC10 | All error responses must include `Content-Type: application/json` | Every `new Response(JSON.stringify(...), {...})` call in every handler includes `headers: { 'Content-Type': 'application/json' }`. Must be verified during implementation. |
