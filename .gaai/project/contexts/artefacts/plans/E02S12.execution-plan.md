## Architecture Analysis

### Key Findings from Codebase Exploration

**Landing Worker constraint:** `src/types/env.ts` only declares `POSTHOG_API_KEY`. The Hono router in `src/index.ts` currently uses only synchronous render functions. All routes for AC1 require async handlers because they must call the backend API over HTTP. The existing `app.get(...)` calls use arrow functions returning synchronous `new Response(...)`. New routes must use `async (c) => ...` lambdas and resolve `c.env.API_BASE_URL` and `c.env.INTERNAL_API_KEY`.

**Backend API routing pattern:** `src/index.ts` uses flat regex-based routing, not Hono. New routes must be inserted before the catch-all `404` return and before the authenticated `experts` block when unauthenticated. The internal endpoint `GET /api/experts/internal/:slug` must be placed before the authenticated `/api/experts/` block (same ordering rule as public endpoints).

**HMAC token pattern:** `lib/bookingToken.ts` already implements `base64url(HMAC-SHA256(...))` using Web Crypto with `importHmacKey(secret, 'sign')`. The direct link token uses a simpler non-JWT HMAC (just signing a payload string, not a JWT triple). A new `lib/directLinkToken.ts` must implement `signDirectLinkToken(expertId, nonce, secret)` and `verifyDirectLinkToken(token, expertId, nonce, secret)` using the same Web Crypto approach but operating on a raw `{expertId}:{nonce}` message (no JWT structure).

**DB types:** `ExpertRow` in `src/types/db.ts` does not yet declare `direct_link_nonce`, `direct_submissions_this_month`, or `direct_submissions_reset_at`. `BookingRow` does not yet declare `lead_source`. Both must be extended after the DB migrations.

**Public expert handler:** `src/handlers/experts/public.ts` currently returns anonymized data without `direct_link_url`. It needs a new field computed server-side from `id + direct_link_nonce + DIRECT_LINK_SECRET`.

**Email validation library:** `lib/emailValidation.ts` already implements the full pre-check pipeline (`preCheckEmail`): syntax → normalize → MX → disposable blacklist. Reuse directly in the direct booking handler.

**Existing availability endpoint:** `GET /api/experts/:id/availability` is already implemented, CORS-gated, and returns slots. The `/book/:slug` page will call this endpoint client-side to populate the slot selector.

**Cron handler dispatch:** `src/handlers/bookings/cron.ts` dispatches on `controller.cron` string. The new monthly reset cron (`0 0 1 * *`) needs a new `else if` branch calling a new `resetDirectSubmissions(env)` function. The cron trigger must also be added to `wrangler.toml` in all three environments.

**Dashboard pattern:** React Router routes in `app/routes/` are flat files. The new section `dashboard.direct-link.tsx` follows the same loader/action/component pattern as `dashboard.availability.tsx`. The `apiPatch` utility is already available in `lib/api.server.ts` for the `PATCH /api/experts/:id/direct-link-token` call.

**CSP on landing worker:** `lib/securityHeaders.ts` sets `HTML_CSP` with `script-src 'self' 'unsafe-inline'` and `connect-src 'self' https://ph.callibrate.io`. The `/book/:slug` page makes client-side fetch calls to `API_BASE_URL` for availability and to submit the booking form. The CSP `connect-src` must be updated to include the API base URL. The CF Turnstile widget also requires `https://challenges.cloudflare.com` in `script-src` and `frame-src`.

**Robots.txt current state:** `src/pages/robots.ts` already blocks `GPTBot`, `Google-Extended`, `anthropic-ai`, `ClaudeBot`. It currently _allows_ `PerplexityBot` and `OAI-SearchBot`. Per AC9, `PerplexityBot` and `OAI-SearchBot` must be blocked from `/experts/`. Googlebot and Bingbot must explicitly be allowed to `/experts/*` (they currently fall under `User-agent: * Allow: /`).

**Sitemap current state:** `src/pages/sitemap.ts` exports a synchronous function returning a static XML string with only the root URL. It must become an async function that accepts the env to call `API_BASE_URL + /api/experts/public` and iterate expert slugs.

**BOOKING_TOKEN_SECRET vs BOOKING_CONFIRM_SECRET:** The existing booking confirmation flow uses `BOOKING_CONFIRM_SECRET`. The direct booking email confirmation (AC16) uses magic link JWTs signed with `BOOKING_TOKEN_SECRET` (referenced in the Story AC definition). This is a new secret distinct from the existing one. Implement it as `DIRECT_BOOKING_SECRET` in env to avoid naming confusion; the magic link signs `{bookingId}:{prospectEmail}` with a 24h TTL using the same `signBookingToken`-style approach.

### Step 3 — New lib: HMAC direct link token (AC7, AC14)

**File:** `workers/backend/api/src/lib/directLinkToken.ts` (new file)

This file implements two functions:

`signDirectLinkToken(expertId: string, nonce: string, secret: string): Promise<string>`
- Message: `${expertId}:${nonce}`
- Signs using Web Crypto `HMAC-SHA256` (same import pattern as `bookingToken.ts`)
- Returns raw `base64url(signature)` — not a JWT, no header/payload structure
- This matches AC7: `base64url(HMAC-SHA256(DIRECT_LINK_SECRET, "{expert_id}:{direct_link_nonce}"))`

`verifyDirectLinkToken(token: string, expertId: string, nonce: string, secret: string): Promise<boolean>`
- Re-signs `${expertId}:${nonce}` and compares using `crypto.subtle.verify` (constant-time, satisfies AC7)
- Returns `true` only on exact match

Helper (shared with bookingToken.ts pattern):
- `toBase64Url(data: ArrayBuffer): string` — same implementation as in bookingToken.ts
- `fromBase64Url(str: string): Uint8Array` — same implementation
- `importHmacKey(secret: string, usage: 'sign' | 'verify'): Promise<CryptoKey>` — same implementation

Do NOT re-export from bookingToken.ts; keep the file self-contained to avoid cross-dependency.

### Step 6 — New internal expert endpoint (AC7)

**File:** `workers/backend/api/src/handlers/experts/internal.ts` (new file)

     id: row.id,
     slug,
     display_name: row.display_name,
     headline: row.headline,
     bio: row.bio,
     skills: profile.skills ?? [],
     languages: profile.languages ?? [],

     rate_max: row.rate_max,

     direct_submissions_this_month: row.direct_submissions_this_month,

### Step 7 — New direct booking POST handler (AC14–AC16)

**File:** `workers/backend/api/src/handlers/bookings/direct.ts` (new file)

Exports `handleDirectBookingSubmit(request: Request, env: Env, ctx: ExecutionContext, slug: string): Promise<Response>`

Logic (in order, matching AC14 checklist):

**Step 7a — HMAC re-validation (AC14):**
1. Parse JSON body. Required fields: `t` (direct link token), `description` (prospect text), `email`, `name`, `honeypot` (expected empty), `submitted_at_ms` (client timestamp in ms).
2. Look up expert by slug (SELECT `id`, `direct_link_nonce`, `direct_submissions_this_month`, `availability` FROM experts WHERE `LEFT(id::text, 8) = shortHash` AND availability != 'unavailable').
3. If not found → 404.
4. Call `verifyDirectLinkToken(t, expertId, nonce, env.DIRECT_LINK_SECRET)`. If false → 422 `{ error: 'invalid_token' }`.

**Step 7b — Quota check (AC14):**
5. If `direct_submissions_this_month >= 100` → 429 `{ error: 'quota_reached' }`.
   - Also fire PostHog `expert.direct_quota_reached` in `ctx.waitUntil`.

**Step 7c — Input pre-validation (AC14):**
6. `description.length < 30` → 422 `{ error: 'description_too_short' }`.
7. Count distinct words in `description.trim().split(/\s+/)`. If `< 3` → 422 `{ error: 'description_too_short' }`.

**Step 7d — Honeypot + timing check (AC14):**
8. If `honeypot !== ''` and `honeypot !== undefined` → silently return 200 (bot; discard without error).
9. If `Date.now() - submitted_at_ms < 3000` → 422 `{ error: 'too_fast' }`.

**Step 7e — CF Rate Limiting (AC14, DEC-120):**
10. Call `env.RATE_LIMITER.limit({ key: `direct-book:${ip}` })`. If `!success` → 429.

**Step 7f — Turnstile validation (AC14, DEC-120):**
11. Validate `turnstile_token` from request body via CF Turnstile API (`POST https://challenges.cloudflare.com/turnstile/v0/siteverify`). If invalid → 403 `{ error: 'turnstile_failed' }`.

**Step 7g — Email pre-check (AC16, DEC-122):**
12. Call `preCheckEmail(email)` from `lib/emailValidation.ts`. If `!result.ok` → 422 `{ error: result.error }`.

**Step 7h — Extract requirements (AC14, DEC-48):**
13. POST to `POST /api/extract` (internal self-call or direct call to `handleExtract`). Pass `{ raw: description }`. On failure → 500. Store `extractedRequirements`.

**Step 7i — Increment quota (AC14):**
14. `UPDATE experts SET direct_submissions_this_month = direct_submissions_this_month + 1 WHERE id = expertId`.

**Step 7j — Create unconfirmed booking record (AC16):**
15. Insert into `bookings`:
    ```sql
    INSERT INTO bookings (expert_id, prospect_name, prospect_email, description, status, lead_source, held_until, duration_min)
    VALUES ($expertId, $name, $normalizedEmail, $description, 'pending_confirmation', 'direct',
            now() + interval '24 hours', 20)
    RETURNING id
    ```
    Note: no `prospect_id`, no `match_id`, no `start_at`/`end_at` yet — slot is confirmed in Step 7k after this endpoint returns. The unconfirmed record exists so it can be purged by cron after 24h if email not confirmed (DEC-129).

    Store the returned `bookingId`.

**Step 7k — Generate magic link + send email (AC16):**
16. Sign a JWT: `signBookingToken(bookingId, 'confirm', env.DIRECT_BOOKING_SECRET, 86400)` → `magicToken`.
    Update `bookings SET confirmation_token = magicToken WHERE id = bookingId`.
17. Send email via `sendEmail` (from `lib/email.ts`):
    - To: `normalizedEmail`
    - Subject: `Confirmez votre demande — Callibrate`
    - Use `buildDirectConfirmationEmail` (new helper in `lib/email.ts`) with the confirm URL: `https://callibrate.io/api/direct-book/${bookingId}/email-confirm?token=${magicToken}`.

**Step 7l — PostHog event (AC19):**
18. `ctx.waitUntil(captureEvent(..., 'expert.direct_form_submitted', { expert_id, slug, lead_source: 'direct' }))`.

**Return:**
```json
{ "status": "pending_email_confirmation", "booking_id": "<uuid>" }
```

### Step 8 — New direct booking email confirmation handler (AC16)

**File:** `workers/backend/api/src/handlers/bookings/direct-confirm.ts` (new file)

Exports `handleDirectBookingEmailConfirm(request: Request, env: Env, bookingId: string, ctx: ExecutionContext): Promise<Response>`

This is a `GET` endpoint (`GET /api/direct-book/:bookingId/email-confirm?token=...`) — no CORS, called directly from the prospect's email click.

Logic:
1. Get `token` from `?token=` param. If missing → redirect to `https://callibrate.io/booking-expired`.
2. `verifyBookingToken(token, bookingId, 'confirm', env.DIRECT_BOOKING_SECRET)`.
   - `invalid` → redirect to `https://callibrate.io/booking-error`
   - `expired` → redirect to `https://callibrate.io/booking-expired`
3. Fetch booking from DB: `id, expert_id, status, confirmation_token, prospect_email, prospect_name, description, lead_source`.
4. If `booking.status === 'confirmed'` → redirect to `https://callibrate.io/book/confirmed/${bookingId}?already=true`.
5. If `booking.confirmation_token !== token` → redirect to `https://callibrate.io/booking-expired` (token was rotated).
6. If `booking.status !== 'pending_confirmation'` → redirect to `https://callibrate.io/booking-error`.
7. Update booking: `SET status = 'awaiting_slot_selection', confirmation_token = NULL` (new status value — see note).
   Note: introduce `'awaiting_slot_selection'` as a valid status in the DB CHECK constraint. This avoids permanently holding GCal slots before the prospect selects one. Alternatively, keep `'pending_confirmation'` and redirect to the slot selection page passing `bookingId` + a signed slot-selection token. Simpler MVP approach: mark `status = 'pending_slot_selection'` (add to constraint).
8. Send PostHog `expert.direct_booking_email_confirmed` (not the same as the final confirmed booking).
9. Redirect to `https://callibrate.io/book/${slug}/select-slot?booking_id=${bookingId}&token=${slotSelectionToken}`.
   - `slotSelectionToken` = `signBookingToken(bookingId, 'confirm', env.DIRECT_BOOKING_SECRET, 3600)` (1h TTL for slot selection).

```sql
-- E02S12: Add pending_slot_selection status to bookings
DO $$
DECLARE constraint_name TEXT;
BEGIN
  SELECT conname INTO constraint_name
  FROM pg_constraint
  WHERE conrelid = 'bookings'::regclass AND contype = 'c' AND conname LIKE '%status%';
  IF constraint_name IS NOT NULL THEN
    EXECUTE 'ALTER TABLE bookings DROP CONSTRAINT ' || quote_ident(constraint_name);
  END IF;
END $$;

ALTER TABLE bookings ADD CONSTRAINT bookings_status_check
  CHECK (status IN (
    'held', 'confirmed', 'cancelled', 'completed', 'no_show',
    'pending_confirmation', 'expired_no_confirmation',
    'cancelled_by_prospect', 'pending_expert_approval',
    'pending_slot_selection'
  ));
```

### Step 17 — Update sitemap.xml (AC10)

**File:** `workers/frontend/landing/src/pages/sitemap.ts`

Change:
- Function signature: `renderSitemapXml(): string` → `renderSitemapXml(apiBaseUrl: string): Promise<string>`
- Implementation: fetch `GET ${apiBaseUrl}/api/experts/public?per_page=50&page=1` (repeat for subsequent pages if `total > 50`; cap at 500 experts for MVP). Build `<url>` entries for each expert slug.
- Cache TTL on the response: 86400s. Since this is a slow-changing list, the route handler should still serve from a cached response. The route handler must store the rendered XML in a module-level variable with a timestamp check (or rely on CF edge cache via `Cache-Control: public, max-age=86400` on the response).
- On API error: return the static fallback (root URL only) and log the error — never throw, never return a 500 to the sitemap consumer.

```ts
export async function renderSitemapXml(apiBaseUrl: string): Promise<string> {
  const slugs: string[] = [];
  try {
    const res = await fetch(`${apiBaseUrl}/api/experts/public?per_page=50`, {
      headers: { 'Accept': 'application/json' },
      cf: { cacheTtl: 300 },
    });
    if (res.ok) {
      const data = await res.json() as { experts: Array<{ slug: string }>; total: number };
      slugs.push(...data.experts.map(e => e.slug));
      // Handle pagination if total > 50 (max 10 pages = 500 experts)
      const pages = Math.min(Math.ceil(data.total / 50), 10);
      for (let p = 2; p <= pages; p++) {
        const r2 = await fetch(`${apiBaseUrl}/api/experts/public?per_page=50&page=${p}`);
        if (r2.ok) {
          const d2 = await r2.json() as { experts: Array<{ slug: string }> };
          slugs.push(...d2.experts.map(e => e.slug));
        }
      }
    }
  } catch { /* fail-open */ }

  const expertUrls = slugs
    .map(slug => `  <url><loc>https://callibrate.io/experts/${slug}</loc></url>`)
    .join('\n');

  return `<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url><loc>https://callibrate.io/</loc></url>
${expertUrls}
</urlset>`;
}
```

## Edge Cases Per AC

**AC2 (standard profile, no PII leak):**
- If `bio_excerpt` is derived from `bio.substring(0, 200)`, ensure it does not cut mid-word by trimming at the last space before position 200. The backend handler should implement `bio.substring(0, 200).replace(/\s\S*$/, '')` if `bio.length > 200`.
- The `direct_link_url` in the API response is safe to include because it does not contain any PII — it is a derived HMAC token. However, if the expert has not yet had their nonce generated (legacy row where `direct_link_nonce` might be NULL), the computation must guard: if nonce is null, return `direct_link_url: null`.

**AC3 (meta/headers on standard mode):**
- `max-snippet:150` is a Google-specific robots directive value, not a standard `robots` meta value. It must appear in BOTH `<meta name="robots">` and `X-Robots-Tag` for full coverage.
- The title format `{tier} Expert {top_skill} — Callibrate` requires a non-null top skill. Guard: if `skills.length === 0`, use headline or the fallback title `Expert IA — Callibrate`.

**AC4 (JSON-LD Person):**
- `knowsAbout` expects an array of strings. Encode as `JSON.stringify(skills).replace(/</g, '\\u003c')` to prevent XSS.
- If `bio_excerpt` is null, omit `description` from JSON-LD (undefined values are dropped by `JSON.stringify`).

**AC5 (CTA satellite funnel):**
- If `outcome_tags` is empty or the vertical cannot be mapped, the primary CTA must still exist with a valid default URL. Never render a broken or empty `href`.

**AC7 (direct mode HMAC validation):**
- Absent `?t=` param: treat as `attribution='callibrate'`, not as an error. The page renders normally with attribution set to `'callibrate'`.
- Expired nonce (expert rotated their link): the old token will fail HMAC verification because the nonce changed. The landing Worker should render the page with `attribution='callibrate'` rather than returning 404, to preserve link usability after rotation from the prospect's perspective.

**AC9 (robots.txt):**
- The ordering of `User-agent` blocks in robots.txt matters. The `User-agent: *` block with `Disallow: /book/` must come AFTER the explicit Googlebot/Bingbot allow blocks. In practice, robots.txt parsers match the most specific agent rule, so this is a belt-and-suspenders measure.

**AC10 (sitemap):**
- The public listing API is paginated. If `total` is 0 or the API fails, the sitemap must still return a valid XML with at least the root URL.
- Experts with `availability = 'unavailable'` are already excluded by the public listing API (its WHERE clause filters them out). So the sitemap will never include unavailable experts.

**AC13 (monthly cron reset):**
- The cron trigger `0 0 1 * *` fires at midnight UTC on the 1st of each month. In CF Workers, `controller.cron` will be the string `"0 0 1 * *"` exactly as declared in wrangler.toml. Verify that the `handleScheduled` dispatch uses exact string matching.

**AC14 (POST /api/book/:slug input validation):**
- The `description` word count check must count distinct words, not total words. Use `new Set(description.trim().split(/\s+/)).size >= 3`.
- The content-type guard in `index.ts` (`checkContentType`) will correctly require `application/json` for this POST endpoint since it is not a webhook. No exception needed.

**AC16 (email confirmation):**
- The `cleanupExpiredConfirmations` cron in `cron.ts` runs every 5 minutes and purges `pending_confirmation` bookings older than 30 minutes. Direct bookings also start in `pending_confirmation` status. The 24h TTL of the magic link token should be matched by a 24h expiry window in the cron. Update `cleanupExpiredConfirmations` to NOT purge `pending_confirmation` bookings where `lead_source = 'direct'` (since they have a 24h window, not 30min). Alternatively, use the `pending_slot_selection` status after email confirmation to differentiate them. The simplest fix: extend the cron purge to check `lead_source IS DISTINCT FROM 'direct' OR created_at < now() - interval '24 hours'`.

**AC18 (dashboard link rotation):**
- After rotation, any old `?t=` links the expert previously shared will fail HMAC validation. The API must return `attribution='callibrate'` for failed validations (not 404), so old links still work — they just route the prospect as a platform-attributed one.
- The expert dashboard must display a clear warning: "En régénérant votre lien, l'ancien lien ne sera plus reconnu comme trafic direct."

**AC19 (PostHog events):**
- `expert.public_profile_viewed` fires on the landing Worker side. `captureEvent` in the landing Worker must use a minimal PostHog client (not `posthog-node`) since it is a Workers environment without Node. Use a direct `fetch` to `https://eu.i.posthog.com/capture/` with the PostHog API key. Alternatively, mirror the pattern from the landing page's client-side snippet: fire from the browser JS, not server-side.
- For purity and reliability, fire `expert.public_profile_viewed` from client-side JS (same pattern as the landing page's `page_view` event), not server-side.

## Rollback Boundaries

The implementation is layered such that each boundary can be deployed or rolled back independently.

**Boundary 1 — DB migrations only (Steps 1, 12)**
- Safe to apply to staging without deploying any code.
- All new columns have defaults. Existing code ignores unknown columns.
- Rollback: manually drop columns with `ALTER TABLE experts DROP COLUMN IF EXISTS ...`. Requires migration rollback script.
- Dependency for: all other steps.

**Boundary 2 — Backend API new handlers (Steps 3–11, 13)**
- Deploy after Boundary 1.
- New endpoints (`/api/experts/internal/:slug`, `/api/book/:slug`, `/api/direct-book/:bookingId/email-confirm`, `/api/experts/:id/direct-link`) are unreachable until the landing worker or dashboard calls them.
- The change to `handleGetPublicExpertBySlug` adding `direct_link_url` is additive (new field). Existing consumers that don't parse `direct_link_url` are unaffected.
- The cron `resetDirectSubmissions` only fires on the 1st of the month. Safe to deploy at any point; first execution is deferred.
- Rollback: remove the new route matches from `index.ts` and remove imports. New handler files can remain dead code. Revert `public.ts` to remove `direct_link_url` field. Revert `env.ts` type additions.

**Boundary 3 — Landing Worker routes + page renderers (Steps 14–19)**
- Deploy after Boundary 2.
- `GET /experts/:slug` returns 404 if the public API returns no expert — safe.
- `GET /book/:slug` returns 404 if the internal API returns no expert — safe.
- `GET /robots.txt` changes are immediate. The new robots.txt is more restrictive for AI bots and more permissive for Googlebot/Bingbot on `/experts/`. No negative SEO impact.
- `GET /sitemap.xml` becomes async. If `API_BASE_URL` is not configured, the sitemap falls back to static. Zero regression risk.
- Rollback: revert `index.ts` routes, revert `robots.ts`, revert `sitemap.ts` to synchronous static version, remove new page files.

**Boundary 4 — Dashboard direct link section (Step 20)**
- Deploy after Boundary 2.
- New route file is only active if added to the navigation. If not linked from the sidebar, it exists but is unreachable.
- Rollback: remove the route file and the nav link entry.

**Deployment order:**
1. Apply migrations (Boundary 1)
2. Deploy backend API (Boundary 2) with secrets: `DIRECT_LINK_SECRET`, `INTERNAL_API_KEY`, `DIRECT_BOOKING_SECRET`
3. Deploy landing Worker (Boundary 3) with vars: `API_BASE_URL`, `INTERNAL_API_KEY`, `TURNSTILE_SITE_KEY`
4. Deploy dashboard Worker (Boundary 4) — no new secrets needed
