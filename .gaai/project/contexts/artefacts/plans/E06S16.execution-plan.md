# Execution Plan — E06S16: Replace n8n Webhooks with CF Workflows

## Approach Rationale

CF Workflows is introduced for the first time in this project. Pattern follows official CF Workflows docs
(DEC-59): `WorkflowEntrypoint` imported from `cloudflare:workers`, `[[workflows]]` top-level only in
wrangler.toml, `this.env` accessible in WorkflowEntrypoint, `step.sleep` accepts string ("7 days") or
number (ms). AC8 sleep acceleration pattern matches DEC-59 specification exactly.

Enriched expert notification dispatches to EMAIL_NOTIFICATIONS queue (new type `booking.confirmed.enriched`)
rather than directly sending via Resend — this preserves retry/idempotency guarantees of the queue consumer.
Survey emails similarly dispatch via `survey.call_experience` / `survey.project_satisfaction` queue types.

---

## Implementation Sequence

### Step 1 — `src/lib/jwt.ts`
**Add:** `signSurveyToken(bookingId, prospectId, secret)` → signed JWT with 30-day TTL.
Reuses existing `toBase64Url`, `importHmacKey` helpers.
Payload: `{ booking_id, prospect_id, exp }`.

### Step 2 — `src/types/queues.ts`
**Add** three new union members to `EmailNotificationMessage`:
- `{ type: 'booking.confirmed.enriched'; booking_id; expert_id; prospect_id; meeting_url; scheduled_at }`
- `{ type: 'survey.call_experience'; booking_id: string; prospect_id: string }`
- `{ type: 'survey.project_satisfaction'; booking_id: string; prospect_id: string }`

### Step 3 — `src/types/env.ts`
**Remove:** `N8N_WEBHOOK_URL: string`
**Add:**
```
BOOKING_CONFIRMED_WORKFLOW: Workflow;
BOOKING_COMPLETED_WORKFLOW: Workflow;
SURVEY_DELAY_7D_MS?: string;
SURVEY_DELAY_38D_MS?: string;
```

### Step 4 — `src/workflows/booking-confirmed.workflow.ts` (NEW)
```
BookingConfirmedWorkflow extends WorkflowEntrypoint<Env, params>
  run(event, step):
    step.do('dispatch-enriched-notification') →
      this.env.EMAIL_NOTIFICATIONS.send({ type: 'booking.confirmed.enriched', ...event.payload })
```
Params type: `{ booking_id; expert_id; prospect_id; meeting_url; scheduled_at }`.

### Step 5 — `src/workflows/booking-completed.workflow.ts` (NEW)
```
BookingCompletedWorkflow extends WorkflowEntrypoint<Env, params>
  run(event, step):
    delay7d = parseInt(this.env.SURVEY_DELAY_7D_MS ?? String(7*24*60*60*1000))
    delay38d = parseInt(this.env.SURVEY_DELAY_38D_MS ?? String(38*24*60*60*1000))
    step.sleep('wait-7-days', delay7d)
    step.do('dispatch-call-experience-survey') →
      this.env.EMAIL_NOTIFICATIONS.send({ type: 'survey.call_experience', booking_id, prospect_id })
    step.sleep('wait-38-days', delay38d)
    step.do('dispatch-project-satisfaction-survey') →
      this.env.EMAIL_NOTIFICATIONS.send({ type: 'survey.project_satisfaction', booking_id, prospect_id })
```

### Step 6 — `src/queues/email-notifications.ts`
**Remove:** `triggerN8nBookingConfirmed()`, `triggerN8nBookingCompleted()` functions.
**Modify `booking.confirmed` case:**
  - Keep `sendBookingConfirmedEmails(body, env)`
  - Replace `triggerN8nBookingConfirmed` → `env.BOOKING_CONFIRMED_WORKFLOW.create({ params: body })`
**Modify `booking.completed` case:**
  - Replace `triggerN8nBookingCompleted` → `env.BOOKING_COMPLETED_WORKFLOW.create({ params: body })`
**Add new cases:**
  - `booking.confirmed.enriched` → `sendBookingConfirmedEnrichedEmail(body, env)` (enriched expert email)
  - `survey.call_experience` → `sendSurveyEmail('call_experience', body, env)`
  - `survey.project_satisfaction` → `sendSurveyEmail('project_satisfaction', body, env)`

**`sendBookingConfirmedEnrichedEmail`:**
  Queries `experts` (gcal_email, display_name) + `bookings` (prospect_name) + `prospects` (requirements).
  Sends single email to expert with: prospect name, requirements summary, scheduled_at, meeting_url.

**`sendSurveyEmail(type, body, env)`:**
  Queries `bookings` by booking_id → prospect_email.
  Calls `signSurveyToken(booking_id, prospect_id, env.PROSPECT_TOKEN_SECRET)`.
  Survey URL: `{env.WORKER_BASE_URL}/api/surveys/{call-experience|project-satisfaction}?token={token}`
  Sends plain text survey email to prospect.

### Step 7 — `wrangler.toml`
**Add** at top-level (before [env.staging]):
```toml
[[workflows]]
binding = "BOOKING_CONFIRMED_WORKFLOW"
name = "booking-confirmed-workflow"
class_name = "BookingConfirmedWorkflow"

[[workflows]]
binding = "BOOKING_COMPLETED_WORKFLOW"
name = "booking-completed-workflow"
class_name = "BookingCompletedWorkflow"
```
No per-env workflows sections — isolation via separate Worker scripts.

### Step 8 — `src/index.ts`
**Add** named exports so CF runtime can find the Workflow classes:
```typescript
export { BookingConfirmedWorkflow } from './workflows/booking-confirmed.workflow';
export { BookingCompletedWorkflow } from './workflows/booking-completed.workflow';
```

## Edge Cases

- AC3: JWT 30-day TTL → uses `signSurveyToken` with `PROSPECT_TOKEN_SECRET`
- AC8: SURVEY_DELAY_*_MS absent in prod → parseInt("604800000") fallback
- `booking.confirmed.enriched`: prospect requirements may be null/empty — handle gracefully
- `bookings.prospect_email` may be null → fallback to `prospects.email` query
- Workflow.create() may be absent in test env → must mock in tests

## Risk Register

| Risk | Mitigation |
|------|-----------|
| `Workflow` type not in `@cloudflare/workers-types@4.20241022.0` | Import from `cloudflare:workers` if global not available |
| `step.sleep` type signature mismatch | Use `number` always (AC8 parseInt pattern) |
| Survey emails go to wrong recipient | Unit tests assert recipient address |
| Workflow not exported from index.ts | CF runtime won't find the class — export is required |
