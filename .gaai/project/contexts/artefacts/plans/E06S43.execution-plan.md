# E06S43 Execution Plan — AI Agent API Endpoints + MCP Server Foundation

## Architectural Decisions (Locked)

**MCP server placement:** Route group `/mcp/*` on core Worker. Rationale: no separate deployment complexity, shares all existing bindings (SESSIONS KV, HYPERDRIVE, POSTHOG_API_KEY, RESEND_API_KEY), no additional wrangler.toml entries needed. The `@modelcontextprotocol/sdk` is NOT available as an npm package compatible with CF Workers edge runtime — implement the Streamable HTTP transport manually (simple JSON-RPC protocol, trivial without SDK). Post-MVP a separate Worker is easy to extract if needed.

**KV namespace for agent rate limiting:** Use existing `SESSIONS` KV namespace. Key schema: `agent-rate:{key_hash}:{endpoint}:{window}`. No new binding needed.

**API key secret:** Add `AGENT_API_KEY_SECRET` to `Env` — a 32-char HMAC secret used to sign reveal confirmation tokens (distinct from PROSPECT_TOKEN_SECRET).

**Reveal confirmation token:** Use the existing HMAC-SHA256 JWT pattern from `lib/jwt.ts`. A new `signRevealConfirmToken` / `verifyRevealConfirmToken` pair with `aud: 'agent:reveal:confirm'` and 24h TTL.

**Reveal state storage:** Store reveal state in `SESSIONS` KV: key `agent-reveal:{match_id}`, value `{ status: "pending"|"confirmed"|"refused", expert_data?: object }`, TTL 48h.

**Step 3.1 — API key generation and hashing utilities**

File: `workers/backend/api/src/lib/agentKey.ts` (new file)

Functions:

```typescript
generateApiKey(): Promise<{ key: string; hash: string }>
// crypto.getRandomValues(new Uint8Array(32)) → hex string (64 chars)
// hash via crypto.subtle.digest('SHA-256', encoder.encode(key))
// return { key: hexKey, hash: hexHash }

hashApiKey(key: string): Promise<string>
// Same SHA-256 hex hash of the raw key string
// Used in agentAuth middleware during verification
```

Pattern follows `lib/otp.ts` hashOtp/generateOtp exactly. Web Crypto only — no external dependencies.

Edge cases:
- Raw key = 64 hex chars (32 bytes × 2). Returned ONCE to prospect. Only hash stored.
- Hash = 64 hex chars (SHA-256 = 32 bytes). `key_hash` column is TEXT.
- No timing-safe comparison needed — lookup is by DB index.

**Step 3.2 — Agent rate limiting utility**

File: `workers/backend/api/src/lib/agentRateLimit.ts` (new file)

```typescript
export type AgentEndpoint = 'extract' | 'match' | 'reveal' | 'booking';

const LIMITS: Record<AgentEndpoint, { count: number; windowSecs: number }> = {
  extract:  { count: 10, windowSecs: 60 },
  match:    { count: 5,  windowSecs: 60 },
  reveal:   { count: 3,  windowSecs: 3600 },
  booking:  { count: 3,  windowSecs: 3600 },
};

export async function checkAgentRateLimit(
  sessions: KVNamespace,
  keyHash: string,
  endpoint: AgentEndpoint,
): Promise<{ allowed: boolean; retryAfter?: number }>
```

Implementation:
- Window key: `agent-rate:${keyHash}:${endpoint}:${windowIndex}` where `windowIndex = Math.floor(Date.now() / (windowSecs * 1000))`
- Get count from KV. If `>= limit` → `{ allowed: false, retryAfter: windowSecs }`
- Otherwise increment, put with `expirationTtl: windowSecs * 2`
- Fail open on KV errors

### Phase 5 — Prospect API Key Management Routes (AC1)

**Step 5.1 — Key management route file**

File: `workers/backend/api/src/routes/agentKeys.ts` (new file)

**`handleCreateApiKey(request, env, prospectId, ctx)`**
1. Verify prospect JWT (`Authorization: Bearer {prospect_jwt}`), verify with `verifyProspectSessionToken`. `claims.prospect_id === prospectId` check (else 403).
2. Parse body: `{ name: string }`. Validate: non-empty, max 64 chars.
3. Count active keys: `SELECT COUNT(*) FROM prospect_api_keys WHERE prospect_id = $1 AND revoked_at IS NULL`. If `>= 3` → 422 `{ error: 'max_keys_reached', max: 3 }`.
4. Generate key: `generateApiKey()`.
5. Insert: `INSERT INTO prospect_api_keys (prospect_id, key_hash, name) VALUES ($1, $2, $3) RETURNING id, name, created_at`.
6. PostHog: `agent.api_key_created` with `{ prospect_id }`.
7. Return 201: `{ id, name, created_at, key: rawKey, key_note: "Store this key securely — it will not be shown again" }`.

**`handleRevokeApiKey(request, env, prospectId, keyId, ctx)`**
1. Verify prospect JWT + ownership.
2. Validate `keyId` as UUID format: `/^[0-9a-f-]{36}$/i`.
3. `UPDATE prospect_api_keys SET revoked_at = now() WHERE id = $1 AND prospect_id = $2 AND revoked_at IS NULL RETURNING id`.
4. If no rows: 404. Else 204.

Edge cases:
- Use `verifyProspectSessionToken` (7-day session token from E03S10), NOT `verifyProspectToken` (24h submit token)
- Revoked keys don't count toward the 3-key limit (COUNT WHERE revoked_at IS NULL)
- `sql.end()` in finally block on each handler

### Phase 6 — Agent Auth Middleware (AC2)

**Step 6.1 — Agent auth middleware**

File: `workers/backend/api/src/middleware/agentAuth.ts` (new file)

```typescript
export type AgentAuthResult =
  | { prospect_id: string; key_id: string; key_hash: string; response?: never }
  | { prospect_id?: never; key_id?: never; key_hash?: never; response: Response };

export async function agentAuth(
  request: Request,
  env: Env,
  ctx: ExecutionContext,
): Promise<AgentAuthResult>
```

Implementation:
1. Extract `Authorization` header. If missing/not `Bearer ` → 401 `{ error: 'missing_api_key' }`.
2. Extract raw key (`header.slice(7)`).
3. Validate format: `/^[0-9a-f]{64}$/` — if not → 401 `{ error: 'invalid_api_key' }`.
4. Hash: `hashApiKey(rawKey)`.
5. DB lookup: `SELECT id, prospect_id, revoked_at FROM prospect_api_keys WHERE key_hash = $1 LIMIT 1`.
6. No row → 401 `{ error: 'invalid_api_key' }`.
7. `revoked_at IS NOT NULL` → 401 `{ error: 'api_key_revoked' }`.
8. Update `last_used_at` async: `ctx.waitUntil(sql\`UPDATE prospect_api_keys SET last_used_at = now() WHERE id = ${row.id}\`)`.
9. Return `{ prospect_id, key_id: row.id, key_hash: hash }`.

Edge cases:
- Format check before DB lookup prevents timing attacks
- `last_used_at` is fire-and-forget (waitUntil)
- Distinct error codes: `invalid_api_key` vs `api_key_revoked`

**Step 7.1 — Agent endpoint handler file**

File: `workers/backend/api/src/routes/agent.ts` (new file)

**`handleAgentExtract(request, env, ctx, agentCtx)`** — AC3 + AC7 + AC10:
1. Check rate limit (`extract`, 10/min).
2. Parse + validate body: `{ freetext, satellite_id? }`.
3. Call `performExtraction(freetext, satelliteId, env)`.
4. PostHog: `agent.extract`.
5. Return 200 with ExtractionResponse.

**`handleAgentMatch(request, env, ctx, agentCtx)`** — AC4 + AC7 + AC10:
1. Check rate limit (`match`, 5/min).
2. Parse + validate body.
3. Call `performExtraction`.
4. Insert `prospect_projects` record (prospect_id from agentCtx).
5. Call `handleMatchCompute` via synthetic Request.
6. Load matches from DB: `SELECT id, score, score_breakdown FROM matches WHERE project_id = $1 ORDER BY score DESC LIMIT 20`.
7. Anonymize: return `{ match_id, score, score_breakdown, tier }` — NO expert_id, NO name.
8. PostHog: `agent.match` with `match_count`.
9. Return `{ project_id, matches: [...] }`.

Edge cases:
- 0 matches → 200 with `matches: []`
- expert_id MUST NOT appear in response (assert in tests)

**`handleAgentRevealRequest(request, env, ctx, agentCtx)`** — AC5 + AC7 + AC10:
1. Check rate limit (`reveal`, 3/hour).
2. Parse body: `{ project_id, match_id }`. Validate UUID format.
3. DB lookup: verify match belongs to agentCtx.prospect_id, join prospects for email, join prospect_projects for freetext.
4. Check KV `agent-reveal:{match_id}` for idempotency. If pending/confirmed → return current status.
5. `signRevealConfirmToken(match_id, prospect_id, env.AGENT_API_KEY_SECRET)`.
6. Store KV: `{ status: 'pending', key_id }` with TTL 48h.
7. Build confirm URL + refuse URL (same token, path distinguishes action).
8. Anonymized expert name: `"Expert ${match_id.slice(0, 4).toUpperCase()}"`.
9. Send email: `buildAgentRevealEmail({...})`.
10. PostHog: `agent.reveal_requested`.
11. Return `{ status: "pending_human_confirmation", confirmation_sent_to: email }`.

Edge cases:
- Null prospect email → 422 `prospect_email_required`
- Duplicate request → idempotent (no second email)

**`handleAgentRevealConfirm(request, env, ctx, matchId)`** — AC5 (GET, email link):
1. Extract `?token=` query param.
2. `verifyRevealConfirmToken(token, env.AGENT_API_KEY_SECRET)`.
3. Invalid/expired → 400 HTML "Ce lien a expiré ou est invalide."
4. Verify token's `match_id` === path param.
5. Load match from DB with expert data (`SELECT m.*, e.display_name, e.bio, e.headline, e.rate_min, e.rate_max FROM matches m JOIN experts e ON e.id = m.expert_id WHERE m.id = $1`).
6. Update KV: `{ status: 'confirmed', expert_data: {...} }`, TTL 48h.
7. PostHog: `agent.reveal_confirmed`.
8. Return 200 HTML "Profil confirmé. Votre assistant IA peut maintenant accéder au profil de [Expert Name]."

**`handleAgentRevealRefuse(request, env, ctx, matchId)`** — AC5:
1-4. Same token verification.
5. Update KV: `{ status: 'refused' }`.
6. Return 200 HTML "Accès refusé."

**`handleAgentRevealStatus(request, env, ctx, agentCtx, matchId)`** — AC5 polling:
1. Load KV `agent-reveal:{matchId}`.
2. If null: check DB for match ownership, return `{ status: 'expired' }` if match exists but KV gone.
3. If no match → 404.
4. Verify `match.prospect_id === agentCtx.prospect_id` (403 if not).
5. If `status === 'confirmed'` → return `{ status: 'confirmed', expert: expert_data }`.
6. Otherwise → `{ status: state.status }`.

**`handleAgentBookingRequest(request, env, ctx, agentCtx)`** — AC6 + AC7 + AC10:
1. Check rate limit (`booking`, 3/hour).
2. Parse body: `{ project_id, expert_id, preferred_dates? }`. Validate UUIDs.
3. Verify expert revealed: `SELECT 1 FROM matches WHERE project_id = $1 AND expert_id = $2 AND prospect_id = $3`. No row → 403 `expert_not_revealed`.
4. Load expert + prospect: `SELECT e.display_name, e.cal_username, p.email, pp.freetext FROM experts e, prospects p, prospect_projects pp WHERE e.id = $expert_id AND pp.id = $project_id AND p.id = $prospect_id`.
5. `cal_username` null → 422 `expert_not_bookable`.
6. Build booking link with optional `preferred_dates` query param.
7. Send email: `buildAgentBookingEmail({...})`.
8. PostHog: `agent.booking_requested`.
9. Return `{ status: "pending_human_action", booking_link: bookingLink }`.

**Step 8.4 — Agent API routes** (after MCP, before final 404):

```javascript
// ── Agent API routes (API key auth) — E06S43 ─────────────────────────────────
if (pathname.startsWith('/api/v1/agent/')) {
  // Public confirm/refuse URLs (email links — no agent auth needed)
  const revealConfirmMatch = pathname.match(/^\/api\/v1\/agent\/reveal\/([^/]+)\/confirm$/);
  if (method === 'GET' && revealConfirmMatch?.[1]) {
    return handleAgentRevealConfirm(request, env, ctx, revealConfirmMatch[1]);
  }
  const revealRefuseMatch = pathname.match(/^\/api\/v1\/agent\/reveal\/([^/]+)\/refuse$/);
  if (method === 'GET' && revealRefuseMatch?.[1]) {
    return handleAgentRevealRefuse(request, env, ctx, revealRefuseMatch[1]);
  }
  // All other routes require API key auth
  const agentAuthResult = await agentAuth(request, env, ctx);
  if (agentAuthResult.response) return agentAuthResult.response;
  const agentCtx = { prospect_id: agentAuthResult.prospect_id, key_id: agentAuthResult.key_id, key_hash: agentAuthResult.key_hash };
  if (method === 'POST' && pathname === '/api/v1/agent/extract') {
    return handleAgentExtract(request, env, ctx, agentCtx);
  }
  if (method === 'POST' && pathname === '/api/v1/agent/match') {
    return handleAgentMatch(request, env, ctx, agentCtx);
  }
  if (method === 'POST' && pathname === '/api/v1/agent/reveal') {
    return handleAgentRevealRequest(request, env, ctx, agentCtx);
  }
  const revealStatusMatch = pathname.match(/^\/api\/v1\/agent\/reveal\/([^/]+)\/status$/);
  if (method === 'GET' && revealStatusMatch?.[1]) {
    return handleAgentRevealStatus(request, env, ctx, agentCtx, revealStatusMatch[1]);
  }
  if (method === 'POST' && pathname === '/api/v1/agent/booking/request') {
    return handleAgentBookingRequest(request, env, ctx, agentCtx);
  }
  return new Response(JSON.stringify({ error: 'Not Found' }), { status: 404, headers: { 'Content-Type': 'application/json' } });
}
```

Critical: `/confirm` and `/refuse` routes BEFORE `agentAuth` call (they are human browser requests).
