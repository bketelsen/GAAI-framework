## Implementation Sequence

### Step 1 ‚Äî Create migration file: core schema (AC1‚ÄìAC9, AC12‚ÄìAC14)

**File to create:** `supabase/migrations/20260220000001_initial_schema.sql`
**Apply via:** `mcp__supabase__apply_migration` (name: `initial_schema`)

**Content (ordered for FK integrity):**
1. `experts` table (FK to `auth.users`) ‚Äî AC2
2. `prospects` table ‚Äî AC3
3. `matches` table (FK to experts, prospects) ‚Äî AC4
4. `bookings` table (FK to matches) ‚Äî AC5
5. `leads` table (FK to bookings, experts, prospects) ‚Äî AC6
6. `satellite_configs` table ‚Äî AC7
7. `call_experience_surveys` table (FK to bookings, prospects) ‚Äî AC12
8. `project_satisfaction_surveys` table (FK to bookings, prospects) ‚Äî AC13
9. `lead_evaluations` table (FK to leads, experts) ‚Äî AC14
10. GIN indexes on experts(profile), experts(preferences), prospects(requirements), matches(score_breakdown) ‚Äî AC8
11. Enable RLS on all 9 tables ‚Äî AC9
12. RLS policies: experts SELECT own row (authenticated), experts UPDATE own row (authenticated) ‚Äî AC9

**FK cascade strategy:** `ON DELETE CASCADE` for child records when parent is deleted.

**CHECK constraint validation (before writing SQL):**
- experts.availability: `IN ('available', 'limited', 'unavailable')`
- prospects.status: `IN ('anonymous', 'identified', 'matched', 'booked')`
- matches.status: `IN ('pending', 'revealed', 'booked', 'expired')`
- bookings.status: `IN ('confirmed', 'cancelled', 'completed', 'no_show')`
- leads.status: `IN ('pending', 'billed', 'disputed', 'refunded')`
- call_experience_surveys.score: `BETWEEN 1 AND 5`
- project_satisfaction_surveys.score: `BETWEEN 1 AND 10`
- lead_evaluations.lead_quality_score: `BETWEEN 1 AND 10`

**Edge cases:**
- `experts.id` must reference `auth.users(id)` ‚Äî this means experts can only be created via Supabase Auth. The FK must use `auth.users`, not `public.users`.
- `matches.id`, `bookings.id`, `leads.id` not explicitly required to have `DEFAULT gen_random_uuid()` by AC, but adding it is safe and expected for UUID PKs not linked to auth.
- `satellite_configs.id` is TEXT PK (not UUID) ‚Äî satellite sites use human-readable IDs like 'default', 'ai-consulting'.
- `bookings.cal_booking_id TEXT UNIQUE` ‚Äî must be unique constraint, not just a column.

### Step 2 ‚Äî Seed default satellite_config (AC11)

**Apply via:** `mcp__supabase__execute_sql` (DML ‚Äî INSERT)

**Content:** One row with `id: 'default'`, a valid `quiz_schema` covering all prospect requirements fields (challenge, budget_range, timeline, company_size, industry, skills_needed, languages), and `matching_weights` covering all scoring dimensions (skills_overlap, budget_fit, timeline_fit, language_match, composite_score_boost).

**Validation:** Weights must sum to 1.0.

### Step 3 ‚Äî Generate TypeScript types (AC10)

**Apply via:** `mcp__supabase__generate_typescript_types`
**Output:** Raw TypeScript types string.
**File to create:** `src/types/database.ts`
**Constraint:** File must compile ‚Äî no manual edits that could break type generation output.

## Risk Register

| Risk | Likelihood | Mitigation |
|------|-----------|-----------|
| FK to auth.users fails if auth schema not accessible | Low | auth.users is always accessible in Supabase public schema FK |
| CHECK constraint syntax error | Low | Validate SQL before applying |
| GIN index on NULL JSONB column | Low | GIN handles NULL safely in PostgreSQL |
| TypeScript gen returns empty output | Low | Verify table count before generating |
| Migration applies partially then fails | Medium | Single atomic migration ‚Äî all tables in one migration block |

---

## Rollback Boundaries

- **If Step 1 fails mid-apply:** Migration is rolled back atomically by Supabase. DB remains clean.
- **If Step 2 fails:** Only the seed INSERT is affected ‚Äî schema is intact. Re-run execute_sql.
- **If Step 3 fails:** DB is correct. TypeScript types can be regenerated independently.
