## 2. PostHog SDK Injection Approach

### Approach chosen: Approach A — Inline stub in `<head>` + custom events in `<body>`

**Rationale for choosing Approach A over Approach B:**

Approach B (`<script defer>` on an external file) is unsuitable for this implementation because:
1. Session replay requires the PostHog SDK to be initialized as early as possible. With `defer`, the script only runs after HTML parsing completes, meaning early user interactions (scroll, hover, first click before DOMContentLoaded) are not recorded.
2. `defer` applies only to external `src` scripts. The PostHog initialization config (api_host, persistence, etc.) must be an inline script, which cannot carry `defer`. This forces a split that negates Approach B's supposed benefit.
3. The inline stub is PostHog's documented best practice and is < 2KB minified. The performance cost on LCP is negligible for a stub of this size on a simple, CSS-only landing page with no above-fold images or heavy resources.

**Approach A detailed breakdown:**

| Concern | Implementation |
|---|---|
| AC1 — defer, non-blocking | The stub script itself is tiny (< 2KB). It immediately inserts an async `<script>` tag for the SDK bundle — that external load is async/non-blocking. LCP is not blocked by the stub. |
| AC2 — proxy host | `api_host: "https://ph.callibrate.io"` in `posthog.init()`. `ui_host: "https://eu.posthog.com"` points the UI back to PostHog's EU cluster for dashboards. |
| AC3 — cookieless | `persistence: "memory"` prevents any cookie or localStorage writes. Session data is held in memory only; cleared on tab close. GDPR-compliant without consent banner for tracking cookies. |
| AC4 — autocapture | `autocapture: true` in init. PostHog will automatically capture clicks, form submissions, and page views on DOM elements. |
| AC5 — session replay | `disable_session_recording: false` in init (this is the default, but stated explicitly for clarity). Session replay begins recording from page load since the stub is in `<head>`. |
| AC6 — custom CTA event | Inline body script attaches a click listener on `.cta` element; fires `satellite.cta_clicked` with `satellite_id` and `cta_text`. |
| AC7 — page view properties | `capture_pageview: false` disables PostHog's automatic pageview; our body script fires `page_view` manually with `satellite_id`, `referrer`, and UTM params parsed from `URLSearchParams`. |
| AC8 — API key from env | `posthogApiKey` arg sourced from `c.env.POSTHOG_API_KEY` (Cloudflare Workers secret); interpolated into HTML via `JSON.stringify()` to produce a safe JS string literal. Never hardcoded. |
| AC9 — tracking toggle | Both `posthogHeadSnippet` and `posthogBodyScript` are empty strings when `config.tracking_enabled === false`. No SDK is loaded; no events fired. |
| AC11 — LCP | The external SDK bundle load is `async` (inserted by the stub). The stub itself is < 2KB synchronous inline code in `<head>`. The landing page has no above-fold images (logo is conditionally rendered, not always present). LCP is the `<h2>` text node, which is not blocked by < 2KB of JS. |

**Why `capture_pageview: false`:** PostHog's automatic `$pageview` event does not include `satellite_id`, `referrer` (in our custom format), or UTM params. We fire our own `page_view` event in the body script with the full property set. This avoids duplicate pageview events and gives us a richer event schema.

```sql
  ADD COLUMN tracking_enabled BOOLEAN NOT NULL DEFAULT true;

  'When false, PostHog SDK is not injected into the satellite landing page (GDPR opt-out or debug mode).';

-- No explicit UPDATE needed; DEFAULT true covers all existing rows via ALTER TABLE.
```

**Applied via:** `mcp__supabase__apply_migration` with `name: "satellite_configs_tracking_enabled"`.

**Post-migration steps:**
1. Run `mcp__supabase__generate_typescript_types` to regenerate TypeScript types.
2. Write updated types to the canonical DB types file (verify path — likely `src/types/database.ts` at repo root).
3. Manually verify the `satellite_configs` type includes `tracking_enabled: boolean`.

**KV cache consideration:** Existing KV-cached config objects (TTL = 3600s) will NOT include `tracking_enabled` until they expire or are purged. The `SatelliteConfig` TypeScript type will have `tracking_enabled: boolean`, but deserialized KV objects will have `tracking_enabled: undefined` until the cache refreshes. The body script and head snippet guards use `config.tracking_enabled` — `undefined` is falsy, so tracking would be incorrectly disabled for cached configs.

**Mitigation:** The `resolveConfig` middleware in `config.ts` deserializes KV JSON directly as `SatelliteConfig`. After the migration, trigger a cache purge for all active satellites via the `/admin/cache/purge` endpoint, or treat `tracking_enabled` as `config.tracking_enabled !== false` (explicit false check) rather than a truthy check in `landing.ts`. The plan recommends the explicit `!== false` guard to handle the transition window safely without requiring a coordinated purge.

## 4. Edge Cases per AC

**AC1 (defer/non-blocking):**
- Edge case: If the PostHog EU CDN (`ph.callibrate.io/static/array.js`) is slow or unreachable, the async SDK load hangs silently. The stub queues all calls; they are never flushed. This is acceptable — tracking failure must never break the user journey.
- Edge case: Content Security Policy headers. If a CSP is ever added to the satellite response, `script-src 'unsafe-inline'` would be required for the inline stub. Document for future CSP implementation.

**AC2 (proxy host):**
- Edge case: If the reverse proxy Worker (E07S01, `ph.callibrate.io`) is down, all PostHog events fail silently. This is by design — tracking must not impact UX.
- Edge case: The `ui_host` must point to `https://eu.posthog.com` (not the proxy) so PostHog's dashboard UI functions correctly. The stub uses the proxy only for event ingestion.

**AC3 (cookieless):**
- Edge case: `persistence: "memory"` means distinct_id is regenerated on every page load (no cross-session identity). Funnel analysis will show higher "new users" counts than actual unique visitors. This is expected and acceptable for GDPR compliance.
- Edge case: If `persistence` is accidentally omitted or set to `"localStorage"`, cookies/localStorage entries will be written. The type-check will not catch this — must be verified in smoke test.

**AC4 (autocapture):**
- Edge case: PostHog autocapture captures all clicks including the CTA. This means a CTA click produces both an autocapture event AND our custom `satellite.cta_clicked` event. This is intentional — autocapture provides the raw signal; our custom event provides enriched properties. Dashboards must account for this duplication if counting CTA clicks.

**AC5 (session replay):**
- Edge case: Session replay records the full DOM. If the expert's `content` fields contain PII (names, emails in value props), they will appear in replay recordings. Current content is marketing copy only — low risk. Document for future content governance.
- Edge case: `disable_session_recording: false` enables replay by default. If a satellite owner needs replay disabled (e.g., for a regulated vertical), `tracking_enabled: false` is the current mechanism. A future story could add `session_recording_enabled` as a separate toggle.

**AC6 (CTA click event):**
- Edge case: The selector `.cta` is an `<a>` tag. If the user middle-clicks or Ctrl+clicks to open in a new tab, the click event fires but the page navigates away before PostHog flushes the event. PostHog uses `sendBeacon` for event flushing which handles navigation correctly.
- Edge case: `ctaEl.textContent.trim()` could be an empty string if the element has no text (e.g., icon-only button). The fallback `''` is intentional.
- Edge case: If `.cta` is absent from the DOM (edge case in future template variants), the `if (ctaEl)` guard prevents a null pointer exception.

**AC7 (page view properties):**
- Edge case: `document.referrer` is empty string `''` when the user navigates directly (no referrer). The `|| null` coercion normalizes this to `null` in PostHog, avoiding empty-string pollution in the referrer property.
- Edge case: UTM params are only present if the satellite is accessed via a UTM-tagged link. `params.get()` returns `null` for absent params; `|| null` is a no-op in that case. PostHog omits null properties from the event payload (verify this behavior — if not, null values are benign noise).
- Edge case: The `page_view` event fires on every page load, including bot traffic. PostHog's bot filtering should handle this, but satellite_id filtering in dashboards is the fallback.

**AC8 (API key from env):**
- Edge case: `POSTHOG_API_KEY` is not set in the Worker environment (e.g., newly deployed env without the secret). The TypeScript type is `string`, but Cloudflare Workers return `undefined` for missing secrets at runtime. This would produce `posthog.init(undefined, ...)` in the HTML — PostHog would fail to initialize but would not throw (the stub handles gracefully). Add a runtime guard in `landing.ts`: if `!posthogApiKey`, return empty strings for both snippets regardless of `tracking_enabled`.
- Edge case: `JSON.stringify(posthogApiKey)` produces `"phc_..."` (a quoted JS string literal). This is correct and safe. If the key somehow contained a double-quote character (it never should per PostHog's key format), `JSON.stringify` would escape it correctly.

**AC9 (tracking toggle):**
- Edge case: KV cache staleness (see section 3 — mitigation described).
- Edge case: `tracking_enabled: false` in DB but `true` in KV cache — tracking continues until TTL expires (max 1 hour). Use `!== false` guard to handle `undefined` from stale cache, but cannot avoid the window where a cached `true` is served after a DB update to `false`. The `/admin/cache/purge` endpoint is the operational control.

**AC10 (existing tests pass):**
- Edge case: No automated tests exist in the satellite worker (`package.json` has only `type-check`, `dev`, `deploy:*` scripts). "Existing tests" effectively means `tsc --noEmit` passes and the Worker deploys without runtime errors. The updated function signature is a breaking change to `renderLandingPage` — the single call site in `index.ts` must be updated atomically in Step 8.

**AC11 (LCP not impacted):**
- Edge case: The inline stub is ~1.5KB minified. If PostHog updates their snippet in future (it has happened), the new snippet must be re-evaluated for size before adoption.
- Edge case: The Worker applies `Cache-Control: public, max-age=300`. The PostHog API key is embedded in the HTML. A key rotation would require a cache purge to propagate to all CDN-cached pages.

## 7. Rollback Boundaries

### Boundary 1 — DB migration (Step 1)

**Rollback SQL:**
```sql
ALTER TABLE satellite_configs DROP COLUMN IF EXISTS tracking_enabled;
```

**Safe to roll back independently.** The column is purely additive. If removed, `SatelliteConfig.tracking_enabled` becomes `undefined` at runtime (TypeScript type would be stale until reverted). Must revert Step 3 (config.ts) in tandem.

**Dependency:** Steps 1, 2, 3 are a coupled unit. Roll back all three together or none.

### Boundary 2 — TypeScript type changes (Steps 3 and 4)

**Rollback:** Revert `config.ts` and `env.ts` to their original content (remove added fields). These changes have zero runtime impact on their own — they only cause compile errors if the DB column or env secret is absent.

**Can be reverted independently from Steps 6–8** only if `renderLandingPage` is also reverted (Step 6 introduced a new parameter that depends on `Env.POSTHOG_API_KEY`).

### Boundary 3 — PostHog instrumentation in `landing.ts` and `index.ts` (Steps 6, 7, 8)

**Rollback:** Revert `landing.ts` to original signature `renderLandingPage(config: SatelliteConfig): string` and remove both snippet variables. Revert `index.ts` call site to `renderLandingPage(config)`.

**This is the primary rollback unit.** Steps 6, 7, 8 are tightly coupled and must be reverted as a unit. Rolling back these steps eliminates all PostHog instrumentation from the rendered HTML without touching the DB or types.

**Safe at any time.** The `tracking_enabled` column in the DB is harmless if the instrumentation is not rendered. The `POSTHOG_API_KEY` secret is harmless if unused.

### Boundary 4 — `wrangler.toml` comment update (Step 5)

**Rollback:** Remove the added comment line. No runtime impact at any time.

### Full rollback order (if catastrophic failure)

1. Revert Steps 6, 7, 8 (landing.ts + index.ts) — eliminates PostHog from production HTML
2. Revert Steps 3, 4 (types) — restores original interfaces
3. Revert Step 5 (wrangler.toml comment) — cosmetic
4. Apply rollback SQL for Step 1 — removes DB column
5. Regenerate TypeScript types (Step 2 inverse)

Steps 1–4 can be deployed independently; no coordinated downtime required.
