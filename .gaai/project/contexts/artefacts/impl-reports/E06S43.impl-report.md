## Summary

Implemented a full AI agent API surface on top of the existing Callibrate Core worker, enabling AI agents (Claude, GPT-4, n8n agents) to discover, evaluate, and initiate contact with experts on behalf of a prospect — while preserving a mandatory human-in-the-loop gate for all identity reveals and bookings.

The implementation covers:

1. **Agent API key management** — prospects create/revoke Bearer tokens (max 3 active) via their session-authenticated dashboard.
2. **Agent authentication middleware** — validates 64-hex-char Bearer keys via SHA-256 hash lookup against the new `prospect_api_keys` DB table.
3. **Agent extraction endpoint** — delegates to the new `lib/extraction.ts` library (refactored from `routes/extract.ts`) with per-key rate limiting.
4. **Agent match endpoint** — creates a `prospect_projects` record then calls `handleMatchCompute`, returning anonymized results (match_id + score only, no `expert_id`).
5. **Reveal flow (human-in-the-loop)** — prospect receives a confirm/refuse email; state stored in KV (48h); reveal confirm/refuse handlers return HTML (email link clicks); status endpoint for polling.
6. **Booking request** — sends a booking email to prospect with the expert's Cal.com link; requires prior reveal confirmation.
7. **Per-key KV rate limiting** — `lib/agentRateLimit.ts` enforces window-based limits per endpoint (extract: 10/min, match: 5/min, reveal: 3/hr, booking: 3/hr), with fail-open on KV errors.
8. **MCP JSON-RPC 2.0 server** — `POST /mcp` implements the Model Context Protocol with 4 tools: `callibrate_describe_project`, `callibrate_find_experts`, `callibrate_request_reveal`, `callibrate_request_booking`.
9. **Full test suite** — 35 new tests across 5 new test files (agentKey, agentRateLimit, jwt extensions, agentKeys route, agent handlers, mcp). All 660 tests pass.

## Known Risks and Limitations

1. **Reveal token reuse** — `signRevealConfirmToken` generates the same-format token for both confirm and refuse URLs in `handleAgentRevealRequest`. Both tokens are valid HMAC signatures; the semantics (confirm vs refuse) are determined by which URL the user clicks. This is correct behavior but the tokens are technically interchangeable — a user could use the confirm token URL to refuse. A future improvement would add an `action` claim to the payload.

2. **Expert email lookup join** — In `handleAgentRevealRequest`, the prospect email is fetched via a cross-table query (`FROM experts e LEFT JOIN prospects p ON p.id = $prospect_id`). This is a workaround because the join on prospect_id via the prospects table is cleaner than two separate queries, but the query structure is non-obvious. Refactored to two separate queries in the booking handler for clarity.

3. **`prospect_projects` insert in agent match** — The agent match handler inserts a `prospect_projects` record then calls `handleMatchCompute`. The `handleMatchCompute` function loads the prospect from DB using `prospect_id`, but agent-created prospects have full `requirements` set on the prospect record (from prior extractions), not on the project record. The match compute uses `prospect.requirements` not `project.requirements`. This is technically correct for the current matching implementation but means the project-level requirements are not used for scoring — only for storage.

4. **Rate limit window granularity** — The window index is `Math.floor(Date.now() / (windowSecs * 1000))`. At window boundaries, a user could make `2 * limit` requests in a short period spanning two windows. This is acceptable for MVP but a sliding window would be more precise.

5. **`AGENT_API_KEY_SECRET` not in `.dev.vars`** — The new secret must be manually added to `.dev.vars` for local development. The wrangler.toml comment documents this.
