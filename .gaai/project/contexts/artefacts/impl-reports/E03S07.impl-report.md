# E03S07 Implementation Report

## Status: COMPLETE

## Summary

E03S07 inserts an email-confirmation step between the inline booking form and the creation of a GCal event. The flow is:

1. Prospect fills the inline hold form → `POST /api/bookings/:id/confirm`
2. `confirm.ts` sets status to `pending_confirmation`, generates HMAC-signed tokens, sends a confirmation email with Confirm/Cancel links
3. Prospect clicks "Confirm" in email → `GET /api/bookings/:id/email-confirm?token=...`
4. `email-confirm.ts` verifies the token, creates the GCal event, then either sets status to `confirmed` (auto-confirm path) or `pending_expert_approval` (manual approval path)
5. On the widget side, the success screen now shows "Check your email" with a countdown-based resend button

## Key Design Decisions

### Token Revocation
The `confirmation_token` column stores the most recently issued token. On resend, a new token is generated and stored — this revokes any prior token because `email-confirm.ts` checks `booking.confirmation_token !== token` before proceeding.

### GCal Creation Timing
GCal event creation was moved from `confirm.ts` to `email-confirm.ts`. This ensures a GCal event is only created when the prospect has actually confirmed via email, preventing orphaned calendar events from prospects who never confirm.

### Auto-confirm vs Expert Approval
`expert.booking_auto_confirm` defaults to `true`. When `false`, after email confirmation the booking enters `pending_expert_approval` and the expert receives an approval email. The cron auto-confirms after 24h if no response.

### Widget Resend
The widget stores `bookingId` in the existing closure variable. The resend fetch uses `apiUrl` (the satellite's configured API base URL) so the CORS origin is correct.

### Reminder Email j1/h1 Detection
The `sendReminderEmail` function in `email-notifications.ts` determines whether the reminder is j1 or h1 by checking `hoursUntilStart >= 3`. This is a heuristic based on the cron windows: j1 fires 23-25h before, h1 fires 50-70min before.
