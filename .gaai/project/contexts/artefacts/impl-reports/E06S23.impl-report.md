## Deviations from Plan

### Deviation 1: `(createSql as Mock)` pattern instead of `vi.mocked(dbModule.createSql)`

**Problem:** `vi.mocked(dbModule.createSql).mockReturnValue(mockSql as ReturnType<...>)` fails because `Mock<Procedure>` does not overlap with `Sql<{}>` (postgres.js tagged-template type).

**Fix:** Followed existing codebase convention from `bookings.test.ts`:
```typescript
vi.mock('./db', () => ({ createSql: vi.fn() }));
import { createSql } from './db';
// ...
(createSql as Mock).mockReturnValue(mockSql);
```

**Justification:** Exact match with the established test pattern. No functional impact.

### Deviation 2: `as unknown as Env` for test mock envs with optional `EXPERT_DB`

**Problem:** `exactOptionalPropertyTypes` TSConfig flag treats `EXPERT_DB: undefined` differently from `EXPERT_DB` absent ‚Äî direct spread assignment fails type check.

**Fix:** `const mockEnvWithDb = { ...baseEnv, EXPERT_DB: {} as D1Database } as unknown as Env;`

**Justification:** Test-only cast; no runtime impact. Matches `bookings.test.ts` pattern for mock envs.

### Deviation 3: Removed `writeCachePool` call for empty expert array in Hyperdrive fallback

**Problem:** Test expected `writeCachePool` not called when Hyperdrive returns empty array. Original implementation called it with `[]`. Calling `writeCachePool([])` would pollute the cache with an empty pool.

**Fix:** Added early return in `loadFromHyperdrive` ‚Äî do not call `writeCachePool` or `upsertToD1` when result is empty.

**Justification:** Semantically correct ‚Äî writing an empty pool to cache/D1 could cause brief denial-of-service for new expert registrations.
