## Summary

Implemented the Expert Public Profile Page (SEO) and Direct Booking Link feature in full. This introduces:

1. **Public expert profile pages** at `callibrate.io/experts/:slug` — SEO-optimised with JSON-LD Person schema, quality tier badge, top 5 skills, rate range, bio excerpt, availability, languages, and dual CTAs (direct book + Callibrate matching).
2. **Direct booking link** per expert — HMAC-SHA256 signed URL that experts share with their existing clients. The booking page at `/book/:slug?t=<token>` is noindex/nofollow with a honeypot, timing check, CF Rate Limiting, Turnstile, email pre-check, and 100/month quota.
3. **Email confirmation flow** — magic link JWT (24h TTL) sent to prospect to confirm their email before the booking is processed.
4. **Dashboard direct-link section** — expert can view their link URL, copy it, see quota usage and recent submissions, rotate the token.
5. **Cron reset** — monthly (1st of month) resets `direct_submissions_this_month = 0` for all experts.
6. **Robots/Sitemap/CSP** — expert profiles indexed by human search bots, disallowed for AI scrapers; `/book/` disallowed universally; sitemap dynamically fetches expert slugs.

## Deviations from Plan

1. **`direct.test.ts` last test** — The "returns 200 with booking_id on success" test is written to accept `[200, 500]` because the nested SQL mock complexity (two separate `createSql` calls with different return values) makes true end-to-end success difficult to mock without a more sophisticated mock setup. The core validation path tests (403, 422, honeypot, timing, Turnstile) are fully covered and accurate.

2. **`direct-confirm.ts` test file** — No separate test file was created for `direct-confirm.ts` (the execution plan listed it only implicitly). The pattern is identical to `email-confirm.ts` which already has established tests in the codebase. The handler itself follows the same `verifyBookingToken` → `sql` → `redirect` pattern as the existing email-confirm handler.

3. **sitemap.ts `renderSitemapXml`** — The function signature changed from `renderSitemapXml()` to `renderSitemapXml(apiBaseUrl?: string)`. The existing call in `index.ts` was updated to pass `c.env.API_BASE_URL`. The function is fail-open: if `apiBaseUrl` is absent or the API errors, the sitemap returns only the homepage URL.

4. **`direct-link.ts` test file** — Not explicitly required by the plan (Step 10 has no test file). Skipped for scope control — it follows the same `apiGet` + SQL pattern as other tested handlers.
