## Notable Decisions

1. **Migration timestamp**: Two migrations already had `20260228000008` (`e03s09_satellite_locale` and `e06s41_prospect_projects`). Used `20260228000009` as specified тАФ no conflict.

2. **`sql.array()` mock**: The `makeMockSql` helper adds `array: vi.fn((arr) => arr)` to the mock object. Since postgres.js template literals pass `sql.array(ids)` as an interpolated parameter, the mock just needs to return the array unchanged тАФ the sql function mock ignores the values and returns whatever is in the response queue.

3. **`purgeAbandonedFunnels` early return**: When no abandoned projects are found, the function returns immediately without inserting a `data_purge_log` row. This matches the execution plan: only insert audit log when there is something to report.

4. **`cleanupOtpKvKeys` audit log always inserted**: Even when 0 keys are deleted, one audit row is written with `records_affected=0`. This was specified in the plan to provide a heartbeat trail of the cron running.

5. **PostHog event**: Used `captureEvent(env.POSTHOG_API_KEY, { ... }).catch(() => {})` pattern, consistent with the existing `cron.ts` pattern (the `captureEvent` function already swallows errors internally, but the `.catch()` makes the fire-and-forget intent explicit).

6. **Match.ts consent notice placement**: Added inside `<form id="match-form">` immediately after the submit button, not outside. This ensures the notice is semantically associated with the form action.
