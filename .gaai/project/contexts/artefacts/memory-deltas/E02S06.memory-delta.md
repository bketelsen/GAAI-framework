## New Patterns Established

### 1. SSR-Safe Date Formatting via Loader Pre-computation

**Pattern:** Format all dates server-side in the loader, not in the component. Store the pre-formatted string as a field on the data object (e.g., `formatted_datetime`).

**Why:** Avoids hydration mismatches between server and client `Intl.DateTimeFormat` output. Using `timeZone: "UTC"` on the server ensures determinism. Components receive a plain string and render it without any date logic.

**Location:** `dashboard.bookings.tsx` â€” `formatBookingDatetime()` called in loader, result stored in `Booking.formatted_datetime`.

**Applicable to:** Any future route rendering user-facing dates or times (billing, analytics, etc.).

### 2. Mutually Exclusive Dialog State Pattern

**Pattern:** When a page has multiple dialogs that share context (a `selectedItem`), use a separate boolean state per dialog and gate the detail/overview dialog as:
```ts
open={!!selectedItem && !actionDialog1Open && !actionDialog2Open}
```

**Why:** Prevents two dialogs from being open simultaneously. When an action dialog (cancel, reschedule) opens from within the detail dialog, the detail dialog automatically closes.

**Location:** `dashboard.bookings.tsx` lines 537â€“542.

**Applicable to:** Any future route with an overview dialog + nested action dialogs (e.g., billing disputes, lead actions).

### 3. `apiDeleteWithBody` â€” DELETE with JSON body

**Pattern:** Added `apiDeleteWithBody<T>` to `api.server.ts` alongside existing `apiDelete`. Delegates to internal `apiFetch` with `"DELETE"` method and a body argument.

**Why:** Some REST APIs (including the bookings cancel endpoint) accept a request body on DELETE to pass optional metadata (e.g., `cancel_reason`). Standard `apiDelete` does not support a body.

**Location:** `api.server.ts` lines 81â€“88.

**Rule:** `apiDeleteWithBody` must always delegate to internal `apiFetch`, not call `fetch` directly, to preserve the shared Authorization header and error handling logic.

### 4. Graceful API Degradation in Loaders

**Pattern:** Wrap `apiGet` calls in `.catch(() => ({ ...emptyState }))` to return a zero-state rather than throwing. This prevents the full page from crashing when the API is temporarily unavailable.

**Location:** `dashboard.bookings.tsx` line 164.

**Precedent:** Same pattern used in leads page (E02S05). This is now confirmed as the standard loader error handling pattern for this project.

---

### 5. `satisfies ActionData` on All Action Return Branches

**Pattern:** Every `Response.json(...)` call in the action function includes `satisfies ActionData` on the object literal. This catches type mismatches at compile time rather than at runtime.

**Why:** TypeScript's `satisfies` keyword validates the shape without widening the type. If a branch returns an object missing a required field, `tsc` will catch it.

**Applicable to:** All future action functions. Treat this as a non-negotiable convention.

---

### 6. Reschedule Duration Normalization

**Decision:** Reschedule action auto-computes `new_end_at = new_start_at + 20 minutes`. The original booking duration is not preserved.

**Rationale:** Platform uses fixed 20-minute slots. This is intentional and consistent with the booking creation model.

**Implication:** If the platform ever supports variable-duration bookings, the reschedule action must be updated to accept `new_end_at` as a user input or derive it from the original booking's duration.
