## New Patterns Established

### 1. Third-Party Portal URL Pattern — `LS_CUSTOMER_PORTAL_URL`

**Pattern:** When a third-party service (Lemon Squeezy) provides a separate customer portal URL distinct from the checkout URL, expose it as a dedicated env var (`LS_CUSTOMER_PORTAL_URL`), type it in `worker-configuration.d.ts` as `optional string`, pass it through the loader as `lsCustomerPortalUrl: string | null`, and render it conditionally in the component.

**Why:** The checkout URL and the customer portal URL serve different intents. Conflating them (e.g., using a billing management query param on the checkout URL) would be fragile. Separate env vars make each URL independently configurable and testable.

**Location:** `worker-configuration.d.ts` line 9, `dashboard.billing.tsx` loader line 153, `LoaderData` type line 70, component lines 442–453.

**Applicable to:** Any future third-party integration that exposes both an activation flow and a post-activation management URL (e.g., Stripe billing portal).

---

### 2. Lemon.js Client-Side Script Injection Pattern

**Pattern:** Inject the Lemon Squeezy script (`app.lemonsqueezy.com/js/lemon.js`) via `useEffect` with a guard on `document.getElementById("lemon-js")` to avoid re-injection on re-renders. Call `window.createLemonSqueezy?.()` and `window.LemonSqueezy?.Setup(...)` both on first load and on already-loaded cases. Wire `Checkout.Success` event to `window.location.reload()` to force loader revalidation.

**Why:** The Lemon.js script is a client-side overlay SDK. SSR-safe guard (`typeof document === "undefined"`) ensures it does not run on server. The reload-on-success strategy is simpler and more reliable than manual React Router revalidation for third-party overlay flows.

**Location:** `dashboard.billing.tsx` lines 287–318.

**Note:** If a CSP header is added in the future, `app.lemonsqueezy.com` must be in `script-src`.

---

### 3. Cents-to-EUR Formatting Exported as Named Helper

**Pattern:** Export `centsToEur(cents: number): string` as a named function from the route file so it can be imported in test files and reused across components.

**Why:** Formatting logic should be deterministic and testable in isolation. Keeping it as a named export avoids duplicating the `/ 100).toFixed(2)` logic across the component.

**Location:** `dashboard.billing.tsx` lines 95–97.

**Applicable to:** Any future billing or financial display route.

---

### 4. Progress Bar Color Injection via Tailwind Arbitrary Class on Child Element

**Pattern:** To color the filled portion of a `<Progress>` Shadcn component, use the arbitrary variant `[&>div]:bg-{color}-500` on the `className` prop of the `<Progress>` element itself (not on a child). Apply conditionally based on threshold state variables (`spendWarning`, `spendCaution`).

**Why:** Shadcn `<Progress>` renders an inner `<div>` for the filled track. There is no direct `indicatorClassName` prop. The `[&>div]` Tailwind arbitrary selector targets that inner div without modifying the component source.

**Location:** `dashboard.billing.tsx` lines 497–504.

---

### 5. Graceful API Degradation in Loader — Confirmed Pattern (Third Occurrence)

**Pattern:** Wrap `apiGet` calls in `.catch(() => ({ ...emptyState }))` to return a zero-state when the billing API is unavailable, rather than throwing and breaking the route.

**Note:** This pattern was first established in E02S05 (leads), confirmed in E02S06 (bookings), and confirmed again here. It is now the **mandatory** loader error handling pattern for all dashboard routes. Deviation requires explicit justification.

**Location:** `dashboard.billing.tsx` lines 122–136.

## Architectural Observations

- **`satisfies ActionData` on all action branches is confirmed standard.** Three routes now use this pattern (leads, bookings, billing). Treat as non-negotiable.

- **Overlay checkout vs. page navigation.** The billing page uses `window.LemonSqueezy?.openUrl(overlayUrl)` (client-side overlay) rather than a `<Link>` navigation. This is intentional — the checkout is an in-page flow that returns to the same page on success. Do not convert to a navigation link in future refactors.

- **PostHog event naming:** `expert.billing_viewed` fires on every loader call with `has_subscription` and `subscription_status` as properties. Both action types fire `expert.spending_control_updated` with `control_name`, `old_value`, `new_value`. This continues the `expert.*` event namespace established in E02S03.
