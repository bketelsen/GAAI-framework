## New Patterns Worth Persisting

### Pattern 1: Locale Resolution in Server-Side HTML Rendering

`effectiveLocale` is resolved at render time using `config.locale?.startsWith('fr') ? 'fr' : 'en'`. This is the established pattern for multi-locale support in satellite HTML pages:

- Uses `startsWith` to support sub-locale codes (`fr-BE`, `fr-CH`, etc.)
- Guards against `null` / `undefined` via `!= null` check before `.startsWith()`
- Falls back to `'en'` for any non-fr locale, null, or absent value
- Locale comes from `satellite_configs.locale` (TEXT column, nullable, BCP 47)

This pattern is now live in `match.ts` and can be replicated to other satellite pages (`landing.ts`, `results.ts`, etc.) when per-locale content is needed.

### Pattern 2: Static i18n Object for Page-Level Strings

All locale-variant strings are collected into a single `const i18n = { ... }` object at the top of `renderMatchPage`, keyed by string name. Each value is a ternary on `effectiveLocale`. This keeps all translatable strings co-located and avoids scattered conditionals in the HTML template.

### Pattern 3: Soft Warning UX (Non-Blocking Client-Side Feedback)

The soft warning pattern (E03S09 AC4) is now established:
- Constants: `SOFT_WARN_CHARS` (threshold) + `SOFT_WARN_DELAY` (debounce ms)
- Timer variable initialized as `null`
- Input listener: `clearTimeout` + `hideSoftWarning` on every keystroke, then schedule if within range
- Submit handler: `clearTimeout` + `hideSoftWarning` before any processing
- Element style toggled via `display: none` / `display: block` (not CSS class)
- Not blocking — does not prevent submission

This pattern can be reused for other form fields requiring non-blocking length nudges.

### Pattern 4: Middle Dot Separator Convention

Middle dot U+00B7 (`\u00b7`) is the established separator in hint/instructional text for satellite pages. In French, non-breaking space U+00A0 (`\u00a0`) precedes `:` and `·` separators (typographically correct for French). Both are encoded as Unicode escapes in TypeScript source strings.

### Pattern 5: Persistent Hint Text vs Placeholder

The pattern for persistent instructional text is `<p id="hint-id">` rendered server-side (not placeholder attribute), always visible, with `aria-describedby` linking textarea to hint + counter + error. This is now the canonical approach for match page textarea guidance.
