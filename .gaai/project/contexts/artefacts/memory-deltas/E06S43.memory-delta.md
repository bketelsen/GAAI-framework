## Decision Alignment Confirmations

### DEC-126 (AI agents: allow with human confirmation gates) — IMPLEMENTED

The full tiered access model from DEC-126 is now live in code:

- **Tier 1 (free, agent-callable):** `POST /api/agent/extract`, `POST /api/agent/match` — no billing, no identity exposure.
- **Tier 2 (human gate):** `POST /api/agent/reveal` → email sent to prospect → `GET /api/agent/reveal/confirm?token=` / `GET /api/agent/reveal/refuse?token=` → KV state.
- **Tier 3 (booking, human gate):** `POST /api/agent/booking` → requires `reveal.status === 'confirmed'` → sends Cal.com link to prospect.
- **MCP server:** `POST /mcp` exposes `callibrate_describe_project`, `callibrate_find_experts`, `callibrate_request_reveal`, `callibrate_request_booking` — all 4 tools from DEC-126 long-term vision delivered.
- **API key auth:** `Authorization: Bearer {64-hex}` tied to verified prospect — agent identity model per DEC-126.

### DEC-14 / DEC-30 (Billing at booking) — CONFIRMED INTACT

## Technical Facts for Project Memory

The following facts are worth recording for future agent context:

1. **Agent API key storage:** SHA-256 hash stored in `prospect_api_keys.key_hash`. Raw key returned once at creation (201 response). Never logged, never stored.

2. **Reveal KV schema:** Key `agent-reveal:{match_id}`. Payload: `{ match_id, prospect_id, expert_id, status: 'pending'|'confirmed'|'refused', requested_at, confirmed_at?, refused_at? }`. TTL: 48h.

3. **Rate limit KV schema:** Key `agent-rate:{keyHash}:{endpoint}:{windowIndex}`. Counter-based, tumbling window. Fails open on KV error.

4. **MCP transport:** HTTP Streamable (not SSE, not WebSocket). Single POST per request. Standard JSON-RPC 2.0. Protocol version `2024-11-05`.

5. **Route naming note:** Implementation uses `/api/agent/*` (not `/api/v1/agent/*` as specified in DEC-126 medium-term vision). The `agent-keys` suffix is used for key management routes instead of `api-keys`. Paths are consistent within the implementation. Update AC/DEC if `/api/v1/` versioning is required.

6. **Token reuse risk (known, low priority):** The confirm and refuse JWT tokens use the same format. A user could use either token at either endpoint. Future improvement: add `action: 'confirm'|'refuse'` claim to the payload.
