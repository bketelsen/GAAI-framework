# E06S23 Memory Delta

## New Patterns

### 3-tier read chain for hot-path data (Cache API Ôćĺ D1 Ôćĺ Hyperdrive)
The expert pool read path uses a layered strategy with write-back at each tier:
```
1. Cache API L1 (60s TTL, per-datacenter) ÔÇö ~0ms
2. D1 edge database ÔÇö <5ms
3. Hyperdrive (Supabase PostgreSQL) ÔÇö ~20-50ms
```
Each tier falls through on miss. On a D1 or Hyperdrive hit, the result is written back up to faster tiers (non-blocking `void` + `try/catch`). Never write back an empty array ÔÇö this protects against caching a temporarily-empty state.

File: `src/lib/expertPool.ts` ÔÇö `loadExpertPool` + `loadFromHyperdrive`

### Cache API L1 with synthetic URL ÔÇö graceful degradation on workers.dev
Cache API only works on custom domains. Use a synthetic `https://cache.internal/...` URL and wrap all Cache API calls in `try/catch`. A miss (including `caches is not defined`) is normal and non-fatal. Set TTL via `Cache-Control: max-age=N` response header.

File: `src/lib/d1ExpertPool.ts` ÔÇö `getCachedPool` + `writeCachePool`

### Optional Cloudflare binding for graceful degradation
When a new binding (D1, KV, Queue, etc.) may not be provisioned yet, declare it as optional in `Env`:
```typescript
EXPERT_DB?: D1Database;
```
Guard all usage with `if (env.EXPERT_DB)`. In cron jobs, return early if the binding is absent. This avoids hard failures during incremental rollout.

File: `src/types/env.ts` line `EXPERT_DB?: D1Database`, `src/cron/syncExpertPool.ts` early-return guard

### Incremental sync pattern with `_sync_meta` table
For periodic sync jobs that must avoid re-processing unchanged rows:
1. Read `last_sync_at` from a `_sync_meta(key TEXT PRIMARY KEY, value TEXT)` table
2. If null Ôćĺ full load (no timestamp filter)
3. If set Ôćĺ incremental (`WHERE updated_at > $ts`)
4. Always update `last_sync_at` to `syncStartedAt` (captured before the query), even if no rows changed
5. Wrap entire sync in `try/catch` ÔÇö failures are logged, never propagated

File: `src/cron/syncExpertPool.ts` + `src/lib/d1ExpertPool.ts` ÔÇö `getLastSyncAt`/`setLastSyncAt`

### D1 batch upsert pattern
Use `db.batch(stmts)` with `INSERT OR REPLACE` or `ON CONFLICT (id) DO UPDATE SET` for efficient multi-row upserts. Build all statements in a `.map()` Ôćĺ pass array to `db.batch()`. All rows land in a single D1 round-trip.

File: `src/lib/d1ExpertPool.ts` ÔÇö `upsertToD1`

## Architectural Decisions

### D1 as edge read cache (not source of truth)
`expert-pool-edge` D1 is a pure edge cache populated by the `syncExpertPoolToD1` cron. Supabase (via Hyperdrive) remains the source of truth. D1 is always writable from the cron and readable from any request handler ÔÇö no user-facing writes go to D1 directly.

### KV (`EXPERT_POOL`) retained in wrangler.toml but removed from hot read path
The `EXPERT_POOL` KV binding is kept in wrangler.toml for backward compatibility (other code may reference it) but is no longer consulted by `loadExpertPool`. The migration to D1+Cache API is complete for the hot read path (AC14).

### `syncStartedAt` captured before queries
The `last_sync_at` timestamp stored in `_sync_meta` is captured with `new Date().toISOString()` **before** running the Hyperdrive queries, not after. This ensures any rows that change during the sync window are picked up in the next incremental run (prevents silent gaps caused by query latency).

## Infrastructure Note

`expert-pool-edge` D1 database requires manual provisioning before first deploy:
```bash
wrangler d1 create expert-pool-edge --env staging
# update PLACEHOLDER_D1_DATABASE_ID in wrangler.toml
wrangler d1 execute expert-pool-edge --file=d1/expert-pool-edge.schema.sql --env staging
```
The `EXPERT_DB?` optional binding means the worker deploys and runs correctly even before D1 is provisioned (falls back to Hyperdrive).
