## New Patterns Established

### 1. Satellite page pattern — results page

`results.ts` confirms the satellite page pattern established by `confirm.ts` and further validates it as the canonical pattern for all satellite pages that do NOT require Turnstile:

```typescript
export function renderResultsPage(
  config: SatelliteConfig,
  posthogApiKey: string,
  coreApiUrl: string
): string
```

Pattern elements now consistent across `landing.ts`, `match.ts`, `confirm.ts`, `results.ts`:
- Server-side `escapeHtml()` helper for config values injected into HTML
- Client-side `escHtml()` helper inside IIFE for dynamic DOM content
- `posthogHeadSnippet`: identical stub in `<head>`, `capture_pageview:false`, `persistence:"memory"`, `autocapture:true`
- `satConfigScript`: `window.__SAT__` with `replace(/</g, '\\u003c')` XSS guard
- CSS vars from `config.theme` (primary, accent, font, radius)
- Conditional logo rendering from `theme.logo_url`

**Rule clarified**: `window.__SAT__` MUST NOT include `turnstileSiteKey` on pages that do not render a Turnstile widget. Only `confirm.ts` (which hosts the human verification challenge) includes `turnstileSiteKey`. All other pages expose only `{ apiUrl, satelliteId }`.

### 2. Email gate pattern (satellite funnel)

The email gate pattern for the match results funnel is now established:

1. Cards rendered anonymized first (rank, scores, skills, rate — no PII)
2. Email gate section revealed AFTER card render (not before)
3. Inline email regex validation before API call
4. `POST /api/prospects/{id}/identify` with `{ email, token }`
5. 409 (already identified) → re-fetch full profiles via `GET /matches?identified=true`
6. On success: `revealFullProfiles()` replaces anonymized cards in-place (DOM mutation, not re-render)
7. This pattern is consistent with anti-pattern note in conventions.md: "Email gate before match reveal (DEC-15 — kills conversion)" — the gate is placed BELOW the anonymized cards, not blocking them

### 3. PostHog event schema — E03 match funnel

Five new satellite events established for the match results funnel. Add to conventions.md PostHog event schema table:

| Event | Properties | Trigger |

| `satellite.booking_cta_clicked` | `satellite_id`, `prospect_id`, `expert_rank` | On "Réserver un appel" button click |

### 4. Exponential backoff pattern for async match computation

`handle202()` implements the pattern for async API operations that return 202 while computing:
- Show "computing" message while waiting
- Exponential backoff: `delay = base * 2^retryCount` where `base = estimated_seconds * 1000` (fallback 3000ms)
- `MAX_RETRIES = 3`
- After max retries: show "not available" fallback + stop polling

This pattern is applicable to any future Worker page that needs to poll a 202-returning API endpoint.

### 5. CustomEvent dispatch as integration contract

`window.dispatchEvent(new CustomEvent('booking-open', { detail: { expertId } }))` on the booking CTA establishes the pattern for satellite page → embedding page communication. The satellite page fires a CustomEvent; the host page (or future booking modal overlay) listens on `window`. This decouples the satellite page from the specific booking implementation (E03S06).
