## New Knowledge

### 1. Supabase auth.users — NULL token columns cause sign-in failure

**Context:** When creating a Supabase auth user via raw SQL (bypassing the Supabase Auth API), the `confirmation_token`, `recovery_token`, `email_change_token_new`, `email_change_token_current`, `phone_change_token`, `reauthentication_token`, and `email_change` columns must be set to `''` (empty string), **not** NULL.

**Symptom:** `POST /auth/v1/token?grant_type=password` returns HTTP 500 with `"Database error querying schema"`. Auth logs reveal: `"sql: Scan error on column index 3, name \"confirmation_token\": converting NULL to string is unsupported"`.

**Fix:**
```sql
UPDATE auth.users SET
  confirmation_token = '', recovery_token = '', email_change_token_new = '',
  email_change_token_current = '', phone_change_token = '', reauthentication_token = '',
  email_change = ''
WHERE email = 'your-user@domain.com';
```

**Scope:** Any future test user creation, seeding scripts, or migration that inserts directly into `auth.users` must include these empty string values.

---

### 2. Supabase auth — auth.identities record required for password sign-in

**Context:** A user in `auth.users` without a corresponding `auth.identities` row cannot sign in with email/password. The sign-in returns HTTP 500.

**Required identity record:**
```sql
INSERT INTO auth.identities (provider_id, user_id, identity_data, provider, last_sign_in_at, created_at, updated_at, id)
VALUES (
  'user@domain.com',
  '<user-uuid>',
  jsonb_build_object('sub', '<user-uuid>', 'email', 'user@domain.com'),
  'email',
  NOW(), NOW(), NOW(),
  gen_random_uuid()
);
```

**Scope:** Any future test user or seed data creation via raw SQL.
