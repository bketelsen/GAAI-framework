# E06S21 Memory Delta

## New Patterns

### ctx.waitUntil fire-and-forget for non-critical async side effects
Embedding upserts use a synchronous `void`-returning function that wraps all async work in `ctx.waitUntil(async () => { try { ... } catch (err) { console.error(...) } })`. The function returns immediately — the caller is never blocked. This is the canonical pattern for any optional, failure-tolerant side effect in this codebase (e.g., AI inference, vector store writes, analytics). Failure is logged but not propagated.

File: `src/lib/vectorize.ts` — `upsertExpertEmbedding`

### Admin route namespace with service-key auth
`/api/admin/` routes bypass JWT authentication entirely. Auth is an exact string match: `request.headers.get('Authorization') === 'Bearer ' + env.SUPABASE_SERVICE_KEY`. Returns 401 on mismatch. The admin block is registered in `index.ts` after all user-facing route groups and before the catch-all 404.

File: `src/index.ts` lines 235-243, `src/handlers/admin/vectorize.ts`

### Vectorize metadata: omit null values (do not include null-keyed metadata)
Vectorize metadata values must be `string | number | boolean`. Null values must not be set as metadata keys. Use conditional inclusion: `if (value != null) metadata.key = value`. The `expert_id` key is always set (never null). Rate and availability keys are omitted when null rather than stored as null.

File: `src/lib/vectorize.ts` lines 44-49, `src/handlers/admin/vectorize.ts` lines 48-53

### Admin bulk operation: inline AI logic, do not call fire-and-forget helpers
When a handler needs to loop over entities and await each embedding step sequentially (for per-entity error isolation), it should inline the AI + Vectorize logic directly rather than delegating to `upsertExpertEmbedding`. This avoids nested `ctx.waitUntil` calls and ensures each iteration is properly awaited.

File: `src/handlers/admin/vectorize.ts` — reindex loop

### mockCtx.waitUntil test pattern for fire-and-forget functions
To test fire-and-forget functions that use `ctx.waitUntil`, implement the mock as:
```typescript
const mockCtx = {
  waitUntil: vi.fn((p: Promise<void>) => p),
} as unknown as ExecutionContext;
```
After calling the function under test, extract and await the background promise:
```typescript
const [bgPromise] = vi.mocked(mockCtx.waitUntil).mock.calls[0] as [Promise<void>];
await bgPromise;
// now assert on mocked side-effects
```
This pattern synchronously executes the background async work in tests without needing timers or workarounds.

## Architectural Decisions

### Embedding model: `@cf/baai/bge-base-en-v1.5` (768 dimensions, cosine)
Selected for the expert profile embedding model. The embedding text format is a fixed-structure sentence: `"Skills: {csv}. Industries: {csv}. Project types: {csv}. Languages: {csv}."` Empty arrays produce empty segments (e.g., `"Skills: ."`) to maintain consistent sentence structure across all profiles. The Vectorize index must be created with `--dimensions=768 --metric=cosine`.

### Registration produces an initial sparse embedding
On expert registration, `upsertExpertEmbedding` is called with `profile: {}` (empty JSONB). This creates a valid but sparse 768-dim vector (`"Skills: . Industries: . Project types: . Languages: ."`). The vector is overwritten on the first profile PATCH with actual skills/industries data. This design avoids a conditional "skip embedding if profile empty" branch and ensures every registered expert has a vector from day one.

### Reindex endpoint filters out `unavailable` experts
`POST /api/admin/vectorize/reindex` fetches experts using `.neq('availability', 'unavailable')`. This includes `'available'`, `'limited'`, and `NULL` (newly registered). Experts who have set themselves as unavailable are excluded from the vector index — they should not appear in semantic search results.

## Infrastructure Note

Vectorize index `expert-profiles` requires manual creation before first deploy to any environment:
```
npx wrangler vectorize create expert-profiles --dimensions=768 --metric=cosine [--env staging|production]
```
This is a one-time operation per Cloudflare account/environment. The wrangler.toml binding (`index_name = "expert-profiles"`) will fail at deploy time if the index does not exist.
