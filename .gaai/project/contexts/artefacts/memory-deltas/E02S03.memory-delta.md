## New Knowledge

### Pattern 1 — JSONB full-replace via partial PATCH (E02S03)

When updating a JSONB column (`preferences`, `admissibility_criteria`) via `PATCH /api/experts/:id/profile`, the action sends the entire reconstructed JSONB object. This is "full replace" semantics on the JSONB field — not a deep merge. This is the correct approach given the Layer 2 API `PATCH` endpoint accepts partial profile updates (field-level, not key-level within a JSONB column).

**Implication for future stories:** Any story adding fields to `preferences` or `admissibility_criteria` MUST ensure all existing keys are re-submitted in the form — otherwise a partial form save will silently drop unrendered keys. The current implementation avoids this by rendering all known keys and reconstructing the full object on every save.

### Pattern 2 — Percentage → fraction conversion at action boundary (E02S03)

UI-facing range sliders for 0–100% values are stored internally as 0.0–1.0 fractions in the DB (matching engine contract). Conversion occurs at the action boundary: `parseFloat(overlapStr) / 100`. The inverse (fraction → percentage for display) occurs at component mount: `Math.round(v * 100)`. This is the canonical conversion pattern for any future slider-based criteria fields.

### Pattern 3 — GcalSection intentionally has no save form (E02S03)

The Google Calendar section in the profile settings page is a read-only status display with a navigation link to `/dashboard/gcal`. This is by design: OAuth flow is a multi-step redirect sequence that cannot be embedded as a simple form action. The settings page shows connection status only; the actual OAuth connect/disconnect is handled by the dedicated gcal route (future story E02S0x).
