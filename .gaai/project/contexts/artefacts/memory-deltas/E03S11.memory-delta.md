## Pattern Confirmed: PostHog Capture Conventions by Page

The satellite frontend uses two distinct PostHog capture patterns, intentionally split by page:

- **confirm.ts** → direct `posthog.capture('event', props)` inline (no helper)
- **results.ts** → `firePostHog('event', props)` helper function defined in IIFE scope

This is a deliberate consistency convention, not a discrepancy. Any future delivery on either page should follow the same pattern as the existing code in that page. Cross-page inconsistency is acceptable given the pattern is internal to each file's IIFE.

## Pattern Confirmed: `satellite.matching_error` Event Schema

Canonical schema validated across E03S11:

```js
posthog.capture('satellite.matching_error', {
  satellite_id: string,       // window.__SAT__.satelliteId
  prospect_id: string|null,   // null on confirm page (no prospect yet), prospect_id on results page
  error_type: string,         // one of: 'network'|'server_5xx'|'validation_422'|'rate_limit_429'|'turnstile'|'computing_timeout'
  page: string,               // 'confirm' | 'results'
  retry_count: number         // 0 on confirm, 1 (hardcoded after silent retry) or retryCount (202 timeout) on results
});
```

Note: `server_5xx` error_type does not appear on the confirm page (all non-422/429 errors collapse to `network` there). This is correct — server errors on the confirm API endpoint are generic network failures from the client's perspective.

## Known Frontend/Backend Gap: AC6 Email Promise

The `no-available-msg` fallback on results.ts displays:
> "Le calcul des correspondances prend plus de temps que prévu. Vous recevrez vos résultats par email dès qu'ils sont prêts."

This is a UI promise. The actual email delivery (when matching computation completes) requires a backend notification system not yet implemented. If and when that backend story is planned, it must reference this UI text as the expected UX contract.
