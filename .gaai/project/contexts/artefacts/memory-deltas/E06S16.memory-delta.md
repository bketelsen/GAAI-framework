# Memory Delta — E06S16

## Verdict: NEW_KNOWLEDGE_FOUND

## New Patterns to Persist

### CF Workflows Pattern (First Use in Project)

**Pattern: WorkflowEntrypoint exports**
- Workflow classes MUST be named exports from the main entrypoint (`src/index.ts`) — the CF runtime discovers them by class name
- `[[workflows]]` in `wrangler.toml` is TOP-LEVEL ONLY — `[[env.staging.workflows]]` does not exist
- `this.env` is accessible in `WorkflowEntrypoint` (same as DurableObject)
- `step.sleep(label, duration)` accepts string ("7 days") or number (ms)
- Sleep acceleration pattern: `parseInt(this.env.DELAY_MS ?? String(realDurationMs))`

**Pattern: Workflow → Queue dispatch loop**
- Workflows dispatch back to the EMAIL_NOTIFICATIONS queue (new message types) rather than calling Resend directly
- This preserves retry/idempotency semantics of the queue consumer
- New message types (`booking.confirmed.enriched`, `survey.*`) are handled by the same consumer switch

### JWT Pattern Extension

- `signSurveyToken(bookingId, prospectId, secret)` uses same HMAC-SHA256 pattern as `signProspectToken`
- Survey tokens have 30-day TTL (vs 24h for prospect tokens)
- Uses `PROSPECT_TOKEN_SECRET` (no separate survey secret at MVP)

### bookings Table Denormalization

- `bookings.prospect_email` and `bookings.prospect_name` are denormalized columns (populated during booking confirm)
- Prefer querying `bookings` for prospect contact data in queue consumers (single join avoided)
- Always fallback to `prospects.email` if `bookings.prospect_email` is null
