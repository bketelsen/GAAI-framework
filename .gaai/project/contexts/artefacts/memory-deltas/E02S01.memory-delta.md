# E02S01 Memory Delta

## Verdict: NEW_KNOWLEDGE_FOUND

E02S01 introduces the first React Router v7 + CF Workers frontend worker in this project. Several new patterns are confirmed through working implementation that are not yet in `conventions.md`.

---

## New Patterns Identified

### 1. React Router v7 SSR on CF Workers — config pattern

`react-router.config.ts` must have:
```ts
export default {
  ssr: true,
  future: { unstable_viteEnvironmentApi: true },
} satisfies Config;
```
`unstable_viteEnvironmentApi: true` is required for `@cloudflare/vite-plugin` integration.

`vite.config.ts` plugin order matters:
```ts
plugins: [
  cloudflare({ viteEnvironment: { name: "ssr" } }),
  tailwindcss(),
  reactRouter(),
  tsconfigPaths(),
]
```
`cloudflare()` must come first, `reactRouter()` after `tailwindcss()`.

### 2. React Router v7 CF Workers entrypoint pattern

`workers/app.ts` canonical form:
```ts
import { createRequestHandler } from "react-router";

declare module "react-router" {
  export interface AppLoadContext {
    cloudflare: { env: Env; ctx: ExecutionContext };
  }
}

const requestHandler = createRequestHandler(
  () => import("virtual:react-router/server-build"),
  import.meta.env.MODE,
);

export default {
  fetch(request: Request, env: Env, ctx: ExecutionContext) {
    return requestHandler(request, { cloudflare: { env, ctx } });
  },
} satisfies ExportedHandler<Env>;
```

The virtual module `virtual:react-router/server-build` requires a type stub in `worker-configuration.d.ts`:
```ts
declare module "virtual:react-router/server-build" {
  import type { ServerBuild } from "react-router";
  const build: ServerBuild;
  export = build;
}
```

### 3. TailwindCSS 4 CSS-first configuration pattern

No `tailwind.config.ts`. All configuration in the CSS file:
```css
@import "tailwindcss";
@import "tw-animate-css";

@custom-variant dark (&:is(.dark *));

:root {
  --primary: oklch(0.55 0.18 250);
  --radius: 0.5rem;
  /* ... other design tokens */
}

@theme inline {
  --color-primary: var(--primary);
  --font-sans: "Inter", ui-sans-serif, system-ui, sans-serif;
  /* ... maps CSS vars to Tailwind tokens */
}
```

The `@theme inline` block bridges CSS custom properties to Tailwind utility generation. No `tailwind.config.ts` file should exist for Tailwind v4 projects.

Vite plugin: `@tailwindcss/vite` (not `@tailwindcss/postcss`) — installed as devDependency.

### 4. `@supabase/ssr` cookie-based auth pattern for CF Workers

Uses `createServerClient` from `@supabase/ssr` (not `createClient`):
```ts
import { createServerClient, parseCookieHeader, serializeCookieHeader } from "@supabase/ssr";

const supabase = createServerClient(env.SUPABASE_URL, env.SUPABASE_ANON_KEY, {
  cookies: {
    getAll() {
      return parseCookieHeader(request.headers.get("Cookie") ?? "") as { name: string; value: string }[];
    },
    setAll(cookiesToSet) {
      cookiesToSet.forEach(({ name, value, options }) => {
        responseHeaders.append("Set-Cookie", serializeCookieHeader(name, value, options));
      });
    },
  },
});
```

Session validation uses `supabase.auth.getUser()` (server-side JWT validation via Supabase), NOT `getSession()` (local-only, no server validation). Token is retrieved separately via `supabase.auth.getSession()` after getUser confirms validity.

`requireSession` pattern:
```ts
export async function requireSession(request, env): Promise<{ session: Session; responseHeaders: Headers }> {
  const { session, responseHeaders } = await getSession(request, env);
  if (!session) throw redirect("/login");
  return { session, responseHeaders };
}
```
The `throw redirect("/login")` is the React Router v7 pattern — loader returns a Response redirect via thrown value.

### 5. React Router v7 auth layout + dashboard shell two-layer pattern

Two nested layout routes serve distinct roles:
- `_layout.tsx`: pure auth guard — calls `requireSession()`, returns loader data, renders `<Outlet />`. No UI.
- `_layout.dashboard.tsx`: UI shell — calls `requireSession()` (independent, for its own data), renders sidebar/header/`<Outlet context={{ user }}>`.

Child routes (e.g., `dashboard.tsx`) access user via `useOutletContext<{ user: SessionUser }>()` from the dashboard shell, not the auth guard.

This double-call to `requireSession` is intentional: `_layout.tsx` guards all authenticated routes, `_layout.dashboard.tsx` provides the dashboard shell with user context.

### 6. React Router v7 NavLink active state pattern

```tsx
<NavLink
  to={to}
  end={end}
  className={({ isActive }) =>
    isActive ? "bg-primary text-primary-foreground" : "text-muted-foreground hover:bg-accent"
  }
/>
```
`end={true}` required for the index route (`/dashboard`) to prevent it matching all child routes. Other nav items use `end={false}`.

### 7. shadcn/ui v2 with `radix-ui` unified package

shadcn/ui now imports from the unified `radix-ui` package (not individual `@radix-ui/*` packages):
```ts
import { Slot } from "radix-ui"         // instead of @radix-ui/react-slot
import { Dialog as DialogPrimitive } from "radix-ui"  // instead of @radix-ui/react-dialog
import { Tabs as TabsPrimitive } from "radix-ui"      // etc.
```
The `radix-ui` package is the umbrella package that re-exports all Radix UI primitives. This is the current shadcn/ui CLI output (2026).

### 8. Vitest config for React Router v7 (Workers runtime, no JSX transform needed)

```ts
// vitest.config.ts
export default defineConfig({
  plugins: [tsconfigPaths()],
  test: {
    environment: "node",
    exclude: ["node_modules/**", "**/node_modules/**"],
  },
});
```
Tests for loader functions (pure async functions) run in node environment. No jsdom needed — loaders return `Response` objects, not DOM. No React Testing Library required for loader-level tests.

Test pattern for protected loaders:
```ts
vi.mock("../app/lib/session.server", () => ({ requireSession: vi.fn() }));
// mock throws redirect → catch as Response → assert status 302 + Location header
```

## Conventions Drift

None detected. All implementation choices are consistent with:
- Workers entrypoint `satisfies ExportedHandler<Env>` pattern
- `Env` interface in `worker-configuration.d.ts`
- Monorepo workspace structure `workers/frontend/*`
- Naming convention `callibrate-dashboard` + `-staging`/`-prod` env suffix

The naming deviation note: default worker name is `callibrate-dashboard-dev` (not `callibrate-dashboard-staging` or `callibrate-dashboard-prod` per DEC-32 suffix convention). The `-dev` suffix on the default environment name is for the local/dev binding only — staging and prod correctly use `-staging`/`-prod`. This is consistent with how other workers handle their default (local dev) naming.
