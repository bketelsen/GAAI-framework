## Pattern 1: Vitest module mocks in Cloudflare Workers environment

**Pattern:** When mocking a module with `vi.mock(...)` where the mock needs to be overridden per-test using `vi.mocked(fn).mockImplementation(...)`, the exported function in the factory MUST be a `vi.fn()`, not a plain arrow function.

**Wrong:**
```ts
vi.mock('../../lib/db', () => ({
  createSql: () => { ... }  // plain arrow — cannot be vi.mocked()
}));
```

**Correct:**
```ts
vi.mock('../../lib/db', () => {
  const defaultFactory = () => { ... };
  return { createSql: vi.fn().mockImplementation(defaultFactory) };
});
```

**Applies to:** Any test file where a module mock needs to be overridden in a specific test case.

## Pattern 2: `globalThis` instead of `global` in Workers-typed environments

**Pattern:** In TypeScript files using `@cloudflare/workers-types`, `global` is not defined. Use `(globalThis as any).fetch = ...` to assign globals in tests.

**Wrong:**
```ts
global.fetch = vi.fn(); // TS2304: Cannot find name 'global'
```

**Correct:**
```ts
(globalThis as any).fetch = vi.fn();
```

**Applies to:** All landing and API test files that mock fetch.

## Pattern 4: Dashboard TSC pre-existing baseline

**Pattern:** The dashboard cloudflare TSC (`tsconfig.cloudflare.json`) produces `type 'never'` errors on all `useLoaderData<typeof loader>()` calls. As of E02S12, the baseline is **109 errors** (97 pre-existing + 12 from dashboard.direct-link.tsx following identical pattern).

**Assessment:** These are a known structural limitation of how React Router typed loaders interact with the Cloudflare Workers runtime types. They do not indicate implementation bugs and do not affect runtime behavior. New dashboard routes will add ~12 errors of this pattern per route file.

## Pattern 5: Mock type safety for `sql.end` in postgres.js mocks

**Pattern:** `postgres.js`'s `Sql<{}>` type includes `.end()`, `.CLOSE`, `.END`, etc. Vitest's `Mock<Procedure>` does not. To add `.end` to a mock sql object without TSC errors:

```ts
const mockSql = vi.fn().mockImplementation(...)
;(mockSql as any).end = vi.fn().mockResolvedValue(undefined)
return mockSql as any  // pass as any to satisfy Sql<{}> parameter
```

## Architectural note: Direct Booking Security Pipeline

The E02S12 direct booking POST (`/api/bookings/direct/:slug`) implements a 10-stage security pipeline in order:

1. Slug format validation (404 if invalid)
2. JSON parse (400 if invalid)
3. DB lookup → expert exists (404 if not)
4. HMAC token re-validation (403 if invalid)
5. Monthly quota check ≥ 100 (429 if exceeded)
6. Honeypot fields (silent 200 — do not reveal)
7. Timing check < 3000ms (silent 200)
8. CF Rate Limiting (429 if exceeded)
9. Turnstile verification (400 if failed)
10. Input validation (422 if invalid)

Order matters: honeypot/timing check comes BEFORE rate limiting and Turnstile to avoid wasting Turnstile verifications on bots.

## Convention: lead_source column

`bookings.lead_source` is nullable text: `'direct'` for direct bookings, `'callibrate'` for standard matched bookings, `NULL` for pre-E02S12 bookings.

`cleanupExpiredConfirmations` in `cron.ts` branches on `lead_source IS DISTINCT FROM 'direct'` (30min TTL) vs `lead_source = 'direct'` (24h TTL). Future cleanup logic involving `pending_confirmation` bookings must preserve this distinction.
