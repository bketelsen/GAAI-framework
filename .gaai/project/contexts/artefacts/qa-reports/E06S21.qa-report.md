## Notable Deviations from Plan

### Deviation 1 â€” AC1: wrangler.toml dimensions not specified in TOML

**Scope:** AC1 wording says "768 dims reference, cosine metric".

**Observation:** `wrangler.toml` contains only `index_name = "expert-profiles"`. Dimensions (768) and metric (cosine) are not expressed as TOML fields â€” they are provisioned at index-creation time using `wrangler vectorize create expert-profiles --dimensions=768 --metric=cosine`.

**Assessment:** This is correct Cloudflare Vectorize behavior. The TOML binding does not support `dimensions` or `metric` fields â€” those are properties of the index itself, set once at creation. The plan's AC1 text "768 dims reference" was describing the index spec, not a TOML field requirement. The plan explicitly documents the CLI command under "Human Action Required Before Deploy". No action needed.

### Deviation 2 â€” AC5: Metadata omits null values (conditional inclusion)

**Scope:** AC5 requires `expert_id`, `rate_min`, `rate_max`, `availability` in Vectorize metadata.

**Plan spec (Step 2c):** Used `?? null` pattern, which would include keys with `null` values unconditionally.

**Implementation:** Uses conditional `if (profileData.rate_min != null) metadata.rate_min = profileData.rate_min` â€” keys are only added when the value is non-null. `expert_id` is always set.

**Assessment:** The implementation is superior. Vectorize metadata values must be `string | number | boolean` â€” `null` is not a valid Vectorize metadata type. Omitting null keys is the correct pattern and avoids a potential runtime type error. All 4 metadata keys from AC5 are still present when the underlying data is non-null. Tests confirm this behavior explicitly. No action needed.

### Deviation 3 â€” Admin reindex inlines AI logic (does not call `upsertExpertEmbedding`)

**Scope:** `src/handlers/admin/vectorize.ts` reindex loop.

**Observation:** Rather than delegating to `upsertExpertEmbedding` (which is fire-and-forget internally), the reindex handler inlines `buildEmbeddingText`, `env.AI.run`, and `env.VECTORIZE.upsert` directly inside its own `ctx.waitUntil` loop.

**Plan context:** The execution plan explicitly addressed this in Step 5 â€” it noted the semantic mismatch of nested `waitUntil` calls and revised the implementation to inline the logic. The implementation matches the plan's revised final version.

**Assessment:** Correct and intentional. The inline approach ensures the reindex loop properly `await`s each expert embedding before moving to the next, providing per-expert error isolation without nested `waitUntil` complexity.

## Pattern Observations

### New Pattern: `ctx.waitUntil` + fire-and-forget for non-critical side effects

The embedding pipeline establishes a new codebase pattern for handling optional, non-critical async side effects: synchronous function returning `void`, with all async work wrapped in `ctx.waitUntil(async () => { try { ... } catch { console.error(...) } })`. This pattern is used for both the inline embedding (`upsertExpertEmbedding`) and the batch reindex.

### New Pattern: Admin routes under `/api/admin/` with service-key auth

`index.ts` introduces a new route namespace `/api/admin/` with its own auth strategy (exact match against `SUPABASE_SERVICE_KEY` via `Authorization: Bearer`). This is distinct from the JWT-based auth used by `/api/experts/` routes.
