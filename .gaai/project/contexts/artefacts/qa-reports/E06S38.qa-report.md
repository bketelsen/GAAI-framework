## Deviations Noted (from impl-report)

All 5 deviations from the execution plan are acceptable:

1. `handleEvaluateLead` takes `ctx: ExecutionContext` as 5th param — required for `ctx.waitUntil()`, router passes it correctly.
2. Bookings uses `COUNT(*) OVER()` window function instead of separate COUNT — valid optimization, reduces round-trips.
3. `POSTHOG_API_KEY` omitted in leads.test.ts `mockEnv` — captureEvent is a no-op without key; production behavior unaffected.
4. `lib/qualityTier.ts` defines its own `QualityTier` type — avoids circular dependency risk; type is identical (4-value union).
5. Skills filtering done in TypeScript after SQL fetch — documented MVP choice; functional at MVP scale.

## Minor Observations (non-blocking)

- **`total` accuracy with skills filter**: When `?skills=` is supplied, the SQL `COUNT(*) OVER()` total reflects the pre-filter count (not the post-filter count). The filtered array shrinks but `total` stays at the SQL value. This is a known MVP limitation documented in the execution plan and is not an AC violation — AC17 does not specify total accuracy for the skills filter case.
- **`cancel_reason` format for public endpoint**: The `/public` and `/public/:slug` endpoints correctly omit `cancel_reason` (it is a private field on bookings, not surfaced in AnonymizedExpert shapes).
