## Detailed Findings

### PKCE Implementation Correctness

The PKCE implementation is correct per RFC 7636 S256 method:

- `code_verifier`: 32 random bytes encoded as BASE64URL (no padding, `-` and `_` replacing `+` and `/`). Length is 43 characters (32 bytes * 4/3 rounded, no padding), which satisfies the RFC 7636 requirement of 43-128 characters.
- `code_challenge`: `BASE64URL(SHA256(ASCII(code_verifier)))` — implemented via `crypto.subtle.digest('SHA-256', encoder.encode(codeVerifier))` then `base64urlEncode(digest)`.
- Both use the Web Crypto API (`crypto.getRandomValues`, `crypto.subtle.digest`), which is available in Cloudflare Workers.

### CSRF State Token

- 32 bytes from `crypto.getRandomValues` hex-encoded = 64-character lowercase hex string.
- Stored as KV key `oauth-state:{stateToken}` with TTL 600 seconds.
- Deleted immediately after first read in callback (one-time use, preventing replay attacks).
- State param in auth URL is the random token, not the expert UUID.

### KV Lifecycle

The KV entry lifecycle is correct:
1. Created in `handleGcalAuthUrl` with TTL 600s.
2. Read in `handleGcalCallback` via `env.SESSIONS.get(...)`.
3. Immediately deleted (line 133) before any subsequent processing, preventing replay.
4. If missing (never existed, expired, or already consumed), returns `invalid_state` redirect.
