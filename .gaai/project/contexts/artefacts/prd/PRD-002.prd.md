# PRD-002: GAAI-Cloud

## Purpose

Frame GAAI-Cloud as the commercial platform built on the GAAI OSS framework. The product trilogy:

| Artefact | Role | Audience |
|----------|------|----------|
| **GAAI OSS** | Proof-of-concept | Developers, Claude Code users (tech-savvy only) |
| **Callibrate.io** | Proof-of-work | Demonstrates GAAI framework value in production |
| **GAAI-Cloud** | La plateforme | Everyone else — abstraction layer for non-devs + teams |

GAAI OSS is deliberately scoped to technical users (Claude Code). It is NOT packaged as a Cowork plugin — that would blur the audience boundary. GAAI-Cloud is the product for everyone who isn't comfortable with a local `.gaai/` folder and a CLI.

GAAI-Cloud provides cloud-hosted workspace storage with an MCP interface, enabling multi-device portability, team real-time collaboration, and framework versioning — none of which GAAI OSS can provide.

> **Trigger conditions (two stages):**
> - **Gate 2 — Technical build starts:** GAAI OSS published publicly. Low complexity (2–3 months, no resource conflict with Callibrate). OSS community begins forming → early feedback available.
> - **Gate 3 — Commercial launch:** Callibrate PASS (≥10 calls payés, GMV ≥1 500€, ≥3 experts re-payants). Proof-of-work established, GAAI credibility validated, audience exists to sell to.
>
> **Why two stages:** GAAI-Cloud is infrastructure-only (DO + R2 + MCP + auth + billing). The AI tool is the client — no custom algorithm, no frontend, no two-sided acquisition. Build time ~2–3 months vs ~12–18 months for Callibrate. The build can start at Gate 2 without competing for founder time. The commercial launch waits for Gate 3 because without a GAAI community, there are no subscribers.

---

## Problem Statement

**For solo GAAI users:** Working across multiple devices (Mac + Windows, personal + client machine) requires manually copying `.gaai/` folders or managing a Git repo just for framework state. Context, backlog, and memory are lost or out-of-sync across devices.

**For teams on GAAI:** Multiple team members editing the same memory, backlog, or artefacts via Git creates merge conflicts and stale state. Git is not designed for concurrent, session-by-session writes to operational files. There is no real-time coordination layer.

**For GAAI framework maintainers:** Distributing framework updates (agent definitions, core skills) requires every user to manually pull from the OSS repo. No centralized versioning or update delivery mechanism exists.

**Root cause:** GAAI OSS is a local-first framework. Local-first is correct for single-device solo use. It is inadequate for multi-device solo use and untenable for team use.

---

## Goals

- Enable any GAAI user to switch devices without losing context, backlog, memory, agents, or skills
- Enable multiple team members to work on the same GAAI workspace without conflicts
- Deliver framework updates (agents, core skills) to all connected users automatically
- Generate recurring revenue from the GAAI framework, independent of Callibrate
- Establish GAAI OSS as the distribution vector for GAAI-Cloud (adoption → conversion)

---

## Non-Goals

- GAAI-Cloud is NOT a hosted LLM inference provider — inference always runs in the user's AI tool
- GAAI-Cloud is NOT an MCP aggregator for external tools (Notion, Google Drive, Slack, etc.) — those are configured directly at the AI tool level
- GAAI-Cloud does NOT provide a web UI for primary workflow execution — AI is the interface
- GAAI-Cloud does NOT execute agents (Discovery or Delivery) — it is pure storage + MCP
- GAAI-Cloud is NOT a replacement for GAAI OSS — OSS remains complete for solo/single-device use
- No mobile app
- No white-label offering at launch

## Users

### GAAI OSS users (Claude Code — not GAAI-Cloud customers by default)
Developers comfortable with CLI, local files, Git. Single-device. They are the **top of funnel**: they discover the framework, use it, hit multi-device or team pain, and convert to GAAI-Cloud. OSS is the acquisition channel, not the product.

### GAAI-Cloud users

**Primary: Solo practitioners (Solo tier)**
Consultants, founders, non-dev operators who want GAAI without managing a local `.gaai/` folder. Use Claude Cowork or Claude.ai — not Claude Code. Pain: no local framework setup, need multi-device portability, want the abstraction layer.

**Also Solo:** GAAI OSS developers who hit multi-device pain and upgrade. Same tier, different entry path.

**Secondary: Small teams (Team tier)**
2–8 person teams (agencies, startups, consultancies). Multiple members need shared backlog + memory without Git conflicts. Pain: no real-time coordination, inconsistent agent behavior between team members (each has their own local OSS version).

**Tertiary: Enterprises (Enterprise tier)**
Larger organizations needing SSO, audit logs, private deployment, compliance controls. Not day-1 addressable — require Team tier validation first.

**Prerequisite:** GAAI-Cloud Solo requires an existing GAAI OSS install or a first-time Cowork plugin install. GAAI-Cloud is never a cold-start product — GAAI OSS (or the Cowork plugin) is always the entry point.

## Architecture Decisions

### Core principle

> GAAI-Cloud = MCP server hosting `.gaai/` content only.
> Everything else (LLM inference, file generation, agent behavior) stays local in the user's AI tool.

### Three-level architecture

```
Organization  (billing entity — owns all workspaces)
│
├── User Workspace(s)       1 per user — portable identity, travels across all projects
│   ├── agents/
│   │   ├── core/           ← GAAI managed agents (versioned by GAAI-Cloud)
│   │   └── custom/         ← user agents (personal, highest priority)
│   ├── skills/
│   │   ├── core/           ← GAAI managed skills (versioned by GAAI-Cloud)
│   │   ├── blueprints/     ← premium domain packs (Content, SaaS, etc.)
│   │   └── custom/         ← user skills (personal, highest priority)
│   ├── workflows/
│   │   ├── core/           ← GAAI managed workflows
│   │   └── custom/         ← user workflows (personal, highest priority)
│   └── scripts/            ← advanced only, opt-in, hash-verified
│       ├── core/
│       └── custom/
│
└── Project Workspace(s)    Unlimited per org — isolated per project
    ├── contexts/
    │   ├── memory/         ← project knowledge base
    │   ├── backlog/        ← project work queue (authoritative)
    │   ├── artefacts/      ← project deliverables
    │   ├── rules/          ← project-specific governance rules
    │   └── reports/
    │       ├── implementation/
    │       └── qa/
    └── members[]           ← which users have access to this project
```

**Priority rule (User Workspace):** `custom/` always overrides `core/`. GAAI-Cloud updates `core/` freely without touching user customizations.

**Isolation rule (Project Workspace):** Each project is fully isolated. Project A's backlog never bleeds into Project B. Users are assigned per project — a contractor only sees the projects they're invited to.

**Portability rule (User Workspace):** The User Workspace is not inside a project. When switching projects, agents and skills stay identical. Only the active Project Workspace changes.

```
MCP connection flow:
  1. Auth → identify user → sync User Workspace (hash manifest)
  2. List projects → user selects active project
  3. Connect Project Workspace (live reads via DO)

gaai project list                  # all projects in current org
gaai project switch client-b-mvp   # switch active project
gaai project create my-saas        # new isolated project
```

### Read patterns (by level)

**Project Workspace — always live (no local cache)**
Every agent request reads `contexts/` directly from GAAI-Cloud via MCP. Never cached locally. Concurrent writes from multiple team members → Durable Objects (real-time, conflict-free). This is the core differentiator over Git. One DO instance per Project Workspace.

**User Workspace — cloud source of truth, local versioned cache**
On MCP connect, GAAI-Cloud sends a manifest `{ agents: {hash}, skills: {hash}, workflows: {hash} }`. The local client compares hashes against its cache. Changed files are pulled; unchanged files are served from cache at zero latency. Cloud always wins (no bidirectional sync — no conflicts). On a new device: all hashes differ → full pull → ready.

```
On MCP connect:
  1. Auth → identify user
  2. Receive User Workspace manifest → diff → pull changed files
  3. User selects active project → connect Project Workspace (DO)

During session:
  → agents / skills / workflows : local cache (zero latency)
  → contexts/                   : live MCP read per request (DO)

On new device:
  → Empty cache → full User Workspace pull → select project → ready

On project switch:
  → User Workspace unchanged (already cached)
  → New Project Workspace connected (DO switch)
```

### Infrastructure stack (Cloudflare)

| Layer | Service | Use |
|-------|---------|-----|
| Real-time concurrent writes | Durable Objects | **1 DO instance per Project Workspace** — memory, backlog, artefacts, delivery locks. Scales horizontally with project count. |
| User Workspace file storage | R2 | Agents, skills, workflows, scripts (blob storage, versioned per user) |
| Metadata & manifests | D1 | Hash manifests, org/user/project metadata, project membership, subscription state |
| Auth sessions & token cache | KV | JWT sessions, active project per user, rate limiting |
| MCP server | Workers | Single MCP endpoint — routes to correct DO instance based on active project |
| Headless Delivery execution | CF Workflows | Delivery agent triggered by approved backlog Story (future — not day-1) |

### MCP interface

GAAI-Cloud exposes a single MCP server. AI tools connect to it as a standard MCP server. No AI tool–specific adaptation required. Compatible with: Claude Code, Claude Cowork, Claude.ai projects, Cursor, Windsurf, any MCP-compatible AI tool.

GAAI-Cloud does NOT proxy external tool MCP servers. Each external tool (Notion, Google Drive, Slack) is configured directly in the AI tool's MCP config.

### Claude Cowork plugin structure

**Design principle:** GAAI OSS is NOT distributed as a Cowork plugin. The Cowork plugin is a **thin connector to GAAI-Cloud** — no local agents, no local skills. Everything comes from the User Workspace via MCP. This maintains the audience boundary: Cowork users are GAAI-Cloud users, not OSS users.

Verified against official Anthropic plugin documentation (2026). Key constraint: `commands/` lives at plugin ROOT — NOT inside `.claude-plugin/`. Only `plugin.json` goes in `.claude-plugin/`.

```
gaai-cloud/                    ← plugin root (thin connector — installed to ~/.claude/plugins/cache/gaai-cloud/)
│
├── .claude-plugin/
│   └── plugin.json            ← manifest (only file inside .claude-plugin/)
│
├── .mcp.json                  ← GAAI-Cloud MCP server config
│
└── commands/                  ← slash commands (call GAAI-Cloud MCP tools)
    ├── gaai-discover.md       → /gaai-cloud:gaai-discover
    ├── gaai-deliver.md        → /gaai-cloud:gaai-deliver
    ├── gaai-status.md         → /gaai-cloud:gaai-status
    └── gaai-bootstrap.md      → /gaai-cloud:gaai-bootstrap
```

No `agents/` or `skills/` in the plugin. Both come from GAAI-Cloud User Workspace, synced to local cache on MCP connect.

**plugin.json** (`.claude-plugin/plugin.json`):
```json
{
  "name": "gaai-cloud",
  "version": "1.0.0",
  "description": "GAAI framework — connect to your GAAI-Cloud workspace for dual-track AI agent governance",
  "author": { "name": "Frédéric Geens", "url": "https://gaai.io" },
  "commands": "./commands/",
  "mcpServers": "./.mcp.json"
}
```

**Plugin commands are namespaced:** `/gaai-cloud:gaai-discover`, `/gaai-cloud:gaai-status`, etc.

**.mcp.json** (plugin root):
```json
{
  "mcpServers": {
    "gaai-cloud": {
      "command": "npx",
      "args": ["@gaai/cloud-mcp", "--workspace", "${GAAI_WORKSPACE_URL}"],
      "env": {
        "GAAI_TOKEN": "${GAAI_AUTH_TOKEN}"
      }
    }
  }
}
```

**SKILL.md frontmatter** (confirmed schema — for skills living in GAAI-Cloud User Workspace):
```yaml
---
name: generate-stories
description: Generate GAAI Stories with acceptance criteria from an Epic. Activate when a refined Epic needs executable Stories for the backlog.
user-invocable: false
context: fork
agent: general-purpose
allowed-tools: Read, Write, Edit
---
```

### Scripts security model

Scripts contain arbitrary executable code. Auto-sync of unreviewed scripts is a supply chain attack vector.

Rules:
1. `scripts/` is disabled by default — user must opt in at workspace level
2. Scripts synced from cloud are hash-verified against a manifest signed at upload time
3. First execution on a new device requires explicit user confirmation
4. No auto-execution of scripts — manual invocation only

## Key Requirements

### MCP Server
- Single MCP endpoint per workspace (GAAI-Cloud URL)
- Authentication: web browser sign-in flow on first connect, JWT token cached in KV
- Multi-workspace: user can belong to multiple Organization Workspaces (client A, client B, personal)
- MCP operations: read/write for all `contexts/` paths, read for User Workspace (writes go via workspace management UI or CLI)

### Project Workspace
- Unlimited per organization, all tiers
- Real-time concurrent read/write via Durable Objects (1 DO instance per project) — no merge conflicts
- All `contexts/` operations (memory-retrieve, memory-ingest, backlog reads/writes, artefact writes) go through GAAI-Cloud MCP
- Delivery locks managed internally by DO (not file-based)
- Member list per project — users only see projects they're invited to
- Full write history for audit (Enterprise tier)
- Project management: `gaai project list|create|switch|archive`

### User Workspace
- Hash-manifest sync on MCP connect (pull only changed files)
- Cloud = source of truth — local = read-only cache (no write-back from local to cloud during session)
- Update path: user edits skills/agents via workspace management → new hash → pulled on next connect
- Blueprint pack activation: user subscribes to pack → files appear in User Workspace `skills/blueprints/` → synced to all devices on next connect

### Authentication & workspace management
- Sign-in: web browser OAuth (GitHub, Google) — opens browser from AI tool, returns token
- Workspace switching: MCP tool `gaai_workspace_switch` — user selects from their workspace list
- Team invitation: email-based, workspace admin role required
- No CLI-only signup — browser required for initial auth (security baseline)

### Framework update delivery
- When GAAI publishes a new version, User Workspace hash manifest is updated server-side
- On next MCP connect, user receives updated agents/skills automatically
- Version pinning available (Team+): workspace admin can pin to a specific GAAI version
- Changelog delivered via MCP notification on connect

### Migration: GAAI OSS → GAAI-Cloud

A developer migrating from GAAI OSS to GAAI-Cloud must not lose any of their custom agents, skills, or workflows. The migration is **atomic** — local files are never modified until cloud upload is fully verified.

**Migration flow (`gaai cloud migrate`):**

```
1. Scan local .gaai/
2. Create workspace on GAAI-Cloud (or join existing org workspace)
3. Upload contexts/ → new Project Workspace (named after the repo by default)
4. Diff skills/ against GAAI OSS baseline:
     → unmodified core skills  : User Workspace skills/core/ (GAAI-Cloud managed version)
     → modified core skills    : User Workspace skills/custom/ (user version preserved, takes priority)
5. Upload agents/, workflows/, scripts/ → User Workspace (intact, unchanged)
6. Hash-verify all uploads (integrity check)
──── local files untouched until this point ────
7. Update .mcp.json with cloud workspace endpoint
8. Update CLAUDE.md canonical files section (contexts/ → MCP)
9. Archive .gaai/ → .gaai.backup/ (not deleted — user can restore)
```

**Custom agents/skills/workflows:** always preserved in User Workspace. They are the user's intellectual property — GAAI-Cloud hosts them, does not replace them.

**Conflict on modified core skill:**
```
⚠ skills/validate-artefacts — modified locally vs GAAI OSS baseline
  [k] Keep yours  (→ skills/custom/validate-artefacts, overrides core)
  [c] Take GAAI-Cloud version  (may have received updates)
  [d] Show diff
```

**Multi-user org migration** (two OSS users joining the same org):
Each user migrates their own `.gaai/` as a separate Project Workspace by default. If they were working on the same project (same repo), the second user sees a merge report:
```
Merge report — contexts/ (Alice → project: acme-main)
✓ memory/decisions/_log.md    — 12 entries Alice + 8 Bob → 20 merged (append)
✓ backlog/active.backlog.yaml — 4 stories Alice + 6 Bob → 10 merged
⚠ artefacts/prd/PRD-001.prd.md — EXISTS IN BOTH → [Alice] [Bob] [diff]
✓ rules/orchestration.rules.md — identical in both
```
Each user's User Workspace remains separate — portable identity, no merge needed.

**Exit / export (`gaai cloud export`):**
Full workspace export to local `.gaai/` folder, OSS-compatible. Anti-lock-in guarantee. Always available regardless of subscription status.

### User Workspace sharing & fork

**Visibility model** (per item: agent, skill, workflow):

| Level | Who can see | Who can import |
|-------|-------------|----------------|
| `private` (default) | Owner only | Nobody |
| `org-shared` | All org members | All org members |
| `public` | Anyone (future marketplace) | Anyone |

**Fork semantics** (not sync):
- A user browses org-shared items from colleagues
- Import = snapshot copy into their own User Workspace
- After import: their copy is independent — edits by the original owner do NOT propagate
- No surprise overwrites, no automatic updates

**Import commands:**
```bash
gaai workspace browse                          # list org-shared items
gaai workspace import bob/skills/code-review   # fork one item
gaai workspace import bob --filter skills      # fork all shared skills from Bob
```

**Conflict on import** (item already exists in own workspace):
```
Conflict: skills/code-review already exists in your workspace.
  [k] Keep yours   [t] Take colleague's   [d] Show diff   [r] Rename to code-review-bob
```

## Monetization Model

### Tiers

| Tier | Monthly | Annual | Devices | Users | Project Workspaces | User Workspace |
|------|---------|--------|---------|-------|--------------------|----------------|
| **GAAI OSS** | Free | — | 1 (local) | 1 | 1 (local `.gaai/`) | Local only |
| **Solo** | $9/month | $84/year ($7/mo) | Unlimited | 1 | Unlimited | Included |
| **Team** | $18/seat/month | $168/seat/year ($14/mo) | Unlimited | N | Unlimited | 1 per user, included |
| **Enterprise** | Custom | Custom | Unlimited | N | Unlimited | 1 per user, included |

**Projects are free.** Revenue lever = seats only. A solo freelancer with 10 client projects pays $9/month.

**Minimum team bill:** 2 seats annual = $28/month — deliberate impulse-buy threshold.

### Seat ownership model

**Billing is at the Organization level, not the individual.**

```
Contractor Caio (freelancer)
├── Client A → Organization Workspace A (Team)
│   Client A pays: 3 seats × $14 = $42/month  ← Caio is seat #3
│
├── Client B → Organization Workspace B (Team)
│   Client B pays: 2 seats × $14 = $28/month  ← Caio is seat #2
│
└── Caio's User Workspace → his personal agents, skills, workflows
    Travels with him across all orgs. Isolated: Client A ≠ Client B.
    Covered by Team membership — no personal subscription needed.
```

Each organization pays for its active members. A contractor/freelancer is a seat in each client's org — each client pays. The contractor never double-pays.

### User Workspace coverage rule

| Situation | Who pays | User Workspace synced |
|-----------|----------|-----------------------|
| Active seat in ≥1 Team workspace | The organization (per seat) | ✓ Included |
| Solo (no Team workspace) | The user ($9/month) | ✓ Included |
| No subscription | — | ✗ Sync disabled (local cache only) |

**Transition:** When a user leaves all Team workspaces → notification: *"No active Team workspace. Subscribe to Solo to keep your User Workspace synced across devices."* Local cache remains accessible offline — nothing is deleted.

### Account sharing defense: feature gating only

No technical enforcement (device counting, IP checks, fingerprinting). Defense is structural:

- Solo = 1 User Workspace for the entire account. A "team" sharing 1 Solo account shares 1 set of agents/skills — one person's changes overwrite everyone's.
- No team invitation system on Solo. No admin roles. No agent consistency lock.
- Concurrent sessions → second session read-only ("Another session is active on this account"). Natural protection for legitimate Solo users too.
- If a team prefers a broken shared experience over $28/month → value problem, not enforcement problem.

### Blueprint Packs

Domain-specific skill packs: curated knowledge base (AKUs) + designed skills for a vertical.

| Pack | Content | Solo add-on | Team+ |
|------|---------|-------------|-------|
| Content Blueprint | CNT-001→CNT-011, 141 AKUs | $29 one-time | Included |
| SaaS GTM Blueprint | positioning, launch, growth skills | $29 one-time | Included |
| [additional verticals] | Legal/Compliance, Marketing, etc. | $29 one-time | Included |

$29 = impulse purchase threshold (no budget approval needed). Included in Team+ = strong upgrade incentive for Solo users who want multiple packs.

### Distribution logic

```
GAAI OSS (free — Claude Code, tech-savvy, single-device)
    ↓  pain: multi-device or non-technical user
Solo $9/month — unlimited devices, any AI tool, Cowork abstraction
    ↓  pain: 2+ people need the same workspace
Team $14/seat/month annual — real-time collaboration (DO), per-user workspaces
    ↓  pain: compliance, audit, private infra
Enterprise — custom
```

GAAI OSS is complete for its intended audience. GAAI-Cloud is a different product for a different audience — not a hierarchy, not a stripped-down teaser.
