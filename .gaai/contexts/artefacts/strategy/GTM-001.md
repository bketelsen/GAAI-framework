## Phase 2 — Premier cohort + première boucle réelle (J21–J45)

**Objectif :** 10 experts réels onboardés. Premiers prospects réels
qualifiés. Premiers appels bookés. Signal de qualité produit.

**Prérequis :** Gate 0 PASS + Gate 1 PASS

**Actions :**
- Contacter les hand raisers experts de Phase 0 (founder-led)
- Onboarding manuel des 10 premiers experts (profil + Cal.com)
- Lancer les 3 premiers satellite sites — Tier 1 verticals (DEC-84, MARKET-001):
  - V3: AI Chatbot / Conversational AI (score expert pain 27/35, gap compétitif 5/5)
  - V1: Workflow Automation — n8n, Make, Zapier (score 26/35, gap 5/5)
  - V8: AI Integration for SaaS — LLM, RAG, agents (score 26/35, gap 5/5)
  - Supply overlap: un expert peut couvrir les 3 (ex: Caio = n8n + Python + Claude)
- Contacter les hand raisers prospects de Phase 0
- Premiers quiz soumis → premiers matchings → premiers bookings
- Collecter les premiers post-call surveys (prospects + experts)

**Proof of Quality Mode (DEC-82)**

Le premier cohort d'experts (10 cibles) opère en mode **proof of quality** — ils reçoivent des leads matchés **gratuitement** pendant Phase 2. Objectif : démontrer la qualité du matching AVANT d'activer le billing (Phase 3). Les experts voient le matching rationale et notent la qualité des leads. Ce n'est PAS un free tier permanent — c'est une période de preuve limitée au premier cohort.

Pourquoi : le field data Discovery (E01S01) révèle un scepticisme structurel envers les plateformes payantes de leads. Les experts ont été échaudés par Upwork et similaires. Le barrier to adoption n'est pas le prix — c'est la confiance. Montrer la qualité d'abord, facturer ensuite.

**Gate 2 — SMART**

| Critère | Mesure | Seuil |
|---|---|---|
| Experts vérifiés | Profils complets + calendrier connecté | ≥ 10 |
| Prospects soumis | Quiz completions + extraction AI | ≥ 20 |
| Appels bookés | Cal.com bookings confirmés | ≥ 5 |
| Score appel moyen | Post-call survey prospects | ≥ 4/5 |
| Lead quality rating | Expert rating of matched leads (proof of quality) | ≥ 4/5 |
| Conversion | Prospects → booking | ≥ 35% |
| Deadline | J45 | Hard |

```
PASS → Phase 3 débloquée. Le produit délivre de la valeur réelle.
FAIL → Analyser le goulot :
  → Peu de soumissions ? → Problème acquisition prospect / funnel
  → Peu de matchings ? → Problème engine / profils experts
  → Peu de bookings ? → Problème reveal / booking UX
  → Score < 4/5 ? → Problème qualité experts / brief prospect
  Ajuster le point de friction. Ne pas tout changer en même temps.
```

## Build in Public — stratégie de lancement orchestré (transversal)

> Références : DEC-2026-02-24-76, DEC-2026-02-24-77, DEC-2026-02-24-78
> Plan détaillé : voir `artefacts/strategy/COMMS-001.strategy.md` (empathy maps, channel strategy, personal branding, launch calendar, comment response, content pipeline)

**Principe :** Le build-in-public n'est pas du contenu à côté des produits — c'est le **funnel d'acquisition de chaque lancement**. Même contenu, 3 audiences chauffées en parallèle :

```
"Voici comment mes agents AI ont livré 35 stories"
  → tease .gaai (le framework)
  → tease Callibrate (le produit construit avec)
  → tease le founder (personal brand)
```

**Automatisation :** Skill `generate-build-in-public-content` (SKILL-CRS-021) — se déclenche automatiquement après chaque story `done`. Génère des drafts (X threads, blog snippets, metrics) dans `content/drafts/`. Le founder review en batch (15 min/semaine). Golden Circle appliqué : WHY → HOW → WHAT.

**Protection copycats :** L'obscurité est un risque plus grand que les copycats. Protections naturelles : auteur original (git history), vitesse d'itération, communauté, marque personnelle. Publier > protéger.

## Règles d'exécution

1. **Un seul gate à la fois.** Ne pas commencer Phase 3 si Gate 2 est
   incertain. Les gates sont binaires : PASS ou FAIL.

2. **Phase 0 et Phase 1 sont parallèles.** Le build ne dépend pas
   du signal — mais Phase 2 dépend des deux.

3. **Ne pas over-engineer entre les gates.** Chaque phase livre
   le minimum nécessaire pour valider le gate suivant.

4. **Chaque FAIL est une décision, pas un échec.** Il indique
   quoi ajuster. Un seul levier à la fois.

5. **La mémoire GAAI est mise à jour à chaque gate.** Les décisions
   prises en réponse à un FAIL sont capturées comme décisions validées.
