## New Patterns to Persist

### 1. AES-256-GCM Token Encryption Pattern (crypto.subtle)

**File:** `src/lib/gcalCrypto.ts`

**Pattern:**
- Key format: base64-encoded 32-byte raw key (256-bit), stored as Worker secret `GCAL_TOKEN_ENCRYPTION_KEY`.
- Storage format: `base64(IV[12 bytes] || ciphertext)` — IV prepended to ciphertext, combined and base64-encoded.
- API: `crypto.subtle.importKey('raw', keyBytes, { name: 'AES-GCM' }, false, [usage])` then `crypto.subtle.encrypt/decrypt({ name: 'AES-GCM', iv }, key, data)`.
- Error handling: `GcalDecryptionError` custom class — callers distinguish decryption failures from other errors.
- This is the canonical pattern for any token/secret stored encrypted in the DB. No third-party crypto library needed — `crypto.subtle` is available in all CF Workers environments.

**Memory target:** `conventions.md` — encryption section.

### 2. OAuth2 Callback Route Placement Rule

**File:** `src/index.ts:64`

**Pattern:**
- The OAuth callback (`GET /api/gcal/callback`) MUST be registered BEFORE the authenticated `/api/experts/` block.
- Google does not send a Bearer token — it redirects with `code` and `state` query params only.
- If the callback were inside the auth block, `authenticate()` would return `401` before the handler runs, breaking the entire OAuth flow silently.
- **Rule:** Any unauthenticated external callback (OAuth, webhook) must be registered in the unauthenticated section of the router, not inside protected blocks.

**Memory target:** `conventions.md` — routing section; `orchestration.rules.md` as a named rule.

### 3. OAuth2 Refresh Token Preservation Pattern (AC3)

**File:** `src/handlers/experts/gcal.ts:141-155`

**Pattern:**
- When updating stored OAuth tokens, build the update object conditionally: only add `gcal_refresh_token` key when a new refresh token is present in the response.
- Do NOT set the key to `null` or `undefined` — omit it entirely from the update payload.
- This preserves the existing stored refresh token across access-token-only refreshes (standard Google behavior: refresh token returned only on first consent, not on subsequent access token refreshes).
- Paired requirement: `gcal_connected` boolean must be set to `encryptedRefreshToken !== undefined`, not unconditionally `true`. This keeps the boolean consistent with actual refresh token presence.

**Memory target:** `conventions.md` — OAuth token storage section.

### 4. connected: true Derivation Rule (AC4)

**File:** `src/handlers/experts/gcal.ts:181-196`

**Pattern:**
- The `connected` field in `GET /api/experts/:id/gcal/status` MUST be derived from the actual `gcal_refresh_token IS NOT NULL` check, not from a stored `gcal_connected` boolean.
- Reason: The stored boolean can drift from the actual token state (e.g., if the DB is patched directly, or if the callback stored an access token without a refresh token). Deriving from actual column is the ground truth.
- Implementation: Select `gcal_refresh_token` in the status query; derive `connected: data.gcal_refresh_token != null`.
- The `gcal_connected` column is still maintained (used for indexing/filtering by E06S11+), but the status API response always reflects the actual token state.

**Memory target:** `conventions.md` — OAuth status conventions section.
