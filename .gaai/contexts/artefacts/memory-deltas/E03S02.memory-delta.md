# Memory Delta — E03S02: AI-Extracted Requirements Confirmation

## Verdict: NEW_KNOWLEDGE_FOUND

## New Patterns and Architectural Decisions

### 1. Satellite Funnel Architecture — Step 2 Confirmed

E03S02 establishes the second step of the satellite prospect funnel:
- Step 1 (E03S01): `/match` — freetext input → `POST /api/extract` → sessionStorage(`match:extraction`)
- Step 2 (E03S02): `/confirm` — read `match:extraction` → edit/confirm → `POST /api/prospects/submit` → sessionStorage(`match:prospect_id`, `match:token`) → `/results`

The funnel uses **sessionStorage as the inter-page state transport**. All data passed between steps flows through named keys:
- `match:text` — original freetext (persists for back navigation)
- `match:utm` — captured UTM params (written by match.ts on every load)
- `match:extraction` — full extraction API response
- `match:prospect_id` — created prospect ID after submit
- `match:token` — JWT for results page access

### 2. UTM Passthrough Pattern

UTM parameters are captured at the `/match` page load (not at landing), stored in `match:utm`, and passed to `POST /api/prospects/submit` only if non-null. This is a deliberate "write once at entry" pattern:

```js
// match.ts — fires on every /match load
var utmData = { utm_source, utm_campaign, utm_medium, utm_content };
sessionStorage.setItem('match:utm', JSON.stringify(utmData));
```

This means UTM params are always re-captured on `/match` load, overwriting any prior value. If a user navigates back to `/match` with different query params, UTMs are refreshed.

### 3. Turnstile Integration Pattern on Satellite Worker

The satellite Worker injects the Turnstile site key at render time via `window.__SAT__.turnstileSiteKey`. The SDK is loaded async from `challenges.cloudflare.com`. The widget render is deferred until the "Confirmer" click (not on page load), using a polling loop (250ms intervals, 10s timeout) to handle async SDK load.

Key characteristics:
- `appearance: 'interaction-only'` — zero friction for legitimate users
- `error-callback` and `expired-callback` both handled gracefully
- CSP requires both `script-src` and `frame-src` for `challenges.cloudflare.com`

### 4. Confidence Indicator Thresholds

Standardized thresholds (match AC1 exactly):
- `c >= 0.7` → ✅ (high confidence)
- `c >= 0.4` → 🟠 (medium confidence)
- `c < 0.4` → ❓ (low confidence)

These thresholds are defined solely in the client-side IIFE. If they need to be reused in future stories, they should be documented as a shared constant.

### 5. Field Edit Pattern (Toggle Display/Edit)

The "Modifier" pattern uses CSS class toggling rather than DOM replacement:
- Display: `<span id="display-{field}">value</span>` + `<button class="modifier-btn">Modifier</button>`
- Edit: `<div class="field-edit" id="edit-{field}">` (hidden by default)
- On click: display span `display:none`, modifier btn `display:none`, edit div gains `.visible` class
- The Modifier button disappears once clicked (no "Cancel edit" affordance)

### 6. Max-3 Questions Constraint Implementation

The max-3 constraint is enforced in two places:
- Render: `displayed` counter + early return in `renderConfirmationQuestions()`
- Collection: `questions.slice(0, 3)` in `collectMergedReqs()`

Note: These two approaches are slightly asymmetric (render filters by `needs_confirmation` membership; collect uses positional slice). The API contract assumption is that `confirmation_questions` only contains entries for fields in `needs_confirmation`. If future API changes break this assumption, `collectMergedReqs` should be aligned with the render logic.

### Turnstile Field Name (cf-turnstile-response vs cf_turnstile_response)

The satellite implementation sends `'cf-turnstile-response'` (Cloudflare canonical form-post name). The story spec AC5 listed `cf_turnstile_response` (underscores). E06S07/E06S20 implementation should be checked to confirm which name the Core API reads. This is an integration surface risk — if mismatched, prospect submit will silently fail Turnstile verification.
