# Memory Delta — E06S34

## New Patterns

### Billing Filter Architecture
The billing filter is implemented as a pure function `applyBillingFilters` in `src/lib/billingFilter.ts` with a companion `loadBillingData` Hyperdrive query helper. This separation enables:
- Unit testing the filter logic without any I/O (pure function over a `Map<string, BillingRecord>`)
- Reuse across both local fallback paths (main package) and the Matching Worker (which reimplements inline)

### Billing Filter Applied Pre-Scoring
The billing exclusion step is positioned BEFORE the Vectorize semantic pre-filter and scoring loop in `workers/matching/src/index.ts`. This is intentional: experts who cannot receive a lead should not consume Vectorize query budget or scoring CPU.

### Monthly Spend SQL Pattern
The canonical monthly spend query pattern for `credit_transactions`:
```sql
SELECT expert_id, COALESCE(SUM(ABS(amount)), 0)::integer AS monthly_spend
FROM credit_transactions
WHERE type = 'lead_debit'
  AND expert_id = ANY(${expertIds})
  AND created_at >= ${monthStart.toISOString()}
GROUP BY expert_id
```
- `COALESCE(..., 0)` handles experts with no transactions in the period (GROUP BY produces no row for them; join to experts table must default to 0).
- `expert_id = ANY(${expertIds})` is the index-range entry point; `created_at >=` uses the compound index on `(expert_id, created_at)`.
- `ABS(amount)` because lead_debit transactions are stored as negative values.

### billing_excluded Propagation Pattern
The Matching Worker returns `billing_excluded: BillingExclusion[]` in its response payload. The calling route (matches.ts, prospects.ts) consumes this array to fire EMAIL_NOTIFICATIONS via `ctx.waitUntil()`. This keeps notification responsibility in the Core Worker where queue bindings live, while billing logic stays in the Matching Worker.

## Decisions

### `applyBillingFilters` is pure in main package but inlined in Matching Worker
The Matching Worker reimplements the filter inline rather than importing from `billingFilter.ts`. This is intentional: the Matching Worker has its own type definitions (`workers/matching/src/types.ts`) and avoids a cross-package dependency. The logic is identical — both use the same three-check sequence: `credit_balance < leadPrice` → `max_lead_price exceeded` → `spending_limit reached`.

### Zero-balance experts have no free pass
An expert with `credit_balance = 0` is excluded for any `leadPrice > 0`. The default for experts missing from the billing map is also `credit_balance: 0`, meaning unknown experts are excluded rather than given free leads. This is the conservative/safe default.

### `spending_limit` check uses strict greater-than
`monthly_spend + leadPrice > spending_limit` means an expert can receive a lead that brings them exactly to their limit. This matches the AC4 boundary condition: "exactly equals spending_limit → included".
