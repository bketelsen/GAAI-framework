# Memory Delta: E06S14 — Satellite Worker Multi-Tenant Scaffold

## Verdict: NEW_KNOWLEDGE_FOUND

## New Knowledge

### 1. Multi-Worker Project Structure (NEW)
The project now has a second Worker at `workers/satellite/`. This establishes a pattern for future Workers (e.g., `workers/matching/` for E06S24 Service Bindings).

**Convention to record:**
- Secondary Workers live in `workers/{name}/` subdirectory
- Each has its own `package.json`, `wrangler.toml`, `tsconfig.json`
- GitHub Actions deploy all Workers in parallel
- KV namespaces use `callibrate-satellite-kv-{resource}-{env}` naming

### 2. Hono Framework Pattern (NEW)
The satellite Worker uses Hono for routing (first use in the project). Hono provides:
- Type-safe context variables via `c.set()` / `c.get()`
- Middleware composition via `app.use('*', ...)`
- Route-level response helpers (`c.json()`, `c.redirect()`)
- JSX support enabled in tsconfig (not used in MVP but available)

**Convention to record:**
- Satellite Worker: Hono for routing
- Core Worker: regex-based routing (unchanged)
- Both patterns are valid — Hono for new Workers, no migration of core

### 3. Satellite Config Resolution Pattern (NEW)
`resolveConfig(hostname, env)` — KV cache-aside with Supabase REST fallback:
- KV key: `satellite:config:${hostname}`
- TTL: 3600s
- DB query: Supabase REST API with anon key
- Cache purge: `POST /admin/cache/purge` with `x-admin-secret`
