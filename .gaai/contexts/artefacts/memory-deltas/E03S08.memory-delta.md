# Memory Delta: E03S08

## NEW_KNOWLEDGE_FOUND

### 1. Funnel page deprecation pattern — redirect-only page approach

When deprecating a funnel page (E03S02 /confirm), the pattern used is:
- Keep the route registered in `index.ts` with the same 4-arg signature
- The page function returns pure redirect HTML: `<meta http-equiv="refresh">` + `window.location.replace()` JS fallback
- Page is marked `noindex, nofollow`
- No tracking, no logo, no functional UI
- Tests verify: redirects to target, does not include old content, no PostHog init

This is the correct approach for backwards compat with browser history and shared links without a 404.

### 2. Merged funnel page architecture (Page 2 = E03S02 + E03S03)

The merged `results.ts` page implements a dual-path client-side state machine driven by `extraction.ready_to_match`:

**Path A — ready_to_match: true:**
1. Check sessionStorage for `match:prospect_id` + `match:token` (back navigation guard)
2. If present: skip Turnstile, call `fetchMatches()` directly
3. If absent: auto-render Turnstile in hidden div (`cf-turnstile-auto`), callback fires `doSubmit()`
4. Show skeleton loading throughout

**Path B — needs_confirmation:**
1. Show questions section + matches-placeholder
2. Confirm button click → collect answers → show visible Turnstile → callback fires `doSubmit()`
3. On success: hide questions, show skeleton, call `fetchMatches()`

**Single Turnstile SDK, two rendering modes:**
- `autoFire=true`: renders in hidden `cf-turnstile-auto` div appended to body (invisible)
- `autoFire=false`: renders in visible `cf-turnstile-container` in questions section

### 3. Re-match endpoint pattern (POST /api/prospects/:id/requirements)

New pattern for edit-triggered re-match:
- **Rate limit:** 5 re-matches per 24h per prospect (KV key `rematch-count:{id}`)
- **Response:** 202 `{ status: "recomputing", message: "..." }` — async, not synchronous
- **Mechanism:** normalizes requirements, expires old matches (status='expired'), triggers `handleMatchCompute` via `ctx.waitUntil()`
- **Client:** after 202, resets polling state (retryCount=0) and calls `fetchMatches()`
- **No Turnstile required** on re-match — user is already authenticated (has prospect session token)
- **Token auth:** same `prospect:submit` JWT audience as all other prospect endpoints

### 4. Extraction summary — client-side escaping discipline

The `results.ts` page uses a client-side `escHtml()` function (identical to server-side `escapeHtml()`) for all innerHTML assignments. This dual-layer escaping is the correct pattern when:
- Server injects a shell HTML page with no data
- Client fetches data from sessionStorage (which contains API response JSON) and renders it client-side

The pattern: ALL innerHTML-bound strings from sessionStorage/API go through `escHtml()`. `textContent` is used where no HTML structure is needed (e.g., `otp-hint`).

### 5. window.__SAT__ injection — turnstileSiteKey added to payload

`results.ts` adds `turnstileSiteKey` to the `window.__SAT__` object (alongside existing `apiUrl` and `satelliteId`). This is how the client-side Turnstile render call accesses the sitekey without server-side HTML template interpolation of the key into script strings. The key travels through `JSON.stringify().replace(/</g, '\\u003c')` which is the canonical XSS-safe pattern for this codebase.

### 6. PostHog event taxonomy update — E03S08 events

New events introduced:
- `satellite.funnel_page2_loaded` — fires on page init, carries `ready_to_match` and `needs_confirmation_count`
- `satellite.extraction_edited` — fires when a field is saved, carries `field_name` and `new_value_hash` (length-based hash, no PII)
- `satellite.rematch_triggered` — fires before re-match request, carries `edit_source: 'extraction_edit' | 'confirmation_answer'`

Pre-existing events still fired from Page 2: `satellite.prospect_created`, `satellite.matches_viewed`, `satellite.profiles_unlocked`, `satellite.email_gate_submitted`, `satellite.matching_error`, `satellite.match_card_expanded`, `satellite.booking_cta_clicked`.
