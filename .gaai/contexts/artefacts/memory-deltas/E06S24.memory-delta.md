## New Patterns

### Service Binding as zero-latency internal RPC
Cloudflare Service Bindings expose a `Fetcher` type on the binding. The bound Worker is called via standard `fetch()` but executes in-process (no network hop, billed at same cost as local function call).

```typescript
// Core's Env:
MATCHING_SERVICE?: Fetcher;

// Core's call site:
const resp = await env.MATCHING_SERVICE.fetch(
  new Request('https://matching/match', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ prospect_id, satellite_id }),
  })
);

// The URL hostname is arbitrary — Service Binding ignores it and routes to the bound Worker
```

Key properties:
- `Fetcher` type (not `KVNamespace`, `D1Database`, etc.) — from `@cloudflare/workers-types`
- URL can be any valid URL — hostname is ignored by the binding; routing is by Worker name
- Always optional (`?: Fetcher`) — enables AC6 fallback when not bound (dev/CI)
- `wrangler.toml`: `[[services]]` section (not `[[kv_namespaces]]` or `[[d1_databases]]`)

Files: `src/types/env.ts`, `wrangler.toml` `[[env.staging.services]]`

### Multi-Worker monorepo structure
Multiple Cloudflare Workers in a single repo at `workers/{name}/` each with their own `wrangler.toml`, `package.json`, `tsconfig.json`. Each deploys independently.

```
workers/
  matching/      ← callibrate-matching (compute)
  satellite/     ← callibrate-satellite (satellite sites)
  posthog-proxy/ ← callibrate-posthog-proxy (analytics proxy)
src/             ← callibrate-core (gateway)
wrangler.toml    ← Core's config (root level)
```

Pattern: `wrangler.toml` per Worker; deploy via `cd workers/{name} && npx wrangler deploy --env {env}`. GitHub Actions must include a deploy step per Worker.

### Type-only optional binding (type compatibility without wrangler.toml declaration)
When code in Worker A references a binding type that has moved to Worker B, declare the type as optional in Worker A's `Env` with no corresponding `wrangler.toml` entry. This makes the property `undefined` at runtime (graceful fallback), while preventing tsc errors from existing usages.

```typescript
// In Core's Env — EXPERT_DB moved to Matching Worker's wrangler.toml
// Kept here so expertPool.ts / syncExpertPool.ts still type-check
EXPERT_DB?: D1Database;
// No [[d1_databases]] entry in Core's wrangler.toml → undefined at runtime → D1 tier skipped
```

This is a transitional pattern: code that references the binding should be updated when refactored, but an immediate refactor isn't required to ship the story.

## Architectural Decisions

### Core = gateway, Matching = compute
The Service Binding split creates a clean separation of concerns:
- **Core** (`callibrate-core`): auth, routing, CORS, rate limiting, security, queue producers, OAuth, expert registration, booking endpoints
- **Matching** (`callibrate-matching`): expert pool cache (Cache API + D1), Vectorize semantic search, Workers AI embedding, scoring pipeline, analytics

This split enables:
1. Independent deployment (Matching can be updated without touching Core)
2. Independent scaling (Matching can be assigned different limits if needed)
3. Independent failure domains (Matching failure → Core falls back to local scoring)
4. Cleaner binding inventory (Core no longer needs AI/Vectorize/D1 bindings)

### Matching Worker uses its own postgres.js + Hyperdrive
The Matching Worker independently connects to Supabase via Hyperdrive for the Hyperdrive-tier of the expert pool read chain. This is intentional: the matching Worker may run in a different CF datacenter than Core, and its Hyperdrive connection is independently pooled.

Both workers declare separate Hyperdrive bindings in their respective `wrangler.toml` files. They connect to the same Supabase database but have independent connection pools.

## Infrastructure Note

Matching Worker requires manual provisioning before first deploy:

```bash
# Matching Worker has its own node_modules
cd workers/matching && npm install

# Deploy order matters: Matching must be deployed BEFORE Core (binding must resolve)
cd workers/matching && npx wrangler deploy --env staging
cd ../../ && npx wrangler deploy --env staging

# Matching Worker reuses the same Hyperdrive config IDs as Core
# Reuse the same KV namespace IDs (EXPERT_POOL) — both Workers read the same KV
# D1 and Vectorize bindings in Matching are the same resources as previously in Core
```
