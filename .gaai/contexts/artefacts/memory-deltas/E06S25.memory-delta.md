# E06S25 Memory Delta: Durable Objects — Expert Pool Write Coordinator

## Patterns Confirmed

### Cloudflare Durable Objects — Vitest Testing
- `vi.mock('cloudflare:workers', () => ({ DurableObject: class { constructor(ctx, env) { this.ctx = ctx; this.env = env; } } }))` is the correct pattern to stub the CF `DurableObject` base class for vitest.
- Pass mock `ctx` as `{ storage: { getAlarm: vi.fn(), setAlarm: vi.fn() } }` to the DO constructor.

### Optional Binding Pattern
TypeScript cannot narrow optional bindings (`Ai | undefined`) across async IIFE closure boundaries even after a guard. Capture into local `const` variables before the `async () => {}` to preserve narrowing:
```typescript
const { AI: ai, VECTORIZE: vectorize } = env;  // capture before async
if (!ai || !vectorize) return;
ctx.waitUntil((async () => {
  await ai.run(...);  // TS narrows correctly
})());
```

### Non-Blocking Fire-and-Forget (ctx.waitUntil)
Pattern for AC5-style non-blocking DO calls:
```typescript
export function notifyExpertPoolDO(env: Env, ctx: ExecutionContext, event: ProfileEvent): void {
  if (!env.EXPERT_POOL_DO) return;
  ctx.waitUntil((async () => {
    try { /* DO call */ }
    catch (err) { console.error(...); }
  })());
}
// Returns void synchronously — HTTP response NOT blocked
```

### DO Singleton — idFromName
Always use `env.EXPERT_POOL_DO.idFromName('expert-pool-coordinator')` — fixed string ensures singleton semantics across all requests regardless of which CF edge node handles the request.

### wrangler.toml — DO Migrations
`[[migrations]]` must appear at top-level in wrangler.toml (not under `[env.staging]` or `[env.production]`). This matches the CF Workers constraint (same as for `[[workflows]]`). The tag `"v1"` with `new_classes = ["ExpertPoolDO"]` auto-provisions the DO on first deploy.

### Parallel Story Branches — vectorize.ts
When multiple stories add the same new utility file (vectorize.ts in E06S21 and E06S25), the merge conflict is trivial if both PRs produce identical content. Strategy: add explicit note in PR description so reviewer knows to keep one copy.

## Architecture Decisions

- **ExpertPoolDO in-memory map** (`Map<string, ProfileEvent>`) chosen over DO storage for pending events because: (1) 5-second window is short enough that eviction risk is low, (2) cron sync provides recovery, (3) storage read/write latency would add overhead to every event.
- **Branch from E06S23 not staging** — required because staging lacks E06S21 (Vectorize) and E06S23 (D1). PR reviewer must merge prerequisite branches first.
