**Suggested additions:**

### Supabase migration pattern
- Migration files stored at `supabase/migrations/{timestamp}_{name}.sql`
- Applied via `mcp__supabase__apply_migration` (DDL) or `mcp__supabase__execute_sql` (DML/seed)
- Migration version in DB = timestamp at apply time (not filename timestamp)
- Verify via `mcp__supabase__list_migrations` after apply

### TypeScript types pattern
- Generated via `mcp__supabase__generate_typescript_types` after each schema migration
- Written to `src/types/database.ts` — commit alongside migration file
- Must be regenerated after any schema change — never manually edited
- Use with `createClient<Database>()` and publishable key (DEC-37)

### RLS pattern for service-role-only tables
- Enable RLS on table → no user-facing policies → service role bypasses automatically
- Supabase security advisor reports INFO `rls_enabled_no_policy` — expected and correct for internal tables
- Only add user-facing policies for tables with direct authenticated user access (e.g., experts)

## NEW KNOWLEDGE: QA tooling observations

1. **`list_tables` CHECK clause:** `mcp__supabase__list_tables` returns `check` field per column directly — preferred over `information_schema` JOIN for CHECK verification. `information_schema.check_constraints` JOIN via `key_column_usage` unreliable for column-level CHECK constraints in this Postgres version.

2. **Migration version auto-assigned:** `apply_migration` assigns a version based on the actual timestamp of apply, not the filename prefix. Plan accordingly when referencing migration versions in documentation.

3. **`rls_enabled` flag on list_tables:** `mcp__supabase__list_tables` returns `rls_enabled: boolean` directly per table — fast way to confirm RLS state without a separate SQL query.
