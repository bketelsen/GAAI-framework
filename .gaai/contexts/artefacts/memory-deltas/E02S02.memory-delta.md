## New Patterns (worth adding to conventions.md)

### PostHog direct fetch pattern for React Router v7 dashboard

The dashboard Worker uses a direct fetch to PostHog EU API for server-side event capture, rather than `posthog-node`. This avoids a Node.js dependency while achieving equivalent functionality:

```ts
// app/lib/posthog.server.ts
export async function captureEvent(env: Env, distinctId: string, event: string, properties = {}): Promise<void> {
  if (!env.POSTHOG_API_KEY) return;
  try {
    await fetch("https://eu.i.posthog.com/capture/", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ api_key: env.POSTHOG_API_KEY, event, distinct_id: distinctId, properties }),
    });
  } catch {}
}
```

File: `workers/frontend/dashboard/app/lib/posthog.server.ts`

### React Router FormData test pattern

When testing React Router actions with FormData in Vitest (Node.js environment): do NOT override `Content-Type` header when passing a FormData body to `new Request()`. The runtime needs to auto-detect the multipart boundary. Setting `Content-Type: application/x-www-form-urlencoded` with a FormData body causes `request.formData()` to fail silently (returns empty data → Zod validation fails spuriously).

**Correct pattern:**
```ts
new Request(url, { method: "POST", body: formData }) // no Content-Type override
```

### vi.mock hoisting constraint

`vi.mock()` calls in Vitest are hoisted to the top of the file before variable declarations. Therefore:
- Never reference module-level variables inside `vi.mock()` factory functions
- Never call `vi.mock()` inside `describe()` or `it()` blocks
- Use `beforeEach` + `vi.mocked(fn).mockResolvedValue()` for per-test mock configuration
