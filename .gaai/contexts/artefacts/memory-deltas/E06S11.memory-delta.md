# E06S11 — Memory Delta

Six new patterns introduced by the booking engine implementation. No drift from existing conventions detected.

## New Patterns to Record in `conventions.md`

### Pattern 1: GCal freebusy + slot computation pattern

**Section:** Code Patterns
**Tag:** E06S11

`computeFreeSlots(params)` in `src/lib/availability.ts` — pure function, no side effects, fully testable. Accepts `{busyIntervals, heldBookings, rules, now}`. Combines GCal freebusy intervals with DB held/confirmed bookings into a single `allBusy` array before checking each candidate slot. Slot spacing = `SLOT_DURATION_MS + BUFFER_MS` (buffer applied as spacing between consecutive slots AND as a padded zone around busy intervals). Working day boundaries derived from `rules.working_hours[dayKey]` — missing key = non-working day (weekend or custom). Min notice: `earliestStart = now + MIN_NOTICE_MS`. All output values are UTC ISO strings.

### Pattern 2: Hold-then-confirm booking flow pattern

**Section:** Code Patterns
**Tag:** E06S11

Two-phase booking lifecycle:
- **Phase 1 — Hold:** `POST /api/bookings/hold` → DB conflict check → insert `status=held`, `held_until=now+10min`, `prep_token=crypto.randomUUID()` → return `{booking_id, held_until}`. 409 on overlap.
- **Phase 2 — Confirm:** `POST /api/bookings/:id/confirm` → fetch booking + assert `status=held` + assert `held_until > now` → freebusy re-check (race guard) → if taken: DELETE row → 409 `{error:'slot_taken'}` → else: `gcalInsertEvent(conferenceDataVersion=1)` → update booking to `status=confirmed` → push `booking.created` → LEAD_BILLING + `booking.confirmed` → EMAIL_NOTIFICATIONS.
- Hold expiry cleanup: cron `*/5 * * * *` deletes `WHERE status=held AND held_until < now()`.

### Pattern 3: CF Worker cron trigger pattern

**Section:** Code Patterns
**Tag:** E06S11

- Worker exports `scheduled(controller, env)` alongside `fetch` and `queue`.
- `handleScheduled(controller, env)` dispatches by `controller.cron` string match (`=== '*/5 * * * *'`, `=== '*/15 * * * *'`).
- Crons declared in `wrangler.toml` at top-level `[triggers].crons`, `[env.staging.triggers].crons`, and `[env.production.triggers].crons` — all three sections required for triggers to fire in all environments.
- Unknown cron values logged as `console.warn` (no-op, non-fatal).

### Pattern 4: GCal API retry/error wrapper pattern (AC10)

**Section:** Code Patterns
**Tag:** E06S11

`withGcalRetry<T>(fn, onTokenRefresh, context, attempt=0)` in `src/lib/gcalClient.ts`:
- **429:** delays `[1000, 4000, 16000][attempt]` ms (1s/4s/16s), `attempt < 3` guard, recursive on `attempt+1`.
- **401:** `attempt === 0` guard → call `onTokenRefresh()` (= `refreshGcalToken`) → retry on `attempt+1`.
- **404 on delete:** normalized to `{status: 200}` inside `gcalDeleteEvent` BEFORE the retry wrapper sees it → treated as success.
- **Other non-2xx:** `throw new GcalApiError(status, message)` → callers return 502 `{error:'gcal_error', status, message}`.
- Pattern: `onTokenRefresh` is a closure that mutates `currentToken` in the outer function scope — no re-injection needed.

### Pattern 5: Prep endpoint token-expiry pattern

**Section:** Code Patterns
**Tag:** E06S11

`GET /api/bookings/:token/prep` — public route, no auth, no CORS check (registered before the CORS-gated `/api/bookings/` block).
- Token stored as UUID in `bookings.prep_token`, generated at hold time.
- Expiry = `start_at + 2 hours` computed at read time (not stored as a column).
- Returns 404 `{error:'prep_token_expired'}` on expiry — same HTTP status as not-found (prevents token enumeration via status code difference).
- Returns full bi-directional prep payload: `{booking, prospect, expert, match}`.

### Pattern 6: Availability route placement rule (extension of OAuth callback rule)

**Section:** Code Patterns
**Tag:** E06S11

Routes that are public/CORS-only but share a URL prefix with JWT-gated routes MUST be registered first in `index.ts`. Applies to:
- `GET /api/experts/:id/availability` (public, satellite-widget-facing) — registered before `/api/experts/` JWT block.
- Extends the existing `OAuth2 callback route placement rule (E06S10)` to cover any no-auth route under an authenticated prefix.
- Violation symptom: silent 401 for all callers (auth middleware intercepts before route match).
