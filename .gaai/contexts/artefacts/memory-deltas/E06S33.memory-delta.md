# E06S33 — Memory Delta

## New Patterns Confirmed

### 1. HMAC-SHA256 Webhook Verification (CF Workers)

Pattern for verifying LemonSqueezy (and similar) webhook signatures in CF Workers:

```typescript
const rawBody = await request.arrayBuffer();
const bodyText = new TextDecoder().decode(rawBody);
const sigHeader = request.headers.get('X-Signature');
if (!sigHeader) return { valid: false, bodyText: '' };

const key = await crypto.subtle.importKey(
  'raw',
  new TextEncoder().encode(secret),
  { name: 'HMAC', hash: 'SHA-256' },
  false,
  ['verify']
);
const hexPairs = sigHeader.match(/../g);
const sigBytes = Uint8Array.from(hexPairs.map(h => parseInt(h, 16)));
const valid = await crypto.subtle.verify('HMAC', key, sigBytes, rawBody);
```

Key: must consume raw `arrayBuffer` BEFORE any other body read. Body text decoded separately for JSON parsing.

### 2. KV Idempotency Pattern (Webhook Dedup)

Key format: `idem:{domain}:{event_id}:{event_name}`, TTL: 86400s.

Includes `event_name` in key to handle same subscription ID across different event types (e.g., `subscription_created` vs `subscription_updated` for same subscription ID).

```typescript
const idemKey = `idem:ls-webhook:${eventId}:${eventName}`;
const alreadyProcessed = await env.SESSIONS.get(idemKey);
if (alreadyProcessed !== null) return deduplicated response;
// ... process ...
await env.SESSIONS.put(idemKey, '1', { expirationTtl: 86400 });
```

### 3. FLAG_WINDOW_MS Acceleration Pattern

All time-window-based staging accelerators follow the same pattern:

```typescript
const windowMs = parseInt(env.FLAG_WINDOW_MS ?? String(DEFAULT_PRODUCTION_MS));
```

- Staging: set to small value (e.g., `60000` = 1 minute) in `[env.staging.vars]`
- Production: absent → defaults to full window (e.g., `7 * 24 * 60 * 60 * 1000` = 7 days)

Prior examples: `SURVEY_DELAY_7D_MS`, `SURVEY_DELAY_38D_MS`. Same pattern in every time-window feature.

### 4. Silent 200 for Unknown/Unmatched Webhook Entities

When a webhook entity (expert, subscription) cannot be found, the handler logs a warning and returns 200 silently. This prevents the external service (LemonSqueezy) from retrying indefinitely on a legitimate "entity not found" situation.

```typescript
if (!expertId) {
  console.warn('handler: entity not found for event', eventId);
  return; // handler returns void; caller returns 200
}
```

### 5. Usage Reporting Pattern — Per-Lead Try/Catch with Continue

For batch operations where individual failures must not abort the batch:

```typescript
for (const lead of leads) {
  try {
    await reportLeadUsage(lead, env);
    await sql`UPDATE leads SET status='usage_reported', usage_reported_at=now() WHERE id=${lead.id}`;
  } catch (err) {
    console.error('failed for lead', lead.id, err);
    // No rethrow — continue to next lead
  }
}
```

Status remains `'confirmed'` on failure → automatic retry on next cron run.

### 6. globalThis vs global in Workers Test Files

CF Workers TypeScript config (`lib: ["ES2022"]`, `types: ["@cloudflare/workers-types"]`) does NOT include `global` (Node.js-only). Test files must use `globalThis` for cross-environment fetch spying:

```typescript
// CORRECT — works in both Node (Vitest) and Workers TS type checking
vi.spyOn(globalThis, 'fetch').mockResolvedValue(...)

// WRONG — TS2304 error with Workers tsconfig
vi.spyOn(global, 'fetch').mockResolvedValue(...)
```

This is consistent with `src/cron/syncExpertPool.test.ts` which correctly uses `globalThis`.

## New Infrastructure

### `credit_transactions` Table

New table tracking balance changes per expert:
- `id` (uuid PK)
- `expert_id` (uuid FK → experts)
- `type` (text — e.g., `'welcome_credit'`)
- `amount` (integer)
- `balance_after` (integer)
- `created_at` (timestamptz)

Idempotency guard via `WHERE NOT EXISTS (SELECT 1 FROM credit_transactions WHERE expert_id = $1 AND type = $2)`.

### New Expert Fields

- `credit_balance` (integer NOT NULL DEFAULT 0)
- `ls_subscription_id` (text)
- `ls_subscription_status` (text) — mapped from LS statuses
- `ls_subscription_item_id` (text) — fetched from LS API after subscription_created

### New Lead Fields

- `confirmed_at` (timestamptz) — set by autoConfirmLeads cron
- `usage_reported_at` (timestamptz) — set after successful LS usage-record POST
- `flag_reason` (text) — reserved for flagging flow
- `flagged_at` (timestamptz) — reserved for flagging flow

### LS Subscription Status Mapping

| LS Status | Internal |
|-----------|----------|
| `active` | `active` |
| `past_due` | `past_due` |
| `cancelled` / `canceled` | `cancelled` |
| `unpaid` | `unpaid` |

Both `cancelled` and `canceled` spellings handled (LS uses both inconsistently).
