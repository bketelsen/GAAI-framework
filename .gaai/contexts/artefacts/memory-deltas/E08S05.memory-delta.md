## New Patterns Established

### 1. JWT `iss` + `aud` as mandatory claims in all tokens

Both prospect tokens and survey tokens now carry `iss: "callibrate"` and a typed `aud` claim.

- Prospect tokens: `aud` is one of `'prospect:submit' | 'prospect:matches' | 'prospect:identify'` (caller-provided, enforced at verification)
- Survey tokens: `aud: 'survey'` (hardcoded)
- `verifyProspectToken` signature now requires `expectedAud: string` as 4th parameter — all call sites must pass the correct audience value

This establishes the pattern: **every JWT issued by this system must carry both `iss` and `aud` claims, and every verification must validate both**.

---

### 2. CORS origin check uses `SUPABASE_ANON_KEY`, not `SUPABASE_SERVICE_KEY`

`cors.ts#validateOrigin` reads from `satellite_configs.domain` using the anon key. This is correct and intentional:
- The `satellite_configs` table has an `anon` SELECT RLS policy (migration `20260224200001_satellite_configs_anon_select.sql`)
- Using the service key for CORS validation is over-privileged; anon key is sufficient
- Pattern: **use the minimal-privilege Supabase key for read operations that are legitimately public**

---

### 3. LemonSqueezy webhook idempotency is two-layer (KV + DB)

The webhook handler now implements dual-layer idempotency:
- **L1 (KV):** Fast dedup via `SESSIONS.get(idem:ls-webhook:{event_id}:{event_name})` — ephemeral, TTL 86400s
- **L2 (DB):** Persistent dedup via `webhook_events` table with `event_id TEXT PRIMARY KEY` — survives KV eviction

This is now the canonical pattern for all webhook handlers that need strong idempotency guarantees. KV handles the common case cheaply; DB handles retries after KV TTL expiry.

The `webhook_events` table is minimal: `event_id (PK)`, `event_name`, `processed_at`. RLS enabled; service role bypasses (Worker has service key).

---

### 4. Ownership assertions in `SECURITY DEFINER` RPC functions

Both `debit_lead_credit` and `restore_lead_credit` now assert cross-entity ownership before executing:
- `debit_lead_credit`: rejects if `booking.expert_id != p_expert_id`
- `restore_lead_credit`: rejects if `lead.expert_id != p_expert_id`

Return codes for rejection: `'booking_expert_mismatch'`, `'lead_not_owned'`

Pattern: **every `SECURITY DEFINER` function that takes `p_expert_id` as input must validate that the target record actually belongs to that expert before mutating state**.

---

### 5. Vertical context injection uses an in-code allowlist

`extract.ts` defines `KNOWN_VERTICALS` as a `const` array and validates any DB-returned vertical against it before injecting into the LLM system prompt. This prevents prompt injection via a compromised `satellite_configs.vertical` value.

Pattern: **any DB-sourced string that flows into an LLM prompt must be validated against a static allowlist before use**.

---

### 6. Survey token TTL reduced from 30 days to 7 days

Previous TTL was 30 days (E06S17). Reduced to 7 days in E08S05 as a hygiene measure. Email templates updated to include "Ce lien expire dans 7 jours." in both `call_experience` and `project_satisfaction` survey emails.

---

### 7. Secrets hygiene: `localConnectionString` pattern

`localConnectionString` for Hyperdrive dev override is:
- **Never** committed to `wrangler.toml`
- Documented in `SETUP.md` as a "add locally, remove before commit" workflow
- Indicated by a comment in the `[[hyperdrive]]` block of each wrangler.toml
- Not loadable from `.dev.vars` (Wrangler limitation — documented explicitly)

`TURNSTILE_SECRET_KEY` is a secret (not a var) — managed via `wrangler secret put`. This is indicated by a comment in `[env.staging.vars]` to prevent future re-addition as a plaintext var.

## Architecture Decisions

**DEC-E08S05-01:** Survey token TTL set to 7 days (down from 30). Rationale: survey links are time-sensitive (sent shortly after booking/completion); 30 days is excessive and increases the attack window if a token is stolen.

**DEC-E08S05-02:** `validateOrigin` in CORS uses anon key. Rationale: `satellite_configs.domain` is a public read (satellite domains are known to the browser anyway). Service key usage here would be over-privileged with no security benefit.

**DEC-E08S05-03:** `verticalContext` allowlist validation. Rationale: DB-sourced strings should never be trusted for LLM prompt injection without validation. The `KNOWN_VERTICALS` list represents the universe of values the system knows how to handle.
