## New Facts Confirmed by Implementation (not previously detailed in memory)

**1. OpenAI integration pattern: function calling (tools), not text completion**

The implementation uses the OpenAI `tools` + `tool_choice` pattern with a named function `extract_requirements`. This is the established pattern for structured extraction in this codebase. Response is parsed from `choices[0].message.tool_calls[0].function.arguments` (JSON string). Memory does not document this at the level of protocol details — no update needed, but worth noting for any future AI endpoint additions.

**2. No Cloudflare AI Gateway in the extraction path**

The migration removes CF AI Gateway entirely from the extraction path. `CLOUDFLARE_AI_GATEWAY_URL` is gone from `src/types/env.ts`. The direct `fetch` to `https://api.openai.com/v1/chat/completions` is the canonical pattern going forward for OpenAI calls. If future stories add more AI endpoints, they should follow this pattern (not re-introduce CF AI Gateway) unless a new decision is made.

**3. Error surface convention established**

The three error strings are now: `'OpenAI API error'` (non-200 from API), `'Invalid response from OpenAI API'` (JSON parse failure), `'AI service unreachable'` (fetch-level network failure). This is the canonical error surface for AI service calls.

**4. Technical debt flagged: stale mockEnv keys in two pre-existing test files**

`src/handlers/experts/gcal.test.ts` and `src/handlers/bookings/bookings.test.ts` still contain `ANTHROPIC_API_KEY: ''` and `CLOUDFLARE_AI_GATEWAY_URL: ''` in their mockEnv objects. These compile because both use `as unknown as Env`. They are non-blocking but represent drift between test fixtures and the live Env interface. A cleanup pass is recommended in a future story.
