## Patterns / Knowledge Found

### 1. Auth Middleware Pattern — Discriminated Union Result

The `authenticate()` function in `src/middleware/auth.ts` uses a TypeScript discriminated union as its return type:

```typescript
export type AuthResult =
  | { user: AuthUser; response?: never }
  | { user?: never; response: Response };
```

This pattern allows callers to check `if (authResult.response)` to short-circuit without a thrown exception. The caller in `index.ts` uses:

```typescript
const authResult = await authenticate(request, env);
if (authResult.response) {
  return authResult.response; // 401 short-circuit
}
const user = authResult.user; // TypeScript narrows to AuthUser here
```

This is the established pattern for auth middleware in this codebase. Future middleware (e.g., role checks, subscription checks) should follow the same discriminated union + caller check pattern.

### 2. Zod Validation Pattern for CF Worker Handlers

Consistent pattern across both `register.ts` and `profile.ts`:

1. Parse body with `request.json()` in a try/catch — return `400` on parse failure.
2. Call `Schema.safeParse(body)` — never `.parse()` (throws).
3. On failure: return `422` with `{ error: "Validation failed", details: parsed.error.flatten().fieldErrors }`.
4. On success: destructure `parsed.data` directly.

`z.record(z.string(), z.unknown())` is the correct Zod v3 form for flexible JSONB object fields — `z.record(z.unknown())` is Zod v4+ only.

### 3. Supabase RPC Pattern for JSONB Merge

When a handler needs to perform a partial JSONB update (merge semantics, not replace):

- Create a Postgres function (`SECURITY DEFINER`, `RETURNS SETOF <table>`) with `COALESCE` for scalar fields and `||` operator for JSONB fields.
- Declare the function signature manually in `src/types/database.ts` under `Functions:` — the type generator does not reliably capture function signatures.
- Call via `supabase.rpc('function_name', { ...args })` — returns `data` as an array; check `data.length === 0` for not-found.
- Cast `z.record(z.string(), z.unknown())` results to `Json` type with `as Json` before passing to the RPC.

### 4. Rate Limiter Pattern — KV Sliding Window

Pattern established for per-IP rate limiting in `src/lib/rateLimit.ts`:

- Key: `rate_limit:{action}:{ip}` where `ip` = `CF-Connecting-IP` header (fallback `'unknown'` for local dev).
- KV `get` → check count; KV `put` with `expirationTtl` on every allowed request (sliding window, not tumbling).
- Rate limit check placed at the top of the handler, after auth (unauthenticated requests do not consume rate limit budget).
- `CF-Connecting-IP` is guaranteed present in deployed CF Workers; `'unknown'` fallback is local-dev only.

### 5. Own-Profile Guard Pattern

For user-scoped API resources, ownership is enforced inline in the handler:

```typescript
if (user.id !== expertId) {
  return new Response(JSON.stringify({ error: 'Forbidden' }), {
    status: 403,
    headers: { 'Content-Type': 'application/json' },
  });
}
```

This check runs before any DB query — no unnecessary DB round-trips for unauthorized access. The `expertId` is extracted from the URL path in the router before being passed to the handler.

### 6. Supabase Client Usage Clarification (DEC-37 refinement)

Confirmed in practice: `createServiceClient(env)` uses `SUPABASE_SERVICE_KEY` and is used for:
- Auth validation: `supabase.auth.getUser(token)` (requires service key to validate any user's token)
- DB writes: `supabase.from('experts').insert(...)` (service key bypasses RLS)
- DB reads: `supabase.from('experts').select('*').eq('id', expertId).single()` (service key)
- RPC calls: `supabase.rpc('merge_expert_profile', {...})` (service key)

The `SUPABASE_ANON_KEY` is retained only for the raw `fetch()` call in the health check endpoint (DEC-37 documented this split already).

### 7. PostgREST Error Codes in Worker Context

- `PGRST116` = no rows returned by `.single()` — used to return `404` when expert not found.
- `23505` = Postgres unique violation — used to return `409` for duplicate registration.
- Other errors fall through to generic `500` with `error.message` in details.
