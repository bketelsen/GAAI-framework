## Implementation Notes

### HMAC-SHA256 Verification (AC1)
Uses `crypto.subtle.importKey` + `crypto.subtle.verify` with the raw body `ArrayBuffer`. The `X-Signature` header is hex-decoded to bytes before verification. Missing or invalid signatures return 401.

### Expert Identification (AC2)
1. Primary: `custom_data.expert_id` — SELECT by UUID
2. Fallback: `customer_email` from subscription attributes → match against `experts.gcal_email`
3. If neither matches: logs warning, returns 200 (silent — prevents LS retry loops)

### Subscription Item ID Fetch (AC2)
After identifying the expert, a separate fetch to `GET /v1/subscription-items?filter[subscription_id]={id}` retrieves the `ls_subscription_item_id`. On failure, the field is set to null and a warning is logged. The subscription is still recorded.

### Welcome Credit Idempotency (AC2)
The `INSERT INTO credit_transactions` uses a `WHERE NOT EXISTS` guard on `(expert_id, type='welcome_credit')` to prevent double-crediting across retries.

### Status Mapping (AC3)
LS statuses `cancelled` and `canceled` (both spellings) map to internal `cancelled`. All other values (`active`, `past_due`, `unpaid`) are mapped directly.

### KV Idempotency Key (AC5)
Key format: `idem:ls-webhook:{event_id}:{event_name}` — includes event name to handle same subscription ID across event types. TTL: 86400s.

### FLAG_WINDOW_MS (AC6)
`parseInt(env.FLAG_WINDOW_MS ?? String(7 * 24 * 60 * 60 * 1000))` — staging: 60000ms (1 minute), production: 7 days (default). Same acceleration pattern as `SURVEY_DELAY_*` vars.

### Usage Reporting Error Handling (AC8)
Each lead is processed in an individual try/catch. On LS API failure (non-2xx or throw), the lead's `status` remains `'confirmed'` and `usage_reported_at` remains null. The error is logged with the `lead_id`. Next cron run will retry.
