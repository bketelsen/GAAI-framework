## Design Decisions

- **JWT**: Pure Web Crypto API HMAC-SHA256 — no external library. Payload: `{ prospect_id, exp }`.
- **EXPERT_POOL KV**: cache-aside pattern, 300s TTL. DB fallback when KV miss. Write-back non-blocking (failure doesn't break request). KV IDs are placeholders — human provisions via SETUP.md Step 3.
- **CORS 403**: Blocked origins receive 403 with the origin in `Access-Control-Allow-Origin` — allows the browser to read the error body (useful for satellite developers debugging misconfigured domains).
- **Response envelopes**: `{ matches: [...] }` and `{ experts: [...] }` — consistent with codebase convention (all other endpoints use wrapper objects). ACs specify the *card shape*, not the top-level envelope.
- **domain format**: Confirmed from DB that `satellite_configs.domain` stores bare hostnames (e.g. `callibrate.ai`). CORS validation correctly extracts `originUrl.hostname` for comparison.
- **quiz_schema formats**: Two formats supported — Format A (`{ questions: [{ key, required }] }`) and Format B (`{ fieldKey: { required: true } }`).

## Known Limitations

- `EXPERT_POOL` KV IDs are placeholder — will fallback to DB until provisioned (no functional gap, just KV cache bypass).
- `PROSPECT_TOKEN_SECRET` must be set via `wrangler secret put` before deployment.
- Pre-existing TS errors for `CAL_API_KEY`, `CAL_OAUTH_CLIENT_ID`, `CAL_WEBHOOK_SECRET` — from E06S04 code, out of scope for this story.
