## Known Risks / Limitations

1. **crypto.subtle in test environment**: The `getAccessToken` → `decryptToken` path uses `crypto.subtle` (AES-GCM). In the vitest environment, `crypto.subtle` may or may not be available depending on the Node.js version. Tests involving GCal token handling are designed to accept either the correct response or a 502 fallback, ensuring the test suite remains deterministic.

2. **`booking.cancelled` email queue type cast removed**: After extending `queues.ts`, the `cancel.ts` and `reschedule.ts` handlers send queue messages without type casts — the new union members are properly typed.

3. **profile.ts side-fix**: The updated `database.ts` (which now includes `availability_rules` and `reminder_settings` in the `merge_expert_profile` RPC return type) exposed a pre-existing TS2322/TS2379 issue in `src/handlers/experts/profile.ts` related to `exactOptionalPropertyTypes`. Fixed by using spread with conditional inclusion pattern.

4. **Prep endpoint expiry logic**: The prep token expires at `start_at + 2h`. After the call ends, the prep sheet is no longer accessible — this is intentional per AC13.

5. **GCal calendar ID for cancel**: Uses `'primary'` as the calendar ID for delete operations. This works correctly when `gcal_email` is the primary calendar, which is the standard case.
