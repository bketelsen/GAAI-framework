## Technical Details

### Bug Fixed During Implementation

The email validation regex `[^\s@]+@[^\s@]+\.[^\s@]+` required careful escaping in the TypeScript template literal. Inside a template literal, `\s` is treated as the single character `s` (backslash dropped). The fix was to write `\\s` in the template string, which produces `\s` in the rendered HTML, which the browser interprets correctly as a regex whitespace character class negation. The test caught this: initial output was `[^s@]` (broken), after fix is `[^\s@]` (correct).

### window.__SAT__ Config

Unlike `confirm.ts`, `results.ts` does NOT include `turnstileSiteKey` in `window.__SAT__` — this page has no Turnstile widget.

## Known Risks / Limitations

1. **Email regex is basic**: `[^\s@]+@[^\s@]+\.[^\s@]+` passes many invalid emails. Intentional — matches the spec which calls for "inline email validation before submit" without specifying strict RFC 5322 compliance. Server-side validation (422) is the authoritative guard.

2. **`fetchProfilesDirect()` error handling**: The 409 branch calls `fetchProfilesDirect()` which does not have its own error handling — a network error here would be silently swallowed and `revealFullProfiles` would receive an empty array. This is acceptable for the current scope; a future improvement would add error recovery.

3. **No `window.__SAT__` type declaration**: The `window.__SAT__` property is set via a script tag injection and used in client-side JS. TypeScript doesn't know about it (treated as `any` in the IIFE string). This is consistent with how `confirm.ts` and `match.ts` handle it — not a new risk.

4. **E03S06 dependency**: `booking-open` CustomEvent is a stub for the future booking modal (E03S06). The event is dispatched but nothing listens until E03S06 is implemented.
