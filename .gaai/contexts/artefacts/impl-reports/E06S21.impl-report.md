## Deviations from Plan (with Justification)

### Deviation 1: Vectorize metadata — null values omitted instead of passed as `null`

**Affected files:** `src/lib/vectorize.ts`, `src/handlers/admin/vectorize.ts`

**Plan intent:** Store `rate_min`, `rate_max`, `availability` in Vectorize vector metadata, using `null` when values are absent.

**Problem:** `@cloudflare/workers-types` defines `VectorizeVectorMetadataValue` as `string | number | boolean | string[]`. `null` is not a valid metadata value — passing it causes a TypeScript type error (TS2322).

**Fix applied:** Metadata fields are only included in the object when their value is non-null. A typed `Record<string, string | number | boolean>` accumulator is built conditionally:

```typescript
const metadata: Record<string, string | number | boolean> = {
  expert_id: expertId,
};
if (profileData.rate_min != null) metadata.rate_min = profileData.rate_min;
if (profileData.rate_max != null) metadata.rate_max = profileData.rate_max;
if (profileData.availability != null) metadata.availability = profileData.availability;
```

**Justification:** This is strictly more correct than the plan — Vectorize does not support null metadata values at the API level either. The semantic intent (store filterable metadata per vector) is fully preserved. AC5 is satisfied: present values are always stored; null-valued fields are simply absent rather than set to null. Tests for the AC5 `upsertExpertEmbedding` happy path were updated to match (rate_min/rate_max/availability are non-null in the test, so the metadata object contains all three fields as expected).
