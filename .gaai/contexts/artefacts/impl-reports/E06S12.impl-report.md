## Known Risks and Limitations

### Out-of-scope stale references (pre-existing)

`src/handlers/bookings/bookings.test.ts` and `src/handlers/experts/gcal.test.ts` still reference `ANTHROPIC_API_KEY` and `CLOUDFLARE_AI_GATEWAY_URL` in their mock env stub objects. These are typed as `as unknown as Env`, so they compile without error and do not affect runtime behaviour. They are outside the scope of E06S12 and should be cleaned up in a future housekeeping story.

### No live integration test

All tests are unit tests with mocked `fetch`. The OpenAI API is not called in CI. A real `OPENAI_API_KEY` must be provisioned via `wrangler secret put` for staging verification (documented in SETUP.md).

### tool_calls enforcement

The implementation uses `tool_choice: { type: 'function', function: { name: 'extract_requirements' } }` to force tool use. In the rare case the model returns a non-tool response despite forced tool_choice, the handler returns 500. This is expected to be extremely rare with GPT-4o-mini.

### Vertical context non-blocking

If Supabase is unreachable when `satellite_id` is provided, the extraction proceeds without vertical context. This is intentional (graceful degradation) but means satellite-specific tuning silently degrades under DB outage.
