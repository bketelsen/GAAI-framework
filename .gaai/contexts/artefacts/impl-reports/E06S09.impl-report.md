## Architecture Decisions Made

### D1: Two-step join instead of nested Supabase filter
`call_experience_surveys` and `project_satisfaction_surveys` don't have `expert_id` directly; they join through `bookings → matches`. Instead of a potentially unreliable 2-level PostgREST nested filter, we fetch `matchIds` then `bookingIds` upfront (shared across components), then pass pre-fetched ID arrays to each component function. This eliminates deep join uncertainty.

### D2: Shared match/booking ID fetch
`computeAndSaveCompositeScore` fetches `matchIds` (1 query) and `bookingIds` (1 query) once, then runs all 5 components in parallel via `Promise.all`. Total: 7 queries per score computation (vs. up to 15 with individual joins).

### D3: Skip computation when no matches
If `matchIds.length === 0`, we return early without writing to DB. New experts have `composite_score = 0` from initialization (E06S02 schema). Writing `0` with a recent `score_updated_at` when there's no data would be misleading.

### D4: Profile JSONB key conventions (canonical)
AC7 trust_score uses these keys in `experts.profile`:
- `portfolio_items: unknown[]`
- `linkedin_url: string`
- `years_experience: number`
- `certifications: unknown[]`

These are now canonical. Future profile update handlers must use these same keys for trust_score to function correctly.
