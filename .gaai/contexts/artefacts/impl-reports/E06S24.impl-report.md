## Implementation Notes

- **`EXPERT_DB?` type-only retention**: `expertPool.ts` and `syncExpertPool.ts` in Core still reference `EXPERT_DB` (the D1 tier of the 3-tier chain). Removing it from `Env` caused tsc errors. Resolution: kept as `EXPERT_DB?: D1Database` with no wrangler.toml binding — undefined at runtime, D1 tier silently skipped, Hyperdrive fallback used. AC7 satisfied (binding not declared; Core does not have the D1 resource).
- **`workers/matching/src/index.ts` fix**: Initial implementation destructured `method` from `url` (type URL), which is wrong — `URL` has no `method` property. Fixed to `const { method } = request`.
- **E06S26 merge bug fix**: `matches.ts` used `experts.length` for analytics after E06S26 merge, but the variable was renamed to `expertPool` in E06S23. Fixed to `expertPool.length`.

## Human Action Required

Before deploying Matching Worker to staging:

```bash
# 1. Install Matching Worker deps
cd workers/matching && npm install

# 2. Deploy Matching Worker to staging first (Core must not be deployed before Matching)
cd workers/matching && npx wrangler deploy --env staging

# 3. Set Matching Worker secrets
npx wrangler secret put SUPABASE_URL --env staging  # in workers/matching/
# (HYPERDRIVE, KV, D1, AI, Vectorize are binding references — no secrets needed)

# 4. Deploy Core Worker to staging (binding will now resolve)
cd ../../ && npx wrangler deploy --env staging

# 5. Extend GitHub Actions deploy workflow to include workers/matching/ deploy step
```

Note: Dev environment intentionally has no `MATCHING_SERVICE` binding — AC6 fallback path activates automatically in dev/CI.
