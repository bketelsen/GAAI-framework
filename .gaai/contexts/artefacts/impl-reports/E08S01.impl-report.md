## Notable Implementation Decisions

### Expert DB check retained for defence-in-depth

The original `SELECT id FROM experts WHERE id = ${expertId}` query in `handleGcalCallback` was kept after the KV lookup. Rationale: the KV state entry proves the OAuth flow originated from a legitimate auth URL request, but the expert row check adds a second layer — it guards against edge cases where an expert account is deleted between auth URL generation and callback arrival. The story instructions permitted keeping or removing this check; defence-in-depth was chosen.

### `code_verifier` length

`generateCodeVerifier()` produces 32 random bytes encoded as base64url. Base64url encoding of 32 bytes yields exactly 43 characters (32 * 4/3, no padding), which satisfies the RFC 7636 minimum of 43 characters. The test asserts `length >= 43`.

### KV mock approach

Rather than using `vi.fn()` with per-test `mockResolvedValue` chains, a `Map<string, string>` is used as backing store with `mockImplementation` wired in `beforeEach`. This makes test intent clearer: tests that want a valid KV entry call `mockSessionsStore.set(...)` explicitly, and tests that want a missing entry simply do nothing. The `delete` mock removes from the map, so a second `.get()` for the same key would return null — correctly simulating the one-time-use property.

### State parameter URL encoding

`URLSearchParams` encodes values automatically. Since the state token is pure hex (`[0-9a-f]{64}`), no encoding transformation occurs and the value appears verbatim in the URL. The test regex `/^[0-9a-f]{64}$/` validates this without needing to URL-decode.
