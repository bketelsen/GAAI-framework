## Minor Observations (Non-Blocking)

1. **AC2 Count Badge:** The spec mentions a count badge per tab showing total for the active tab. The implementation shows tabs with active highlighting but no badge. The `total` field from loader data is available and could trivially power this. Given no other delivery artefact marks this as required, this is noted as a minor gap but does not constitute a FAIL. Recommendation: add as a follow-up enhancement if product spec is firm on it.

2. **`isUpcoming` computed from `period`:** `const isUpcoming = period === "upcoming"` (line 334). This means action buttons in the detail dialog use the page-level `period` state, not the individual booking's future/past status. This is correct behavior — the period tab drives whether the user is in "upcoming" view, and only upcoming bookings show action buttons.

3. **Reschedule end time auto-computation:** `new_end_at = start + 20 minutes`. This is a simplification — it does not preserve original booking duration. This appears intentional (hardcoded 20-min slots), consistent with the platform's booking model.

4. **No Google Meet button in detail dialog:** The spec mentions "meeting URL with Google Meet button" in AC5. The implementation shows a plain anchor link with a copy button, not a branded "Open in Google Meet" button. This is a presentation gap, not a functional one — the link is clickable and opens the URL.
