### New patterns introduced by E06S11

The following patterns are new and not yet recorded in `conventions.md`:

1. **GCal freebusy + slot computation pattern** — `computeFreeSlots()` in `src/lib/availability.ts`: pure function, accepts `{busyIntervals, heldBookings, rules, now}`, combines GCal freebusy intervals with DB held/confirmed bookings into a single `allBusy` array, enumerates 20-min slots per working day with buffer spacing, applies min notice filter. Slot spacing = `SLOT_DURATION_MS + BUFFER_MS` (buffer between consecutive slots, not around each individual slot).

2. **Hold-then-confirm booking flow pattern** — Two-phase booking: (1) POST `/hold` creates `status=held` row with `held_until=now+10min` and `prep_token`; (2) POST `/:id/confirm` freebusy re-checks, inserts GCal event, updates booking to `status=confirmed`. Race guard at confirm phase deletes the held row and returns 409 if slot was taken concurrently.

3. **CF Worker cron trigger pattern** — `handleScheduled(controller, env)` dispatches by `controller.cron` string match. Crons declared in `wrangler.toml` under top-level `[triggers]` AND each env block (`[env.staging.triggers]`, `[env.production.triggers]`). Worker exports `scheduled` handler alongside `fetch` and `queue`.

4. **GCal retry/error wrapper pattern (AC10)** — `withGcalRetry<T>(fn, onTokenRefresh, context, attempt)` in `src/lib/gcalClient.ts`: generic wrapper, 429→exponential backoff `[1000,4000,16000][attempt]` max 3 retries, 401→`refreshGcalToken`+retry once, 404 on delete normalized to success before entering retry loop, all other non-2xx → `throw GcalApiError` → caller returns 502 `{error:'gcal_error', status, message}`.

5. **Prep endpoint token-expiry pattern** — `GET /api/bookings/:token/prep` is public (no CORS, no auth). Token stored in `bookings.prep_token` (UUID), generated at hold time. Expiry = `start_at + 2h` enforced at read time (not stored). Returns 404 `{error:'prep_token_expired'}` on expiry — same status as not-found to prevent token enumeration.

6. **Availability route placement rule** — `GET /api/experts/:id/availability` must be registered BEFORE the JWT-gated `/api/experts/` block in `index.ts`. The route is satellite-widget-facing (public CORS, no JWT). Pattern mirrors the OAuth callback placement rule (E06S10) for unauthenticated routes that share a URL prefix with authenticated routes.
