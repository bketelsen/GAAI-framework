## Remediation Log

### Cycle 1 — AC4 Fix

**Issue identified:** The `handleGcalStatus` handler selected `gcal_connected` (stored boolean) to derive the `connected` field. However, `gcal_connected` was set to `true` in the callback regardless of whether a refresh token was present in the token exchange response. This violated AC4's requirement: "connected: true only if non-null refresh token exists."

**Root cause:** Two distinct but related bugs:
1. In `handleGcalCallback` (gcal.ts:145), `gcal_connected: true` was unconditional — it was set even when `encryptedRefreshToken === undefined`.
2. In `handleGcalStatus` (gcal.ts:181), `connected` was derived from `gcal_connected` column rather than from actual `gcal_refresh_token` presence.

**Fixes applied:**
- `gcal.ts:146` — changed `gcal_connected: true` to `gcal_connected: encryptedRefreshToken !== undefined` so the stored boolean is always consistent with refresh token presence.
- `gcal.ts:181-183` — changed `select()` to include `gcal_refresh_token` instead of `gcal_connected`; derive `connected` from `data.gcal_refresh_token != null`.
- `gcal.test.ts:148-175` — updated mock data in status tests to use `gcal_refresh_token` presence (not `gcal_connected` boolean) to match new query.

**Post-remediation result:** All 28 tests pass. AC4 now correctly satisfied by implementation.
