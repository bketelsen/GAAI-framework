### AC8: Error handling — 422, 429, Turnstile failure, generic error

**PASS**

- `handleSubmitError(status, data)` function centralizes all error paths.
- `status === 422`: shows field-level error from `data.error` if available, else generic validation message. Form remains editable (`setFormReadonly(false)`, `confirmBtn.disabled = false`, `isSubmitting = false`).
- `status === 429`: shows "Trop de tentatives. Veuillez patienter." exactly as specified.
- `status === 'turnstile_error'` or `'turnstile_timeout'`: shows security verification failure message, resets Turnstile widget via `turnstile.reset(widgetId)`.
- `else` (network, other HTTP): shows "Une erreur est survenue. Veuillez réessayer."
- In all error cases, form returns to editable state — no data lost.

**INFO-2: Turnstile timeout wait is 10 seconds**
`handleConfirm()` polls for `turnstile` global every 250ms up to 10 seconds. On timeout, falls through to `handleSubmitError('turnstile_timeout', null)`. This is a pragmatic implementation that covers the Turnstile SDK load failure path.

### INFO

**INFO-1: Turnstile field name spelling (`cf-turnstile-response` vs `cf_turnstile_response`)**
- Location: `confirm.ts` line 594
- Current: `'cf-turnstile-response': turnstileToken`
- Story AC5 specifies: `cf_turnstile_response` (underscores)
- Cloudflare canonical field name uses hyphens. If the Core API (E06S07) reads the Cloudflare form-post standard name, the implementation is correct and the story spec contains a typo. If E06S07 was coded to the story spec with underscores, a mismatch exists. Resolution: verify E06S07 implementation at integration time.

**INFO-2: `satellite.confirmation_submitted` fires with `turnstile_passed: false`**
- Location: `confirm.ts` lines 539-543
- The `turnstile_passed` property is always `false` at fire time (event fires before Turnstile resolves).
- This matches the spec intent (event tracks the submission attempt), but if PostHog analytics use `turnstile_passed` to filter successful challenges, a second event (`satellite.prospect_created`) is more reliable. No action required unless analytics require the value to be `true` on success.

**INFO-3: `questions.slice(0,3)` vs `displayed` counter**
- In `collectMergedReqs()`, the question collection uses `questions.slice(0,3).forEach(...)` to limit to 3.
- In `renderConfirmationQuestions()`, the `displayed` counter + `needsConf.indexOf(q.field)===-1` guard is used.
- These two approaches are slightly different: render uses `needsConf` filtering + counter, collect uses positional slice. If `confirmation_questions` has items that are NOT in `needs_confirmation` early in the array, `collectMergedReqs` could process different questions than were rendered. This is an edge case unlikely to occur in practice (API should only return questions for `needs_confirmation` fields), but the two methods are not perfectly symmetric.
