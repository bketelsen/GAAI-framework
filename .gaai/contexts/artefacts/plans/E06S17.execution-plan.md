# E06S17: Execution Plan — Survey Submission Endpoints

## Tier
Tier 2 — Core Team (complexity 4, 7 files affected, 9 ACs)

## Approach Evaluation
No `approach-evaluation` needed. All patterns are established:
- Survey token: Web Crypto HMAC-SHA256 JWT, same as `signProspectToken` pattern (E06S07)
- Input validation: Zod `safeParse()` (E06S03 pattern)
- Queue dispatch: `SCORE_COMPUTATION.send()` (E06S09 pattern)
- Ownership check: fetch + compare expert_id (E06S03 pattern)
- Idempotency: unique constraint → 23505 → 409 (E06S06 partial-failure pattern)

## Key Pre-Condition Finding
E06S16 (PR#6 open, not merged) uses `env.PROSPECT_TOKEN_SECRET` for `signSurveyToken()`.
E06S17 AC4 specifies `SURVEY_TOKEN_SECRET`. These differ.
Resolution: E06S17 implements `SURVEY_TOKEN_SECRET`. E06S16's `sendSurveyEmail` must be updated to use `env.SURVEY_TOKEN_SECRET` before both PRs are deployed together.

## Edge Cases
- Token expired → 401
- Token malformed → 401
- booking_id in token doesn't exist in DB → 404
- prospect_id mismatch between token and booking → 403 (not exposing whether booking exists)
- Lead not found → 404
- Supabase error on INSERT → 500 with sanitized message
- Queue dispatch failure should NOT prevent 201 (survey already saved — dispatch is best-effort; if queue fails, throw and let retry handle it)

Actually: Queue dispatch failure SHOULD fail the request (no 201 without queue dispatch). Pattern from E06S06/E06S09: dispatch is part of the transaction. If it fails, the unique constraint ensures the next retry will return 409 (idempotent). This is the correct behavior.
