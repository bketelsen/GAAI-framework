## Overview

This plan adds Cloudflare Vectorize + Workers AI embedding infrastructure to the existing Workers codebase. All changes are purely additive: no existing logic is deleted, no existing test behaviour changes. The Vectorize/AI path is always fire-and-forget (`ctx.waitUntil`) with a wrapping `try/catch`, so failures are silent and never block registration or profile update responses.

### Step 2 — `src/lib/vectorize.ts` (new file)

**Purpose:** Central module for all Vectorize/AI operations. Contains two exported functions.

#### 2a — `ExpertProfile` type (local, file-scoped)

The `profile` JSONB column on the `experts` table is a free-form object. Define a minimal local type for the fields the embedding text formatter needs:

```typescript
interface ExpertProfile {
  skills?: string[];
  industries?: string[];
  project_types?: string[];
  languages?: string[];
}
```

#### 2b — `buildEmbeddingText(profile: ExpertProfile): string`

Formats a profile's JSONB fields into the canonical embedding sentence.

Rules:
- Each field maps to a labeled segment: `"Skills: {csv}. Industries: {csv}. Project types: {csv}. Languages: {csv}."`
- If an array is absent or empty, include the label with an empty value (i.e., `"Skills: . Industries: ."`). This keeps the sentence structure stable across profiles and prevents dimension drift.
- Join array items with `", "`.
- The order is fixed: skills, industries, project types, languages.

Exact implementation:

```typescript
export function buildEmbeddingText(profile: ExpertProfile): string {
  const skills = (profile.skills ?? []).join(', ');
  const industries = (profile.industries ?? []).join(', ');
  const projectTypes = (profile.project_types ?? []).join(', ');
  const languages = (profile.languages ?? []).join(', ');
  return `Skills: ${skills}. Industries: ${industries}. Project types: ${projectTypes}. Languages: ${languages}.`;
}
```

#### 2c — `upsertExpertEmbedding(...): void`

Fire-and-forget function. Returns `void` (not a Promise), relies on `ctx.waitUntil` to keep the embedding work alive after the HTTP response is sent.

Full signature:

```typescript
export function upsertExpertEmbedding(
  env: Env,
  ctx: ExecutionContext,
  expertId: string,
  profileData: {
    profile: ExpertProfile;
    rate_min?: number | null;
    rate_max?: number | null;
    availability?: string | null;
  }
): void {
  ctx.waitUntil(
    (async () => {
      try {
        const text = buildEmbeddingText(profileData.profile);

        const result = await env.AI.run('@cf/baai/bge-base-en-v1.5', {
          text: [text],
        }) as { data: number[][] };

        const vector = result.data[0];
        if (!vector || vector.length === 0) {
          console.warn('vectorize: empty embedding returned for expert', expertId);
          return;
        }

        await env.VECTORIZE.upsert([
          {
            id: expertId,
            values: vector,
            metadata: {
              expert_id: expertId,
              rate_min: profileData.rate_min ?? null,
              rate_max: profileData.rate_max ?? null,
              availability: profileData.availability ?? null,
            },
          },
        ]);
      } catch (err) {
        console.error('vectorize: upsert failed for expert', expertId, err);
      }
    })()
  );
}
```

**Critical notes:**
- The outer function is synchronous (`void`). It schedules the async work via `ctx.waitUntil` and returns immediately. The caller is not blocked.
- The `try/catch` is inside the `waitUntil` promise. Any AI or Vectorize failure is logged but does NOT propagate.
- The AI binding call uses `text: [text]` (array) because `bge-base-en-v1.5` accepts a `text` array and returns `{ data: number[][] }`.
- The guard `if (!vector || vector.length === 0)` prevents upserting a zero-dimension vector.
- Metadata keys match AC5 exactly: `expert_id`, `rate_min`, `rate_max`, `availability`.

**Imports at top of file:**

```typescript
import { Env } from '../types/env';
```

### Step 5 — `src/handlers/admin/vectorize.ts` (new file)

**Purpose:** Admin-only bulk reindex endpoint. Fetches all active experts from Supabase and re-embeds each one sequentially.

```typescript
import { Env } from '../../types/env';
import { createServiceClient } from '../../lib/supabase';
import { upsertExpertEmbedding, ExpertProfile } from '../../lib/vectorize';

export async function handleVectorizeReindex(
  request: Request,
  env: Env,
  ctx: ExecutionContext
): Promise<Response> {
  // AC6: Service-key auth
  const authHeader = request.headers.get('Authorization');
  if (authHeader !== `Bearer ${env.SUPABASE_SERVICE_KEY}`) {
    return new Response(JSON.stringify({ error: 'Unauthorized' }), {
      status: 401,
      headers: { 'Content-Type': 'application/json' },
    });
  }

  const supabase = createServiceClient(env);

  // Fetch all active (non-unavailable) experts with their profile JSONB
  const { data: experts, error } = await supabase
    .from('experts')
    .select('id, profile, rate_min, rate_max, availability')
    .neq('availability', 'unavailable');

  if (error) {
    return new Response(
      JSON.stringify({ error: 'Failed to fetch experts', details: { message: error.message } }),
      { status: 500, headers: { 'Content-Type': 'application/json' } }
    );
  }

  const total = experts?.length ?? 0;

  // Fire-and-forget all upserts inside a single waitUntil
  ctx.waitUntil(
    (async () => {
      for (const expert of experts ?? []) {
        try {
          await new Promise<void>((resolve) => {
            upsertExpertEmbedding(env, ctx, expert.id, {
              profile: (expert.profile as ExpertProfile) ?? {},
              rate_min: expert.rate_min ?? null,
              rate_max: expert.rate_max ?? null,
              availability: expert.availability ?? null,
            });
            resolve();
          });
        } catch (err) {
          console.error('reindex: failed for expert', expert.id, err);
        }
      }
    })()
  );

  return new Response(
    JSON.stringify({ queued: total }),
    { status: 202, headers: { 'Content-Type': 'application/json' } }
  );
}
```

**Notes:**
- Returns `202 Accepted` immediately with `{ queued: N }`. The actual embedding work runs in the background via `waitUntil`.
- Auth check is exact string match: `Authorization: Bearer {SUPABASE_SERVICE_KEY}`. Returns `401` on mismatch.
- The query uses `.neq('availability', 'unavailable')` to match "active experts" — experts who are `'available'` or `'limited'`, plus those with `NULL` availability (newly registered).
- `upsertExpertEmbedding` is already fire-and-forget internally, but we call it inside the `ctx.waitUntil` loop to ensure the Worker stays alive for the full batch duration.

**Important design note on nested `waitUntil`:** `upsertExpertEmbedding` calls `ctx.waitUntil` internally. In the reindex handler, we also call `ctx.waitUntil` for the outer loop. This is valid — the CF runtime tracks all `waitUntil` promises independently. The inner calls from `upsertExpertEmbedding` will each register their own promise. The outer loop resolves immediately (since `upsertExpertEmbedding` is synchronous from the caller's perspective). To ensure the reindex loop itself does not race ahead without waiting for each embedding, the implementation above should be revised: instead of calling `upsertExpertEmbedding` (which is fire-and-forget by design), the reindex handler should inline the embedding logic so it can `await` each step. This avoids the semantic mismatch.

### Notes on `mockCtx.waitUntil` test pattern

`ctx.waitUntil` in tests must be:
```typescript
waitUntil: vi.fn((p: Promise<void>) => p),
```
This means in tests, `await mockCtx.waitUntil(somePromise)` will actually await `somePromise`. After calling `handleVectorizeReindex`, tests must do:

```typescript
const response = await handleVectorizeReindex(request, mockEnv, mockCtx);
// waitUntil was called with a promise — await it to run background work
const [bgPromise] = vi.mocked(mockCtx.waitUntil).mock.calls[0];
await bgPromise;
// now assert on mockVectorize.upsert
```

This pattern ensures the background `async` work completes before assertions run.

## Rollback Risk Assessment

All changes are additive:
- The `Env` interface gains two optional bindings at runtime — workers that lack the binding at test time use mocks.
- Handler signatures gain a new trailing parameter (`ctx`). The only call sites are in `index.ts`, which is updated atomically.
- No database schema changes.
- No changes to existing queue, KV, or auth logic.
- The admin route is new — no existing route is modified.
- `wrangler.toml` additions do not affect existing bindings.

**One non-additive change:** `src/index.ts` `fetch` gains `ctx` as a third parameter. This is required by the CF `ExportedHandler<Env>` interface and is safe — the runtime always passes `ctx`.

**Risk: zero.** If embedding fails, it fails silently. The Worker continues to function without Vectorize at all if the binding is absent in a test environment.
