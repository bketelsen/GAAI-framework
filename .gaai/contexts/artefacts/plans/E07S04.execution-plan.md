# Execution Plan — E07S04: PostHog Funnel Dashboards via API

## Approach Rationale

First use of PostHog Dashboard + Insights REST API in this project. The Story defines the API endpoint, auth mechanism, and data model explicitly — no approach evaluation required. Using native `fetch` (Node 18+) + `npx tsx` runtime for the one-time setup script. No CF Worker patterns apply.

| `scripts/posthog-setup-dashboards.ts` | CREATE | One-time setup script — creates dashboards + insights via PostHog API |

## Implementation Sequence

### Step 1 — Create `scripts/` directory + script file

Create `scripts/posthog-setup-dashboards.ts` at the repo root (sibling to `src/`).

The script does NOT need a separate `tsconfig.json` — run via `npx tsx scripts/posthog-setup-dashboards.ts`.

### Step 2 — Environment variable reading (AC5)

Read at script entry:
```typescript
const apiKey = process.env.POSTHOG_PERSONAL_API_KEY;
const projectId = process.env.POSTHOG_PROJECT_ID;
```
Fail fast with a clear error if either is missing.

### Step 3 — PostHog API client helpers

**Base URL:** `https://eu.posthog.com` (EU Cloud — matches proxy config)

```typescript
const BASE = `https://eu.posthog.com`;
const headers = {
  Authorization: `Bearer ${apiKey}`,
  'Content-Type': 'application/json',
};
```

**Helper functions:**
- `listDashboards()` → `GET /api/projects/{id}/dashboards/` → returns array of `{ id, name }`
  - PostHog paginates at 100 — script handles pagination via `?limit=100` (well under expected dashboard count)
- `createDashboard(name, description)` → `POST /api/projects/{id}/dashboards/` → returns `{ id, name }`
- `listInsights(dashboardId)` → `GET /api/projects/{id}/insights/?dashboards={dashboardId}` → returns array of `{ id, name }`
- `createInsight(dashboardId, payload)` → `POST /api/projects/{id}/insights/` with insight payload (automatically associates with dashboard via `dashboards: [dashboardId]` in the body)

### Step 4 — Idempotency logic (AC6)

For each of the 3 dashboards:
1. `listDashboards()` — find by exact name match
2. If dashboard exists: log "Dashboard '{name}' already exists (id={id}) — skipping" and skip insight creation entirely
3. If not: create dashboard → create insight(s) on it

This is the correct idempotency scope: entire dashboard (including insights) is treated as a unit. Running twice does not create duplicates.

## Edge Cases

| AC | Edge case | Handling |
|----|-----------|---------|
| AC5 | Missing env vars | `process.exit(1)` with clear message before any API call |
| AC5 | Wrong API key (401) | Non-2xx response → `throw new Error('PostHog API error {status}: {body}')` |
| AC5 | Wrong project ID (404) | Same error surface |
| AC6 | Dashboard exists | Skip entirely — log message — no insight duplication |
| AC6 | Partial state (dashboard exists, insights missing) | Acceptable for one-time setup — idempotency operates at dashboard level |
| AC7 | Dashboard creation failed | Error thrown before URL print — no partial output |
| AC2 | `breakdown_type: "event"` | Required alongside `breakdown: "satellite_id"` for event property breakdowns |

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| PostHog Dashboard API schema differs from documented | Low | Medium | Use generic `{ filters: {...} }` shape; validate response status |
| `funnel_window_interval` field not supported in EU instance | Low | Low | Field is optional; funnel still works with default window |

| `tsx` not available in environment | Low | Medium | Use `npx tsx` — available via npm auto-download |

## Rollback Boundaries

The script is purely additive — it only creates dashboards and insights via POST. No DB mutations, no Worker changes, no wrangler.toml changes. Rollback = delete dashboards in PostHog UI if needed. No git rollback needed.
