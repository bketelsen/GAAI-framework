# Execution Plan — E06S06: Async Queue Workers — Email, Lead-Billing

---

## 1. High-Level Approach

Two Cloudflare Queue consumer modules will be created under `src/queues/`:

- `src/queues/email-notifications.ts` — handles `expert.registered`, `booking.confirmed`, and `booking.completed` message types
- `src/queues/lead-billing.ts` — handles `booking.created` message type

The existing `queue` export stub in `src/index.ts` will be replaced with a dispatcher that routes each batch to the correct consumer module based on `batch.queue` name.

Each consumer module exports a single function: `consumeEmailNotifications(batch, env)` and `consumeLeadBilling(batch, env)`. The dispatcher in `src/index.ts` calls the appropriate function.

Each consumer processes messages individually in a loop. For each message:
1. Check idempotency via `SESSIONS` KV (key: `idem:{queue_name}:{message.id}`, TTL 24h). If key exists, ack and skip.
2. Execute business logic based on `type`.
3. On success: write idempotency key to KV, then `message.ack()`.
4. On recoverable failure (external API error, transient DB error): call `message.retry({ delaySeconds })` with exponential backoff based on `message.attempts`.
5. After max retries (handled by CF at platform level), DLQ receives the message. The consumer logs a structured DLQ entry via `console.error` when `message.attempts >= 3` before calling `message.retry()` — this ensures the log is visible before the message is routed to DLQ.

No new KV namespace is introduced. The existing `SESSIONS` namespace serves idempotency at MVP scale.

### Step 3 — Implement idempotency utility

**File:** `src/lib/idempotency.ts` (NEW)

Two functions:

```typescript
// Returns true if this message_id has already been processed (duplicate).
export async function isAlreadyProcessed(
  kv: KVNamespace,
  queue: string,
  messageId: string
): Promise<boolean>

// Records the message as processed. Call after successful business logic.
export async function markProcessed(
  kv: KVNamespace,
  queue: string,
  messageId: string
): Promise<void>
```

Implementation details:
- Key format: `idem:{queue}:{messageId}`
- `isAlreadyProcessed`: `kv.get(key)` — returns `true` if non-null
- `markProcessed`: `kv.put(key, '1', { expirationTtl: 86400 })` — 24h TTL

No third function. The consumer handles the flow: check → execute → mark.

**Why a utility rather than inline:** Idempotency key format is consistent across both consumers and is independently testable.

### Step 4 — Implement retry/DLQ utility

**File:** `src/lib/retryQueue.ts` (NEW)

One function:

```typescript
export function handleMessageFailure<T>(
  message: Message<T>,
  error: unknown,
  queueName: string
): void
```

Logic:
- `delaySeconds = 4 ** (message.attempts - 1)` — yields 1s, 4s, 16s for attempts 1, 2, 3 (since `4**0=1`, `4**1=4`, `4**2=16`)
- If `message.attempts >= 3` (about to be DLQ'd after this retry): log structured DLQ entry via `console.error`:
  ```json
  {
    "queue": "email-notifications",
    "message_id": "...",
    "type": "...",
    "payload": { ... },
    "error": "...",
    "failed_at": "2026-02-21T..."
  }
  ```
  Note: The payload is `message.body` cast to `Record<string, unknown>`. The `type` field is extracted from body if it exists, else `"unknown"`.
- Call `message.retry({ delaySeconds })`

**Why separate utility:** The retry logic is identical across both consumers. Centralizing avoids drift.

**Note on CF platform behavior:** Cloudflare Queues moves a message to the DLQ after the configured `max_retries` in `wrangler.toml`. The `message.attempts` field starts at 1 on the first attempt. So at `attempts === 3`, this is the third attempt — the DLQ entry log fires before the final retry call, ensuring the log is captured in CF Workers logs before the message leaves the consumer.

**wrangler.toml note:** The existing consumer declarations do not specify `max_retries`. The default is 3. No wrangler.toml changes are required. The backoff delay is applied via `message.retry({ delaySeconds })` in the consumer.

### Step 5 — Implement `email-notifications` consumer

**File:** `src/queues/email-notifications.ts` (NEW)

```typescript
export async function consumeEmailNotifications(
  batch: MessageBatch<EmailNotificationMessage>,
  env: Env
): Promise<void>
```

Consumer loop (process each message individually):

```
for (const message of batch.messages) {
  try {
    if (await isAlreadyProcessed(env.SESSIONS, 'email-notifications', message.id)) {
      message.ack();
      continue;
    }

    const body = message.body; // typed as EmailNotificationMessage

    switch (body.type) {
      case 'expert.registered':
        await sendExpertRegisteredEmail(body, env);
        break;
      case 'booking.confirmed':
        await sendBookingConfirmedEmails(body, env);
        await triggerN8nBookingConfirmed(body, env);
        break;
      case 'booking.completed':
        await triggerN8nBookingCompleted(body, env);
        break;
      default:
        // Unknown type — ack to avoid DLQ accumulation on schema drift
        console.warn('email-notifications: unknown message type', (body as any).type);
        message.ack();
        continue;
    }

    await markProcessed(env.SESSIONS, 'email-notifications', message.id);
    message.ack();
  } catch (err) {
    handleMessageFailure(message, err, 'email-notifications');
  }
}
```

#### Sub-function: `sendExpertRegisteredEmail(body, env)`

Calls Resend API:
- `POST https://api.resend.com/emails`
- Headers: `Authorization: Bearer {RESEND_API_KEY}`, `Content-Type: application/json`
- Body:
  ```json
  {
    "from": "Callibrate <onboarding@callibrate.io>",
    "to": ["{body.email}"],
    "subject": "Welcome to Callibrate",
    "html": "<p>Hi {body.name}, welcome to the Callibrate expert network.</p>"
  }
  ```
- Throws on non-2xx response (triggers retry).

#### Sub-function: `sendBookingConfirmedEmails(body, env)`

Fetches expert email from `experts` table via Supabase service client. Fetches prospect email from `prospects` table.

Sends two emails via Resend:
1. Expert email — subject "New booking confirmed", body includes prospect context placeholder and `body.meeting_url`
2. Prospect email — subject "Your booking is confirmed", body includes `body.meeting_url` and `body.scheduled_at`

Each Resend call is made sequentially. Failure of either throws (triggers retry for the entire message — idempotency guard prevents duplicate sends on retry since the KV mark only writes on full success).

#### Sub-function: `triggerN8nBookingConfirmed(body, env)`

```
POST {env.N8N_WEBHOOK_URL}/booking-confirmed
Content-Type: application/json
Body: { booking_id, expert_id, prospect_id, meeting_url, scheduled_at }
```

Throws on non-2xx response.

#### Sub-function: `triggerN8nBookingCompleted(body, env)`

```
POST {env.N8N_WEBHOOK_URL}/booking-completed
Content-Type: application/json
Body: { booking_id, expert_id, prospect_id, scheduled_at }
```

Throws on non-2xx response.

### Step 6 — Implement `lead-billing` consumer

**File:** `src/queues/lead-billing.ts` (NEW)

```typescript
export async function consumeLeadBilling(
  batch: MessageBatch<LeadBillingMessage>,
  env: Env
): Promise<void>
```

Consumer loop:

```
for (const message of batch.messages) {
  try {
    if (await isAlreadyProcessed(env.SESSIONS, 'lead-billing', message.id)) {
      message.ack();
      continue;
    }

    const body = message.body;

    if (body.type !== 'booking.created') {
      console.warn('lead-billing: unknown message type', (body as any).type);
      message.ack();
      continue;
    }

    await processLeadBilling(body, env);

    await markProcessed(env.SESSIONS, 'lead-billing', message.id);
    message.ack();
  } catch (err) {
    handleMessageFailure(message, err, 'lead-billing');
  }
}
```

#### Sub-function: `processLeadBilling(body, env)`

Steps in order:
1. Create `leads` row:
   ```sql
   INSERT INTO leads (booking_id, expert_id, prospect_id, status)
   VALUES (body.booking_id, body.expert_id, body.prospect_id, 'pending')
   RETURNING id
   ```
   Store returned `lead_id`.

2. Fetch expert email from `experts` table (needed for Lemon Squeezy checkout attribution).

3. Call Lemon Squeezy API to create a one-time checkout:
   ```
   POST https://api.lemonsqueezy.com/v1/checkouts
   Authorization: Bearer {LEMON_SQUEEZY_API_KEY}
   Content-Type: application/vnd.api+json
   Accept: application/vnd.api+json
   Body: {
     "data": {
       "type": "checkouts",
       "attributes": {
         "checkout_data": {
           "email": "{expert_email}",
           "custom": {
             "lead_id": "{lead_id}",
             "booking_id": "{body.booking_id}",
             "expert_id": "{body.expert_id}"
           }
         }
       },
       "relationships": {
         "store": { "data": { "type": "stores", "id": "{LS_STORE_ID}" } },
         "variant": { "data": { "type": "variants", "id": "{LS_VARIANT_ID}" } }
       }
     }
   }
   ```
   **MVP note:** `LS_STORE_ID` and `LS_VARIANT_ID` are not yet secrets in `env.ts`. At MVP, these are hardcoded constants in the consumer file with a `// TODO: move to env secrets` comment. This avoids adding secrets that require coordination with Lemon Squeezy account setup to this story's scope. The implementor must fill in real values from the LS dashboard.

   Extract `ls_checkout_id` from the response: `data.data.id`.

4. On Lemon Squeezy API failure (non-2xx): log error, throw (consumer retries via `handleMessageFailure`). AC8 is satisfied: logs error, keeps status as `'pending'`, routes to DLQ after 3 retries.

5. Update `leads` row:
   ```sql
   UPDATE leads SET ls_checkout_id = '{ls_checkout_id}', status = 'billed' WHERE id = '{lead_id}'
   ```
   Per AC2 + planning guidance: "billed" at MVP means "checkout link created and sent", not "payment confirmed".

**Idempotency edge case on partial failure:** If the `leads` INSERT succeeded but the LS call failed, on retry the INSERT will fail with a unique constraint violation on `booking_id` (assuming `leads.booking_id` has a unique constraint — see Step 7 migration). The consumer catches this case: if the INSERT returns a conflict error, it fetches the existing `lead_id` for this `booking_id` instead of creating a duplicate. This is the only place where the consumer needs special handling beyond the top-level idempotency KV guard.

### Step 7 — Database migration: `leads` table constraint

**Action:** Verify whether `leads.booking_id` has a unique constraint. If not, add one.

The `leads` table schema from `database.ts` does not show a unique index on `booking_id`. A booking should only ever generate one lead. The migration:

```sql
ALTER TABLE leads ADD CONSTRAINT leads_booking_id_unique UNIQUE (booking_id);
```

**Why this matters:** Without this, retries of a failed `processLeadBilling` would insert duplicate `leads` rows. The KV idempotency guard catches full-success retries. This DB constraint catches partial-failure retries where the INSERT succeeded but a later step failed.

**Migration name:** `add_leads_booking_id_unique_constraint`

This migration must be applied to staging before deploying E06S06.

## 4. Edge Cases per Acceptance Criterion

| AC | Scenario | Handling |
|----|----------|----------|
| AC1 | `expert.registered` message missing `email` or `name` | TypeScript type guard catches at compile time; runtime: Resend call fails → retry |
| AC1 | Resend API rate limit (429) | Throws on non-2xx → retry with backoff |
| AC1 | Expert email not in `experts` table (for `booking.confirmed`) | Supabase query returns null → throw → retry |
| AC1 | Prospect email not in `prospects` table | Supabase query returns null → throw → retry |
| AC2 | `leads` row INSERT fails (DB down) | Throws → retry via handleMessageFailure |
| AC2 | LS API returns 401 (invalid key) | Throws → retries → DLQ after 3; structured log includes error |
| AC2 | LS API returns 422 (invalid payload) | Throws → retries → DLQ (same handling — non-recoverable errors also retry to DLQ per AC8) |
| AC2 | Partial failure: INSERT succeeded, LS failed on retry | Unique constraint on `booking_id` prevents duplicate INSERT; consumer catches conflict error and fetches existing lead_id before proceeding to LS call |
| AC4 | First attempt fails, second attempt succeeds | `message.attempts === 2`, `delaySeconds = 4`. Retry fires after 4s. |
| AC4 | All three attempts fail | `handleMessageFailure` fires DLQ log on `attempts === 3`, then calls `message.retry()` one last time. CF platform routes to DLQ. |
| AC5 | DLQ log format | Structured via `console.error(JSON.stringify({ queue, message_id, type, payload, error, failed_at }))` — visible in CF Workers log dashboard |
| AC6 | Same `message.id` received twice (duplicate delivery) | KV key `idem:email-notifications:{id}` exists → `message.ack()` immediately, no side effects |
| AC6 | `markProcessed` KV write fails | Throws → consumer retries business logic again → side effects may repeat. Mitigation: Resend is idempotent by default (same email sent twice is acceptable at MVP); LS checkout creation is protected by the DB unique constraint on booking_id. This is documented as a known MVP limitation. |
| AC7 | n8n webhook returns non-2xx | Throws → message retries. The n8n trigger is in the same try block as the emails — on retry, both emails are re-sent (idempotency issue for emails). Mitigation: Resend deduplication is not natively available per message; at MVP this is acceptable (worst case: duplicate confirmation email on n8n failure). |
| AC8 | LS checkout creation fails on all 3 retries | `leads.status` remains `'pending'` (only updated to `'billed'` after successful LS call). DLQ log is written. |
| AC10 | `booking.completed` event — no email sent, only n8n trigger | Consumer switch correctly routes to `triggerN8nBookingCompleted` only. No Resend call. |

- `wrangler secret put RESEND_API_KEY` / `LEMON_SQUEEZY_API_KEY` / `N8N_WEBHOOK_URL` — secrets added to CF dashboard must be manually deleted if rollback is permanent
- `leads_booking_id_unique` DB constraint — must be explicitly dropped:
  ```sql
  ALTER TABLE leads DROP CONSTRAINT leads_booking_id_unique;
  ```
  Only required if the constraint causes problems with existing data.

**Queue behavior during rollback:**
- Rolling back to the `batch.ackAll()` stub means all messages in flight will be acked without processing. Messages already in the queue at rollback time will be silently dropped. This is acceptable because the stub's documented purpose is exactly this (DLQ prevention during stub period).

**Git boundary:** All work is done on branch `story/E06S06`. Rollback is `git checkout production && git branch -D story/E06S06`.
