  it('rejects a survey token with exp in the past', async () => {
    // Manually craft a token with past exp to simulate expired 7-day token
    const encoder = new TextEncoder();
    const toBase64Url = (data: Uint8Array | ArrayBuffer): string => {
      const bytes = data instanceof Uint8Array ? data : new Uint8Array(data);
      let binary = '';
      for (const byte of bytes) binary += String.fromCharCode(byte);
      return btoa(binary).replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');

    );
    const sigBuffer = await crypto.subtle.sign('HMAC', key, encoder.encode(signingInput));
    const expiredToken = `${signingInput}.${toBase64Url(sigBuffer)}`;

    const result = await verifySurveyToken(expiredToken, SECRET);
    expect(result).toBeNull();

Risk note: Check whether RLS is already enabled on `satellite_configs` before adding the policy. If there is an existing policy that conflicts with this one, the migration will fail. ### 3.3 `supabase/migrations/20260224200003_lead_credit_rpc_ownership.sql`

**Purpose:** AC10 — Add ownership assertion to `debit_lead_credit` and `restore_lead_credit`.

For `debit_lead_credit`: the ownership assertion is that `p_expert_id` matches the expert who should be charged. The function receives `p_expert_id` and `p_booking_id`. The booking's matched expert is the authoritative owner. The assertion should verify that the lead being debited belongs to `p_expert_id` by checking that no lead for this `booking_id` exists with a different `expert_id`.

More precisely: the function already locks `experts WHERE id = p_expert_id FOR UPDATE`. The ownership assertion should verify that `p_booking_id` is not already associated with a lead for a DIFFERENT expert:

```sql
-- Ownership check: reject if this booking already has a lead for a different expert
IF EXISTS (
  SELECT 1 FROM leads
  WHERE booking_id = p_booking_id
  AND expert_id != p_expert_id
) THEN
  RETURN jsonb_build_object('success', false, 'reason', 'booking_expert_mismatch');
END IF;
```

For `restore_lead_credit`: the function receives `p_expert_id` and `p_lead_id`. The ownership assertion verifies that the lead's `expert_id` matches `p_expert_id`:

```sql
-- Ownership check: reject if lead does not belong to this expert
DECLARE
  v_lead_expert_id uuid;
BEGIN
  SELECT expert_id INTO v_lead_expert_id FROM leads WHERE id = p_lead_id;
  IF NOT FOUND OR v_lead_expert_id != p_expert_id THEN
    RETURN jsonb_build_object('success', false, 'reason', 'lead_not_owned');
  END IF;
  ...
END;
```

## 6. Risk Notes

### R1 — Breaking change: `verifyProspectToken` signature (HIGH IMPACT, contained)
Adding `expectedAud` as a 4th parameter is a breaking change. Every call site in the codebase must be updated simultaneously or TypeScript compilation fails. Confirmed call sites: `prospects.ts` lines 437 and 521 only. No other files call `verifyProspectToken` (confirmed by file listing — no other routes or handlers reference it).

### R2 — Breaking change: `signProspectToken` signature (HIGH IMPACT, contained)
Adding `aud` as a 3rd parameter. Every call site: `prospects.ts` lines 268, 331, 394 only. Confirmed no other callers.

### R3 — Tokens issued before E08S05 deployment will fail verification after deployment (ROLLOUT)
Existing prospect tokens in circulation (24h TTL) will lack the `aud` claim. After deployment, `verifyProspectToken` with `expectedAud` check will reject them. This means all in-flight prospect sessions break at deploy time. Impact window: up to 24h. Mitigation: acceptable for a staging-first deploy where no real users hold active sessions. For production deploy, schedule at low-traffic time.

### R4 — Survey tokens in circulation (30d → 7d) (ROLLOUT)
Existing survey tokens issued with 30d TTL remain valid (they have a valid signature and non-expired `exp`). The TTL reduction only affects newly issued tokens. No breakage on existing links. No rollout risk.

### R5 — cors.ts ANON_KEY change depends on Supabase policy (DEPLOYMENT ORDER)
If the anon SELECT policy on `satellite_configs` is not applied before the cors.ts code is deployed, all satellite domain CORS checks will fail (deny-on-error). The deployment order in Phase 1 / Phase 4 must be respected. The safe approach: apply Supabase migration first, then deploy the Worker code.

### R6 — `webhook_events` table must exist before lemonsqueezy.ts L2 code (DEPLOYMENT ORDER)
If the Worker is deployed before the migration, the L2 DB check will throw on every webhook event (table not found), triggering the catch block that returns 500. The KV L1 check happens before the DB L2, so if the event is in KV it will still deduplicate. But new events will 500. Deploy migration first.

### R7 — `satellite_configs.vertical` column name (VERIFY)
The `extract.ts` query selects `.select('vertical')` from `satellite_configs`. The TypeScript type `Database` should have this column. If the column does not exist in the Supabase type definitions, the query will be typed as `unknown`. Verify that `satConfig.vertical` is typed correctly; if not, add a type assertion.

### R8 — `localConnectionString` removal and local dev breakage
Removing `localConnectionString` from `wrangler.toml` means any developer running `wrangler dev` without a `.dev.vars` file will get a Hyperdrive connection error. The SETUP.md update (AC6) must document the `.dev.vars` alternative clearly. Confirm the exact wrangler env var name for overriding `localConnectionString` from `.dev.vars` — as of wrangler 3.x, the pattern is `WRANGLER_HYPERDRIVE_LOCAL_CONNECTION_STRING_HYPERDRIVE=postgresql://...` where `HYPERDRIVE` is the binding name. This should be verified against current wrangler 4.x docs.

### R9 — `webhook_events` RLS and service key access
The `webhook_events` table is queried via `createSql(env)` which uses Hyperdrive / postgres.js directly (not the Supabase client). Hyperdrive bypasses RLS by default (it connects as the postgres superuser via the connection string). So the RLS on `webhook_events` is cosmetic — the actual enforcement is at the application level (only the Worker ever writes to this table). This is the correct pattern for the codebase.

### R10 — Ownership assertion in `restore_lead_credit` and impact on lead flagging
The `restore_lead_credit` function is called by `POST /api/leads/:id/flag` (confirmed from the migration comment). The HTTP handler already validates expert ownership before calling the RPC. Adding ownership assertion at the DB level is defense-in-depth and creates no breakage for well-formed calls. Malformed calls (wrong expert_id) now get a clean `lead_not_owned` JSON error instead of silently restoring credits for the wrong expert.

### R11 — `debit_lead_credit` ownership assertion and retry safety
The ownership check `booking_id + expert_id != p_expert_id` uses `!=` which is correct. On legitimate retries (same expert, same booking), the check passes because `expert_id = p_expert_id`. The check only fires if the same booking_id is submitted with a different expert_id, which would be a logical error in the billing queue.
