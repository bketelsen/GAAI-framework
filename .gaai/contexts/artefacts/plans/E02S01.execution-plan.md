# E02S01 Execution Plan — Dashboard Scaffold

## Story Summary

Scaffold `workers/frontend/dashboard/` — a React Router v7 SSR application on Cloudflare Workers. Provides server-side authentication via Supabase PKCE cookies, a responsive sidebar shell, TailwindCSS 4 design system, shadcn/ui components, typed API client, and full CI/CD pipeline.

---

## Approach Evaluation: React Router v7 + Cloudflare Vite Plugin

**Research conclusion (2026-02-27):** The Cloudflare-maintained template at `cloudflare/templates/react-router-starter-template` is the canonical reference. Key findings:

1. `@cloudflare/vite-plugin` v1 uses `cloudflare({ viteEnvironment: { name: "ssr" } })` — the `name: "ssr"` option is required for React Router framework mode.
2. `react-router.config.ts` must include `future: { unstable_viteEnvironmentApi: true }` to enable compatibility with the Cloudflare Vite plugin.
3. Workers entry point is `workers/app.ts` using `createRequestHandler` from `react-router` (not `@react-router/cloudflare`).
4. TypeScript requires THREE tsconfig files: `tsconfig.json` (root, no include), `tsconfig.node.json` (for vite.config.ts), `tsconfig.cloudflare.json` (for app/ + workers/).
5. A `worker-configuration.d.ts` file is generated by `wrangler types` and provides the `Env` interface.
6. **Vitest**: Do NOT use `@cloudflare/vitest-pool-workers` / `defineWorkersConfig` for this project — known incompatibility with React Router entry points (esbuild error "react cannot be marked as external"). Use plain `vitest` with jsdom environment to test loaders as pure functions.
7. `@supabase/ssr` requires `nodejs_compat` flag (already planned) — uses `parseCookieHeader` + `serializeCookieHeader` from the package itself.
8. TailwindCSS 4 uses `@tailwindcss/vite` plugin (not PostCSS) — no `tailwind.config.ts` needed.

import { createRequestHandler } from "react-router";

declare module "react-router" {
  export interface AppLoadContext {
    cloudflare: {
      env: Env;
      ctx: ExecutionContext;
    };
  }
}

const requestHandler = createRequestHandler(
  () => import("virtual:react-router/server-build"),
  import.meta.env.MODE,
);

export default {
  fetch(request: Request, env: Env, ctx: ExecutionContext) {
    return requestHandler(request, {
      cloudflare: { env, ctx },
    });
  },
} satisfies ExportedHandler<Env>;

import type { AppLoadContext, EntryContext } from "react-router";
import { ServerRouter } from "react-router";
import { isbot } from "isbot";
import { renderToReadableStream } from "react-dom/server";

export default async function handleRequest(
  request: Request,
  responseStatusCode: number,
  responseHeaders: Headers,
  routerContext: EntryContext,
  _loadContext: AppLoadContext,
) {
  let shellRendered = false;
  const userAgent = request.headers.get("user-agent");

  const body = await renderToReadableStream(
    <ServerRouter context={routerContext} url={request.url} />,
    {
      onError(error: unknown) {
        responseStatusCode = 500;
        if (shellRendered) {
          console.error(error);
        }
      },
    },
  );
  shellRendered = true;

  if ((userAgent && isbot(userAgent)) || routerContext.isSpaMode) {
    await body.allReady;
  }

  responseHeaders.set("Content-Type", "text/html");
  return new Response(body, {
    headers: responseHeaders,
    status: responseStatusCode,
  });
}

### STEP 9 — Supabase auth session helpers (AC4)

`workers/frontend/dashboard/app/lib/session.server.ts`:

```typescript
import { createServerClient, parseCookieHeader, serializeCookieHeader } from "@supabase/ssr";
import { redirect } from "react-router";
import type { AppLoadContext } from "react-router";

export type SessionUser = {
  id: string;
  email: string | undefined;
};

export type Session = {
  user: SessionUser;
  token: string;
};

/**
 * Creates a Supabase server client bound to the current request's cookies.
 * Returns both the client and a response headers object that will carry
 * any Set-Cookie headers for token refresh.
 */
function createSupabaseServerClient(request: Request, env: Env) {
  const responseHeaders = new Headers();

  const supabase = createServerClient(
    env.SUPABASE_URL,
    env.SUPABASE_ANON_KEY,
    {
      cookies: {
        getAll() {
          return parseCookieHeader(request.headers.get("Cookie") ?? "") as {
            name: string;
            value: string;
          }[];
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value, options }) => {
            responseHeaders.append(
              "Set-Cookie",
              serializeCookieHeader(name, value, options),
            );
          });
        },
      },
    },
  );

  return { supabase, responseHeaders };
}

/**
 * Reads and validates the session from the request cookies.
 * Returns { user, token } if authenticated, null otherwise.
 *
 * Uses getUser() (not getSession()) — validates JWT server-side via Supabase.
 */
export async function getSession(
  request: Request,
  env: Env,
): Promise<{ session: Session | null; responseHeaders: Headers }> {
  const { supabase, responseHeaders } = createSupabaseServerClient(request, env);

  const {
    data: { user },
    error,
  } = await supabase.auth.getUser();

  if (error || !user) {
    return { session: null, responseHeaders };
  }

  const {
    data: { session: supabaseSession },
  } = await supabase.auth.getSession();

  const token = supabaseSession?.access_token ?? "";

  return {
    session: {
      user: { id: user.id, email: user.email },
      token,
    },
    responseHeaders,
  };
}

/**
 * Requires an authenticated session. Throws a redirect to /login if not found.
 * Use in React Router loaders for protected routes.
 *
 * Usage:
 *   const { session, responseHeaders } = await requireSession(request, context.cloudflare.env);
 */
export async function requireSession(
  request: Request,
  env: Env,
): Promise<{ session: Session; responseHeaders: Headers }> {
  const { session, responseHeaders } = await getSession(request, env);

  if (!session) {
    throw redirect("/login");
  }

  return { session, responseHeaders };
}
```

**Edge cases:**
- `getUser()` validates the JWT against Supabase's public keys on every call (secure). Never use `getSession()` alone for auth guards.
- `responseHeaders` carries any `Set-Cookie` headers for token refresh. Callers must include these in their response or the session will expire without auto-refresh.
- `parseCookieHeader` returns `{ name, value }[]` directly from `@supabase/ssr` — no external cookie library needed in Workers.

### STEP 10 — API client (AC10)

`workers/frontend/dashboard/app/lib/api.server.ts`:

```typescript
export class ApiError extends Error {
  constructor(
    public readonly status: number,
    public readonly body: unknown,
    message: string,
  ) {
    super(message);
    this.name = "ApiError";
  }
}

async function apiFetch<T>(
  env: Env,
  token: string,
  method: "GET" | "POST" | "PATCH",
  path: string,
  params?: Record<string, string>,
  body?: unknown,
): Promise<T> {
  const url = new URL(path, env.CORE_API_URL);
  if (params) {
    Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v));
  }

  const response = await fetch(url.toString(), {
    method,
    headers: {
      Authorization: `Bearer ${token}`,
      "Content-Type": "application/json",
    },
    body: body !== undefined ? JSON.stringify(body) : undefined,
  });

  if (!response.ok) {
    const errorBody = await response.json().catch(() => null);
    throw new ApiError(
      response.status,
      errorBody,
      `API ${method} ${path} failed: ${response.status}`,
    );
  }

  return response.json() as Promise<T>;
}

export function apiGet<T>(
  env: Env,
  token: string,
  path: string,
  params?: Record<string, string>,
): Promise<T> {
  return apiFetch<T>(env, token, "GET", path, params);
}

export function apiPost<T>(
  env: Env,
  token: string,
  path: string,
  body: unknown,
): Promise<T> {
  return apiFetch<T>(env, token, "POST", path, undefined, body);
}

export function apiPatch<T>(
  env: Env,
  token: string,
  path: string,
  body: unknown,
): Promise<T> {
  return apiFetch<T>(env, token, "PATCH", path, undefined, body);
}
```

**Edge cases:**
- `CORE_API_URL` is the base (e.g., `https://callibrate-core-staging.frederic-geens-consulting.workers.dev`). Paths are relative (e.g., `/api/experts/123/leads`).
- `ApiError` is typed for use in loader `catch` blocks.
- Do not use `@supabase/supabase-js` database methods in this client — only the CORE_API_URL (Layer 3 rule).

export async function action({ request, context }: ActionFunctionArgs) {

## Edge Cases per AC

### AC2 — React Router + Cloudflare Vite plugin
- `unstable_viteEnvironmentApi: true` is required in `react-router.config.ts`. Without it, the Cloudflare plugin cannot serve the SSR handler.
- `cloudflare({ viteEnvironment: { name: "ssr" } })` must be FIRST in the plugins array. If `reactRouter()` comes first, the environments are set up in the wrong order.

### AC4 — Supabase Auth
- `getUser()` makes a network call to Supabase on every request. This is intentional — it validates the JWT. Do not cache this in KV for now.
- Token refresh: `@supabase/ssr` automatically refreshes tokens when needed. The `setAll` callback ensures refreshed tokens propagate as `Set-Cookie` headers. If `responseHeaders` is NOT forwarded from the loader to the response, sessions will expire silently.
- `SUPABASE_URL` and `SUPABASE_ANON_KEY` must be real values for auth to work. The placeholder values in `wrangler.toml` must be replaced before testing auth flows. For the smoke test, they are mocked.

### AC5 — Auth layout
- `_layout.tsx` wraps routes named `_layout.*`. In React Router v7 file-based routing, a file named `_layout.tsx` creates a pathless layout route. All routes matching `_layout.*` are children. Verify with `react-router routes` command.
- The `/dashboard` route must be `_layout.dashboard.tsx` (not `dashboard.tsx`) to be inside the auth guard. The `dashboard.tsx` index is inside `_layout.dashboard.tsx` as a child route.

### AC6 — Login
- Supabase `signInWithPassword` returns a user object and sets session cookies via the `setAll` callback. The `redirect("/dashboard", { headers: responseHeaders })` call MUST include the response headers — if omitted, the session cookie is not set and the user will be immediately redirected back to `/login`.

### AC7/AC8 — Dashboard shell
- `useState` in a server-rendered layout: the sidebar `sidebarOpen` state starts as `false` on both server and client, ensuring hydration match. No SSR/hydration mismatch.
- `useOutletContext` in `dashboard.tsx`: the parent `_layout.dashboard.tsx` passes `{ user }` via `<Outlet context={...} />`. If `dashboard.tsx` calls `useOutletContext` before the parent loader resolves, it will be `undefined`. This is safe because React Router guarantees parent loaders resolve before children render.

### AC10 — API client
- `CORE_API_URL` for local dev: `http://localhost:8787` (core API dev server). The `apiGet`/`apiPost` functions construct the full URL via `new URL(path, base)`. Paths must start with `/` for correct URL construction.
- `ApiError` in loaders: loaders should catch `ApiError` and return appropriate responses (e.g., 404 cards, error toasts). Future route loaders will use `try/catch` on `apiGet()`.

### AC12 — TailwindCSS 4
- No `tailwind.config.ts` file is needed. TailwindCSS 4 reads configuration from CSS (`@theme` blocks). The `@tailwindcss/vite` plugin handles discovery.
- `@import "tailwindcss"` must be in the CSS file imported via `?url` in `root.tsx` links. Without `?url`, Vite treats it as a CSS module import.

### AC13 — shadcn/ui
- shadcn/ui CLI modifies `package.json` to add `sonner` (for the Sonner component) and potentially other packages. After running the CLI, re-run `npm install` from repo root.
- Components import from `~/lib/utils` which maps to `app/lib/utils.ts`. The tsconfig `paths` must be configured (Step 3.3) before running the CLI or TypeScript will error.
- If CLI prompts "Would you like to use CSS variables? › yes" — answer yes. This is already set in `components.json`.

### AC18 — Smoke test
- `react-router` must be importable from test context. Since we use plain vitest (not workers runtime), React Router's `redirect()` function works as a normal module import.
- The `_layout.tsx` loader throws a `Response` (not throws an `Error`) when redirecting. The test catches `Response` instances, not generic errors.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `unstable_viteEnvironmentApi` removed/renamed | Low | High | Pin `@react-router/dev` to 7.9.6; check CF changelog before upgrading |
| shadcn/ui CLI interactive prompt blocks CI | Medium | Medium | Use `--yes` flag: `npx shadcn@latest add --yes button card ...`; if still blocked, copy component files manually |
| `@supabase/ssr` `parseCookieHeader` API changes | Low | Medium | Pin `@supabase/ssr` to `^0.5.0`; only the `getAll/setAll` interface is stable |
| `Env` type missing before `wrangler types` run | High | Low | Manual `worker-configuration.d.ts` placeholder (Step 3.4) unblocks tsc |
| React Router generated types (`.react-router/`) missing | High | Medium | Run `npm run cf-typegen` after install; or stub virtual module for tests |
| vitest cannot import `virtual:react-router/server-build` | High | Medium | Tests do not import the virtual module; they import loaders directly |
| `Set-Cookie` not forwarded from loader | Medium | High | Always spread `responseHeaders` into Response; see AC4 edge case |
| wrangler deploy fails: asset directory not found | Medium | High | Run `npm run build` before `wrangler deploy` in CI (Step 18 correction) |
