### Step 4 — `src/queues/score-computation.ts`

**Architecture decision:** The orchestrator function fetches shared intermediate data (matchIds, bookingIds) once, then passes them as arrays to the five component functions. This avoids N-deep Supabase joins and is reliable across PostgREST versions.

#### 4a. Composite score formula (AC2)
`(call_experience_avg × 0.35) + (trust_score × 0.20) + (client_satisfaction_avg × 0.20) + (hire_rate × 0.10) + (recency_score × 0.15)`

#### 4b. `consumeScoreComputation(batch, env)` — exported consumer

Same structure as `email-notifications.ts`:
- Per-message idempotency via `isAlreadyProcessed(env.SESSIONS, 'score-computation', message.id)`
- Switch on `body.type` — all three types route to `computeAndSaveCompositeScore(body.expert_id, env)`
- `markProcessed` + `ack` on success
- `handleMessageFailure(message, err, 'score-computation')` in catch
- Unknown type: `console.warn` + `message.ack()` + `continue`

#### 4c. `computeAndSaveCompositeScore(expertId, env)` — orchestrator

```
1. supabase = createServiceClient(env)
2. Fetch matchIds: SELECT id FROM matches WHERE expert_id = expertId
   → if empty, write composite_score = 0 (no matches yet) and return
3. Fetch bookingIds: SELECT id FROM bookings WHERE match_id IN (matchIds)
   → bookingIds may be empty (matches exist but no bookings yet)
4. Run 5 components in parallel: Promise.all([
     computeCallExperienceAvg(bookingIds, supabase),      // AC3
     computeClientSatisfactionAvg(bookingIds, supabase),   // AC4
     computeHireRate(expertId, supabase),                  // AC5
     computeRecencyScore(matchIds, supabase),              // AC6
     computeTrustScore(expertId, supabase),                // AC7
   ])
5. compositeScore = (callExp × 0.35) + (trust × 0.20) + (satisfaction × 0.20) + (hire × 0.10) + (recency × 0.15)
6. Round to 2 decimal places: Math.round(compositeScore * 100) / 100
7. Write: UPDATE experts SET composite_score = compositeScore, score_updated_at = now() WHERE id = expertId
   → on error: throw (consumer will retry)
8. No dispatch to matching-jobs (DEC-33). AC9: re-ranking skipped at MVP.
```

**Note:** If matchIds is empty, we still write composite_score = 0 to ensure score_updated_at is set. Expert with no matches has trust_score contribution only — but since trust_score alone makes sense, we could still compute it. Alternatively, for MVP simplicity: only write composite_score when there are matches. This is consistent with "new expert starts at 0" per story notes.

**Revised approach:** If matchIds is empty → skip computation (scores all remain 0 from initialization). Only compute when there is at least one match. This keeps the logic simple and avoids a write with no meaningful data.

#### 4d. `computeCallExperienceAvg(bookingIds: string[], supabase): Promise<number>` — AC3

```
if (bookingIds.length === 0) return 0
query: SELECT score FROM call_experience_surveys WHERE booking_id IN (bookingIds) AND score IS NOT NULL
scores = data.map(r => r.score! * 20)
return scores.length > 0 ? avg(scores) : 0
// avg = sum/count. Clamp to [0,100] via min/max of score (1×20=20 to 5×20=100).
```

#### 4e. `computeClientSatisfactionAvg(bookingIds: string[], supabase): Promise<number>` — AC4

```
if (bookingIds.length === 0) return 0
query: SELECT score FROM project_satisfaction_surveys WHERE booking_id IN (bookingIds) AND score IS NOT NULL
scores = data.map(r => r.score! * 10)
return scores.length > 0 ? avg(scores) : 0
// Clamp via score range (1×10=10 to 10×10=100).
```

#### 4f. `computeHireRate(expertId: string, supabase): Promise<number>` — AC5

```
query: SELECT conversion_declared FROM lead_evaluations WHERE expert_id = expertId
if (data.length === 0) return 0
converted = data.filter(r => r.conversion_declared === true).length
return (converted / data.length) * 100
// Returns [0, 100]. null values count as non-converted in denominator.
```

#### 4g. `computeRecencyScore(matchIds: string[], supabase): Promise<number>` — AC6

```
if (matchIds.length === 0) return 0
query: SELECT scheduled_at FROM bookings
  WHERE match_id IN (matchIds) AND scheduled_at IS NOT NULL
  ORDER BY scheduled_at DESC LIMIT 1
if (data.length === 0 || !data[0].scheduled_at) return 0
daysSince = (Date.now() - new Date(data[0].scheduled_at).getTime()) / 86400000
if (daysSince < 30) return 100
if (daysSince >= 180) return 0
return Math.round(100 - ((daysSince - 30) / 150) * 100)
// Linear interpolation: 100 at day 30, 0 at day 180.
// Clamp: Math.min(100, Math.max(0, result)) for edge cases (future dates).
```

#### 4h. `computeTrustScore(expertId: string, supabase): Promise<number>` — AC7

```
query: SELECT verified_at, bio, profile FROM experts WHERE id = expertId SINGLE
if (error) throw it (triggers consumer retry)
score = 0
p = data.profile as Record<string, unknown> | null

// Criterion 1: verified email (20pts)
if (data.verified_at !== null) score += 20

// Criterion 2: bio (non-empty) + ≥3 portfolio items (20pts) — both required
const portfolioItems = p?.portfolio_items
if (data.bio && data.bio.trim().length > 0 &&
    Array.isArray(portfolioItems) && portfolioItems.length >= 3) score += 20

// Criterion 3: LinkedIn URL present (20pts)
const linkedinUrl = p?.linkedin_url
if (typeof linkedinUrl === 'string' && linkedinUrl.trim().length > 0) score += 20

// Criterion 4: years_experience >= 3 (20pts)
const yearsExp = p?.years_experience
if (typeof yearsExp === 'number' && yearsExp >= 3) score += 20

// Criterion 5: ≥1 certification/credential (20pts)
const certs = p?.certifications
if (Array.isArray(certs) && certs.length >= 1) score += 20

return score  // [0, 100] in steps of 20
```

**Profile JSONB key conventions (canonical for this project):**
- `portfolio_items: unknown[]` (array of portfolio item objects)
- `linkedin_url: string`
- `years_experience: number`
- `certifications: unknown[]` (array of certification objects)

These key names are not enforced by the API schema (`profile: z.record(z.string(), z.unknown())`). They are the canonical names established by this story (AC7). Future profile API routes MUST use these same keys for trust_score to function correctly.

## 3. Edge Cases per AC

| AC | Scenario | Handling |
|----|----------|----------|
| AC1 | Unknown message type | `console.warn` + `message.ack()` + `continue` (no retry) |
| AC1 | `expert_id` missing from body | TypeScript prevents at compile time; runtime: `computeTrustScore` would get `undefined`, `.single()` returns PGRST116 → throw → retry |
| AC2 | DB update fails | Throw → consumer retries |
| AC2 | Expert has no matches | Skip computation (matchIds empty guard) |
| AC3 | No surveys | Returns 0 |
| AC3 | Survey `score = null` | Filtered before multiplication |
| AC3 | Empty `bookingIds` | Short-circuit: return 0 |
| AC4 | No satisfaction surveys | Returns 0 |
| AC4 | Empty `bookingIds` | Short-circuit: return 0 |
| AC5 | No evaluations | Returns 0 |
| AC5 | All `conversion_declared = null` | 0 conversions / total = 0 |
| AC6 | No bookings (but matches exist) | Query returns empty → 0 |
| AC6 | No matches | Short-circuit on empty matchIds → 0 |
| AC6 | Future `scheduled_at` | `daysSince < 0` → `100` (< 30 branch triggers) |
| AC6 | Booking at exactly 30 days | `daysSince = 30` → formula: `100 - ((30-30)/150)*100 = 100`. Correct. |
| AC6 | Booking at exactly 180 days | `daysSince = 180` → `100 - ((180-30)/150)*100 = 0`. Correct. |
| AC6 | Booking > 180 days | Clamped to `Math.max(0, ...)` = 0 |
| AC7 | `profile = null` | All profile-based criteria evaluate to false; max score = 20 (verified_at only) |
| AC7 | `bio` present but `portfolio_items` has < 3 items | Criterion 2 = false (both sub-conditions required) |
| AC7 | `years_experience` is string | `typeof yearsExp !== 'number'` → criterion 4 = false |
| AC7 | Expert not found | `.single()` → PGRST116 → throw → consumer retry |
| AC8 | Same `message.id` replayed | KV `idem:score-computation:{id}` exists → immediate ack, no computation |
| AC8 | Two different messages, same `expert_id` | Each has different `message.id` → both compute (different message IDs). Both produce same result if no new data between them. |
| AC9 | No matching-jobs dispatch | Confirmed: no queue send after score write |
| AC10 | Queue name includes env suffix | `batch.queue.includes('score-computation')` matches `*-staging` and `*-prod` |

## 5. Risk Register

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Supabase `.in()` with empty array causes error | Medium | Add `if (array.length === 0) return 0` guards before every `.in()` query |
| `experts.profile` JSONB keys don't match conventions | High | Keys defined here are canonical — document in SETUP.md or profile handler comment |
| CF queue not pre-created before deploy | High | Document as SETUP.md prerequisite; deployment will fail with binding error |
| Supabase `.single()` on non-existent expert → PGRST116 | Low | Thrown → retries → DLQ after 3 attempts. Acceptable (deleted expert = irrelevant score) |
| recency_score > 100 for future-dated booking | Low | `Math.min(100, ...)` clamp |
| TypeScript non-null access on `data[0]` (strict mode) | Medium | Use `data[0]?.scheduled_at` throughout |

**Cleanly reversible (branch delete):**
- `src/queues/score-computation.ts` (new file)

- Type additions in `queues.ts` and `env.ts` (remove 3–5 lines)

**Requires coordination:**
- `wrangler.toml` consumer/producer blocks (remove + redeploy before CF queue deletion)
- CF queue resources (delete via `wrangler queues delete` if rollback permanent)
- Messages in flight when consumer removed → `batch.ackAll()` (silently dropped)

**DB state:** scores already written to `experts.composite_score` remain. No rollback migration needed (values stay at 0 for unprocessed experts, updated values for processed ones). No data corruption.

3. **No new secrets required.** `SCORE_COMPUTATION` is a queue binding in `wrangler.toml`, not a secret.

4. **`experts.composite_score` write:** The update does not use `.single()` — just `.eq('id', expertId)`. If expert not found, no rows updated and no error thrown by Supabase (it returns empty data, not an error). Add explicit check: `if no rows updated AND trust_score computation succeeded → expert was deleted between message enqueue and processing. Log and ack.`
