# Story: JWT claims hardening + secrets hygiene + defense-in-depth fixes

## User Story

**As the** platform operator,
**I want** JWT tokens to include proper claims, secrets to be properly managed, and defense-in-depth measures applied across the codebase,
**so that** token reuse attacks are prevented and the codebase follows security best practices.

---

## Context

SEC-AUDIT-001 findings SEC-08 through SEC-16 (MEDIUM + LOW). Batch story grouping 9 related fixes that are individually small but collectively important for production readiness.

---

## Acceptance Criteria

### JWT Claims (SEC-10, SEC-11)

- [ ] AC1: `signProspectToken` includes `iss: "callibrate"` and `aud` claim. The `aud` value is passed as parameter by the caller: `"prospect:submit"` for submit, `"prospect:matches"` for matches, `"prospect:identify"` for identify.
- [ ] AC2: `verifyProspectToken` accepts an `expectedAud` parameter and validates `claims.aud === expectedAud`. Mismatched audience → reject.
- [ ] AC3: `signSurveyToken` TTL is reduced from 30 days to 7 days (7 * 86400). Survey emails include text: "Ce lien expire dans 7 jours."
- [ ] AC4: Survey tokens include `iss: "callibrate"` and `aud: "survey"`.

### Secrets Hygiene (SEC-08, SEC-09)

- [ ] AC5: `TURNSTILE_SECRET_KEY` is removed from `[env.staging.vars]` in `workers/backend/api/wrangler.toml`. A comment is added: `# Set via: wrangler secret put TURNSTILE_SECRET_KEY --env staging`. The staging secret is set via CLI (founder action).
- [ ] AC6: `localConnectionString` is removed from both `workers/backend/api/wrangler.toml` and `workers/backend/matching-engine/wrangler.toml`. Instructions added to SETUP.md to set it in `.dev.vars`.

### CORS Optimization (SEC-12)

- [ ] AC7: `validateOrigin` in `cors.ts` uses `SUPABASE_ANON_KEY` instead of `SUPABASE_SERVICE_KEY`. A SELECT RLS policy on `satellite_configs` for `anon` role is added via Supabase migration (or the query is replaced with a Hyperdrive/postgres.js query selecting only `domain`).

### Prompt Injection Guard (SEC-13)

- [ ] AC8: In `extract.ts`, `verticalContext` is validated against a hardcoded allowlist of known verticals (e.g., `['ai-consulting', 'automation', 'data-science', 'general']`). Unknown values are rejected (fallback to no vertical context).

### Webhook Idempotency (SEC-14)

- [ ] AC9: LemonSqueezy webhook idempotency is backed by a DB table `webhook_events` (columns: `event_id TEXT PRIMARY KEY`, `event_name TEXT`, `processed_at TIMESTAMPTZ DEFAULT now()`). INSERT with ON CONFLICT DO NOTHING. KV idempotency is kept as L1 check (fast path), DB as L2 (persistent).

### RPC Defense-in-Depth (SEC-16)

- [ ] AC10: `debit_lead_credit` and `restore_lead_credit` PostgreSQL functions add an ownership assertion: `IF NOT EXISTS (SELECT 1 FROM leads WHERE id = p_lead_id AND expert_id = p_expert_id) THEN RAISE EXCEPTION 'ownership_check_failed'; END IF;` at the start of each function.

### Tests & Housekeeping

- [ ] AC11: All existing vitest tests pass. New tests cover: (a) prospect token with wrong `aud` → rejected, (b) survey token expired after 7 days → rejected.
- [ ] AC12: tsc clean across all workers.

---

## Out of Scope

- Prospect token migration to Authorization header (SEC-15 — breaking change, deferred to frontend epic)
- KV-backed token revocation via jti (post-MVP optimization)
- CORS caching in KV (performance, not security-critical)

## Notes

- AC1-AC4 are backward-incompatible for existing tokens. Since no production users exist yet, this is acceptable. If tokens were in the wild, a migration period with dual-validation would be needed.
- AC5 founder action: `wrangler secret put TURNSTILE_SECRET_KEY --env staging` with the CF test key `1x0000000000000000000000000000000AA`.
- AC7 alternative: if Hyperdrive is available, use `SELECT domain FROM satellite_configs WHERE domain = $1` via postgres.js (avoids Supabase JS entirely in CORS path).
- AC8 allowlist will grow as new satellites are created. Keep it in a const array in extract.ts.
- AC9 migration creates a small table — no performance impact.
