# Story E03S04: Expert Directory + Profile Pages

## User Story

**As a** prospect exploring the platform,
**I want** to browse available experts by specialty and see detailed profiles,
**so that** I can discover qualified experts even without submitting a specific project description.

---

## Context

A secondary entry point to the platform, complementing the quiz funnel (E03S01–S03). The directory serves three purposes: (1) SEO — indexable expert pages drive organic traffic, (2) social proof — shows the breadth and quality of the expert pool, (3) discovery — prospects can explore before committing to the funnel.

Directory pages are server-rendered by the satellite Worker for SEO. Expert data is fetched from the Core API at request time and CF-edge-cached. All expert information on directory pages is anonymized (no full names, no contact info) to protect against disintermediation. To see full profiles or book, prospects go through the identify flow (email gate).

The directory is scoped to the satellite's vertical: a "Workflow Automation" satellite shows only automation experts.

- [ ] AC5: **API dependency — public expert listing endpoint.** Required contract:
  - `GET /api/experts/public?vertical={v}&skills={s1,s2}&page={n}&per_page=12` → `{ experts: AnonymizedExpert[], total: number, page: number }`

  - `AnonymizedExpertDetail`: extends AnonymizedExpert with `{ bio_excerpt, availability_status, availability_rules_summary }`
  - **If this endpoint does not exist in Core API, this story is BLOCKED.** Delivery flags the gap → Discovery creates a Layer 2 story (E06S38 or equivalent).

- [ ] AC6: Pagination: "Charger plus d'experts" button at bottom of card grid. Loads next 12 experts via client-side JS fetch. Appends new cards without page reload. Initial page is server-rendered; subsequent pages are client-side.

- [ ] AC7: SEO:
  - `/experts`: `<meta name="robots" content="index, follow">`, canonical URL, `<title>`: "Experts en {vertical} — {brand.name}", JSON-LD ItemList structured data listing expert entries

## Notes

- **Anonymization is non-negotiable on directory pages.** Expert full names and contact info are never rendered in server-side HTML. Protects against disintermediation.
- **Slug format:** anonymized unique ID (e.g., `exp-{short_hash}`), NOT derived from expert name. Prevents reverse-engineering identity from URLs.
- **Vertical scoping:** directory shows only experts matching the satellite's `config.vertical`. API filtering is server-side via the `vertical` parameter.
- **Server-rendered for SEO.** Unlike the quiz funnel (client-side interactive), directory pages must be fully server-rendered for search engine indexing. Client-side JS is progressive enhancement only (filters, load-more).
- **Sitemap extension:** E06S14 AC7 currently lists `/`, `/match`, `/experts`. Individual expert profile pages (`/experts/:slug`) need to be added. Requires the satellite Worker to fetch expert slugs from Core API during sitemap generation (CF-edge-cached 86400s). Delivery may fold this into E03S04 or create a separate micro-task.
- **"Débloquer ce profil" from directory (AC4):** This path requires creating a prospect record without going through the full funnel. Delivery to assess: either a lightweight `POST /api/prospects/create-from-directory` endpoint, or reuse the identify flow with minimal required fields. The billing model applies — unlocking a profile from directory also counts as a lead.
- **AC11 — Direct CTA note:** The CTA links to `callibrate.io/book/:slug?t={token}`. The HMAC-signed token is fetched server-side from the Core API (field `direct_booking_url` on AnonymizedExpertDetail). The full direct mode implementation (HMAC validation, noindex rendering, freetext form, booking, quota, magic link email, DB schema) is owned by **E02S12** in the callibrate.io landing Worker (`workers/frontend/landing/`). E03S04 only adds the outbound link.
