# Story E06S34: Balance / max_lead_price / spending_limit Enforcement at Matching

## User Story

**As a** prospect,
**I want** to only be matched with experts who can actually accept my lead,
**so that** every match result is actionable and no booking fails due to billing constraints.

**As an** expert,
**I want** my billing controls (max_lead_price, spending_limit) to automatically filter me out of matches I can't afford or don't want,
**so that** I'm never charged for a lead that exceeds my configured limits.

---

## Context

DEC-67/68 define three billing filters that must apply at matching time:
1. **Balance check:** expert.credit_balance >= lead price — expert must have enough credits
2. **max_lead_price:** if set, lead price must not exceed it (hard deal-breaker per DEC-67)
3. **spending_limit:** if set, monthly cumulative spend + this lead must not exceed the cap

These filters run BEFORE scoring. An expert excluded by billing constraints never appears in match results. The prospect sees fewer results but every result is actionable. When an expert is excluded, they receive a notification about the missed lead opportunity.

---

## Acceptance Criteria

- [ ] AC1: Before scoring experts, calculate `leadPrice = calculateLeadPrice(prospect.budget_max, qualificationData)` once for the current prospect
- [ ] AC2: Expert excluded from candidate pool if `credit_balance < leadPrice` (insufficient balance)
- [ ] AC3: Expert excluded from candidate pool if `max_lead_price IS NOT NULL AND leadPrice > max_lead_price` (price ceiling exceeded)
- [ ] AC4: Expert excluded from candidate pool if `spending_limit IS NOT NULL` AND (monthly debit total from credit_transactions WHERE expert_id = expert.id AND type = 'lead_debit' AND created_at >= first day of current UTC month) + leadPrice > spending_limit
- [ ] AC5: Monthly spend query uses the index on `credit_transactions(expert_id, created_at)` from E06S31
- [ ] AC6: When expert excluded by any billing constraint: fire-and-forget notification via EMAIL_NOTIFICATIONS queue with reason ('insufficient_balance' | 'max_lead_price_exceeded' | 'spending_limit_reached') and missed lead context (prospect vertical, budget tier). Dispatched via ctx.waitUntil() — non-blocking.
- [ ] AC7: If ALL experts in the pool are excluded by billing constraints, match returns empty results (empty array, HTTP 200). This is correct behavior — not an error.
- [ ] AC8: Billing filters apply in BOTH matching paths: `POST /api/matches/compute` (core) and `POST /api/prospects/submit` (satellite funnel)
- [ ] AC9: When none of the billing columns are set on an expert (credit_balance=0, max_lead_price=NULL, spending_limit=NULL), the expert is still excluded if balance < leadPrice. No implicit "free pass" for unsubscribed experts.
- [ ] AC10: `tsc --noEmit` — zero errors
- [ ] AC11: `vitest run` — all tests pass + new tests: balance sufficient (included), balance insufficient (excluded + notification queued), max_lead_price exceeded (excluded), spending_limit reached (excluded), all experts excluded (empty results), multiple constraints active simultaneously

## Notes

The billing filter should be applied AFTER loading the expert pool but BEFORE scoring:
1. Load expert pool (Cache API → D1 → Hyperdrive chain)
2. **NEW: Filter by billing constraints**
3. Score remaining experts via scoreMatch()
4. Return ranked results

The monthly spend query for spending_limit could be expensive per-expert. Two options:
(a) Single batch query: `SELECT expert_id, SUM(ABS(amount)) FROM credit_transactions WHERE type='lead_debit' AND created_at >= date_trunc('month', now()) GROUP BY expert_id` — one query for all experts in pool
(b) Denormalize `monthly_spend` on experts table, updated by lead-billing consumer

Option (a) is simpler and correct. Option (b) is an optimization for later if needed.

For notifications: reuse EMAIL_NOTIFICATIONS queue with new message type `expert.billing.lead_missed`. The email consumer (E06S06) dispatches by type — add the new type handler.
