# Story E02S12: Expert Public Profile Page — SEO & Direct Booking Link

## User Story

**As a** prospect discovering an expert via search or a shared link,
**I want** to view that expert's profile and book a call directly,
**so that** I can connect with the right person without friction.

**As an** expert on Callibrate,
**I want** a shareable booking link I can put anywhere (my website, LinkedIn, Google Business Profile),
**so that** my existing audience can reach me through a seamless, pre-qualified booking flow.

---

## Context

DEC-131 establishes the "expert direct traffic" model: experts share their personal booking URL (`callibrate.io/book/{slug}?t={token}`) anywhere they are present online. The link from an expert's real website to callibrate.io is a dofollow editorial backlink — legitimate SEO value for callibrate.io. The booking page is the primary value; the backlink is a natural by-product.

This story adds two distinct public expert routes to the **callibrate.io landing Worker** (`workers/frontend/landing/`, DEC-115, E02S09):

- **Standard mode** (`/experts/:slug`) — indexed, anonymized, SEO-optimized. Ranks for "[specialty] AI expert for hire" queries. If a prospect arrives here via Google or AI search and books → `lead_source='callibrate'` → expert pays (DEC-67). Callibrate drove discovery.
- **Direct mode** (`/book/:slug?t={token}`) — noindex, full profile, booking form. HMAC-signed token (`t`) validates expert-originated traffic. Valid token → `lead_source='direct'` → free. Expert drove discovery.

**LLM/AI search protection:** The most dangerous scenario is an AI search engine (Perplexity, ChatGPT Search, Gemini, Google AIO) crawling profile pages and answering "who are the best n8n experts?" by giving names and contacts directly. Protection is layered:
1. Indexed pages are anonymized — no name, no email, no external URL → nothing to extract for direct contact
2. AI answer/search bots are blocked in robots.txt — they cannot index profiles at all
3. `max-snippet:150` limits what Google AI Overview can extract even from standard search results
4. The full identity (name, contact) is only revealed server-side after email gate (OTP) in the main funnel — no crawlable HTML ever contains real identity on public pages
5. Direct mode pages are `noindex` + blocked in robots.txt → full profile never indexed

**E06S38 status:** Already delivered (PR #42 merged). The public expert API endpoints (`/api/experts/public`, `/api/experts/public/:slug`) are available.

### Direct Mode Booking Flow

- [ ] AC14: Freetext form submission (`POST /api/book/:slug`) in direct mode. On submit:
  1. **HMAC token re-validation (primary attribution gate):** Re-validate token from form payload (same logic as AC7). Set `lead_source = 'direct'` if valid, `lead_source = 'callibrate'` if invalid/absent. This is the authoritative attribution decision — not the HTTP referrer.
  2. **Quota check (secondary guard):** Check `experts.direct_submissions_this_month < 100` — if exceeded → return graceful message: "Agenda complet pour ce mois-ci — contactez [expert_name] directement." No error code. PostHog: `expert.direct_quota_reached`. Applies regardless of `lead_source`.
  3. Input pre-validation: min 30 chars, min 3 distinct words (client-side + server-side).
  4. Honeypot field + min 3s timing check.
  5. CF Rate Limiting (DEC-120): existing binding applies.
  6. Turnstile invisible (DEC-120): on submit button.
  7. `POST /api/extract` (DEC-48, GPT-4o-mini) → extraction requirements.
  8. Increment `experts.direct_submissions_this_month`.

## Notes

- **Anonymization is the primary LLM protection.** Standard mode pages never contain real identity. Even if all AI bots were to ignore robots.txt, they could not extract actionable contact data. The `noindex` + robots.txt blocking on AI bots is defense-in-depth, not the primary protection.
- **`max-snippet:150` rationale.** Google AI Overviews can extract and synthesize content from indexed pages even without full indexing of that specific page. Limiting snippet length reduces synthesis quality for queries like "best n8n expert on Callibrate." This is SEO-001 AKU-007 (Dual Visibility Model) applied defensively.
- **HMAC token design.** `t = base64url(HMAC-SHA256(DIRECT_LINK_SECRET, "{expert_id}:{direct_link_nonce}"))`. `DIRECT_LINK_SECRET` is a Cloudflare Worker secret (env binding, never in source). Token is stateless — validation requires only recomputing the HMAC, not a DB lookup. DB lookup is only needed for rotation (to read the current nonce). Token uniqueness is per-expert (nonce), not per-session — same token is always shared in the expert's bio/website. **Security model:** security is NOT based on the token being secret (the expert shares it publicly). It's based on: (1) noindex preventing search discovery of `/book/` URLs, (2) robots.txt blocking all bot crawling of `/book/`, (3) HMAC preventing URL forgery (can't generate a valid token without the secret), (4) quota limiting abuse if a token is widely diffused, (5) rotation available if token is compromised. The "full profile display" in direct mode is not sensitive — the expert explicitly chose to share it.
- **Outbound links in direct mode** use `rel="nofollow noopener"` — prevents Callibrate from leaking link equity to expert's external sites, and prevents opener exploitation.
- **DB migration AC11-AC13 are prerequisites** for AC14 (quota check) and AC16 (lead_source field). Deliver DB migration first in the implementation order.
- **E03S04 relationship:** E03S04 AC11 adds a "Prendre un rendez-vous direct" link on satellite profile pages pointing to `callibrate.io/book/{slug}?t={token}`. The token is fetched server-side from the Core API (expert record includes pre-computed signed URL). E02S12 owns the implementation; E03S04 only renders the outbound link.
