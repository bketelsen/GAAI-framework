# Story E06S06: Async Queue Workers — Email, Lead-Billing

## User Story

**As a** platform operator,
**I want** all background jobs (emails, billing, matching computation) processed reliably and asynchronously,
**so that** no user-facing request blocks on slow external calls and failures are retried automatically.

---

## Context

Two Cloudflare Queue consumers handle all async work. This decouples the request lifecycle from slow external API calls (Resend, Lemon Squeezy). Each consumer has retry logic and a Dead Letter Queue for observability. n8n is triggered via webhook for complex multi-step business workflows (e.g., expert notification with prospect context summary). `matching-jobs` queue dropped — matching is synchronous (DEC-33). Cal.com setup handled synchronously at expert registration (DEC-34).

---

## Acceptance Criteria

- [ ] AC1: `email-notifications` consumer is a registered Cloudflare Queue consumer that processes messages by `type`:
  - `expert.registered` → sends welcome onboarding email to expert via Resend (template: onboarding)
  - `booking.confirmed` → sends booking confirmation email to expert (with prospect context) and to prospect (with meeting link) via Resend
- [ ] AC2: `lead-billing` consumer processes `{ type: "booking.created", booking_id, expert_id }`: creates a `leads` row with `status: 'pending'`, calls Lemon Squeezy API to create a checkout session for the expert, stores `ls_checkout_id` in `leads` table, updates `leads.status` to `'billed'` on payment confirmation webhook (or keeps `'pending'` if checkout not yet completed)
- [ ] AC3: ~~`matching-jobs` consumer~~ — SUPPRIMÉ (DEC-33). Matching synchrone. Cal.com setup synchrone à l'inscription expert (DEC-34). Voir E06S04 pour l'implémentation cal.setup.
- [ ] AC4: Both consumers implement retry logic: failed message is retried up to 3 times with exponential backoff (1s, 4s, 16s) before routing to the corresponding DLQ (`callibrate-core-queue-email-notifications-{env}-dlq`, `callibrate-core-queue-lead-billing-{env}-dlq`)
- [ ] AC5: DLQ messages produce a structured log entry with: `{ queue, message_id, type, payload, error, failed_at }` — visible in Cloudflare Workers logs
- [ ] AC6: All consumers are idempotent — processing the same `message_id` twice produces no duplicate side effects (check by `message_id` in KV or DB before processing)
- [ ] AC7: `email-notifications` consumer triggers a n8n webhook `POST {N8N_WEBHOOK_URL}/booking-confirmed` on `booking.confirmed` events, passing `{ booking_id, expert_id, prospect_id, meeting_url, scheduled_at }` — n8n handles the expert notification with full prospect context summary
- [ ] AC8: `lead-billing` consumer handles Lemon Squeezy checkout creation failure gracefully: logs error, keeps `leads.status` as `'pending'`, routes to DLQ after 3 retries — does not crash the worker
- [ ] AC9: Each consumer has at least one integration test covering: happy path, retry on failure, idempotency on duplicate message
- [ ] AC10: `email-notifications` consumer on `booking.completed` events: triggers n8n webhook `POST {N8N_WEBHOOK_URL}/booking-completed` passing `{ booking_id, expert_id, prospect_id, scheduled_at }` — n8n orchestrates the J+7 call experience survey email to prospect and J+45 project satisfaction survey email (time-based triggers handled by n8n, not Workers)

## Notes

Lemon Squeezy billing model: expert is billed per booked call (pay-per-lead). The `lead-billing` consumer creates a Lemon Squeezy checkout link — the expert receives the link and completes payment. At MVP, payment is not fully automatic (expert must click through checkout). Full automatic billing (stored payment method) is a post-MVP enhancement. Cloudflare Queues consumer binding is defined in `wrangler.toml` under `[[queues.consumers]]`.
