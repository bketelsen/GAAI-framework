# Story E06S27: Staging — Provisionnement infra CF (Hyperdrive + D1 + Vectorize + Workflows + Rate Limiter + Workers)

## User Story

**As a** platform operator,
**I want** all Cloudflare infrastructure resources provisioned and new workers deployed to staging,
**so that** the full backend stack runs on staging without silent fallbacks or missing bindings.

---

## Context

Plusieurs stories E06 ont été livrées (code QA PASS) avec des déferred human actions : ressources CF à créer + IDs à injecter dans wrangler.toml. Aucune de ces ressources n'existe encore :

- Hyperdrive config → requis par E06S18 (toutes les queries postgres.js passent en direct connection sans pooling/caching)
- D1 `expert-pool-edge` → requis par E06S23 (pas de cache edge, matching fallback sur Hyperdrive only)
- Vectorize index `expert-profiles` → requis par E06S21 (semantic scoring désactivé, fallback deterministic)
- CF Workflows → requis par E06S16 (surveys J+7/J+38 ne se triggent pas)
- CF Rate Limiter namespace → requis par E06S20 (rate limiting désactivé)
- `workers/matching` non déployé → requis par E06S24 (Service Binding MATCHING_SERVICE absent)
- `workers/posthog-proxy` non déployé → requis par E07S01 (proxy PostHog absent)

Cette story est **founder-executed** : pas de code à écrire, uniquement des commandes wrangler CLI + dashboard CF.

---

## Acceptance Criteria

- [ ] AC1: Hyperdrive config créé — `npx wrangler hyperdrive create callibrate-core-hyperdrive-staging --connection-string "postgresql://postgres.[ref]:[password]@db.[ref].supabase.co:5432/postgres"` → ID noté et inséré dans `workers/matching/wrangler.toml` section `[[env.staging.hyperdrive]]`
- [ ] AC2: D1 database créé — `npx wrangler d1 create expert-pool-edge-staging` → ID inséré dans `wrangler.toml` (root + `[env.staging.d1_databases]`); idem pour prod: `npx wrangler d1 create expert-pool-edge-prod`
- [ ] AC3: Vectorize index créé — `npx wrangler vectorize create expert-profiles --dimensions=768 --metric=cosine` (index unique, pas de variante staging/prod — Vectorize n'a pas de namespaces par env)
- [ ] AC4: CF Workflows créés dans CF Dashboard (Workers & Pages → Workflows) : `BookingConfirmedWorkflow` + `BookingCompletedWorkflow` pour staging. Noms exacts alignés avec les `[[workflows]]` de `wrangler.toml`
- [ ] AC5: CF Rate Limiter créé — `npx wrangler rate-limit create --name callibrate-core-rate-limiter-staging` → `namespace_id` inséré dans `wrangler.toml [env.staging.rate_limits]`; idem pour prod
- [ ] AC6: `workers/matching` déployé en staging — `cd workers/matching && npm install && npx wrangler deploy --env staging`; smoke test: `POST /match` sur l'URL du worker → 401 (auth expected, binding reachable)
- [ ] AC7: `workers/posthog-proxy` déployé en staging — `cd workers/posthog-proxy && npm install && npx wrangler deploy --env staging`; smoke test: `GET /health` ou `OPTIONS /ingest` → répond
- [ ] AC8: CF API token scope (GitHub Actions `CLOUDFLARE_API_TOKEN`) étendu pour inclure `callibrate-matching-*` et `callibrate-posthog-proxy-*` → CI/CD peut déployer ces workers
- [ ] AC9: `wrangler.toml` (core + matching + posthog-proxy) mis à jour avec tous les IDs provisionnés, committé sur `staging` et poussé
- [ ] AC10: Smoke test infra — `curl https://callibrate-core-staging.frederic-geens-consulting.workers.dev/api/health` → HTTP 200

## Notes

**Connexion Hyperdrive :** utiliser le Connection Pooler Supabase (Transaction mode, port 6543) ou la connexion directe (port 5432). DEC-66 précise Direct port 5432. Récupérer la connection string dans Supabase Dashboard → Settings → Database → Connection string.

**CF Workflows :** si l'API wrangler ne supporte pas encore la création CLI, utiliser le Dashboard CF → Workers & Pages → Workflows → New Workflow. Vérifier que les noms correspondent exactement aux `class_name` dans le code.

**Ordre de déploiement E06S24 :** le Matching Worker doit être déployé AVANT le Core Worker (le Service Binding `MATCHING_SERVICE` dans Core ne peut pas résoudre un worker inexistant).
