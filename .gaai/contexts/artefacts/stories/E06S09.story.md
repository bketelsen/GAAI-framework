# Story E06S09: Expert Composite Score Computation Worker

## User Story

**As a** platform operator,
**I want** expert composite scores to update automatically after each feedback event,
**so that** match rankings reflect real-world performance without any manual intervention.

---

## Context

The composite score is the quality signal that differentiates experts with equal criteria match scores. It aggregates feedback from three streams: call experience (prospect J+7), project satisfaction (prospect J+45), and lead evaluation (expert J+48). A Cloudflare Queue consumer recomputes the score after each feedback event and writes it back to `experts.composite_score`. Re-ranking of pending matches for the updated expert is triggered as a best-effort side effect.

Formula: `composite_score = (call_experience_avg × 0.35) + (trust_score × 0.20) + (client_satisfaction_avg × 0.20) + (hire_rate × 0.10) + (recency_score × 0.15)` — all components normalized to 0–100 before weighting.

---

## Acceptance Criteria

- [ ] AC1: `score-computation` Cloudflare Queue consumer processes three message types: `{ type: "feedback.call_experience", expert_id: string }`, `{ type: "feedback.project_satisfaction", expert_id: string }`, `{ type: "feedback.lead_evaluation", expert_id: string }` — all dispatched by the API routes that receive survey/evaluation submissions
- [ ] AC2: On any message: queries all three feedback tables for the given `expert_id`, computes each component, calculates composite_score, writes result atomically to `experts.composite_score` and `experts.score_updated_at = now()`
- [ ] AC3: `call_experience_avg` = average of `call_experience_surveys.score × 20` (converts 1–5 to 0–100) across all bookings for this expert's matched prospects. Returns 0 if no surveys yet
- [ ] AC4: `client_satisfaction_avg` = average of `project_satisfaction_surveys.score × 10` (converts 1–10 to 0–100) across all bookings for this expert. Returns 0 if no surveys yet
- [ ] AC5: `hire_rate` = `(count of lead_evaluations WHERE conversion_declared = true) / (count of lead_evaluations submitted for this expert)` × 100. Returns 0 if no evaluations yet
- [ ] AC6: `recency_score` = 100 if most recent booking `scheduled_at` < 30 days ago; decreases linearly to 0 at 180 days with no booking; 0 if no bookings at all
- [ ] AC7: `trust_score` computed from expert profile completeness: verified email (20pts) + bio + 3 portfolio items (20pts) + LinkedIn URL present (20pts) + years_experience declared ≥ 3 (20pts) + at least one certification/credential in profile JSONB (20pts). Static — recalculated only when expert updates profile, not on each feedback event
- [ ] AC8: Consumer is idempotent — recomputing the same expert's score twice with no new feedback data produces the same result and no duplicate writes
- [ ] AC9: After score update: no async dispatch to `matching-jobs` (queue supprimée — DEC-33). Re-ranking best-effort inline si le temps de réponse le permet, sinon ignoré au MVP. Le score mis à jour sera pris en compte lors de la prochaine recherche prospect.
- [ ] AC10: Consumer registered in `wrangler.toml` alongside existing consumers from E06S06. Queue names follow convention (DEC-32): `callibrate-core-queue-score-computation-{env}` + `callibrate-core-queue-score-computation-{env}-dlq`
- [ ] AC11: Unit tests for each score component: `call_experience_avg`, `client_satisfaction_avg`, `hire_rate`, `recency_score`, `trust_score` — each with known input → expected normalized output

---

## Out of Scope

- ML-based weight optimization (post-MVP — requires sufficient data volume)
- Real-time score propagation to active browser sessions (cache invalidation — post-MVP)
- Expert score history / timeline chart (post-MVP)
- Batch recomputation of all experts (post-MVP maintenance job)

---

## Dependencies

- E06S02 (all feedback tables + `experts.composite_score` + `experts.score_updated_at` fields must exist)
- E06S06 (`score-computation` queue registered in `wrangler.toml`; consumer co-located with other queue workers)
- Survey/evaluation API endpoints (deliver `POST /api/surveys/call-experience`, `POST /api/surveys/project-satisfaction`, `POST /api/evaluations/lead` — these dispatch messages to the `score-computation` queue)

---

## Notes

Score initialization: new expert starts with `composite_score = 0`, displayed as `quality_tier: 'new'` in match results. The worker does not block on re-ranking — it fires-and-forgets the `compute.matches` messages after writing the score. For MVP the `trust_score` computation happens inline during profile PATCH (not as a separate worker) — it is fast enough to be synchronous. The composite score formula weights can be adjusted in a single constant in code post-M3 once sufficient data informs calibration.
