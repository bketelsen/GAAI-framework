# Story E06S32: Lead-Billing Consumer — Credit Debit + Flag/Restore

## User Story

**As an** expert,
**I want** lead billing to deduct credits from my balance instantly when a call is booked, and to restore credits if I flag the lead as non-qualified within 7 days,
**so that** I only pay for leads I accept, with full transparency and zero friction.

---

## Context

DEC-68 replaces the per-lead LS checkout flow with internal credit accounting. The current `lead-billing` queue consumer (E06S06) creates an LS checkout per lead with hardcoded store/variant IDs. This story rewrites it to:
1. Calculate lead price (DEC-67 grid via `calculateLeadPrice` from E06S31)
2. Debit credits from expert's balance atomically
3. Record transaction in audit trail (`credit_transactions`)
4. Create lead with amount and status tracking

It also adds flag/restore endpoints so experts can dispute within the 7-day window, and an explicit confirm endpoint.

---

## Acceptance Criteria

- [ ] AC1: `processLeadBilling` rewritten — on `booking.created` message: fetch prospect data (budget_range, requirements) from DB → `calculateLeadPrice()` → verify `expert.credit_balance >= amount` → INSERT `credit_transactions` (type='lead_debit', amount=-price, balance_after) → UPDATE `experts` SET credit_balance = credit_balance - price → INSERT/UPDATE `leads` with amount, status='pending'
- [ ] AC2: If expert credit_balance < lead price at debit time: INSERT lead with status='insufficient_balance', skip debit, queue email notification to expert via EMAIL_NOTIFICATIONS queue (type 'expert.billing.insufficient_balance')
- [ ] AC3: Credit debit is atomic — credit_transaction INSERT and experts.credit_balance UPDATE in same SQL transaction (BEGIN/COMMIT via postgres.js)
- [ ] AC4: `POST /api/leads/:id/flag` — expert JWT auth, ownership check (lead.expert_id = user.id) — validates flag window (lead.created_at + 7 days > now(), else 422 "Flag window expired") — validates lead.status='pending' (else 422) — atomic transaction: INSERT credit_transactions (type='lead_credit', amount=+lead.amount, balance_after) → UPDATE experts SET credit_balance = credit_balance + amount → UPDATE lead SET status='flagged', flagged_at=now(), flag_reason=body.reason
- [ ] AC5: `POST /api/leads/:id/confirm` — expert JWT auth, ownership check — validates lead.status='pending' (else 422) — UPDATE lead SET status='confirmed', confirmed_at=now()
- [ ] AC6: LS checkout creation REMOVED from consumer entirely. No LS API call in this consumer.
- [ ] AC7: Idempotency pattern preserved (KV `idem:lead-billing:{message.id}`, TTL 86400s)
- [ ] AC8: Retry + DLQ pattern preserved (exponential backoff 1s/4s/16s, max 3 attempts)
- [ ] AC9: `leads.booking_id` UNIQUE constraint still enforced; 23505 recovery pattern handles partial failures
- [ ] AC10: `tsc --noEmit` — zero errors
- [ ] AC11: `vitest run` — all tests pass. Existing lead-billing tests rewritten for credit debit flow + new tests for flag endpoint (happy path, expired window, wrong owner, already flagged, already confirmed) + confirm endpoint (happy path, wrong status)

## Notes

Files to modify: `src/queues/lead-billing.ts`, `src/queues/lead-billing.test.ts`, `src/index.ts` (route registration)
Files to create: `src/handlers/leads/flag.ts`, `src/handlers/leads/confirm.ts` (or inline in routes)

Route registration: `POST /api/leads/:id/flag` and `POST /api/leads/:id/confirm` inside JWT-authenticated block.

The flag/restore is a credit reversal. Both debit and credit are recorded in `credit_transactions` for full audit trail. `balance_after` on each transaction enables point-in-time balance reconstruction without re-querying the experts table.

postgres.js transactions: `await sql.begin(async (tx) => { ... })` — all writes inside the callback are atomic.
