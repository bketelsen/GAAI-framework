# Story: OAuth security — CSRF state token + PKCE

## User Story

**As an** expert connecting my Google Calendar,
**I want** the OAuth flow to be protected against CSRF and authorization code interception,
**so that** no attacker can link their Google account to my Callibrate profile or steal my calendar tokens.

---

## Context

SEC-AUDIT-001 findings SEC-01 (CRITICAL) and SEC-03 (HIGH). The current OAuth flow uses the plain expert ID as the `state` parameter and does not implement PKCE. This allows CSRF attacks (attacker forges auth URL with victim's expert_id) and authorization code interception.

---

## Acceptance Criteria

- [ ] AC1: `handleGcalAuthUrl` generates a cryptographically random state token (32 bytes, `crypto.getRandomValues`, hex-encoded) and stores it in SESSIONS KV with key `oauth-state:{token}`, value `{expert_id}`, TTL 600 seconds (10 min).
- [ ] AC2: `handleGcalAuthUrl` generates a PKCE `code_verifier` (43+ chars, base64url-safe random) and `code_challenge = BASE64URL(SHA256(code_verifier))`. The `code_challenge` and `code_challenge_method=S256` are included in the Google auth URL params.
- [ ] AC3: The `code_verifier` is stored alongside the expert_id in the same KV entry: `{expert_id, code_verifier}`.
- [ ] AC4: `handleGcalCallback` reads `state` from query params, looks up `oauth-state:{state}` in SESSIONS KV. If not found or expired → redirect to gcal-error with `reason=invalid_state`. If found → extract `expert_id` and `code_verifier`, then DELETE the KV entry (one-time use).
- [ ] AC5: `handleGcalCallback` includes `code_verifier` in the token exchange POST to `oauth2.googleapis.com/token`.
- [ ] AC6: The `state` parameter in the auth URL is the random token, NOT the expert ID.
- [ ] AC7: All existing vitest tests pass. New tests cover: (a) valid state → success, (b) missing state → 302 error, (c) expired state → 302 error, (d) reused state → 302 error.
- [ ] AC8: tsc clean across the worker.

---

## Out of Scope

- Rate limiting on OAuth callback (deferred — low severity)
- Token revocation improvements
- Additional OAuth scopes

---

## Dependencies

- None — standalone fix on existing gcal.ts

---

## Notes

- PKCE is recommended by Google even for confidential clients (Workers have a client_secret). See: https://developers.google.com/identity/protocols/oauth2/native-app#pkce
- The SESSIONS KV namespace is already bound in all environments.
- The state token must be URL-safe (hex encoding is sufficient).
- Remediation priority: CRITICAL — must be delivered before any expert connects GCal in production.
