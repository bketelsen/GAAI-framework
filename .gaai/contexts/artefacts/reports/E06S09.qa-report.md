## Architecture Review

### Two-step join (D1/D2)
Correct. `call_experience_surveys` and `project_satisfaction_surveys` lack `expert_id`; joining through `bookings → matches` via pre-fetched ID arrays is reliable and efficient vs. 2-level PostgREST nested filter. 7 queries/computation vs. up to 15 with individual joins.

### Composite formula
```
composite_score = (call_experience_avg × 0.35) + (trust_score × 0.20)
               + (client_satisfaction_avg × 0.20) + (hire_rate × 0.10)
               + (recency_score × 0.15)
```
Weights sum to 1.00 ✅. All component values are [0,100] → composite is [0,100] ✅.

### Skip on no matches (D3)
Correct. New experts retain `composite_score = 0` from E06S02 initialization. Writing 0 with a fresh `score_updated_at` when there's no data would be misleading. Early return is the right choice.

### Trust score profile keys (D4)
Keys `portfolio_items`, `linkedin_url`, `years_experience`, `certifications` are now canonical. Correctly documented as implementation constraint for future profile update handlers.
