## Known Risks and Limitations

### LS_STORE_ID / LS_VARIANT_ID placeholders
`src/queues/lead-billing.ts` has hardcoded constants `LS_STORE_ID = 'YOUR_LS_STORE_ID'` and `LS_VARIANT_ID = 'YOUR_LS_VARIANT_ID'`. These must be replaced with real Lemon Squeezy IDs once the LS account is configured, and moved to env secrets. The TODO comment is in place.

### Expert email via gcal_email
The `experts` table has no `email` column. Expert contact email is resolved via `gcal_email` (the Gmail address used for Google Calendar OAuth). This is consistent with the database schema. If an expert has not connected Google Calendar, `gcal_email` will be null, and the consumer throws (and retries). This is intentional — it incentivizes completing the GCal connection flow before booking.

### n8n webhook retry semantics
The n8n webhooks for `booking.confirmed` and `booking.completed` are called sequentially within the consumer. If the Resend emails succeed but the n8n webhook fails, the entire message retries. On retry, Resend will be called again (duplicate emails) because the KV idempotency mark is only set after full success. Mitigation: Resend is idempotent by design (duplicate emails are a UX issue, not a data integrity issue). A finer-grained idempotency per-step was not implemented at MVP scope.

### Lead INSERT uniqueness on retry
If the leads INSERT succeeds but the LS API call fails, on retry the INSERT would fail with a unique constraint violation (`23505`). This is handled by catching that error code, fetching the existing lead by `booking_id`, and continuing with the LS API call using the existing `lead_id`. The DB migration applying `leads_booking_id_unique` is required for this recovery path.

### 'billed' status semantics at MVP
'billed' means "a Lemon Squeezy checkout link was created." It does not mean the expert has paid. The actual payment flow (expert clicks the link → LS processes payment → LS webhook → lead status update) is out of scope for this story.
