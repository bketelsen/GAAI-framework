## Known Risks and Limitations

1. **Token encryption key rotation:** No rotation mechanism exists. If `GCAL_TOKEN_ENCRYPTION_KEY` is rotated, all stored tokens become unreadable. A future migration story would need to re-encrypt all tokens.

2. **State parameter is not CSRF-protected:** The OAuth `state` param contains only the expert UUID. A production hardening could add a short-lived signed token as state to prevent CSRF on the callback. Not in scope for this story.

3. **Callback redirect URLs are frontend paths:** `/onboarding/gcal-connected` and `/onboarding/gcal-error` are hardcoded. These must be implemented in `callibrate-io` (the expert UI).

4. **Email fetch is non-blocking:** If the Google userinfo call fails, `gcal_email` is not stored. This is intentional per the implementation plan.

5. **Token revocation is fire-and-forget:** Google revocation failures during disconnect are silently swallowed. The DB cleanup always proceeds regardless.
