## New Patterns Found

### Pattern 1 — Queue Consumer Idempotency via KV

**Source:** `src/lib/idempotency.ts`, `src/queues/email-notifications.ts`, `src/queues/lead-billing.ts`

**Pattern:** Queue consumers check idempotency at the top of the message loop, before any side effect. On success (after all side effects), `markProcessed` writes to KV with 24h TTL. On failure, the KV key is never written, allowing safe retry.

- KV key format: `idem:{queue}:{message_id}` — uses SESSIONS KV namespace
- TTL: 86400s (24 hours) — matches CF Queue message retention window
- Implementation: `isAlreadyProcessed` / `markProcessed` in `src/lib/idempotency.ts`
- Placement: idempotency check is the first operation in each message's try block; `markProcessed` is the last operation before `message.ack()`

**Recommended addition to conventions.md under "Code Patterns":**
> **Queue consumer idempotency pattern (E06S06):** `isAlreadyProcessed(env.SESSIONS, queueName, message.id)` at the top of each message loop iteration. `markProcessed(env.SESSIONS, queueName, message.id)` only after all side effects succeed and before `message.ack()`. Key format: `idem:{queue}:{message_id}`, TTL: 86400s. Never mark on failure — ensures safe retry on any partial failure.

### Pattern 2 — Exponential Backoff + Structured DLQ Logging

**Source:** `src/lib/retryQueue.ts`

**Pattern:** `handleMessageFailure(message, error, queueName)` is the single shared function for all queue consumer error paths. Computes `delaySeconds = 4 ** (message.attempts - 1)` → 1s, 4s, 16s. At `attempts >= 3`, logs structured JSON via `console.error` before calling `message.retry()`. CF Queue runtime routes to DLQ after max retries.

- Backoff formula: `4^(attempts-1)` — base 4 chosen so delays are 1s/4s/16s (fast enough for transient failures, slow enough to avoid thundering herd)
- DLQ log fields: `{ queue, message_id, type, payload, error, failed_at }` — all string-serializable
- `message.retry({ delaySeconds })` called on every failure including attempt >= 3 (CF runtime handles DLQ routing after max retries)
- Shared util prevents per-consumer retry logic duplication

**Recommended addition to conventions.md under "Code Patterns":**
> **Queue retry + DLQ pattern (E06S06):** `handleMessageFailure(message, err, queueName)` in `src/lib/retryQueue.ts`. Exponential backoff: `4^(attempts-1)` seconds (1s, 4s, 16s). On `attempts >= 3`: `console.error(JSON.stringify({ queue, message_id, type, payload, error, failed_at }))` before `message.retry()`. CF Queue runtime routes to DLQ after max retries. All consumers call this shared util from their catch block — no per-consumer retry duplication.

### Pattern 3 — Partial-Failure Retry Safety via Unique Constraint Recovery

**Source:** `src/queues/lead-billing.ts`, lines 60–77

**Pattern:** When a multi-step queue consumer (INSERT → external API → UPDATE) fails partway through, retrying naively would re-attempt the INSERT and fail on a unique constraint. The pattern catches error code `23505` (unique violation), fetches the existing row by the business key (`booking_id`), and continues processing from where it left off. This makes the consumer safely retriable without manual intervention or separate deduplication tables.

- Applicable when: a message creates a DB row in step 1, then calls an external API in step 2
- Unique constraint required: `ALTER TABLE leads ADD CONSTRAINT leads_booking_id_unique UNIQUE (booking_id)` applied as a migration
- Recovery: `catch 23505 → SELECT existing row → continue with existing id`
- Complements KV idempotency: KV idempotency prevents re-processing of fully completed messages; unique constraint recovery handles the partial-completion case (completed step 1, failed step 2+)

**Recommended addition to conventions.md under "Supabase Patterns":**
> **Partial-failure retry recovery pattern (E06S06):** In multi-step queue consumers that INSERT then call external APIs, catch INSERT error code `23505` (unique violation), fetch the existing row by business key, and continue with the existing row ID. Requires a unique constraint on the business key column. Complements KV idempotency: KV prevents re-processing fully completed messages; unique constraint recovery handles partial completion (step 1 succeeded, step 2+ failed).
