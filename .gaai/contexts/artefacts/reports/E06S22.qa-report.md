### AC1: Prospect submit embeds requirements, queries Vectorize for top-100 candidates

**PASS**

`src/lib/vectorize.ts` exports `buildProspectEmbeddingText()` and `queryVectorizeForProspect()`.

`buildProspectEmbeddingText()` constructs a text representation from `skills_needed`, `industry`, and `languages`:

```ts
// vectorize.ts:68-77
export function buildProspectEmbeddingText(requirements: {
  skills_needed?: string[];
  industry?: string;
  languages?: string[];
}): string {
  const skills = (requirements.skills_needed ?? []).join(', ');
  const industry = requirements.industry ?? '';
  const languages = (requirements.languages ?? []).join(', ');
  return `Skills: ${skills}. Industries: ${industry}. Languages: ${languages}.`;
}
```

`src/routes/prospects.ts` invokes `env.AI.run('@cf/baai/bge-base-en-v1.5')` on the prospect requirements text, then calls `queryVectorizeForProspect`:

```ts
// prospects.ts:186-195
const embeddingText = buildProspectEmbeddingText(requirements);
const embeddingResult = await env.AI.run('@cf/baai/bge-base-en-v1.5', {
  text: [embeddingText],
}) as { data: number[][] };
const vector = embeddingResult.data[0];

if (vector && vector.length > 0) {
  const topK = Math.max(experts.length, 100);
  similarityMap = await queryVectorizeForProspect(env, vector, topK);
```

The `topK` formula `Math.max(experts.length, 100)` ensures at least 100 results are requested.

### AC4: scoreSkillsOverlap() blends: 0.7 × exactMatch + 0.3 × vectorSimilarity

**PASS**

The blend is implemented in `src/matching/score.ts`:

```ts
// score.ts:111-116
// AC4: blend 0.7 × exact + 0.3 × vector when semanticSimilarity available
const blendedRatio = semanticSimilarity !== undefined
  ? 0.7 * exactRatio + 0.3 * semanticSimilarity
  : exactRatio;

return blendedRatio * weight;
```

The blend uses `semanticSimilarity !== undefined` (not a falsy check), so a value of `0.0` still triggers blending. This is correct: a zero similarity should result in `0.7 * exactRatio + 0.3 * 0 = 0.7 * exactRatio`, not a pure exact fallback.

Test AC4 in `score.test.ts` (line 270-278) verifies the math:
- 1/2 exact (0.5) + vector 0.6 → `0.7*0.5 + 0.3*0.6 = 0.53` → `0.53*40 = 21.2` — passes.

## Logic Review — Edge Cases and Potential Issues

### Observation 1 (Low severity): Blend reduces score for full exact match + vector < 1.0

When an expert has a perfect exact skill overlap (ratio = 1.0) but vector similarity is, say, 0.9, the blend produces `0.7*1.0 + 0.3*0.9 = 0.97` → 97% of the weight rather than 100%.

This is a known, intentional design trade-off — the AC4 formula is fixed as `0.7 × exact + 0.3 × vector`. It is tested explicitly at score.test.ts lines 281-292. No remediation needed; the behavior is documented and intentional.

### Observation 2 (Low severity): `buildProspectEmbeddingText` omits `project_types`

The expert embedding at `buildEmbeddingText()` includes `project_types`. The prospect embedding at `buildProspectEmbeddingText()` does not include `project_types` (because `ProspectRequirements` has no `project_types` field).

This asymmetry is structurally correct — the prospect quiz does not ask for project type preference. The embedding text formats are intentionally different and the vector space remains coherent as long as both texts were embedded with the same model. No defect.

### Observation 3 (Low severity): `similarityMap.size > 0` guard still falls through to all experts if index unpopulated

If Vectorize returns results but none of the IDs match any expert in the pool (index populated with different IDs), `filtered.length === 0` and `candidates` falls back to all experts with `similarityMap` still populated. This means `scoreMatch()` would be called with `semanticSimilarity = undefined` for all experts (no ID matches), degrading gracefully to deterministic-only scoring. This is correct behavior and is commented in the code. Not a defect.
