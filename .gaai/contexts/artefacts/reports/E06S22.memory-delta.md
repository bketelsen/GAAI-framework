## New Patterns Established

### Pattern: Semantic Scoring Architecture

**Established in E06S22. Should be recorded in project context.**

The matching pipeline now has two distinct scoring paths. This is a fundamental architectural fact for all future scoring-related stories.

**Semantic path (when VECTORIZE + AI bindings are available):**
1. `buildProspectEmbeddingText(requirements)` — formats skills, industry, languages into a single embedding text string
2. `env.AI.run('@cf/baai/bge-base-en-v1.5', { text: [text] })` — embeds the prospect requirements
3. `queryVectorizeForProspect(env, vector, topK)` — queries the index, returns `Map<expert_id, cosine_similarity>`
4. `topK = Math.max(experts.length, 100)` — the canonical formula to avoid filtering loss when pool < 100
5. `scoreMatch(..., semanticSimilarity)` — blend formula: `0.7 × exactMatch + 0.3 × vectorSimilarity` for skills; proximity-or-vector for industry
6. `semantic_similarity` field stored in `ScoreBreakdown` as raw cosine similarity (0.0–1.0)

**Deterministic fallback path (when VECTORIZE/AI unavailable or query throws):**
- `scoreMatch(..., undefined)` — exact matching only, no blend
- `semantic_similarity` field absent from `ScoreBreakdown` (not `undefined` key, genuinely absent via spread)
- Candidates = all experts (no pre-filter)

**Design decision logged:**
- The blend `0.7 × exact + 0.3 × vector` is intentional even when exact = 1.0 (full match). This slightly reduces the score when vector < 1.0, trading a small score penalty for consistent formula behavior. This is documented and tested.
- Known proximity pairs in `getIndustryProximity()` always win over vector similarity — vector is only used when proximity is 0 (unknown pair). This prevents semantic drift from overriding curated knowledge.

### Pattern: Embedding Text Format Asymmetry

Expert embedding text (`buildEmbeddingText`): includes `skills`, `industries`, `project_types`, `languages`.

Prospect embedding text (`buildProspectEmbeddingText`): includes `skills_needed`, `industry`, `languages`. **No `project_types`** — because `ProspectRequirements` has no project type field.

This asymmetry is correct and intentional. Both sides are embedded with the same model (`@cf/baai/bge-base-en-v1.5`), preserving vector space coherence. Do not "fix" this asymmetry without first adding `project_types` to the prospect quiz schema.

### Pattern: ScoreBreakdown Extensibility

`ScoreBreakdown` uses optional fields for path-dependent data (`semantic_similarity?`, `deal_breaker?`, `reliability_modifier?`). Fields are conditionally added via spread (`...(condition ? { field: value } : {})`), ensuring they are genuinely absent (not null/undefined) in the fallback path. This is the canonical pattern for extending `ScoreBreakdown` in future stories.

## New Decisions to Log

| Decision | Context | Rationale |
|---|---|---|
| Blend weights: 0.7 exact + 0.3 vector | scoreSkillsOverlap() | Prioritizes deterministic signal; vector boosts partial matches without overriding strong exact matches |
| topK = Math.max(pool.length, 100) | Vectorize query | Guarantees no filtering loss when pool < 100; cap at 100 prevents unnecessary large queries when pool is small |
| Known proximity wins over vector (industry) | scoreIndustryMatch() | Curated domain knowledge (proximity table) is more reliable than embedding similarity for known pairs |
| Fallback resets similarityMap to empty Map | try/catch in prospects.ts | Clean reset ensures scoreMatch() receives undefined similarity for all experts in fallback — no partial semantic state |
