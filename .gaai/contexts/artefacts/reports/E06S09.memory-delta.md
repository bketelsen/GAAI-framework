# Memory Delta — E06S09: Expert Composite Score Computation Worker

## New Patterns

### Score computation consumer pattern
`src/queues/score-computation.ts` follows the same consumer pattern as `email-notifications.ts` (batch loop, KV idempotency, `handleMessageFailure`). The score consumer dispatches all 3 feedback event types to a single `computeAndSaveCompositeScore` function.

### Two-step join pattern for surveys
`call_experience_surveys` and `project_satisfaction_surveys` join to experts through `bookings → matches`. The canonical approach: fetch `matchIds` (via `matches.expert_id`) then `bookingIds` (via `bookings.match_id IN matchIds`) once in the parent function, then pass ID arrays to component functions. Avoids unreliable 2-level PostgREST nested filter.

### Parallel component computation
`computeAndSaveCompositeScore` runs all 5 score components concurrently via `Promise.all([...])`. Total: 7 DB queries per invocation (1 matchIds + 1 bookingIds + 5 component queries, some reusing the same pre-fetched arrays).

## New Conventions

### Profile JSONB canonical keys (trust_score)
`experts.profile` JSONB uses these keys for trust_score computation:
- `portfolio_items: unknown[]` — 20pt if length > 0
- `linkedin_url: string` — 20pt if present
- `years_experience: number` — 20pt if >= 2
- `certifications: unknown[]` — 20pt if length > 0
- `bio` or `description: string` — 20pt if present

Future profile update handlers MUST use these same keys.
